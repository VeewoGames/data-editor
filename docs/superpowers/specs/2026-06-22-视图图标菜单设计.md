# 视图图标菜单设计

## 方案概述

### 1. 总体目标和范围

本方案的目标，是在当前 `ViewTabs` 视图菜单中，明确收敛两个容易混淆的入口语义：

- `显示为`：定义为未来“切换视图类型”的预留入口，本轮不实现真实功能，继续保持禁用
- `更换图标`：新增独立入口，用于修改当前共享视图的视觉图标，并立即反映到视图 tab

本次范围包括：

- 视图菜单信息架构调整
- 共享视图叶子项增加 `icon` 元数据字段
- `ViewTabs` 中 tab 图标、菜单图标选择 UI 和拖拽幽灵图标渲染接线
- 视图复制、重命名、删除、分组场景下的图标行为收敛
- 对应状态层测试与 e2e 交互测试补齐

本次不包含：

- 真实的视图类型切换能力，例如 `table / board / gallery / timeline`
- emoji、自定义上传图片、图标颜色、图标搜索
- 组级图标配置
- 基于图标反推视图语义或业务能力

### 2. 各阶段任务概要

#### 阶段一：菜单语义收敛

主要工作：

- 明确 `显示为` 是“视图类型”的预留入口
- 在其旁边或其后新增 `更换图标`
- 调整菜单文案和禁用态说明，避免用户把两者混为一谈

预期成果：

- 用户能区分“视图类型”和“视图图标”是两个独立概念
- 当前未实现的 `显示为` 不再被误解为“修改图标”

执行顺序：

- 先固定菜单语义
- 再实现图标选择能力

#### 阶段二：数据模型与状态层扩展

主要工作：

- 给共享视图叶子项增加可选 `icon` 元数据
- 在 normalize / duplicate / 结构变换流程中接入该字段
- 为缺省视图定义统一回退图标

预期成果：

- 每个共享视图都可以稳定拥有自己的图标标识
- 未配置图标的旧视图无需迁移也能正常显示默认图标

执行顺序：

- 先扩 schema 和 normalize
- 再补 duplicate / structure / active view 相关读写

#### 阶段三：交互层实现

主要工作：

- 在 `ViewTabs` 菜单中增加 `更换图标` 子菜单或二级列表
- 提供受控图标集合供用户选择
- 让 tab、拖拽幽灵、必要的菜单预览都读取当前视图图标

预期成果：

- 图标修改后立即反馈到当前 UI
- 视图切换、拖拽、重开页面后图标保持一致

执行顺序：

- 先做静态选择 UI
- 再接状态写回
- 最后补菜单细节和选中态

#### 阶段四：验证与契约更新

主要工作：

- 更新 shared view 状态层测试
- 更新 `ViewTabs` 结构断言
- 增加图标选择的 e2e 用例

预期成果：

- 图标字段不会破坏现有共享视图主流程
- rename / duplicate / delete / group 场景行为都有明确测试保护

执行顺序：

- 先补单测
- 再补端到端测试

### 3. 整体结构框架

本方案把“视图菜单中的图标能力”拆成三层：

1. 存储层：共享视图叶子项上的 `icon` 元数据
2. 解析层：把共享配置解析为当前视图的有效图标，缺省回退 `borderAll`
3. 交互层：`ViewTabs` 菜单中的 `更换图标` 入口，以及 tab / 拖拽态图标渲染

核心边界如下：

- 视图类型和视图图标是两个独立维度
- `icon` 只负责视觉标识，不承载筛选、排序、布局或数据语义
- `显示为` 未来即使启用，也不能通过 `icon` 字段来偷渡实现
- 图标属于共享视图身份的一部分，应进入团队共享配置，而不是只存本地 profile
- 图标不能混入 `CollectionViewDraft` 语义链，否则会和当前“个人 draft 只承载 query / filters / sorts / layout”的边界冲突

## 背景与问题

当前 `src/components/ViewTabs.tsx` 中，视图 tab 图标和视图菜单里的 `显示为` 都还处于半成品状态：

- 视图 tab 图标固定写死为 `borderAll`
- 拖拽中的 ghost 图标同样固定为 `borderAll`
- 菜单中的 `显示为` 是禁用态，没有任何状态来源或事件绑定
- 当前 shared view 配置中没有可表达“视图图标”的字段

因此，用户从截图中自然会把 `显示为` 理解成“更换视图图标”的入口；但从代码事实看，这个入口目前既没有图标状态，也没有视图类型状态，属于纯占位。

如果不先收敛语义，后续会有两个问题：

- 用户认知混乱：不清楚“显示为”和“更换图标”到底谁负责什么
- 状态模型混乱：未来真正引入视图类型时，容易把图标字段误用成类型字段

所以本轮要先把概念拆开：

- `显示为` 专门保留给未来视图类型
- `更换图标` 独立承接本轮要实现的视觉标识能力

## 设计原则

### 1. 先拆语义，再做交互

不能为了快速实现一个菜单而把“类型”和“图标”合并。两者虽然都影响视图呈现，但职责不同：

- 类型决定视图能力和布局形态
- 图标只决定用户在标签上的识别方式

### 2. 图标属于共享视图，不属于个人偏好

当前系统中，共享视图是团队对象。视图名称、顺序、分组都属于共享结构；图标同样应被视为共享视图身份的一部分。

如果图标只保存在 profile 或浏览器本地，会出现：

- 同一个共享视图在不同用户眼里长得不一样
- 复制视图、分享链接、录屏沟通时缺少一致性
- 后续要迁回共享配置时还要二次搬运

因此图标应直接写入 shared view 配置。

### 3. 图标元数据不能直接并入 `CollectionViewDraft`

当前实现里，shared view 的核心内容对象是 `CollectionView`，而本地个人 draft 也是围绕 `Partial<CollectionView>` 展开。如果直接把 `icon` 粗暴加进 `CollectionView` 并默认复用 draft normalize 流程，就会产生错误语义：

- 图标会看起来像一个可以进入个人 draft 的字段
- `为所有人保存` 和 `重置` 的行为边界会被污染
- 本轮明明要求“图标是共享视图身份，不是个人临时改动”，实现却会反过来

因此本方案明确约束：

- `icon` 是 shared view leaf metadata
- `icon` 不进入 `normalizeCollectionViewDraft()` 的输入输出语义
- `icon` 不通过 `viewDrafts`、`saveSharedViewDraftsToConfig()` 或 `重置` 这条链路流转
- `icon` 的修改是直接改 shared view config 的独立动作

### 4. 本轮只做受控图标，不做开放式图标系统

项目当前还没有 emoji、外链图片或图标上传体系。如果本轮一并开放，会把问题从“视图图标”扩大成“完整资源管理和兼容策略”，偏离目标。

本轮只允许从现有 `icons` registry 中选择有限集合，保证：

- 无需引入新资产格式
- 渲染组件和测试边界清晰
- 未来可以平滑扩展，不影响现有字段

### 5. 旧数据零迁移成本

`icon` 必须是可选字段。旧视图没有 `icon` 时，统一按默认图标显示，不做强制迁移，不增加升级负担。

## 方案对比

### 方案 A：图标写入共享视图叶子项元数据

做法：

- 在 shared view leaf item 上增加 `icon?: string`
- 所有人看到同一个视图时，共享同一个图标

优点：

- 与共享视图身份一致
- duplicate / 分享 / 恢复语义最稳定
- 状态源单一，后续维护成本低

缺点：

- 要改 shared view schema 和相关测试

### 方案 B：图标只存 profile / localStorage

做法：

- 图标只作为个人偏好存本地，不进入 shared view 配置

优点：

- 表面上实现更快

缺点：

- 同一个共享视图在不同用户下表现不一致
- 复制视图和跨用户沟通语义不稳定
- 与当前 shared view 的团队对象定位冲突

### 方案 C：先只做菜单占位，不做持久化

做法：

- 增加 `更换图标` 菜单，但只改内存态或临时 UI

优点：

- 改动最小

缺点：

- 刷新即丢，几乎没有实际价值
- 后续还要重做一遍完整状态链

### 推荐方案

采用方案 A。

原因：

- 它和当前共享视图的产品定位一致
- 可以一次把菜单语义、状态模型和复制语义全部收敛干净
- 后续真的做“显示为 = 切换视图类型”时，不会和图标状态互相污染

## 详细设计

### 一、菜单信息架构

当前视图菜单中，与本方案相关的结构调整为：

1. `重命名`
2. `显示为`
3. `更换图标`
4. `编辑视图`
5. `来源`
6. `拷贝视图链接`
7. `创建视图副本`
8. `删除视图`

约束如下：

- `显示为` 继续保留当前禁用态，本轮不新增弱提示文案，也不改它的交互结构
- `显示为` 的视觉样式可以保留右箭头，表示未来会扩展成子菜单
- `更换图标` 为可用入口，承担本轮真实能力
- `来源` 继续保留现有只读语义

本轮用新增 `更换图标` 入口来完成语义拆分，不额外扩大 `显示为` 的视觉范围，避免把一次简单能力扩成菜单布局重构。

### 二、图标数据模型

建议在共享视图叶子项上增加：

```ts
type SharedViewIconId =
  | "borderAll"
  | "filter"
  | "adjust"
  | "edit";

type SharedViewLeafItem = {
  kind: "view";
  icon?: SharedViewIconId;
  view: CollectionView;
};
```

设计约束：

- `icon` 可选
- 缺省值不写入存储层，由解析层回退
- 写入时必须校验为受控集合中的合法值
- 非法值按缺省回退处理，不让脏数据污染渲染
- `CollectionView` 自身的内容字段与 `icon` 分层存放
- `normalizeCollectionViewDraft()` 不接收也不产出 `icon`

### 三、默认图标与候选集合

默认图标固定为：

- `borderAll`

首批候选图标建议进一步收窄到 4 个，直接复用现有组件：

- `borderAll`
- `filter`
- `adjust`
- `edit`

选择标准：

- 全部已存在于 `src/components/icons.ts`
- 都是当前产品语义里相对中性的导航/操作图标
- 不依赖新增资源和样式体系

本轮不开放：

- 颜色选择
- emoji
- 任意 icon id 输入
- 按 collection 或 group 自动推荐图标

### 四、状态流与持久化语义

#### 1. 读取

- 视图加载时，解析 shared view 配置中的 `icon`
- 若字段缺失或非法，回退 `borderAll`
- `ViewTabs` 渲染和 drag ghost 全部读取解析后的有效值

#### 2. 修改

- 用户点击 `更换图标`
- 在候选列表中选择某个图标
- 当前视图的共享配置立刻更新
- 当前 tab 图标和菜单中的选中态立即刷新

补充约束：

- 这是直接写 shared view config 的动作
- 不进入 `viewDrafts`
- 不需要点击 `为所有人保存`
- `重置` 不负责回滚图标修改

#### 3. 复制

`创建视图副本` 时：

- 复制来源视图当前有效 `icon`
- 新视图初始图标与来源保持一致

原因是图标属于视图身份快照的一部分，而不是当前用户本地草稿。

#### 4. 重命名 / 删除 / 分组 / 排序

- `重命名`：只改名称，不改图标
- `删除视图`：随视图一起删除图标字段
- 拖入组 / 拖出组 / 排序：不影响图标
- 视图组本身不持有图标字段

#### 5. 新建与组路径

- 顶层 `创建视图副本`：复制来源视图当前有效 `icon`
- 组内创建新视图：使用默认图标，不继承所在组或当前活动视图图标
- 创建新分组并附带首个视图：该首个视图使用默认图标，除非入口语义明确是“复制当前视图进入新分组”
- 复制整个视图组时：组内每个视图各自保留并复制自己的 `icon`

这样定义可以避免“普通新建”和“复制来源快照”语义被混在一起。

### 五、交互形态

`更换图标` 建议采用菜单内二级列表，而不是打开单独弹窗。

推荐交互：

- 主菜单项显示当前图标预览
- 点击后展开二级列表
- 每一项包含：
  - 图标预览
  - 文本名称
  - 当前选中标记

这样做的原因：

- 交互规模小，不值得额外引入 modal
- 用户是在视图 tab 的局部操作上下文中修改图标，菜单内完成更符合心智
- 现有 `显示为` 和 `来源` 都已是“带右箭头的子层级感知”，信息架构保持一致

如果当前 `Popover` 菜单组件不方便做真正嵌套菜单，也可以退一步做成同层列表区块：

- 点击 `更换图标` 后，在当前菜单底部展开候选图标列表

但不建议为本轮引入独立对话框。

### 六、与未来视图类型的边界

本轮必须显式约束：

- 不能因为已经有 `icon`，就让未来视图类型默认等于某个图标
- 未来若增加 `viewType`，也必须是单独字段
- `显示为` 启用后，负责修改的是 `viewType`
- `更换图标` 负责修改的是 `icon`

两者的推荐关系可以在未来再讨论，例如：

- 新建 `board` 视图时默认推荐某个图标

但那只能是默认值策略，不能变成字段合并。

- `ViewTabs` 只负责展示和触发交互
- `shared-view-structure` 负责保证组内新建、复制、结构变换时不丢失 `icon`

## 测试策略

### 1. 状态层测试

需要补这些契约：

- 旧配置没有 `icon` 时，解析结果回退默认图标
- `icon` 非法值时，解析结果回退默认图标
- 修改图标后，只更新目标视图，不污染其他视图
- duplicate 视图时，图标一起复制
- rename / delete / reorder / group move 不意外改写图标
- `normalizeCollectionViewDraft()` 不接收 `icon`，避免把图标纳入个人 draft
- `saveSharedViewDraftsToConfig()` 与图标修改解耦，图标不依赖 save-for-everyone
- 复制整个视图组时，组内每个视图的图标都被保留

### 2. 结构断言测试

更新 `tests/view-state.test.mjs` 中对 `ViewTabs` 源码结构的断言，覆盖：

- `显示为` 仍存在
- `更换图标` 新入口存在
- tab 图标不再固定写死为唯一图标常量，而是从视图数据读取

### 3. e2e 交互测试

至少补两条端到端主流程：

1. 打开某个共享视图菜单
2. 选择 `更换图标`
3. 选中一个非默认图标
4. 断言当前 tab 图标变化
5. 刷新或重新打开后图标仍存在
6. 创建视图副本后，新视图继承同一图标

以及：

1. 在组内视图上更换图标
2. 执行拖出组、拖回组或组内重排中的一种结构变换
3. 断言图标不丢失
4. 如覆盖组复制，则断言复制出的组内视图图标保持一致

如果当前 e2e 不方便直接断 SVG 组件，可为图标节点补稳定的 `data-*` 标识，避免写脆弱选择器。

## 风险与控制

### 风险一：菜单复杂度上升

新增 `更换图标` 后，单个视图菜单项目变多。控制方式：

- 不额外新增弹窗层级
- 图标集合保持小而稳定
- 只把本轮必要能力加进去

### 风险二：把图标误当成类型字段

如果后续开发者看到 `显示为` 与 `icon` 靠得近，可能误把二者耦合。控制方式：

- 在方案、类型命名和代码注释里明确 `icon` 与 `viewType` 分离
- `显示为` 本轮不接入任何 `icon` 状态

### 风险三：旧配置兼容处理不稳

shared view 配置是已有线上/本地真实数据。控制方式：

- `icon` 保持可选
- normalize 负责吞掉缺失值和非法值
- 测试覆盖旧配置读取场景

### 风险四：实现时把 `icon` 错挂到 `CollectionView` / draft

这是本轮最容易返工的技术风险。控制方式：

- 在类型设计中显式区分 shared view leaf metadata 与 `CollectionView`
- 不修改 `normalizeCollectionViewDraft()` 去接纳 `icon`
- 把“不走 save-for-everyone / reset”写成测试契约

## 实施建议

推荐按以下顺序落地：

1. 先补状态层测试，定义 `icon` 默认值、合法值和 duplicate 语义
2. 再扩 shared view leaf metadata、normalize 与 group structure helper
3. 接着补“直接更新图标、不走 drafts/save-for-everyone”的写入入口
4. 再修改 `ViewTabs` 菜单和 tab 图标渲染
5. 最后补 e2e，验证完整交互链与组场景

这样可以确保“图标是共享视图身份的一部分”这条主线先被测试锁定，再做 UI 接线，不会把逻辑写散在菜单事件里。

## 结论

本方案把当前容易误解的 `显示为` 入口正式定义为“未来视图类型切换”的预留位，同时新增一个独立的 `更换图标` 能力，并将图标状态纳入 shared view 配置。

这样做的结果是：

- 当前用户诉求可以被准确满足
- 菜单语义更清楚
- 未来引入真实视图类型时不会和图标模型互相污染
- 本轮实现边界可控，能够沿现有 shared view 架构自然演进

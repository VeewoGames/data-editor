# 共享视图团队保存入口弱化与个人草稿持久化设计

## 方案概述

### 1. 总体目标和范围

本方案的目标，是在不改变“团队共享视图仍需显式发布”这一语义边界的前提下，解决当前 shared view 交互里的两个核心问题：

- `为所有人保存` 按钮在筛选栏中过于显眼，和普通编辑/个人保存混在一起，容易形成错误心理模型
- 使用命名 profile（当前用户主要使用 `Lans`）时，个人 shared draft 的持久化稳定性不足，用户会因为未发布团队配置而误以为个人状态也被丢失

本次范围包括：

- 将团队保存入口从 `ViewFilterBar` 移到右上角 `Toolbar` 小按钮区
- 团队保存入口改为纯图标按钮，并提供明确的可用/不可用态和 tooltip
- 保留 `shared-views.json` 仍需显式发布的现有语义
- 加固命名 profile 模式下 shared draft 的持久化链路，重点覆盖 query / filters / sorts / viewOrderDrafts / structureDrafts
- 补齐对应状态层、组件层和 e2e 回归测试

本次不包含：

- shared view 自动发布到团队配置
- shared draft 与团队共享配置语义合并
- 新增多人冲突检测、版本比较或发布历史
- 改动业务数据 autosave、project config autosave 的既有保存策略

### 2. 各阶段任务概要

#### 阶段一：入口位置与视觉层级重构

主要工作：

- 从 `ViewFilterBar` 移除 `为所有人保存` 文本按钮
- 在 `Toolbar` 中新增团队保存图标按钮，位置位于 `Hidden fields` 按钮左侧
- 调整按钮样式、禁用态、可用态、tooltip 和无障碍标签

预期成果：

- shared view 团队发布入口不再和筛选编辑动作混排
- 用户可以继续显式发布团队配置，但不会被高频文本按钮反复打断

执行顺序：

- 先收敛 Toolbar / ViewFilterBar 的职责边界
- 再落 UI 位置和样式

#### 阶段二：命名 profile 下 shared draft 持久化加固

主要工作：

- 审视 `updateActiveViewDraft()`、`updateSharedViewDraftState()`、`commitProfileSave()`、`flushPendingProfileSave()` 之间的保存时序
- 消除“内存已更新，但 profile 文件未可靠落盘”导致的 shared draft 丢失窗口
- 让 shared draft 的 profile 持久化走统一、串行、可等待的保存队列

预期成果：

- 在 `Lans` profile 下，shared draft 在刷新页面、切文件、切视图和连续编辑后都能稳定恢复
- 用户即使没有点击团队发布，也不会丢掉自己的个人 shared draft

执行顺序：

- 先确认当前不稳定点
- 再统一 shared draft 的保存路径和 flush 时机

#### 阶段三：测试与回归验证

主要工作：

- 更新现有 ViewFilterBar / Toolbar / App 契约测试
- 增加 profile 模式下 shared draft reload / flush / 连续编辑回归
- 覆盖团队保存按钮的新布局和按钮状态

预期成果：

- shared view 入口迁移后，测试仍能证明“团队发布显式、个人草稿稳定”这两个核心约束成立

执行顺序：

- 先补失败测试定义目标行为
- 再实现并跑通验证

### 3. 整体结构框架

本方案按三层收敛：

1. 交互层：
   `Toolbar` 承载低频、全局、命令式动作；`ViewFilterBar` 只承载当前 view 的筛选/排序编辑与 draft 重置
2. 状态层：
   `App.tsx` 负责 shared draft 脏态、按钮可用性、命名 profile 保存队列和 flush 时机
3. 持久化层：
   shared draft 继续保存到 profile 的 `lastActiveViews` / `viewDrafts` / `viewOrderDrafts` / `structureDrafts`，团队发布继续单独写入 `shared-views.json`

## 当前问题与证据

### 当前 shared view 的语义边界

当前实现已经明确把两类保存拆开：

- 业务数据、项目级配置、命名 profile 属于 autosave 域
- shared view draft dirty 独立存在，不自动发布到团队共享配置

对应证据：

- `docs/07_校验与保存机制.md`
- `docs/04_信息架构与交互.md`
- `tests/view-state.test.mjs` 中“shared view drafts behind explicit save-for-everyone actions”的契约

### 当前 UI 结构问题

当前 `ViewFilterBar` 同时承载：

- 排序
- 筛选
- `重置`
- `为所有人保存`

这会把“当前 view 的编辑动作”和“覆盖团队共享配置的发布动作”放在同一视觉层级里。对用户来说，`为所有人保存` 很容易被误解为“把我当前改动保存一下”，而不是“把当前个人草稿发布给所有人”。

### 当前稳定性问题

当前 shared draft 的两条保存路径不一致：

- 浏览器本地模式：直接 `writeLocalSharedViewDrafts(window.localStorage, next)`，同步落盘
- 命名 profile 模式：先更新内存中的 `selectedViewProfileRef.current`，再 `void commitProfileSave(...)`

这意味着在 `Lans` profile 下，shared draft 保存存在以下风险窗口：

- 快速连续编辑时，UI 已更新但 profile 文件落盘尚未完成
- 页面刷新、服务恢复、文件切换等操作发生在异步保存真正完成之前
- 某些 shared draft 更新路径只 `setViewDraftDirty(true)`，但没有形成足够强的“必须等待 profile save 完成”的约束

## 设计决策

### 决策一：保留“团队发布必须显式触发”

本方案不把团队保存改成自动发布。

原因：

- 当前 shared draft 和 shared config 已有明确分层
- 用户要的是弱化发布按钮，以及增强个人草稿稳定性
- 如果改成自动发布，会把个人试验性筛选/排序直接升级成团队默认视图，风险过高

结论：

- `shared-views.json` 仍只通过显式团队保存动作更新
- 个人 draft 继续独立存在，并优先修复其可靠保存问题

### 决策二：Toolbar 承载团队保存，ViewFilterBar 只保留 draft 编辑与重置

调整后的职责边界：

- `Toolbar`
  - 全局搜索
  - autosave 状态反馈
  - profile 切换
  - 团队保存按钮
  - hidden fields
  - reset view
  - theme / rebuild / restart / close 等全局动作
- `ViewFilterBar`
  - 排序
  - 筛选 chips
  - 高级筛选
  - `重置`

结论：

- 团队保存不再是筛选栏的一部分
- 视觉上把它降级为“全局小按钮中的一项低频命令”

### 决策三：团队保存按钮采用纯图标态

按钮行为：

- 位置：`Toolbar` 中，位于 `toolbar-hidden-fields` 左侧
- 形式：`ghost-button icon-button toolbar-action-button`
- 图标：复用现有 `save` 或更贴近“publish/shared save”语义的图标，但本轮优先复用仓内现成图标资源
- tooltip：
  - 可用时：`保存团队共享视图`
  - 不可用时：`当前没有可发布的团队视图改动`
- aria-label：
  - 可保持固定值 `保存团队共享视图`

按钮状态：

- 可用：当前存在 `dirty || viewOrderDirty`
- 不可用：没有可发布 draft，或 `commandSaving === true`
- 进行中：复用现有 `commandSaving` 禁用态，不新增第三种视觉系统

视觉要求：

- 默认弱化为小图标按钮
- 有可发布改动时，用颜色或描边提升可见性，但不恢复成大橙色 CTA

### 决策四：命名 profile 下 shared draft 持久化改成可等待的统一队列

目标不是改变存储介质，而是强化保存一致性。

核心要求：

- 所有 shared draft 更新入口都走同一套 profile 保存调度
- 不能再出现“部分入口 fire-and-forget，部分入口同步写本地”的分裂语义
- 在需要切文件、切项目、刷新、重启、关闭或恢复时，可以可靠等待 pending shared draft profile save 完成

推荐收敛方式：

- 为命名 profile 下的 shared draft 更新提供统一 helper
- helper 在内存状态更新后立即把 profile 保存任务加入现有串行队列
- `flushPendingProfileSave()` 继续作为对外等待点，但 shared draft 相关路径必须确保真的进入这条队列

这里不引入“profile + localStorage 双写镜像”。

原因：

- 双写虽然能增强兜底，但会带来恢复优先级、去重、冲突归属的新复杂度
- 当前问题更像保存时序不稳，而不是 storage 介质完全错误

### 决策五：保持 `重置` 仍在 ViewFilterBar

`重置` 处理的是当前 shared draft，而不是团队共享配置本身。

它仍属于当前 view 编辑上下文，放在 `ViewFilterBar` 里是合理的：

- 用户刚编辑完筛选/排序，最自然的撤销动作仍在附近
- `重置` 是局部草稿动作，不应与 Toolbar 的全局命令混在一起

## 交互与状态设计

### Toolbar 新增字段

`ToolbarSnapshot` 需要新增至少以下语义：

- `sharedViewPublishEnabled: boolean`
- `sharedViewPublishBusy?: boolean`
- `sharedViewPublishTooltip: string`

也可以只传：

- `sharedViewPublishDirty: boolean`
- `commandSaving`

再由 Toolbar 内部推导按钮状态。

推荐优先保持 snapshot 语义显式，避免 Toolbar 再去理解 shared view 脏态细节。

### App 状态计算

`App.tsx` 需要继续维持现有 shared draft dirty 判断：

- active view 内容 draft
- `viewOrderDrafts`
- `structureDrafts`

并把这组状态同时服务于：

- Toolbar 团队保存按钮是否可用
- ViewFilterBar 的 `重置` 是否可见
- 全局 dirty 防护

### ViewFilterBar 收缩

`ViewFilterBar` 去掉：

- `onSaveForEveryone`
- `.view-filter-actions .save-shared`

保留：

- `onResetView`
- 与 dirty / viewOrderDirty 相关的 `重置` 显示逻辑

如果当前没有任何 shared draft，也不显示 `重置`。

## 测试策略

### 组件 / 源码契约测试

需要更新：

- `tests/view-state.test.mjs`

重点断言：

- `Toolbar` 暴露 shared view publish 按钮
- 按钮位置在 `toolbar-hidden-fields` 左侧
- `ViewFilterBar` 不再暴露 `onSaveForEveryone`
- `App` 把 `handleSaveViewForEveryone()` 接到 `Toolbar` 而不是 `ViewFilterBar`

### 状态层测试

重点补充：

- shared draft 更新后，命名 profile 保存任务确实进入串行队列
- 连续两次 shared draft 更新时，后一次不会被前一次覆盖掉
- flush 前后 `selectedViewProfileRef.current` 与落盘内容一致

### e2e 测试

重点覆盖：

- `Lans` / 命名 profile 模式下，普通 filter draft 刷新后仍存在
- 高级筛选 draft 在 profile 模式下刷新后仍存在
- 视图组结构 draft 在 profile 模式下刷新后仍存在
- Toolbar 团队保存按钮在有 draft 时可点击，无 draft 时禁用
- 团队保存后按钮回到不可用态，且 shared config 正常更新

## 风险与边界

### 风险一：只换入口位置，不修保存时序

结果会是视觉问题解决了，但 `Lans` profile 下个人 draft 还是会丢。

因此本轮必须把“入口迁移”和“profile 持久化加固”绑定交付。

### 风险二：误把团队按钮状态与 autosave dirty pill 混用

团队保存是 shared view 发布动作，不应复用普通 autosave 的“待保存/保存中/保存失败”语义。

本轮按钮只能表达“有无可发布草稿”和“当前命令保存是否忙碌”。

### 风险三：为了兜底引入 profile / localStorage 双写

这会把问题从“保存时序”升级成“多源恢复”问题，不适合本轮。

本轮应先把单一 profile 保存链路做稳。

## 验收标准

- `为所有人保存` 不再出现在 `ViewFilterBar`
- Toolbar 中新增团队保存图标按钮，位置在 `Hidden fields` 左侧
- 没有 shared draft 时按钮禁用；有 shared draft 时按钮可用且弱高亮
- 点击团队保存后仍更新团队共享配置，语义不变
- `Lans` profile 下 shared draft 在刷新、切文件和连续编辑后稳定保留
- 现有“团队发布必须显式触发”的测试契约仍成立

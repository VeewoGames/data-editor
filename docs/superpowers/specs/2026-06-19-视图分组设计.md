# 视图分组设计

## 方案概述

### 1. 总体目标和范围

本方案的目标，是把当前 `data-editor` 中扁平的共享视图标签，升级为“顶层视图 + 视图组”并存的两层结构，用来解决视图数量增长后横向标签过长、难以整理、难以定位的问题。

本次设计明确采用“方案 2”：

- 视图组是视图容器，不是新的业务视图类型
- 真正承载筛选、排序、查询、列布局和保存语义的，始终是具体视图
- 第一行负责顶层结构导航
- 第二行负责当前组内视图导航

本次范围包括：

- 共享视图数据模型升级
- `ViewTabs` 两层结构重构
- 顶层与组内双 `+` 创建机制
- 视图拖入组、拖出组、组内排序、顶层排序
- active view / draft / 恢复状态规则收敛
- 旧扁平共享视图配置到新结构的迁移
- 对应单测与 e2e 契约更新

本次不包含：

- 组级筛选、组级排序、组级保存
- 三层及以上嵌套分组
- 组级权限、组级来源类型
- 视图组作为独立页面或独立工作对象

### 2. 各阶段任务概要

#### 阶段一：数据模型与迁移

主要工作：

- 把当前 collection 下的扁平 `views[]` 升级为顶层 `items[]`
- 引入 `group` item 结构
- 保留旧数据自动迁移能力

预期成果：

- 新旧共享视图配置都能被正确读取
- 新结构能稳定表达顶层视图、视图组和组内视图

执行顺序：

- 先定义新结构
- 再补 normalize / save / migration

#### 阶段二：状态层与解析层收敛

主要工作：

- 保持 active 对象仍然只能是具体 view
- 新增“当前展开组”状态
- 升级顺序 draft 结构，能同时表达顶层顺序和组内顺序

预期成果：

- 第一行与第二行都能从统一状态源稳定渲染
- draft、恢复、保存的语义边界保持清晰

执行顺序：

- 先定义状态 contract
- 再替换现有视图解析链

#### 阶段三：交互层重构

主要工作：

- `ViewTabs` 改为两层导航
- 顶层与组内双 `+` 创建入口
- 实现入组、出组、组内排序、顶层排序

预期成果：

- 用户可以像使用 Chrome 标签组一样整理视图
- 但组本身不承载业务语义，仍由具体视图工作

执行顺序：

- 先做静态结构和选中态
- 再接入创建
- 最后补拖拽

#### 阶段四：验证与契约更新

主要工作：

- 更新 shared views normalize / state / e2e 契约
- 验证迁移、恢复、双层高亮、拖拽命中与删除组行为

预期成果：

- 新结构替换旧扁平模型后，关键工作流仍可稳定使用

执行顺序：

- 先补单元测试
- 再补端到端交互测试

### 3. 整体结构框架

本方案把共享视图系统拆成三层：

1. 存储层：`SharedViewsConfig`
2. 解析层：把共享配置和个人 draft 解析成可渲染的顶层项、当前展开组和当前 active view
3. 交互层：`ViewTabs` 两层导航 + 创建入口 + 拖拽

核心边界如下：

- 组只负责收纳和导航整理
- 视图才是真正的工作对象
- draft 只挂具体 view，不挂 group
- 保存“为所有人保存”仍然保存具体视图内容和视图顺序，不产生组级业务配置

## 背景与问题

当前实现中，`ViewTabs` 基于扁平 `views[]` 渲染一排横向标签。随着共享视图数量增加，出现以下问题：

- 标签过长后横向拥挤，扫描成本高
- 语义相近的视图无法收纳
- 新建视图后只能继续平铺，长期会退化
- 当前排序模型只能表达“线性顺序”，无法表达“某个视图属于某个组”

现有代码中，这种扁平假设已经写进多个层面：

- `src/api/client.ts` 的 `SharedViewsConfig`
- `src/view/shared-view-normalize.mjs` 的 normalize 规则
- `src/App.tsx` 中 active view、view order draft 的解析
- `src/components/ViewTabs.tsx` 的渲染和拖拽命中
- `tests/data-editor.spec.ts` 中大量对扁平 tab 数量和顺序的断言

因此本次不是单纯的视觉增强，而是共享视图结构升级。

## 设计原则

### 1. 组是容器，不是视图

组不拥有自己的筛选、排序、查询、列布局和保存语义。所有这些都仍然属于组内具体视图。

### 2. active 只能是具体视图

无论用户当前是否点中了某个组，真正驱动表格、筛选栏和保存行为的，必须是一个具体 view。

### 3. 两层结构，职责清楚

- 第一行负责顶层结构
- 第二行负责当前组内容

不能让一个入口同时隐式承担两层创建职责。

### 4. 平滑迁移旧数据

现有所有扁平视图配置必须自动进入新结构，不能要求用户手工重建。

### 5. 不做防御性兼容的双轨长期保留

项目仍处于早期阶段。迁移完成后，新的两层结构应成为唯一主模型；旧扁平结构只作为读取入口和一次性迁移来源，不长期共存两套业务实现。

## 信息架构

### 顶层结构

第一行展示两类顶层 item：

- 未分组普通视图
- 视图组

它们并列出现。

### 组内结构

点击某个组后，在第一行下方展开第二行。第二行展示：

- 当前组内的子视图
- 组内 `+`

第二行只服务当前展开组，不同时展示多个组的内容。

## 数据模型设计

### 共享配置结构

推荐把当前：

```ts
collections: Record<string, { views: CollectionView[]; defaultViewId: string | null }>
```

升级为：

```ts
collections: Record<string, {
  items: CollectionViewItem[];
  defaultViewId: string | null;
}>

type CollectionViewItem = CollectionViewLeaf | CollectionViewGroup;

type CollectionViewLeaf = {
  kind: "view";
  view: CollectionView;
};

type CollectionViewGroup = {
  kind: "group";
  id: string;
  name: string;
  views: CollectionView[];
};
```

说明：

- 普通视图作为顶层 `kind: "view"` item 出现
- 视图组作为顶层 `kind: "group"` item 出现
- 组内仍然直接承载 `CollectionView[]`
- 组内不再继续嵌套 group

### ID 不变量

为了让现有状态索引、链接恢复和本地布局缓存继续稳定工作，本方案必须把以下约束提升为 collection 级不变量：

- `view.id` 在当前 collection 范围内全局唯一，不区分是否位于组内
- `group.id` 在当前 collection 范围内全局唯一
- `view.id` 与 `group.id` 属于不同命名空间，但各自都必须可稳定反查

这条约束必须在：

- normalize
- create
- duplicate
- migration
- save 前校验

几个环节同时成立，不能只靠 UI 层“尽量不撞”。

### 为什么不直接给每个 view 加 `groupId`

虽然给 view 增加 `groupId` 看起来改动更小，但那只是“扁平数据模拟层级关系”。后续一旦需要：

- 顶层排序
- 组内排序
- 删除组后提升子视图
- 组重命名
- 组展开与选中恢复

逻辑就会不断围绕“从扁平列表推导树结构”打补丁。长期维护成本高于直接建真结构。

### 迁移规则

旧结构读取时：

- 旧 `views[]` 中的每个视图，都迁移成顶层 `kind: "view"` item
- 不自动生成默认组
- `defaultViewId` 保持原值

这样迁移后，旧项目行为不会突然变化；只有用户主动创建组或拖拽入组时，才开始使用新能力。

迁移后的默认不变量：

- 所有旧视图初始都属于未分组顶层视图
- 不生成空组
- 不生成隐式“未分组”系统组

## 状态模型设计

### active view

active 仍然记录具体 `viewId`，不允许 group 作为 active 对象。

理由：

- `ViewFilterBar` 只认识具体视图
- 表格渲染只认识具体视图
- draft 保存和“为所有人保存”只应该落到具体视图

### expanded group

新增“当前展开组”状态，表示第二行当前显示哪个组的内容。

它和 active 的关系：

- active 是当前工作视图
- expanded group 是当前导航容器

当 active 是组内视图时，expanded group 必须与该视图所属组一致。

### 状态归属

本次新增状态必须先定义 owner，避免实现时分散到多套存储：

- 团队共享真相：
  - 顶层结构
  - 组名
  - 组成员关系
  - 顶层顺序
  - 组内顺序
- 个人共享视图草稿：
  - 结构草稿
  - 具体 view 内容草稿
- 页面级本地上下文：
  - 当前展开组
  - 每个组最近一次激活的子视图

具体约束如下：

- `expandedGroupId` 不写入共享配置
- `expandedGroupId` 不写入共享视图 draft
- `expandedGroupId` 只属于当前用户、当前页面上下文恢复信息
- `lastActiveViewIdByGroupId` 属于页面级本地上下文，不写入共享配置，也不写入共享视图 draft
- `lastActiveViews` 继续只承担“collection 级最后一个 active view”职责，不复用为 group 级最近访问记录

也就是说：

- 共享层只关心结构真相
- draft 层只关心可发布改动
- page context 只关心当前浏览位置

### last active in group

当用户点击组时，需要决定第二行里默认激活哪个视图。推荐规则：

- 先恢复该组在页面级本地上下文中的上次激活子视图
- 若没有历史记录，则激活组内第一个视图

固定 fallback 顺序如下：

1. `lastActiveViewIdByGroupId[groupId]`
2. 当前组第一个子视图

这条顺序应直接进入实现和测试，不留自由推导空间。

### draft 规则

保留现有原则：

- `viewDrafts` 只挂具体 `viewId`
- group 不产生自己的内容 draft

但顺序 draft 需要升级。

当前 `viewOrderDrafts: string[]` 只能表达线性顺序，无法表达：

- 某个 item 是顶层 group
- 组内有一串 view

因此需要升级成能表达层级顺序的结构，例如：

```ts
type ViewOrderDraft = {
  topLevel: Array<
    | { kind: "view"; viewId: string }
    | { kind: "group"; groupId: string; viewIds: string[] }
  >;
};
```

具体字段名可以调整，但必须满足两个条件：

- 能表达顶层顺序
- 能表达每个组内部顺序

同时要把“结构草稿”和“内容草稿”在概念上拆开：

- 内容草稿：具体 view 的 `query`、`filters`、`sorts`
- 结构草稿：组名、顶层顺序、组内顺序、成员归属关系

虽然实现上可以继续放在同一个 profile/local draft 容器里，但语义上必须分清，避免后面把 group 结构错误地绑到某个 active view 上。

本方案中，结构草稿统一命名为：

- `shared view structure draft`

它至少包含：

- 顶层顺序
- 组内顺序
- 组成员关系
- 组名

### 解析层中间模型

为了避免 group 判断散落到 UI、保存、恢复与拖拽各处，本方案要求在 `App` 和视图状态层之间先引入统一的解析结果结构，再开始改 UI，例如：

```ts
type ResolvedCollectionViews = {
  topLevelItems: ResolvedTopLevelItem[];
  activeViewId: string | null;
  activeGroupId: string | null;
  expandedGroupId: string | null;
  activeView: CollectionView | null;
  viewsById: Record<string, CollectionView>;
  parentGroupIdByViewId: Record<string, string | null>;
};
```

关键要求不是字段名，而是所有下游都从同一份 resolved 结构消费：

- `ViewTabs`
- 保存/发布
- reset
- URL 恢复
- 拖拽

不能在每一层各自重新从 `items[]` 推导。

## 交互设计

### 第一行：顶层导航

第一行展示：

- 未分组普通视图 tab
- 视图组 tab
- 顶层 `+`

点击普通视图：

- 直接切换 active view
- 若第二行之前展开的是某个组，可保持该组展开，也可在切到未分组视图时收起第二行

本方案推荐：

- 当切到未分组视图时，收起第二行

理由是当前工作对象已经离开组上下文，继续保留第二行会制造视觉噪声。

点击组：

- 展开第二行
- 激活该组上次使用的子视图；若没有则激活第一个

### 第二行：组内导航

第二行仅在某个组展开时出现，展示：

- 当前组的子视图 tab
- 组内 `+`

点击子视图：

- 切换 active view
- 第一行组 tab 保持选中态

### 高亮规则

当 active 为组内视图时：

- 第一行对应组 tab 高亮，表示当前所在组
- 第二行对应子视图高亮，表示当前具体工作视图

两者都高亮，但视觉主次要区分：

- 第一行强调“导航容器”
- 第二行强调“当前工作对象”

### Group 操作入口

组不能只存在视觉壳，而没有完整操作入口。推荐 group 与普通 view 一样，拥有自己的 tab 菜单入口，但菜单项按 group 类型裁剪。

group 菜单至少应包含：

- 重命名组
- 删除组
- 在组内创建视图

group 菜单不应包含：

- 编辑视图
- 创建视图副本
- 拷贝视图链接

原因是 group 不是工作视图，不拥有具体视图语义。

### 创建机制

创建入口明确分为两个 `+`。

#### 顶层 `+`

位置：

- 第一行最右侧，永远显示

行为：

- 打开 action popover
- 菜单项为：`创建视图`、`创建视图组`

规则：

- 通过顶层 `+` 创建的普通视图，默认是未分组视图
- 顶层 `+` 不根据当前是否展开组而隐式改成组内创建

#### 组内 `+`

位置：

- 第二行最右侧，永远显示
- 仅在某个组展开时出现

行为：

- 只负责在当前组内创建子视图

规则：

- 不提供“创建视图组”
- 新建子视图默认插入当前组末尾

#### 创建视图组

当用户通过顶层 `+` 选择“创建视图组”时：

- 创建一个新组
- 自动生成一个默认子视图
- 进入命名态

原因：

- 空组在当前产品里没有可操作性
- 用户最终工作的对象仍然是具体视图

### 空组不变量

本方案明确规定：组不能为空。

所以任何可能导致空组出现的操作，都必须立刻收敛成非空结果，而不是留下悬空 group。

具体规则：

- 创建组时，必须自动创建一个默认子视图
- 若用户删除组内最后一个子视图：
  - 不保留空组
  - 直接删除该组
- 若用户把组内最后一个子视图拖出到顶层：
  - 不保留空组
  - 拖拽完成后自动删除该组

这条规则能让“组只是容器”的定义保持一致，避免后续又为“空组展示什么、怎么保存、怎么恢复”追加复杂度。

### 拖拽机制

#### 顶层排序

第一行支持顶层 item 拖拽排序，包括：

- 未分组普通视图之间排序
- 组之间排序
- 普通视图与组的相对顺序调整

#### 拖入组

普通视图拖到组 tab 本体时：

- 该组高亮为接收目标
- 松手后视图进入该组

这是快速入组路径。

#### 组内精确插位

组展开时，拖到第二行具体子视图之间：

- 显示 before/after 命中位置
- 松手后按精确位置插入

这是精细排序路径。

#### 拖出组

组内子视图可拖回第一行：

- 松手后成为未分组顶层视图

这条路径必须存在，否则用户只能入组不能出组。

若该视图原本是组内最后一个子视图，则：

- 拖拽完成后自动删除原组
- 顶层落点以最终拖拽位置为准

## 删除与重命名规则

### 删除视图组

已确认规则：

- 删除组时，不删除组内子视图
- 组内子视图提升到顶层，位置尽量落在原组附近

理由：

- 最安全
- 最不容易造成误删
- 更符合“组是容器，不是数据所有者”的定义

### 重命名

- 普通视图可重命名
- 组也可重命名
- 组内默认新建视图和新建组后的默认子视图，都应支持直接进入命名态

### 复制规则

当前系统已有“创建视图副本”，分组后必须补齐层级语义：

- 复制未分组普通视图：
  - 副本仍为未分组视图
  - 默认插在源视图后面
- 复制组内普通视图：
  - 副本仍留在原组内
  - 默认插在源视图后面
- 视图组不支持“创建组副本”

这样可以让 duplicate 继续是“复制当前工作视图”，而不是偷偷演变成结构复制功能。

## 保存与发布规则

“为所有人保存”在新方案下的主规则是：

- 发布当前 active view 的内容草稿
- 发布当前 collection 的 `shared view structure draft`

其中 `shared view structure draft` 不再是简单扁平顺序，而是包含：

- 顶层 item 顺序
- 各组内部顺序

组本身不产生独立业务配置，仅产生结构配置。

### 纯结构变更的发布语义

本次新增大量“没有 active view 内容变化，只有结构变化”的操作：

- 创建组
- 重命名组
- 入组
- 出组
- 顶层排序
- 组内排序
- 删除组

因此必须明确：

- `shared view structure draft` 可以独立于内容草稿存在
- 即使当前 active view 没有内容变更，只要 `shared view structure draft` 存在，也允许“为所有人保存”
- `viewOrderDirty` 需要升级为能覆盖全部 `shared view structure draft` 的 dirty 标志，而不是只表示扁平 tab 顺序脏
- 组名修改明确属于 `shared view structure draft`，不允许直接绕过 draft 即时写入共享配置

### reset 语义

`重置` 也必须同时覆盖两类草稿：

- 当前 active view 的内容草稿
- 当前 collection 的结构草稿

若当前只有结构草稿，没有内容草稿：

- `重置` 仍应可见
- 执行后丢弃组结构相关改动

若当前 active view 位于某个新建但未发布的组内，且用户执行 `重置`：

- 该组及其未发布结构变化一并回退
- active 与展开状态按回退后的 resolved 结构重新计算

## 恢复与页面上下文

刷新或回到页面时，需要恢复：

- 当前 active view
- 当前展开组
- 该 active view 对应的第二行状态

若恢复到的是未分组视图：

- 第二行不显示

若恢复到的是组内视图：

- 自动展开对应组

### 链接恢复规则

当前“拷贝视图链接”仍然只以 `viewId` 作为链接真相，不新增 `groupId` 作为外部链接参数。

恢复时的规则是：

- 先根据 `viewId` 解析具体视图
- 再由解析层反查所属组
- 若该视图位于组内，则自动展开对应组

这样可以避免 URL 同时携带 `viewId` 与 `groupId` 形成双真相。

## 风险与约束

### 1. 扁平假设扩散较深

当前视图系统不只是组件层扁平，状态与测试也都建立在扁平假设上。因此需要避免“UI 先行，状态后补”的半成品改法。

### 2. draft 结构升级是关键风险点

如果顺序 draft 还停留在线性 `string[]`，就会不断出现“组内顺序发布后丢失”或“顶层顺序与组内顺序互相覆盖”的问题。

### 3. active 语义不能漂

一旦允许 group 作为 pseudo-active，筛选栏、保存按钮和恢复逻辑都会开始出现边界混乱。必须坚持“active 只能是具体 view”。

### 4. 删除组必须防止误删心智

虽然设计上已经决定“提升子视图而不是删除”，实现时仍要确保确认文案和结果一致。

## 推荐结论

本方案推荐采用“组作为容器”的两层视图结构，而不把组升级成真正的导航页面层级。

这是因为当前 `data-editor` 的核心工作对象仍然是具体视图，用户需要解决的是“视图太多，需要整理”，而不是“视图系统要升级成完整多级导航”。

因此最合适的长期结构是：

- 数据上建真层级
- 语义上保持 view 为唯一工作对象
- 交互上提供两层导航和双 `+`
- 状态上只为 group 引入必要的容器状态，不引入组级业务状态

这能以最低必要复杂度支持：

- 分组收纳
- 双层导航
- 双层创建
- 拖入组
- 拖出组
- 组内排序
- 顶层排序

同时为未来继续扩展预留空间，但不过早把系统升级成重型导航框架。

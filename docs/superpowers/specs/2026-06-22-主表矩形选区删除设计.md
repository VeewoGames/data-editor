# 主表矩形选区删除设计

## 概述

### 1. 总体目标和范围

本设计的目标是在 data-editor 主表中增加最小可用版矩形选区删除能力，让用户可以通过鼠标拖拽建立连续矩形选区，并用 `Delete` 一次性清空选区内多个单元格的内容。

本次范围只覆盖主表 `tbody` 内当前可见的数据单元格，不包含表头、行删除按钮列、尾部占位列，也不扩展到右侧 detail panel。第一版必须覆盖以下字段类型：

- `Text`
- `Number`
- `Checkbox`
- `Select`
- `Multi-select`

本次不追求完整 Excel 风格模型，不包含复制粘贴、跨视口自动扩展、键盘扩选、批量填充、右键菜单或跨 detail/main 的统一选区。

### 2. 各阶段任务概要

#### 阶段一：建立主表选区模型

主要工作：在表格层引入单元格锚点、拖拽过程、矩形范围和命中单元格集合。  
预期成果：主表能稳定识别“按下-拖拽-抬起”的矩形选区，并提供稳定 DOM 标记和选中样式。  
执行顺序：先补表格坐标与命中模型，再接入鼠标事件。

#### 阶段二：接入删除语义与写回

主要工作：将 `Delete` 绑定到当前矩形选区，按字段类型映射为清空值，再复用现有写回链路批量提交。  
预期成果：选区内支持类型的单元格可一次性清空；不支持或不可编辑单元格被安全跳过。  
执行顺序：先定义清空值合同，再接入批量遍历和跳过策略。

#### 阶段三：处理交互冲突与回归验证

主要工作：明确拖拽优先于 cell 内部点击行为，避免误开 detail、误进入文本编辑、误弹出离散值弹层，并补测试。  
预期成果：拖拽删除可用，同时不破坏现有单击交互。  
执行顺序：先收敛优先级规则，再补自动化断言。

### 3. 整体结构框架

整体结构采用“表格层集中管理选区状态，单元格 renderer 只消费结果”的方式：

1. `DataTable` 负责选区状态、拖拽事件、键盘删除和批量清空调度。
2. `table-columns` / `CellRenderer` 继续负责各类单元格渲染，不自行维护矩形选区逻辑。
3. 现有 `onEditCell` / option draft commit 继续作为写回入口，不新增独立批量存储协议。
4. 样式层只根据表格层暴露的稳定属性渲染选中态，不反向驱动业务逻辑。

## 背景与现状

当前主表只有“行选中”，没有“单元格选中”或“矩形选区”：

- 行点击由 `DataTable` 的 `selectRow` 处理，主状态仍是 `selectedRow` / `selectedRowId`。
- 主表全局键盘逻辑当前只处理保存和搜索快捷键，没有 `Delete` 批量清空入口。
- 文本格、标题格、`Select`、`Multi-select`、`Relation` 等单元格各自有点击语义，若直接在各 renderer 内分散实现选区，后续会产生大量冲突。

因此，这次功能本质上是给主表补一层新的交互模型，而不是给现有 editor 追加一个快捷键。

## 设计目标

### 必须满足

1. 用户可通过鼠标拖拽在主表建立连续矩形选区。
2. 松开鼠标后选区保留，按 `Delete` 可批量清空。
3. `Select` 与 `Multi-select` 必须包含在第一版内。
4. 真正发生拖拽时，不触发单元格原有点击行为。
5. 单击且未形成拖拽时，现有主交互保持不变。

### 明确不做

1. 不做复制、剪切、粘贴。
2. 不做 `Shift + 方向键` 或 `Shift + 点击` 扩选。
3. 不做跨滚动区域自动延展。
4. 不做 detail panel 与主表共享同一套选区。
5. 不做 relation/document/nested/backlink 的第一版批量清空。

## 方案比较

### 方案 A：表格层集中管理选区状态

在 `DataTable` 维护：

- 锚点单元格
- 当前悬停终点单元格
- 是否处于拖拽中
- 当前矩形范围
- 当前命中的单元格集合

所有鼠标拖拽和 `Delete` 调度都在表格层完成，各类单元格 renderer 只接收“当前是否被选中”和“当前是否允许自身点击继续执行”的结果。

优点：

- 选区模型集中，和现有 `activeTextCellId`、滚动窗口、列拖拽等运行时状态处于同一层级。
- 可以统一处理拖拽优先级，避免 `TextCellSurface`、`SelectCellEditor`、`MultiSelectCellEditor` 各自实现不一致。
- 更容易做矩形命中、批量遍历和测试。

缺点：

- 需要给主表单元格补稳定的行列坐标和 DOM 标记。
- 要在表格层补一层“拖拽和单击区分”的事件治理。

### 方案 B：各类单元格 renderer 自行上报选区事件

每个 cell surface 单独处理 `mousedown` / `mouseenter` / `mouseup`，再把结果往上汇总。

优点：

- 局部接入看起来直接。

缺点：

- 现有 cell 类型多，点击语义分散。
- `title`、文本编辑态、离散值弹层、relation 打开目标等都容易互相打架。
- 后续维护成本高，测试覆盖也更碎。

### 结论

采用方案 A。矩形选区属于主表交互协议，应该由 `DataTable` 集中管理，而不是下沉到每个 cell editor。

## 交互合同

### 建立选区

1. 用户在某个数据格按下鼠标，记录为锚点单元格。
2. 鼠标拖过其他数据格时，根据锚点和当前命中格的“可见行索引 + 可见列索引”实时计算矩形范围。
3. 鼠标抬起后，选区固定保留。
4. 再次单击其他数据格时，旧选区被新的单格选区替换。
5. 单击 `title` cell 时，仍保留单格选区可见态，然后继续执行打开 detail 的既有行为。

### 拖拽与单击的判定

1. `mousedown` 后如果没有跨出锚点格，按普通单击处理。
2. 只要指针移动导致命中格变化，即认定为拖拽选区。
3. 一旦进入拖拽态，本次手势内不再触发单元格原有点击行为。

### Delete 清空语义

已确认的第一版清空合同如下：

- `Text` -> `""`
- `Number` -> `""`
- `Checkbox` -> `false`
- `Select` -> `null`
- `Multi-select` -> `[]`

以下单元格在第一版中直接跳过：

- `Relation`
- `Document`
- nested object summary
- backlink
- 不兼容单元格
- 非可编辑状态下不允许写回的单元格

### 与现有交互的优先级

1. 拖拽优先于 cell 内点击行为。
2. 真正发生拖拽时：
   - 不打开 title detail
   - 不进入文本编辑态
   - 不打开 `Select` / `Multi-select` 弹层
3. 单击且未拖拽时：
   - `title` cell 先呈现单格选区，再按现有行为打开 detail
   - 文本格仍可进入编辑态
   - `Select` / `Multi-select` 仍按现有方式工作

## 架构与职责拆分

### 1. `DataTable`

新增职责：

- 保存矩形选区运行时状态
- 维护从 DOM 单元格到“可见行索引/字段名/显示类型”的映射
- 处理 `mousedown` / `mouseover or pointerenter` / `mouseup`
- 处理 `Delete`
- 按类型生成批量清空操作并分发到现有写回入口

不新增职责：

- 不解析各类 editor 内部细节
- 不承担新的持久化协议

### 2. `table-columns` / `CellRenderer`

保留各字段类型渲染职责，只做最小配合：

- 提供表格层可识别的稳定 cell 元信息
- 在拖拽态时允许外层阻断自身点击链路
- 必要时暴露当前字段 display type 给表格层

### 3. 写回链路

复用现有入口：

- 普通值类型通过 `onEditCell`
- `Select` / `Multi-select` 不通过 `onEditCell` 直写，而是走独立的 option field 清空适配层
- option field 清空适配层负责把“清空选中值”翻译成现有 draft commit 协议，再分别调用 `onCommitSelectDraft` / `onCommitMultiSelectDraft`

第一版不新增新的 batch API。批量删除仍是“选区遍历 + 多次现有写回调用”的组合。

#### option field 清空适配合同

由于当前主表对 `Select` / `Multi-select` 暴露的是 draft commit 入口，而不是通用 `onEditCell` 值写回入口，批量删除必须显式定义一个值级适配层，避免 `DataTable` 直接理解离散值编辑器的内部草稿结构。

建议适配层职责如下：

1. 输入：单元格当前字段类型、当前值、当前 options。
2. 输出：
   - `Select`：生成“下一选中值为空”的 commit patch，并调用 `onCommitSelectDraft`
   - `Multi-select`：生成“下一选中值为空数组”的 commit patch，并调用 `onCommitMultiSelectDraft`
3. 约束：
   - 不新增新的批量存储协议
   - 不要求 `DataTable` 理解 `OptionFieldEditor` 的完整 UI 草稿细节
   - 只收敛“清空当前选中值”这一条最小语义

## 数据模型

建议在表格层引入一个轻量运行时结构：

```ts
type TableCellCoord = {
  rowId: string;
  visibleRowIndex: number;
  fieldName: string;
  visibleColumnIndex: number;
};

type TableSelectionRect = {
  anchor: TableCellCoord | null;
  focus: TableCellCoord | null;
  dragging: boolean;
};
```

派生结果而不是长期持久化：

- 当前矩形边界
- 当前命中的 row/field 组合
- 当前是否为单格选区
- 当前是否存在需先关闭的活跃 option field popover

该状态只存在于当前表格运行时，不进入项目配置、profile 或 local storage。

## DOM 合同与样式建议

主表数据格需要补稳定标记，至少包括：

- `data-row-id`
- `data-column-field`
- `data-view-row-index`
- `data-visible-column-index`
- `data-cell-selected="true|false"`

其中：

- `data-view-row-index` 指当前过滤/排序后的整张视图里的可见行索引，不是当前虚拟窗口切片内的局部索引。
- 列索引同样按当前视图下的可见列顺序计算，不包含行操作列与尾部占位列。

样式目标：

1. 单元格选中态要明显，但不能破坏现有 issue/icon/button 可见性。
2. 矩形内部和边界尽量有区分，便于用户识别选区范围。
3. 视觉样式只挂在数据格，不影响表头和行操作列。

第一版可以先用统一背景 + 细边框，不要求一次做到 Excel 级四边框细节。

## 删除流程

建议流程如下：

1. 用户完成拖拽，表格层得到当前选区。
2. 用户按 `Delete`。
3. 表格层过滤选区内所有可清空单元格。
4. 按单元格 display type 计算目标值。
5. 若命中单元格存在已打开的 `Select` / `Multi-select` popover，先关闭 popover，再执行删除。
6. 逐个调用现有写回入口或 option field 清空适配层。
7. 如果选区内没有任何可清空单元格，则直接忽略本次按键。

这样可以最大程度沿用现有 `mutate`、校验刷新和 dirty 标记链路，避免另起一套批量写回协议。

## 风险与处理

### 1. 拖拽误触发单击行为

风险最高。title、文本格、离散值编辑器目前都依赖点击进入交互。  
处理：把“是否已形成拖拽”作为最高优先级状态，一旦成立，阻断本次点击链路。

### 2. 虚拟滚动与可见行坐标

当前表格有窗口化和 wrapped 行高测量。  
处理：矩形计算基于当前可见窗口内的稳定 `visibleRowIndex`，不直接依赖 source row index 的连续性。

### 3. `Select` / `Multi-select` 的清空方式与现有 editor 状态冲突

这些单元格当前有本地草稿和弹层状态。  
处理：第一版规则改为“若当前命中的 `Select` / `Multi-select` popover 已打开，则先关闭，再执行删除”；因此实现上必须补一条可被表格层消费的活跃 option field 状态，而不是继续假设它们和文本编辑一样已有全局注册入口。

### 4. 性能与大选区遍历

第一版只处理当前可见区域，且复用现有写回函数，性能压力可控。  
后续若要支持跨大范围或不可见区域扩展，再考虑真正的 batch 写回。

### 5. title cell 的单击双重语义

`title` cell 既是普通数据格，也是打开 detail 的入口。  
处理：第一版明确保留“单格选区可见态 + 打开 detail”的双重结果，不把 `title` 排除在单击选区展示之外；只有发生真实拖拽时，才阻断其打开 detail。

## 测试策略

### 单元测试 / 结构断言

至少补以下断言：

1. 表格层存在矩形选区状态与稳定数据格标记。
2. `Delete` 对不同 display type 映射到正确清空值。
3. 不支持类型会被跳过，不产生写回。

### 交互测试

至少覆盖：

1. 鼠标拖拽可建立矩形选区。
2. `Delete` 可批量清空 `Text`、`Number`、`Checkbox`、`Select`、`Multi-select`。
3. 单击 title 时仍打开 detail。
4. 拖拽经过 title 区域时不会误开 detail。
5. 拖拽经过文本格时不会误进入编辑态。
6. 拖拽经过 `Select` / `Multi-select` 时不会误开弹层。

## 验收标准

满足以下条件即可视为第一版完成：

1. 主表支持鼠标拖拽建立可见矩形选区。
2. 选区在鼠标抬起后保留。
3. `Delete` 能正确清空支持类型。
4. 不支持类型被稳定跳过。
5. 非拖拽单击交互不回归。
6. 代码结构仍保持“表格层管选区，renderer 管渲染”的职责边界。

## 暂不扩展的后续方向

本设计故意把范围控制在最小可用版。若后续继续扩展，可以在第二阶段考虑：

- `Shift + 点击` / 键盘扩选
- 复制/粘贴
- relation/document 的批量清空语义
- 跨不可见区域的更完整选区模型
- 批量写回事务优化

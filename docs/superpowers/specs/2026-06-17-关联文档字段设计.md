# 关联文档字段设计

## 概述

### 1. 总体目标和范围

本方案目标是在 `data-editor` 中增加一种新的特殊字段能力「关联文档」。

它服务的不是结构化数据之间的 `relation`，而是“记录主键 ID 与本地 Markdown 文档资产”的一一对应关系。用户可以在表格和详情面板中像查看普通字段一样查看这个字段，但系统会基于字段值自动定位唯一的 Markdown 文档，并在详情面板中额外展示该文档的渲染预览。

在架构前提上，本方案新增一条明确决策：`data-editor` 不自行重新发明 Markdown 文档栈，而是以 `C:\Code\md-reader` 的文档子系统为 source of truth，但接入按阶段收敛推进。本次关联文档字段只在第一阶段接入只读渲染语义、必要 transform、文档排版样式和受控读取链路；可视编辑、写回与冲突处理明确留到第二阶段。本次关联文档字段是两产品未来合并前的第一条正式集成链路。

本方案范围包括：

- 新字段语义与配置模型
- 每个数据文件独立配置文档根目录 `docRoot`
- 基于主键 ID 定位唯一 Markdown 文档
- 表格与详情字段的显示规则
- 详情面板额外文档预览区
- 以 `C:\Code\md-reader` 为 source of truth 的只读 Markdown 渲染接入
- 为后续复用 `md-reader` 可视编辑器预留接入边界
- 最小可行校验、候选列表、缓存策略

本方案不包括：

- 第一阶段不在 `data-editor` 内开放 Markdown 正文编辑
- 双栏源码/预览模式
- 全文搜索
- 多文档关联
- 用户手填任意文件路径

### 2. 各阶段任务概要

#### 阶段一：定义字段与配置语义

主要工作：

- 定义新的字段类型 `Document`
- 定义数据文件级 `docRoot`
- 明确字段值存储为业务主键 ID，而不是文件路径
- 明确 `docRoot` 下 `<id>.md` 必须唯一匹配

预期成果：

- 关联文档字段从概念上与 `Text`、`Relation`、`Backlink` 清晰分离
- 配置边界稳定，后续实现不需要在 `Text` 或 `Relation` 上堆特判

执行顺序：

1. 先补配置模型
2. 再补字段语义
3. 最后定义解析规则与错误状态

#### 阶段二：接入读取、索引、第三面板与展示链路

主要工作：

- 让后端在受控范围内允许读取 `docRoot` 下的 `.md`
- 生成 `docRoot` 下的 Markdown 文档索引
- 提供标题提取与正文读取能力
- 新增详情区左侧文档面板及其持久化状态
- 在表格、详情字段、详情额外文档区接入显示

预期成果：

- 字段可显示文档名称
- 详情面板可显示 Markdown 渲染预览
- 同一条记录的文档读取不会无意义重复请求

执行顺序：

1. 先打通后端文件访问与索引
2. 再接前端字段显示
3. 再接第三面板壳层与状态持久化
4. 最后接详情文档预览区

#### 阶段三：补齐校验、交互与异常状态

主要工作：

- 处理空值、缺失文档、重名冲突
- 限制候选值只能来自现存 Markdown 文件名
- 明确详情联动规则
- 明确缓存和回退策略

预期成果：

- 用户能看到明确、可诊断的状态
- 不会出现“隐式命中某个同名文档”的不确定行为

执行顺序：

1. 先补后端校验与解析状态
2. 再补前端状态展示
3. 最后补测试与文档

### 3. 整体结构框架

- 配置层：每个数据文件声明 `docRoot`，字段声明为 `Document`
- 解析层：字段值 `id` -> `docRoot` 下唯一 `<id>.md`
- 数据层：读取 Markdown 原文，提取一级标题与正文
- UI 层：字段显示文档名称，详情面板额外展示 Markdown 预览
- 校验层：空值、缺失、冲突三类状态可见化

## 背景与当前事实

### 当前 relation 机制的边界

现有 `relation` 机制表达的是“某个字段引用另一份结构化数据中的主键”，配置核心是：

- `targetFile`
- `targetCollection`
- `targetKey`
- `mode`
- `titleFields`

保存时字段只写回原始 id 或 id 数组，显示 label 由运行时根据目标数据生成。

这说明当前系统已经接受“存原始值，显示层再补 label”的模型，但 `relation` 的目标仍然是结构化记录，不是 Markdown 文件。

### 当前文档能力的边界

当前后端文件服务主要围绕数据文件读写展开，文档 API 读取链路默认按 JSON/CSV 文档模型处理；Markdown 文档还没有进入字段语义链路，也没有独立的详情预览表面。

因此，本需求不能简单理解为：

- 给 `Text` 字段套一个显示插件
- 或给 `Relation` 字段换一个目标类型

它需要补的是一条新的“文档引用字段”能力链路。

### `md-reader` 已有能力的边界

`C:\Code\md-reader` 已经具备以下与本需求直接相关的能力：

- 只读 Markdown 渲染组件
- 可视 Markdown 编辑器封装
- 文档扫描、读取、写回与冲突检测
- 文档 typography token 和阅读表面样式
- 只读渲染与编辑表面的样式一致性测试

这意味着本需求的关键问题不再是“能否做 Markdown 预览/编辑”，而是“如何把 `md-reader` 的文档子系统以稳定边界接入 `data-editor`，而不是复制一份逐渐漂移的实现”；同时也意味着第一阶段必须控制复用边界，避免把依赖整合风险一起带入字段功能首版。

## 需求收敛结果

### 已确认约束

- 每个数据文件可以单独配置自己的固定文档根目录 `docRoot`
- 关联方式不是存路径，而是“字段值的主键 ID 与文档文件名一致”
- 文档解析规则固定为 `<主键ID>.md`
- 每个字段值只允许唯一命中一个文档
- 不允许一个字段关联到多个文档
- 字段显示名称优先取 Markdown 首个一级标题，没有标题时回退文件名
- 打开详情面板时，需要额外独立展示关联文档的 Markdown 预览
- 字段候选值严格限制为 `docRoot` 下已存在 Markdown 文件名去掉 `.md` 之后的结果

### 由这些约束推导出的结论

1. 字段值不应该存 Markdown 链接字符串  
   否则会把引用目标和显示标题揉到一个文本值里，增加解析、重命名和校验复杂度。

2. 字段值也不应该存相对路径  
   因为需求已经明确文档定位由“主键 ID = 文件名”决定，路径应该从 `docRoot` 与文件索引推导，而不是写入业务数据。

3. 该能力不应挂到现有 `relations` 配置上  
   因为它没有 `targetCollection/targetKey` 这套目标记录结构，而是“当前字段值就是目标文档 ID”。

## 方案选择

### 方案 A：新增独立字段类型 `Document`

字段语义独立存在，配置上增加文档根目录与文档字段声明，解析逻辑独立于 `relation`。

优点：

- 语义最清晰
- 与 `Relation` / `Backlink` 边界明确
- 详情面板独立文档页的职责自然
- 后续扩展缓存、失效提示、文档元数据更顺

缺点：

- 需要补完整的新字段链路

### 方案 B：复用 `Relation` 外壳，增加 `document` kind

把文档字段塞进现有 relation 框架，用特别配置表示文档目标。

优点：

- 可复用部分选择器外观和打开目标行为

缺点：

- `relation` 语义被污染
- 目标文件不再是结构化集合，会导致配置模型失真
- 后续维护会在解析、校验、菜单、详情联动上不断出现分叉特判

### 方案 C：保留 `Text`，增加“按文档解释”的显示插件

字段仍是普通文本，只在 UI 上按文档引用解析。

优点：

- 初期改动最小

缺点：

- 数据层没有结构约束
- 字段菜单、候选值、校验、详情文档页都要到处写特判
- 会把正式能力做成脆弱的显示 hack

### 推荐方案

采用方案 A：新增独立字段类型 `Document`。

原因：

- 需求已经不只是“显示一个可点击名字”，而是完整的文档引用能力
- “主键 ID -> 唯一 Markdown 文档”的模型天然是结构化字段能力
- 详情面板额外文档页也更适合挂在独立字段语义之下

## 架构补充决策

### 文档子系统来源

`Document` 字段功能仍然归属于 `data-editor`，但 Markdown 文档域本身不再由 `data-editor` 重新实现，而应明确以 `C:\Code\md-reader` 为 source of truth。

本次先做的不是“照着 md-reader 再做一套像它的界面”，而是“沿用同一套文档能力语义”。对产品体验来说，这意味着未来两个产品里的 Markdown 阅读和编辑应被用户感知为同一种文档系统，而不是两个长得相似但细节不同的模块。

第一阶段推荐复用范围：

- 只读渲染：`src/document-renderer/readonly-markdown-renderer.tsx`
- 只读渲染实现：`src/document-renderer/readonly-markdown-renderer-impl.tsx`
- 文档排版样式：`src/index.css` 中 `.markdown-document` / `.readonly-markdown-renderer` / `--doc-*`
- 文档相关 transform 与兼容规则

第二阶段再扩展到：

- 可视编辑：`src/editor/visual-markdown-editor.tsx`
- 写回与冲突处理：`server/project-reader.mjs`、`server/project-writer.mjs`

### 复用方式决策

第一阶段不采用“短期复制模块”方案，也不先做完整独立 package 抽离；而是采用更务实的中间路径：以 vendoring/shared-source 的方式把 `md-reader` 中与只读文档域直接相关的模块引入 `data-editor`，并在仓库内明确标记其来源与同步边界。

这样做的理由是：

- 比完整抽公共 package 更快落地
- 比短期复制模块更不容易漂移
- 符合后续两产品合并时“先统一文档子系统，再统一外壳”的节奏

第一阶段允许进入 `data-editor` 的只限只读文档子系统本体，不包括 `md-reader` 的完整工作区壳、文件树、顶部栏、项目切换 UI、可视编辑器以及写回链路。

这条收窄是明确的工程决策，而不是实现时再看情况：

- 第一阶段目标是先让 `Document` 字段和文档阅读面可靠落地
- 只读 renderer、样式 token、必要 transform 可以先统一
- 编辑器、writer、冲突恢复涉及 React/Node/依赖版本差异，应延后到第二阶段统一处理

### 本次集成原则

1. `data-editor` 负责记录、字段、详情面板、配置模型和文档字段解析
2. 第一阶段的 `md-reader` 负责只读渲染语义、阅读样式真相与必要兼容规则
3. 首期先接只读预览，不在 `data-editor` 中直接开放 Markdown 编辑
4. 二期若接编辑，再把 `md-reader` 的编辑、写回和冲突处理正式纳入统一文档栈

### 为什么不直接整产品合并

本次需求的范围仍然应收敛在“关联文档字段”与“文档子系统复用”上，而不是把 `md-reader` 的完整工作区壳层、文件树、顶部栏和项目模型一次性并入 `data-editor`。

原因：

- 一次性整壳合并会把任务从字段能力膨胀成产品级整合
- 当前最有价值的是先统一文档域能力，而不是先统一所有 UI 外壳
- 先对齐文档子系统边界，后续做产品合并时返工最少

## 目标设计

## 一、字段与配置模型

### 1. 字段值模型

`Document` 字段的实际值仍然是业务主键 ID，例如：

```json
{
  "keyword_id": "burn",
  "keyword_doc": "burn"
}
```

其中 `keyword_doc` 不是路径，不是 Markdown link，也不是标题文本；它只是“目标文档 ID”。

系统运行时根据该值去对应数据文件的 `docRoot` 中查找唯一的：

```text
burn.md
```

### 2. 文件级配置模型

每个数据文件都可以独立声明文档根目录。推荐在项目级 view config 中增加单独的文档配置区，而不是塞进 `fields` 或 `relations` 的旧结构。

建议结构：

```ts
type DocumentFileConfig = {
  docRoot: string;
};

type DocumentFieldConfig = {
  enabled: true;
};

type ViewConfig = {
  // existing fields...
  documentFiles: Record<string, DocumentFileConfig>;
  documentFields: Record<string, DocumentFieldConfig>;
};
```

说明：

- `documentFiles` 的 key 对应数据文件路径，例如 `data/keywords.json`
- `documentFields` 的 key 对应字段级路径，建议沿用当前 relation key 的“源文件 + 集合 + 字段路径”风格，避免顶层和嵌套字段冲突

配置落盘契约也需要明确：

- `ViewConfig` 向后兼容扩展，旧配置文件没有 `documentFiles/documentFields` 时应按空配置处理
- `view-config.mjs` 的 load / normalize / save 必须同时支持这两块新结构
- `documentFields` 的 key 正式沿用 relation key 风格，而不是单纯 collection field key，避免后续嵌套字段时再次迁移 key 体系

### 3. 解析规则

对于某个被配置为 `Document` 的字段：

1. 先找到其所在数据文件的 `docRoot`
2. 取字段值作为 `documentId`
3. 在 `docRoot` 下查找所有名为 `<documentId>.md` 的文件
4. 解析结果只能是以下三种状态之一：
   - `resolved`：恰好命中一个文件
   - `missing`：没有命中
   - `conflict`：命中多个文件

不允许“取第一个”或模糊匹配。

### 4. 显示名称规则

对于 `resolved` 文档：

1. 优先读取 Markdown 首个一级标题 `# Title`
2. 如果没有一级标题，回退到文件名，例如 `burn.md`

字段显示名不写回业务数据，只在运行时解析。

### 5. 主键改名联动

`Document` 字段应被视为强绑定关系，而不是弱引用关系。

这意味着当主键 ID 改名时：

- 对应 `Document` 字段值默认进入影响分析
- 用户应看到“哪些文档引用会被一起更新”
- 在用户确认后，系统与 relation maintenance 一样执行批量同步

第一阶段不要求把这套能力完全做到和 relation maintenance 同级复杂度，但必须在方案上明确：主键改名不是放任文档引用断链，而是默认沿用“影响分析 + 一起更新”的产品方向。

## 二、候选值与编辑交互

### 1. 候选值来源

候选值严格来自 `docRoot` 下现有的 Markdown 文件名：

- `docRoot/status/burn.md` -> 候选值 `burn`
- `docRoot/combat/freeze.md` -> 候选值 `freeze`

不允许自由输入任意字符串作为正式可选值。

这样可以保证：

- 字段值与真实文档资产同步
- 不会制造“数据里填了 ID，但根本没有文档”的脏状态作为常态

### 2. 编辑器形态

推荐交互形态与现有 `RelationCellEditor` 接近，但语义独立：

- 单值选择
- 支持按候选 ID、标题、文件名搜索
- 选中后字段值写回原始 ID
- 不提供多选能力

它可以复用现有的 token/popover 外观，但不应复用 relation 配置语义。

这里的“编辑器”仅指文档引用选择器，不是 Markdown 正文编辑器。第一阶段术语上应统一叫“文档引用选择器”或“Document picker”，避免与第二阶段的 Markdown 可视编辑器混淆。

### 3. 字段菜单能力

对于可配置字段，列头菜单应支持：

- 设为关联文档字段
- 取消关联文档字段

不需要像 `relation` 那样弹出目标文件/目标集合/目标主键配置，因为文档字段没有这些概念。

### 4. 字段显示状态

表格和详情字段层面应展示以下状态：

- 已解析：显示标题或文件名
- 空值：显示未设置
- 缺失：显示缺失文档
- 冲突：显示文档冲突

## 三、详情面板与独立文档预览区

### 1. 设计目标

当用户打开记录详情时，如果该记录存在 `Document` 字段，就在详情面板中额外展示该字段关联文档的 Markdown 预览，而不是要求用户再点一次字段才能看正文。

### 2. 联动规则

对于每个 `Document` 字段：

- 该字段是单值
- 该字段最多只能解析到一篇文档
- 详情面板随当前记录刷新而同步刷新文档预览

展示规则固定为：

- 如果当前记录只有一个 `Document` 字段，则显示一个独立文档预览区
- 如果当前记录有多个不同的 `Document` 字段，则在同一文档面板中按字段维度切换，而不是把多个长文档区块堆叠在主详情区里

这里的“多个”指多个字段，不是一个字段多篇文档。

### 2.1 预览区宿主决策

详情文档预览区第一阶段不复用 `secondary panel`，而是新增一个独立的文档面板，放在主详情面板左侧。它属于“详情区内部的第三面板”，不是整个应用主工作区级别的第三栏。

这条决策背后的产品语言是：

- 左侧文档面板代表“当前记录关联出去的文档阅读区”
- `primary panel` 继续代表“记录属性编辑区”
- `secondary panel` 保持“嵌套字段深入查看区”的单一职责
- 用户在空间上会自然理解为“左边看文档，中间改数据，右边看嵌套细节”

这样可以避免以下问题：

- 文档预览和嵌套详情争抢同一个 `secondary panel`
- 主详情区被拉成长页面
- 为了复用 `secondary panel` 而引入复杂切换规则

从实现契约上，第三面板与现有 primary / secondary panel 属于同一个 detail shell：

- 点击第三面板内部不应触发详情关闭
- 第三面板、主面板、二级面板都属于同一组 outside-click 豁免区域
- 切换记录时，第三面板与二级面板都按记录级上下文同步刷新
- 文档面板不是浮层，不走临时 popover 语义

### 2.2 文档面板开关与定位规则

文档面板需要有三种入口，但职责不同：

- 主面板上的文档按钮：显式打开/关闭文档面板
- 关联文档字段点击：如果面板关闭则打开；如果面板已打开，则切换显示到该字段对应的文档
- 文档面板自身的关闭按钮：只负责关闭文档面板

这里的“定位到对应文档”只表示：

- 点击哪个关联文档字段，文档面板就切换显示哪个字段对应的文档

它不表示滚动到文档内部某个标题或锚点。

### 2.3 默认开关状态与个人配置

文档面板的默认打开/关闭状态需要记入个人配置，而不是只存在当次会话内。

规则固定为：

- 用户上一次关闭详情页时文档面板是打开还是关闭，作为下次打开详情页时的默认状态
- 文档按钮和文档面板关闭按钮都会更新这个默认状态
- 点击关联文档字段时，如果面板原本关闭，立即打开并写回默认状态
- 因字段点击而打开文档面板，不算临时态，而视为用户明确表达“我需要文档面板”

默认开关状态与记录自动展示之间的优先级也固定为：

1. 若用户记忆状态为关闭，则打开任何详情记录时默认保持关闭
2. 若用户记忆状态为打开，则打开详情时立即展示当前记录的默认文档
3. 点击关联文档字段始终可以覆盖当前关闭状态并立即打开

也就是说，“有 `Document` 字段”不自动凌驾于用户上次明确关闭过文档面板的偏好之上。

### 2.4 文档面板宽度

文档面板需要支持自由调节宽度，并保存在个人配置中，方式与现有 `detailPanelWidth` 保持一致。

新增个人配置项：

- `detailDocumentPanelOpen`
- `detailDocumentPanelWidth`

这两个状态都应放进现有 view profile 体系，而不是只放 localStorage。

并且这里不是“只新增两个字段”这么简单，而要明确纳入现有状态链路：

- profile mode 下：从 view profile 读取、保存、复制和重置
- local mode 下：从 localStorage fallback 读取、保存和重置
- `emptyViewProfile` / `normalizeViewProfile` / `serializeViewProfile` 需要承载这两个字段
- `emptyLocalViewState` / `readViewLayoutState` / `writeLocalViewLayoutState` 需要承载这两个字段
- `buildProfileFromCurrentView` 需要把这两个字段一起带入 profile 快照

也就是说，这两个状态应与现有 `detailPanelWidth` 一样，是完整的“profile + local fallback”双通路契约，而不是单点缓存。

### 2.5 第三面板宽度与三栏共存规则

第三面板引入后，详情区可能同时出现：

- 左侧文档面板
- 中间 primary panel
- 右侧 secondary panel

因此必须在方案里提前写死宽度与降级规则：

- 文档面板有独立的 `min/max/default width`
- primary panel 继续沿用现有 `detailPanelWidth` 语义
- secondary panel 保持当前“有嵌套详情时才出现”的条件
- 当总宽度不足以同时舒适容纳三栏时，优先压缩文档面板到最小宽度
- 若仍不足，再让 secondary panel 进入更紧凑的窄栏态；不要先挤压 primary panel 到不可编辑
- 若当前记录没有任何可解析或可展示的文档，左侧文档面板不强行显示空壳，而是自动隐藏内容区

首版把文档面板宽度约束写成与 primary panel 同级的显式常量，而不是留给 CSS 自行碰撞。

### 2.6 默认文档选择规则

当一条记录存在多个 `Document` 字段时，详情首次打开的默认显示规则固定为：

1. 若本次详情会话中用户已经点过某个 `Document` 字段，则优先显示该字段对应文档
2. 否则按详情字段排序后的第一个 `Document` 字段显示
3. 若第一个字段为空值或缺失，则仍显示该字段对应状态，而不是静默切到别的字段

这样做的原因是：文档面板的默认选择必须跟字段顺序和用户点击保持一致，不能偷偷替用户做“更聪明”的跳转。

### 3. 预览内容

预览区只显示 Markdown 渲染结果，不提供源码编辑。

视觉方向上，预览样式不采用仓库里普通表单/详情块的默认文本样式，而应明确对齐 `C:\Code\md-reader` 的阅读型样式语言：把它当作“文档阅读表面”，不是普通字段值延伸区。

实现上，详情面板预览区应优先直接宿主 `md-reader` 的只读渲染组件，或按同一套只读渲染语义接入，而不是自己单独用另一套 `react-markdown + CSS` 重搭一个近似版本。

预览头部建议显示：

- 字段名
- 文档标题
- 文档相对 `docRoot` 的路径
- 状态标识（正常 / 缺失 / 冲突）

### 4. 预览样式约束

Markdown 预览区的样式目标不再是抽象“参考 md-reader”，而是尽量直接复用或对齐 `C:\Code\md-reader` 里的以下实现来源：

- 只读渲染入口：`src/document-renderer/readonly-markdown-renderer.tsx`
- 真实渲染实现：`src/document-renderer/readonly-markdown-renderer-impl.tsx`
- 文档排版样式：`src/index.css` 中 `.markdown-document`、`.readonly-markdown-renderer`、`--doc-*` tokens
- 样式基线测试：`tests/app/document-typography.test.tsx`

这意味着 data-editor 侧应优先考虑两种实现路径：

1. 直接抽取/复用 `md-reader` 的只读 Markdown 渲染组件与配套样式
2. 若不能直接复用，则至少按同一套 DOM 类名和 typography tokens 对齐

首选是 1，因为它能最大限度减少“看起来像，但细节不一致”的漂移。

但第一阶段的工程边界也要写清楚：

- 首期目标是复用只读链路，不要求当场把 `md-reader` 的 React 版本、Vite 版本、编辑器依赖一起并入
- 如果直接宿主整段只读组件会带来版本耦合，则允许先 vendoring 只读 renderer 相关最小闭包
- 允许为第一阶段补入 `react-markdown`、`remark-gfm` 与必要 transform 依赖
- 不允许为了“顺手统一”而把 `@milkdown/*`、writer 或 editor 依赖一并拉进第一阶段

至少需要对齐以下具体样式契约：

- 以阅读体验为主，不复用表单输入区或属性块的紧凑排版
- 使用与 `md-reader` 一致的文档宽度和排版 token，尤其是 `--doc-max-width`、`--doc-body-line-height`、`--doc-block-spacing`、`--doc-surface-inline-padding`
- 标题层级、段落间距、列表缩进、引用块、分隔线要与 `.markdown-document` 的规则对齐
- 代码块、行内代码、表格、链接、图片需要独立视觉规则
- 只读预览区的 DOM 应优先兼容 `.markdown-document` 或 `.readonly-markdown-renderer` 的样式作用域
- 预览渲染结果应尽量与 `md-reader` 的 `ReadonlyMarkdownRenderer` 保持一致，而不是只复用 CSS、不复用渲染规则
- 内容宽度、行高、字号、留白应服务于连续阅读，而不是表格密度
- 预览区应与记录属性区视觉分层，避免看起来像“字段展开文本”
- `synced_block` 这类 `md-reader` 已处理的只读隐藏规则也应纳入兼容范围，避免相同文档在两个工具里显示不一致

其中 `tests/app/document-typography.test.tsx` 可以作为 data-editor 后续实现时的对齐依据。即使不直接搬测试代码，也应按它覆盖的层级、列表、代码块、表格和 spacing 契约做回归验证。

### 5. Markdown 编辑的阶段边界

#### 第一阶段

第一阶段只做：

- 关联文档字段
- 文档索引
- 标题提取
- 第三面板壳层、开关状态与宽度持久化
- 详情面板只读预览

不在 `data-editor` 中直接编辑 Markdown 正文。

#### 第二阶段

第二阶段再考虑开放 Markdown 编辑，并明确直接复用 `md-reader` 的编辑器封装，而不是为 `data-editor` 另造一套：

- `VisualMarkdownEditor`
- 相关样式与 loading/error shell
- 文档写回与磁盘冲突检测
- 编辑态下的依赖统一与运行时版本对齐

#### 为什么要拆阶段

如果把“字段引用 + 只读预览 + Markdown 编辑”一次性绑定，会显著放大本次任务范围。真正额外高成本的部分不是编辑器控件本身，而是：

- 详情面板中的编辑/预览模式切换
- 文档保存与 `data-editor` 业务数据保存的边界
- 记录切换/详情关闭时的未保存保护
- 编辑中的冲突检测与恢复

这些都属于第二阶段更适合承接的复杂度。

### 6. 异常展示

- `missing`：显示“未找到 `<id>.md`”
- `conflict`：显示“在 `docRoot` 下找到多个 `<id>.md`，请清理冲突”
- 空值：不显示正文，显示“未设置关联文档”

## 五、后端读取与索引设计

### 1. 访问边界

后端不能开放任意 Markdown 文件路径读取，而应只允许：

- 某个数据文件配置的 `docRoot` 下的 `.md`

也就是说，Markdown 文件访问应遵守和数据源类似的受控根目录边界。

### 2. 需要的基础能力

最小可行实现至少需要：

- 列出某个 `docRoot` 下所有 `.md`
- 建立 `documentId -> file` 索引
- 读取 Markdown 原文
- 提取首个一级标题
- 明确复用 `md-reader` 只读读取语义，而不是平行发明另一套路径边界模型

第一阶段后端范围也要收窄：

- 只解决受控读取与索引
- 不把 `md-reader` 的 writer、冲突恢复、编辑保存协议一起引入
- API 设计优先服务只读预览，而不是预埋完整编辑工作流

### 3. 索引结构建议

建议按数据文件缓存一份文档索引：

```ts
type DocumentIndexEntry =
  | { status: "resolved"; id: string; relativePath: string; title: string | null }
  | { status: "conflict"; id: string; matches: string[] };
```

缺失项不必预先存入索引，可在查询时按 `id` 返回 `missing`。

### 4. 读取时机

- 字段列表显示阶段：优先使用索引中的标题摘要
- 详情面板打开阶段：需要时再加载 Markdown 正文

这样可以避免表格首屏就把所有 Markdown 正文全读进来。

索引刷新规则也需要一并定义：

- 打开或切换数据文件时，刷新该文件对应 `docRoot` 的索引
- 手动刷新项目或重新加载视图时，允许重扫 `docRoot`
- 切换记录本身不触发全量重扫，只消费当前缓存

这样可以在性能和可见性之间取得基本平衡，不把“外部新增/重命名文档”永远卡死在旧缓存里。

## 六、校验与状态模型

### 1. 空值

允许为空，不报错。

### 2. 缺失文档

字段值非空，但在 `docRoot` 下找不到 `<id>.md`。

建议级别：

- 默认作为 warning 显示
- 是否阻止保存可后续再收紧

理由：

- 早期项目中可能存在“先录数据、后补文档”的工作流

### 3. 文档冲突

在 `docRoot` 下发现多个同名 `<id>.md`。

建议级别：

- 直接作为 error

理由：

- 这违反了本能力最核心的唯一性前提
- 系统不应自动决定取哪一篇

### 4. 非法配置

如果字段被标记为 `Document`，但其所在数据文件没有配置 `docRoot`，则视为配置错误。

这属于结构性问题，应在 UI 中明确暴露，而不是退回普通文本显示。

## 七、缓存与性能策略

### 1. 最小可行缓存

首版建议只做前端内存缓存：

- 文档索引缓存
- 文档标题缓存
- Markdown 原文缓存

key 建议包含：

- 数据文件路径
- `docRoot`
- 文档相对路径

文档面板开关状态和宽度不属于这类内容缓存，而属于用户视图偏好，应走 profile/local 状态链路，不混入文档正文缓存。

### 2. 失效策略

以下场景应使相关缓存失效：

- 项目切换
- 数据文件切换
- `docRoot` 配置变更
- 明确触发刷新或重载

### 3. 不做的优化

首版不建议做：

- 持久化缓存
- 文件系统监听
- 大规模预热全文档内容

## 八、实现边界与模块影响

预计会影响的层次包括：

- 字段类型定义与兼容判断
- view config 模型与持久化
- 列头菜单与字段类型切换
- 单元格/详情字段渲染
- 文档引用选择器
- 详情区内部新增文档面板及其开关/宽度状态
- 第三面板与 primary / secondary 的宽度协商和 outside-click 边界
- 详情面板额外预览区
- 后端 Markdown 索引与读取 API
- `md-reader` 文档子系统在 `data-editor` 中的宿主边界
- 校验与状态展示
- 测试与文档

但本方案强调：

- 不修改现有 `relation/backlink` 核心语义
- 不把 Markdown 文档当作 `DocumentModel` 集合去硬套 JSON/CSV 编辑逻辑
- 不把“文档引用字段”实现成 `Text` 字段上的局部特判
- 不在第一阶段复制一份脱离 `md-reader` 的 Markdown 渲染/编辑实现
- 不在第一阶段把 `md-reader` 的编辑器、writer、冲突恢复与运行时依赖整包并入

## 九、测试建议

至少覆盖以下场景：

1. 正常命中  
   `burn` -> 唯一 `burn.md`

2. 缺失文档  
   字段值存在，但没有对应 `.md`

3. 冲突文档  
   同一个 `id` 对应多个 `<id>.md`

4. 无一级标题回退  
   文档没有 `#` 标题时回退文件名

5. `docRoot` 缺失配置  
   字段启用但文件级配置不存在

6. 候选值来源约束  
   编辑器只允许从真实 Markdown 文件名选择

7. 详情联动  
   打开记录详情时文档预览自动刷新

8. `md-reader` 样式对齐  
   只读预览与 `md-reader` 的 `.markdown-document` 排版契约一致

9. `md-reader` 兼容规则  
   `synced_block` 等已处理内容在 `data-editor` 中显示一致

10. 主键改名联动  
   `Document` 字段值进入影响分析，并按确认后的同步方案更新

11. 文档面板开关记忆  
    详情页重新打开时，文档面板按上次的打开/关闭状态出现

12. 文档面板宽度持久化  
    用户调整后的文档面板宽度写入个人配置，并在下次详情打开时恢复

13. 字段点击切换文档  
    多个 `Document` 字段存在时，点击哪个字段，文档面板就显示哪个字段对应的文档

14. 独立关闭按钮  
    文档面板自身关闭按钮可关闭面板，并同步更新默认开关状态

15. profile/local 双通路一致性  
    有 profile 与无 profile 两种模式下，文档面板开关与宽度恢复行为一致

16. 宽度降级规则  
    三栏共存时，文档面板、primary、secondary 的压缩优先级符合预期

17. 默认文档选择规则  
    多个 `Document` 字段存在时，详情首次打开默认落到排序后的第一个字段

18. outside-click 边界  
    点击左侧文档面板内部不会误触发整个详情关闭

## 十、推荐落地顺序

1. 明确第一阶段只复用 `md-reader` 的只读链路，锁定 vendoring/shared-source 的最小闭包
2. 扩展 `data-editor` 配置模型，支持文件级 `docRoot` 与字段级 `Document` 声明
3. 补齐 profile/local 双通路状态模型，纳入 `detailDocumentPanelOpen/detailDocumentPanelWidth`
4. 增加后端 Markdown 文件索引与正文读取 API
5. 增加前端 `Document` 字段显示、第三面板壳层与详情预览区
6. 接入 warning/error 状态展示与三栏宽度降级规则
7. 补测试与仓库文档
8. 第二阶段再接 `md-reader` 编辑器、写回链路与冲突处理

## 十一、最终结论

「关联文档」应被实现为新的独立字段类型 `Document`，而不是 `Relation` 的特例，也不是 `Text` 的显示插件。

其核心模型是：

- 业务字段值存主键 ID
- 每个数据文件配置自己的 `docRoot`
- 文档解析规则固定为在 `docRoot` 下唯一定位 `<id>.md`
- 表格与详情字段显示标题
- 详情面板额外展示由 `md-reader` 文档子系统提供的 Markdown 预览

这个设计与当前 `data-editor` “存原始值，显示层补语义”的整体思路一致，同时避免把文档资产能力污染到现有 relation 结构里；并且它把 `md-reader` 明确设为文档域的 source of truth，为后续两产品合并提供了稳定前提。

## 十二、当前首版实现对齐结果（2026-06-17）

本次实际落地已经完成以下首版闭环：

- 新增独立字段类型 `Document`
- `ViewConfig` 正式扩展 `documentFiles` 与 `documentFields`
- profile / local 双通路持久化 `detailDocumentPanelOpen` 与 `detailDocumentPanelWidth`
- 后端新增文档索引与正文读取 API：`/api/document-index`、`/api/document-content`
- 详情区落地为左文档 / 中详情 / 右嵌套 的三栏结构
- 主面板文档按钮、字段点击开关、文档面板独立关闭按钮都已接通
- 文档面板宽度可拖拽并持久化
- 详情页重新打开与页面 reload 后，文档面板开关和宽度都会按上次状态恢复
- 主键改名影响分析已把同文件同集合的 `Document` 字段值纳入 rewrite 计划

首版刻意未做的部分也保持与方案一致：

- 还没有在 `data-editor` 内开放 Markdown 正文编辑
- 还没有做独立的 Document picker / 候选值选择器
- 还没有把 `md-reader` 的完整编辑器、writer、冲突恢复整包接入

样式实现上，首版采用的是“最小 vendoring + 风格对齐”路径：

- 只读渲染使用 `react-markdown + remark-gfm`

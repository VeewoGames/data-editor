# 双击单元格开启全局编辑模式设计

## 概述

### 1. 总体目标和范围

本方案的目标是在 `data-editor` 主表中新增一个更顺手的入口：当用户双击普通 `Text/Number` 单元格时，如果当前全局 `编辑` 模式尚未开启，则自动将其开启。该行为只负责打开表级编辑模式，不负责把当前单元格直接切入输入态，也不负责关闭编辑模式。

本方案范围仅限主表文本类单元格的双击入口、全局 `textEditable` 状态切换链路，以及相应的自动化验证。标题列、`Select`、`Multi-select`、`Relation`、详情页输入、现有 `TableTextCellEditor` 的输入提交逻辑均不在本轮范围内。

### 2. 各阶段任务概要

第一阶段是确认现有主表文本编辑协议的入口边界，重点核对 `TextCellSurface`、`DataTable`、`ViewTabs/App` 中全局 `编辑` 状态的传递关系，确保新增双击动作只改变表级开关，不挤进单元格局部状态机。预期成果是明确接入点和非目标边界。

第二阶段是在主表文本单元格的 display layer 上接入双击事件，当用户双击 `Text/Number` 单元格且全局 `编辑` 尚未开启时，触发一次“开启全局编辑模式”的上行回调。预期成果是双击只能开启、不能关闭，且开启后当前单元格仍停留在 `editable-idle`。

第三阶段是补齐回归验证，覆盖双击开启成功、双击后不自动进入 `editable-active`、已开启时重复双击不产生副作用、非文本类单元格不受影响等场景。预期成果是有稳定 e2e 保护这次交互调整。

### 3. 整体结构框架

```text
主表双击事件
-> TextCellSurface / 文本 display layer
-> 表格运行时回调
-> 上层全局 textEditable 状态
-> DataTable 重新渲染为 editable-idle
-> 同一双击手势结束后当前 cell 仍保持 idle
-> 用户后续再按现有方式激活具体 cell
```

---

## 一、背景与现状

当前主表普通文本/数字单元格并不是“编辑模式一开就直接出现输入框”，而是三态结构：

- `readonly`：全局 `编辑` 关闭
- `editable-idle`：全局 `编辑` 开启，但当前 cell 尚未激活
- `editable-active`：全局 `编辑` 开启，且当前 cell 已激活，挂载输入 overlay

现有实现中：

- `TextCellSurface` 负责 display layer 与 overlay 的切换。
- `DataTable` 持有当前 active text cell id，并把全局 `textEditable` 下发给 cell。
- `TableTextCellEditor` 只负责已进入 `editable-active` 后的输入、提交、`Enter/Escape/blur` 语义。

因此，本次需求不应改写文本输入组件，而应新增一个“更早一步的表级入口”。

---

## 二、需求定义

本轮需求固定为以下语义：

1. 双击仅对主表 `Text/Number` 单元格生效。
2. 双击时，如果全局 `编辑` 尚未开启，则将其开启。
3. 双击只能开启，不能关闭；若全局 `编辑` 已开启，重复双击不执行关闭动作。
4. 双击开启后，当前被双击的单元格仍保持 `editable-idle`，不自动进入 `editable-active`。
5. 同一双击手势结束后，不允许顺带触发当前 cell 的激活逻辑或挂载 editor overlay。
6. 具体 cell 的激活方式、输入方式、退出方式保持现有行为不变。

---

## 三、非目标

本轮明确不做以下事情：

- 不支持双击后直接进入输入态。
- 不支持双击反向关闭全局 `编辑`。
- 不改变标题列双击或标题列编辑策略。
- 不把该行为扩展到 `Select`、`Multi-select`、`Relation`、`Document`、`Checkbox`。
- 不实现 Excel 式双击后选中文本、跨 cell 跳转等增强编辑协议。

---

## 四、推荐方案

### 方案 A：仅把双击作为“开启全局编辑模式”的表级入口（推荐）

做法：

- 在 `TextCellSurface` 的 display layer 增加 `onDoubleClick`。
- 当 `editable === false` 且当前 display type 为 `Text/Number` 时，触发一个上行回调，请求上层开启全局 `编辑`。
- 上层切换 `textEditable` 后，表格重新渲染为 `editable-idle`；当前 cell 不自动激活。

优点：

- 与当前三态架构一致，只改 `readonly -> editable-idle`。
- 不碰 `TableTextCellEditor` 的输入与提交协议。
- 不把“表级开关”和“cell 级激活”绑在一起，语义更清楚。
- 双击只承担单一职责，测试边界清晰。

缺点：

- 用户双击后还需要一次后续激活动作才能开始输入，不是最快路径。

### 方案 B：双击同时开启全局编辑并激活当前 cell

做法：

- 双击后同时打开全局 `编辑`，并把当前 cell 设为 `editable-active`。

不推荐原因：

- 把表级状态切换和 cell 级输入激活耦合到同一个双击动作里。
- 更容易和现有单击选区、失焦、caret 放置、overlay 测高约束混在一起。
- 需求已明确不需要“直接进入编辑”，没有必要扩大实现面。

---

## 五、接入点设计

### 1. `TextCellSurface`

这里是最合适的双击入口，因为它已经掌握：

- 当前 cell 是否可编辑
- 当前是否 active
- 当前是文本 display layer 还是 overlay

新增一个窄接口，例如：

```ts
onRequestEnableEditing?: () => void;
```

行为规则：

- `editable === true` 时，双击不负责切换任何全局状态。
- `editable === false` 时，双击触发 `onRequestEnableEditing`。
- 不在这里直接操作全局 store，只发出请求。
- 双击回调只挂在 `TextCellSurface` 的 display layer，不挂到 `td`、`CellRenderer` 外层或其他 cell 容器上，避免误伤非文本类单元格。

### 2. `CellRenderer` / `table-columns`

沿现有文本编辑装配链，把“请求开启全局编辑模式”的回调从表格运行时继续传到 `TextCellSurface`。

这一层只做装配，不承载状态机判断。

### 3. `DataTable` 与更上层表级状态

`DataTable` 当前只消费 `snapshot.textEditable`，并不拥有最终真相。因此真正的“开启全局编辑模式”动作需要由拥有该状态的更上层处理，再通过 `snapshot.textEditable` 回流。

这里的要求是：

- `App` 侧应提供一个语义明确的 `enableTableTextEditMode()` 或等价专用回调。
- 接收到请求时，只执行“置为 true”。
- 若当前已是 `true`，保持幂等，不做额外切换。
- 不得直接复用当前 `onToggleTableTextEditMode={() => setTableTextEditMode((value) => !value)}` 这类 toggle 入口，否则双击在已开启状态下会带来误关风险。
- 不在该动作里顺带设置 `activeTextCellId`。

## 六、交互约束

为避免实现时语义漂移，本轮必须遵守以下约束：

1. 双击开启后，当前 cell 不自动挂载 `.table-text-cell-editor`。
2. 全局 `编辑` 开启后，文本 cell 仍需按现有方式从 `editable-idle` 进入 `editable-active`。
3. 同一双击手势内，不允许因为状态切换时机问题把当前 cell 顺带推进 `editable-active`。
4. 双击不能影响现有矩形选区的拖拽清除、单击选中与 click suppression 机制；允许保留普通单格选中，但不能误进入拖拽扩选。
5. 双击开启全局模式后，用户下一次普通单击当前或其他文本 cell，仍应按现有方式进入 `editable-active`。
6. 已经 active 的 cell、已开启全局编辑模式的页面，重复双击不做关闭或重置。
7. 双击期间不能误伤非文本类单元格现有交互。
8. 触控双击 / 双 tap 不在本轮目标内，桌面鼠标双击为唯一要求。

---

## 七、验证策略

建议补以下自动化验证：

1. `double click text cell enables global table edit mode`
   - 初始 `编辑` 按钮为关闭
   - 双击文本 cell 后，按钮变为开启

2. `double click text cell does not immediately activate the editor`
   - 双击后当前 cell 仍无 `.table-text-cell-editor`
   - 当前 cell 仍显示 display layer
   - 同一双击手势结束后，当前 cell 的 `data-text-cell-mode` 为 `editable-idle`

3. `double click number cell enables global table edit mode`
   - 数字单元格也遵守同样规则

4. `double click does not disable global table edit mode once enabled`
   - 已开启后再次双击，不会把按钮切回关闭

5. `double click enable keeps normal click activation available afterwards`
   - 双击开启全局模式后
   - 后续普通单击文本 cell 仍可正常进入 `.table-text-cell-editor`

6. `non text cells do not use double click to enable table edit mode`
   - `Select` / `Relation` 等场景保持现有行为

---

## 八、风险与控制

主要风险有两个。

第一，双击事件可能和现有单击选区逻辑互相干扰。控制方式是把行为限定在文本 display layer，并验证双击不会破坏既有 pointer down / click capture 语义。

第二，容易有人误复用现有 toggle 回调，导致双击在已开启状态下把全局 `编辑` 重新关闭。控制方式是新增 `enable-only` 专用回调，并把“不允许复用 toggle”写成明确实现约束。

第三，容易有人顺手把“双击开启全局编辑”继续扩成“双击直接进入输入”。控制方式是把“不开启 active、不挂 overlay”写成显式验收标准，并在测试里单独覆盖。

---

## 九、验收标准

本方案完成时，应满足：

1. 双击主表 `Text/Number` 单元格可把全局 `编辑` 从关闭切到开启。
2. 双击不会把全局 `编辑` 从开启切回关闭。
3. 同一双击手势结束后，当前 cell 仍保持 `editable-idle`，不自动进入输入态。
4. 双击开启后，后续普通单击文本 cell 仍可按现有方式进入输入态。
5. 现有文本输入、退出、wrapped overlay、选区和非文本 cell 行为不回归。

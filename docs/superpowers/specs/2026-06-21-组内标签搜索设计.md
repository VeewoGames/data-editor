# 组内标签搜索设计

## 概述

### 1. 总体目标和范围

本方案的目标，是在 `data-editor` 的共享视图两层导航中，为当前展开的视图组第二层增加一个轻量搜索入口，用于快速筛选该组右侧的子视图标签，降低组内标签数量增多后的扫描成本。

本次范围只包含第二层组内标签行的导航筛选能力，不改动当前 active view 的查询语义，不改动表格数据搜索，不改动共享视图配置结构，也不引入新的持久化状态。

### 2. 各阶段任务概要

1. 明确语义边界  
   主要工作：区分“当前视图 query 搜索”和“第二层组内标签筛选”两类能力。  
   预期成果：组内搜索仅影响第二层标签显示，不影响表格内容。  
   执行顺序：优先完成。

2. 增加第二层搜索入口与本地状态  
   主要工作：在 `ViewTabs` 第二层最左侧增加搜索图标，点击后展开为输入框，并维护组件本地过滤词。  
   预期成果：用户可在当前展开组内快速输入并筛选标签。  
   执行顺序：在语义边界明确后实现。

3. 接入组内标签过滤规则  
   主要工作：基于过滤词筛选 `expandedGroup.views`，并确保 active 子标签始终保留可见。  
   预期成果：视觉上只收窄第二层标签范围，且不会把当前正在使用的标签筛掉。  
   执行顺序：在搜索入口可用后接入。

4. 补充验证与回归检查  
   主要工作：验证切组重置、active 保留、空关键字恢复、组内 `+` 保持可用、拖拽未出现明显回归。  
   预期成果：交互行为符合预期，且不干扰现有组内导航。  
   执行顺序：最后执行。

### 3. 整体结构框架

- UI 容器：`src/components/ViewTabs.tsx`
- 复用控件：`src/components/ExpandableSearch.tsx`
- 数据来源：`expandedGroup.views`
- 本地状态：`ViewTabs` 内部组内标签过滤词
- 过滤结果：仅参与第二层标签渲染
- 结构约束：第二层搜索入口与组内 tabs 分属不同子容器，`tablist` 只包裹真实 tabs
- 不涉及模块：`App.tsx` 共享视图数据结构、active view `query`、page context、shared config

## 背景

当前两层导航中，第一层负责顶层普通视图和视图组切换，第二层负责当前展开组内的子视图导航。随着某个组内子视图数量增加，用户需要从左到右扫描整排标签，查找成本会明显升高。

现有系统已经有一个“展开式搜索”控件，但它服务的是当前 active view 的 `query`，会影响表格数据结果。第二层需要的是一个纯导航筛选器，只用于缩小当前展开组中的标签可见范围。因此，这不是给现有视图搜索换位置，而是新增一个独立的组内标签筛选能力。

## 现状证据

- `docs/04_信息架构与交互.md` 已定义：点击视图组后，会在第二行展开该组内的具体视图标签。
- 同一文档已定义：现有“展开式搜索”写入当前 active view 的 `query`。
- `src/components/ViewTabs.tsx` 当前第二层仅渲染 `expandedGroup.views` 和组内 `+`，没有独立搜索状态。
- `src/components/ExpandableSearch.tsx` 已提供可复用的“点击图标后展开输入框”的 UI 行为。

这说明本需求适合落在 `ViewTabs` 内部，以“复用控件、隔离语义”的方式实现，而不是修改现有视图搜索链路。

## 设计目标

1. 在第二层最左侧提供一个紧邻标签行的搜索入口。
2. 输入关键字后，只筛选当前展开组的子视图标签显示。
3. 当前 active 子视图即使不匹配关键字，也必须继续保留显示。
4. 切换到其他组时，组内搜索词直接清空，不保留组级记忆。
5. 不改 shared view 配置、不改页面恢复状态、不改 active view 的 `query`。
6. 不污染现有全局 `.search-box` 语义，不影响“编辑视图”菜单跳转到当前表格搜索框的行为。

## 非目标

1. 不把第二层搜索接入表格内容搜索。
2. 不把组内搜索词持久化到 localStorage、page context 或共享配置。
3. 不新增组级业务状态或组级 draft。
4. 不扩展为跨组全局标签搜索。
5. 不在本次顺手调整顶层搜索、筛选栏或视图结构模型。

## 方案对比

### 方案 A：复用 `ExpandableSearch`，绑定组内标签过滤词

做法是在 `ViewTabs` 第二层最左侧插入 `ExpandableSearch`，但其 `value` 与 `onChange` 只连接到 `ViewTabs` 本地的组内过滤词。过滤时只作用于 `expandedGroup.views` 的渲染结果。

优点：

- 与现有展开式搜索视觉和交互一致
- 改动范围集中在 `ViewTabs`
- 不需要复制新的展开/收起/聚焦逻辑

缺点：

- 需要非常明确地区分“组内标签筛选”和“active view query 搜索”两套语义，避免后续维护时误接

### 方案 B：在 `ViewTabs` 内单独实现一个新的组内搜索控件

做法是新写一套局部输入控件，专门服务第二层组内标签筛选。

优点：

- 语义最直观，名字和职责都可以完全独立

缺点：

- 会重复实现展开、收起、聚焦等交互
- 容易与现有展开式搜索产生样式和行为漂移

### 推荐方案

推荐方案 A。当前需求本质是“复用现有交互壳，但接不同语义的数据”，因此更适合以低耦合方式复用 `ExpandableSearch`，同时把组内搜索状态严格限制在 `ViewTabs` 本地。

## 详细设计

### 1. 状态边界

在 `src/components/ViewTabs.tsx` 内新增本地状态，例如：

- `groupTabFilter: string`

这个状态只服务当前 `expandedGroup` 的第二层标签筛选，不进入 `ViewTabsSnapshot`，不向上提升到 `App.tsx`，也不写入 shared views、page context 或 localStorage。

当 `expandedGroupId` 变化时，自动把 `groupTabFilter` 重置为空字符串。这样切换到别的组时不会残留上一个组的筛选词，符合当前确认的交互规则。

### 2. 渲染结构

当前第二层容器是：

- 组内子视图标签列表
- 组内 `+`

改造后结构调整为：

- 第二层外层：普通 flex 容器
- 左侧：组内搜索图标 / 展开输入框
- 中间：仅承载组内 tabs 的 `tablist`
- 最右侧：组内 `+`

这里的搜索入口属于第二层容器的一部分，而不是第一层 actions 区，也不是当前 active view 搜索的替代物。

这样调整的原因是，第二层搜索框和组内 `+` 都不是 tab 本身，不应直接混入 `tablist` 语义容器。实现时应把“搜索入口”“tabs 列表”“组内创建按钮”拆成并列子区域，避免把输入框错误地作为 tab 集合成员。

### 2.1 搜索入口选择器与焦点边界

现有 `ViewTabs` 的“编辑视图”菜单会通过全局 `.search-box` 选择器，把焦点导向当前表格搜索框。因此第二层组内搜索虽然复用 `ExpandableSearch` 组件，但不能复用 `.search-box` 这一 class 语义，也不能让任何现有 DOM 查询把它识别成“当前视图 query 搜索”。

本方案要求：

- 第二层搜索使用独立 class，例如 `group-tab-search`
- 若需要程序化聚焦，应优先使用局部 ref，而不是新的全局 `document.querySelector`
- 现有“编辑视图”菜单继续只指向当前表格搜索框，不得跳到第二层组内搜索

也就是说，本次复用的是展开式搜索的 UI 交互壳，而不是它在当前系统里的搜索身份。

### 3. 过滤规则

过滤逻辑建议保持简单且可预测：

1. 对输入关键字做 `trim()` 和 `toLowerCase()`
2. 对每个组内视图名称做同样的归一化
3. 使用 `includes()` 判断是否匹配

匹配范围只基于 `view.name` 这一展示名，不扩展到拼音、别名、query、filters 或其他 metadata。

若关键字为空，则显示全部组内视图。

若关键字非空，则显示：

- 名称匹配关键字的视图
- 当前 `activeViewId` 对应的视图，即使它不匹配，也强制保留

这样可以避免用户在筛选时把自己当前正在使用的标签“筛没了”，造成定位混乱。

### 4. active 保留规则

这是本次交互的硬约束：

- 当前 active view 属于当前展开组时，必须始终可见
- 即使 active 标签名称不匹配关键字，也不能隐藏

这样可以保证：

- 当前上下文始终有明确视觉锚点
- 用户不会误以为当前视图被切换了
- 搜索只改变“可见候选集”，不改变“当前使用对象”

### 5. 切组行为

当用户点击第一层其他组并切换 `expandedGroupId` 时：

- 第二层搜索词立即清空
- 新组的第二层以默认全量标签展开

这意味着组内搜索不是“组级记忆状态”，而是一次性的导航辅助工具。

### 6. 对现有行为的影响边界

本次能力不应影响以下现有行为：

- `onSelectView` 的调用时机
- `lastActiveViewIdByGroupId` 的维护
- shared view 结构拖拽保存逻辑
- active view 的 `query`
- `ViewFilterBar`、`DataTable`、详情面板的任何数据语义

也就是说，第二层搜索只改变“当前第二层显示哪些标签”，不改变任何业务真相。

### 7. 拖拽与创建按钮

组内 `+` 按钮继续固定存在于第二层最右侧，不参与过滤。

拖拽排序仍沿用当前第二层可见标签的交互模型。本次不额外为“被过滤隐藏的标签”提供特殊拖拽入口。也就是说，搜索开启后，用户当前只能对可见标签进行拖拽交互。这一限制是可接受的，因为组内搜索本身就是“缩窄当前操作范围”的辅助能力。

这里还需要明确 2 个实现边界：

1. 拖拽命中与 drop target 计算，只以过滤后当前可见的组内标签为准。
2. 第二层溢出检测，也以“搜索入口 + 当前可见 tabs + 组内 `+`”组成的实际可见布局为准。

因此实现时如果复用当前第二层 refs 和 overflow 检测逻辑，必须同步确保：

- 被过滤隐藏的标签不会继续作为当前交互目标参与计算
- 隐藏标签的残留 ref 不会影响 drop target 判断
- 溢出状态反映的是过滤后的真实可见布局，而不是过滤前全集

### 8. 焦点、收起与切换行为

组内搜索除了“切组清空”之外，还需要补齐以下交互约束：

1. 点击搜索图标后，输入框展开并自动聚焦。
2. 输入框失焦后，若当前关键字为空，则收起为图标；若关键字非空，则继续保持展开。
3. 点击组内某个 tab 切换视图时，不自动清空当前关键字；搜索结果保持，直到用户手动清空或切组。
4. 切换到其他组时，无论输入框是否还处于聚焦状态，都立即清空关键字并回到新组默认全量标签视图。
5. 如果实现中支持 `Escape`，应把其定义为“在关键字为空时收起输入框；在关键字非空时优先清空关键字”，避免出现不可预期的收起状态。

这些规则的目标，是让组内搜索保持轻量、短时、可预测，而不是变成另一套长期导航状态。

### 主要文件

  - 在第二层最左侧插入组内搜索入口
  - 基于过滤结果渲染第二层标签

- `src/components/ExpandableSearch.tsx`

  - 仅在必要时补充可定制类名或 aria 文案

### 不需要改动的文件

- `src/App.tsx`
- `src/page-context-storage.ts`
- `src/view/shared-view-structure.mjs`
- 共享视图配置结构相关模块

## 验证要点

1. 展开某个组后，第二层最左侧出现搜索图标。
2. 点击搜索图标后，输入框正常展开并自动聚焦。
3. 输入关键字后，只过滤当前展开组内右侧标签。
4. 当前 active 标签不匹配时仍保留显示。
5. 清空关键字后，第二层标签恢复全量显示。
6. 切换到另一个组后，搜索词被清空。
7. 组内 `+` 按钮始终可见且可用。
8. “编辑视图”菜单仍然只聚焦当前表格搜索框，不会跳到组内标签搜索。
9. 第二层搜索输入框不混入 `tablist` 语义容器。
10. 过滤后拖拽、drop target 和横向溢出检测行为稳定，没有被隐藏标签干扰。
11. 点击组内 tab 后，当前关键字保持不变。
12. 不影响表格数据搜索、不影响当前 view 的 `query`。
13. 搜索关闭或切组后，不产生额外持久化状态。

## 风险与约束

1. 语义串线风险  
   如果后续维护时把第二层搜索误接到 active view `query`，会导致标签筛选和数据筛选混在一起。实现时必须保持状态和回调边界清晰。

2. 拖拽可见范围收窄  
   搜索开启后，隐藏标签不会成为当前拖拽目标。这是当前设计接受的结果，但实现时要确保不会导致 drop target 计算异常。

3. 样式拥挤  
   第二层左侧加入搜索入口后，组内标签横向空间会变窄。样式上需要确保搜索入口足够轻量，不抢占主要导航区域。

4. 语义容器混用  
   如果仍把输入框直接塞进 `tablist`，会让第二层的无障碍语义继续恶化。实现时需要显式拆开搜索区与 tabs 区。

## 后续实现建议

实现时优先保持最小改动原则：

1. 先在 `ViewTabs` 内完成本地状态和过滤逻辑
2. 再把 `ExpandableSearch` 挂到第二层最左侧
3. 最后补测试或最小回归验证

不要在这一轮顺手改动顶层搜索语义、page context 结构或 shared view 数据模型，避免把一个局部导航增强扩大成架构级改造。

# Data Editor 信息架构与交互

## 概述

`data-editor` 的界面围绕“选文件、选集合、看主表、改详情、自动保存结果”这个主工作流组织。

## 主界面分区

### Sidebar

`Sidebar` 负责三个任务：

- 选择文件
- 选择 collection
- 展示 metadata 和主键候选提示

它是用户进入具体数据上下文的入口。

### Toolbar

`Toolbar` 负责全局动作：

- 搜索
- 选择个人视图配置 profile
- 展示自动保存状态
- 重置视图
- 刷新构建
- 关闭服务
- 管理隐藏字段

这里承载的是“当前上下文的全局操作”，不是行级编辑。

### ViewTabs

`ViewTabs` 位于主表上方，负责当前 collection 的共享视图切换、视图 tab 管理，以及当前视图的轻量入口：

- view tabs：展示当前 collection 的共享视图列表，当前 tab 作为 active view。
- 新建视图：基于当前 active view 复制出一个新共享视图。
- 重命名、删除：直接修改共享视图配置。
- 拖拽排序：调整共享视图 tab 顺序，先进入个人 draft。
- 当前 active tab：点击时打开 Notion-like 浮框；点击其他 tab 时只切换视图，不额外打开浮框。
- `筛选`：位于当前视图搜索图标左侧，用于显示或隐藏筛选栏；当前视图已有筛选时文字变蓝。
- `编辑`：位于 `筛选` 和 `调整` 之间，用于开启或关闭主表普通文本单元格编辑模式；关闭时普通文本单元格只负责展示和打开详情。
- 展开式搜索：平时只显示放大镜图标，点击后展开并写入当前 active view 的 `query`。

视图 tab 上的未保存标记表示该视图有个人 draft 尚未保存到共享配置。

### ViewFilterBar

`ViewFilterBar` 位于 view tabs 和主表之间，可由 `ViewTabs` 中的 `筛选` 按钮隐藏或显示。它负责当前 active view 的筛选、排序和共享发布入口：

- `排序`：位于筛选栏左侧，打开排序 popover。
- `+ 筛选`：打开字段选择菜单，选择字段后为当前视图新增对应类型的默认筛选规则。
- filter chips：展示当前视图已有筛选规则，点击 chip 打开对应字段类型的编辑 popover。
- 排序 chips：展示当前视图已有排序规则，点击 chip 打开同一个排序 popover。
- `重置`：位于筛选栏右侧；仅在当前 active view 或 view tab 顺序存在个人 draft 时显示，点击后丢弃对应 draft。
- `为所有人保存`：位于筛选栏右侧；仅在存在可发布 draft 时显示，点击后把当前 active view 的个人 draft 和当前 collection 的 view 顺序 draft 覆盖保存为团队共享配置。

筛选当前只支持单层 `and` 规则组；不支持高级筛选、OR 分组或嵌套条件。

### DataTable

`DataTable` 是主表区域，负责：

- 展示顶层字段
- 行内编辑：普通文本字段受 `编辑` 模式控制；Select、Multi-select、Relation 不受该模式影响，仍可在主表内操作
- 排序
- 列宽调整
- 列顺序拖拽
- 字段隐藏
- 新增行、删行
- 打开详情

主表优先承担高频、低上下文切换的编辑操作。标题列仍优先承担打开详情的导航职责，不进入主表文本编辑模式。

### DetailPanel

`DetailPanel` 负责：

- 查看和编辑当前记录的完整字段
- 编辑嵌套对象和数组
- 展示 relation backlink
- 展示 primary key 改名后的影响与同步入口

它是主表之外的深度编辑区。

### RelationConfigDialog

`RelationConfigDialog` 用于把某个字段显式配置为 relation。配置项包括：

- 目标文件
- 目标集合
- 目标主键
- 单值 / 多值模式
- 标题字段优先级
- 是否允许缺失引用

## 主工作流

### 1. 打开文件

用户先在 `Sidebar` 中选择文件，然后选择具体 collection。

### 2. 浏览和筛选

用户先通过 `ViewTabs` 选择当前 collection 的 active view。当前视图搜索在 view tab 行通过展开式搜索完成；筛选栏可由 `筛选` 按钮隐藏或显示。用户通过 `ViewFilterBar` 添加 filter chips、编辑排序 chips，最后在 `DataTable` 中查看、编辑、调整列布局。

当前视图内的搜索、筛选和排序会先写入个人 draft；view tab 左右顺序也会先写入当前 collection 的个人顺序 draft。用户可以继续切换视图或文件；draft 属于个人，不会立刻影响团队共享配置。筛选栏右侧的 `重置` 会丢弃对应 draft；`为所有人保存` 会把当前 active view 的 draft 和当前 collection 的 view 顺序 draft 直接覆盖保存为团队共享视图配置，其他用户下次加载共享视图时看到新的结果。

列隐藏、换行、列顺序、详情字段顺序和列宽仍属于个人布局偏好，由 profile 或浏览器本地状态按当前 active view 独立保存，不随 `为所有人保存` 发布到团队共享视图。

### 3. 编辑数据

- 简单字段：在详情中直接编辑；在主表中需要先开启 `编辑` 模式后才能编辑普通文本字段
- 嵌套字段：进入详情面板继续编辑
- relation 字段：使用 relation editor 选择目标记录

### 4. 处理结构语义

如果字段有 special role：

- relation：显示为引用选择器
- backlink：只读展示引用来源
- primary key：受额外保护

### 5. 自动保存

业务数据默认自动保存，不再要求用户手动点击右上角保存。系统分别处理：

- 业务数据
- 项目级配置
- 个人视图配置
- 团队共享视图配置

详情面板和主表普通文本编辑都先写入本地稳定 draft，再按输入节奏提交到 document model。自动保存只负责后台持久化，不应打断当前输入焦点、输入法组合态或光标位置。切换文件、切换 profile、刷新构建、关闭服务和页面卸载前，会先 flush 当前活动文本编辑器的 draft，再等待 autosave 队列落盘。

## 交互约束

- 主表只承载顶层、适合高频查看的内容。
- 嵌套结构不在主表内强行铺平。
- 主表 `编辑` 模式只控制普通文本字段；Select、Multi-select、Relation 的主表操作保持原有即时交互。
- relation 和 backlink 有独立视觉语义，不等同于普通 select。
- 主键字段不应被当作普通 relation 字段处理。
- 详情面板是深度编辑器，不是简单信息预览层。
- 共享视图保存是直接覆盖团队配置，不做冲突检测。

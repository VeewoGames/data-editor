# Data Editor 跨文件关联字段方案

## 方案概述

本方案定义 `data-editor` 中跨数据文件 relation 字段的完整框架设计。目标是在不改变业务 JSON/CSV 底层存储格式的前提下，让编辑器可以像 Notion database 一样把某个字段解释为“引用另一份数据文件中的记录”，并提供可视化选择、校验、跳转和后续维护能力。

本方案范围包括：

- 项目级 relation 配置格式
- relation 字段的识别、索引、校验和显示规则
- 表格、一级详情、二级详情中的 relation 编辑交互
- relation 配置 UI
- 缺失引用、目标文件异常、类型不兼容等状态处理
- 后续反向引用、主键改名影响分析和批量修复的扩展边界

阶段顺序如下：

1. 第一阶段：把当前写死的 `relationTargets` 升级为项目级配置。
2. 第二阶段：建立统一的 relation path、resolver 和 index 层。
3. 第三阶段：让表格、一级详情、二级详情全部通过同一套 relation 机制渲染。
4. 第四阶段：提供字段菜单中的 relation 配置 UI。
5. 第五阶段：扩展跳转、反向引用、主键改名影响分析和批量修复。

整体结构框架：

```text
view-config.json
└─ relations 配置
   ├─ source file / collection / field path
   ├─ target file / collection / key
   ├─ single / multi mode
   └─ title fields / validation options

relation-config
└─ 读取、保存、规范化配置

relation-path
└─ 匹配顶层字段与嵌套字段路径

relation-index
└─ 加载目标文件并建立 id -> record / option 索引

relation-resolver
└─ 按字段路径解析 relation 配置、标题、缺失状态

UI
├─ DataTable
├─ DetailPanel
├─ NestedEditor
└─ RelationCellEditor
```

## 核心原则

### 不改变源数据格式

relation 是编辑器层能力，不是业务数据结构升级。源文件里仍然只保存 id。

单值 relation：

```json
{
  "skill_id": "skill_slash"
}
```

多值 relation：

```json
{
  "keywords": ["burn", "projectile"]
}
```

嵌套 relation：

```json
{
  "effects": [
    {
      "trigger_skill_id": "skill_ignite"
    }
  ]
}
```

不采用以下格式：

```json
{
  "skill": {
    "id": "skill_slash",
    "name": "斩击"
  }
}
```

原因：

- `name` 属于显示层数据，保存副本会产生同步问题。
- 目标记录改名后，引用方副本会变脏。
- 业务 JSON 会被编辑器 UI 需求污染。
- 后续主键改名、反向引用、批量修复会更难处理。

### 配置独立于业务数据

relation 配置保存到项目级编辑器配置中：

```text
C:\Code\Nocturnel\tools\data-editor\view-config.json
```

业务数据文件只负责保存实际 id，不保存 relation 元信息。

### 缺失引用只提示，不自动修复

MVP 阶段不自动删除缺失 id，不自动替换旧 id，不自动级联改名。

编辑器只负责：

- 标出缺失引用
- 标出目标文件未加载
- 标出字段类型不兼容
- 提供后续手动修复入口

## 配置格式设计

### view-config.json 顶层结构

建议在现有 `view-config.json` 中新增 `relations` 区块：

```json
{
  "fields": {},
  "relations": {}
}
```

其中 `fields` 继续保存单选、多选颜色等字段配置；`relations` 专门保存跨文件引用配置。

### relation 配置示例

```json
{
  "relations": {
    "data/enemies.json:enemies:skills": {
      "targetFile": "data/skills.json",
      "targetCollection": "skills",
      "targetKey": "skill_id",
      "mode": "multi",
      "titleFields": ["skill_name", "name", "*_name"],
      "allowMissing": false
    },
    "data/runes.json:$:effects.*.trigger_skill_id": {
      "targetFile": "data/skills.json",
      "targetCollection": "skills",
      "targetKey": "skill_id",
      "mode": "single",
      "titleFields": ["skill_name", "name", "*_name"],
      "allowMissing": false
    }
  }
}
```

### relation key 规则

relation key 使用统一格式：

```text
sourceFile:sourceCollection:fieldPath
```

示例：

```text
data/enemies.json:enemies:skills
data/runes.json:$:effects.*.trigger_skill_id
data/status_effects.json:$:control.required_keyword_id
```

规则：

- `sourceFile` 使用 `/api/files` 返回的相对路径。
- `sourceCollection` 使用 `DocumentModel.collections[].path`。
- `fieldPath` 使用点分路径。
- 数组中的对象字段使用 `*` 表示任意数组项。

### RelationConfig 类型

```ts
type RelationMode = "single" | "multi";

type RelationConfig = {
  targetFile: string;
  targetCollection: string;
  targetKey: string;
  mode: RelationMode;
  titleFields: string[];
  allowMissing: boolean;
};

type ViewConfig = {
  fields: Record<string, FieldViewConfig>;
  relations: Record<string, RelationConfig>;
};
```

### 标题字段规则

`titleFields` 用于决定 relation 标签显示什么。

推荐默认值：

```json
["name", "*_name", "title", "display_name"]
```

Nocturnel 数据常用值：

```json
["skill_name", "rune_name", "enemy_name", "trait_name", "affix_name", "keyword_id", "name", "*_name"]
```

解析顺序：

1. 按 `titleFields` 从左到右查找。
2. `*_name` 表示第一个以后缀 `_name` 结尾的非空字符串字段。
3. 找不到标题时回退到 `targetKey` 的值。

## 类型系统设计

当前真实字段类型已经收敛为：

```ts
type RealFieldType = "Text" | "Select";
```

relation 框架不应把 `Relation` 重新加入普通字段类型切换菜单。原因是 `Text` 和 `Select` 都是字符串字段的真实编辑行为，而 relation 是“字段引用能力”，它需要目标文件、目标 collection、目标主键、单值/多值模式等额外配置，不能和普通类型并列成一个无上下文的切换项。

建议把字段解释拆成两层：

```ts
type FieldBehavior = "Text" | "Select";

type RelationMode = "single" | "multi";

type RelationConfig = {
  targetFile: string;
  targetCollection: string;
  targetKey: string;
  mode: RelationMode;
  titleFields: string[];
  allowMissing: boolean;
};
```

语义：

- `Text`：自由文本，底层是 `string`。
- `Select`：受控文本选项，底层仍是 `string`。
- `RelationConfig`：字段引用能力，底层仍然是 `string | null | string[]`，但通过独立配置解释为引用关系。

重要边界：

- `Relation` 不进入列头的普通 `Text / Select` 类型菜单。
- relation 只能通过 `设为关联字段`、`编辑关联配置`、`取消关联字段` 管理。
- `Relation(single)` 不会强制把已有字符串改成对象。
- `Relation(multi)` 要求底层是数组；如果当前字段是字符串，则显示“不兼容”，不自动转换。
- 单值/多值必须由 `RelationConfig.mode` 明确决定，不能再由当前单元格值是否为数组临时推断。

## 模块拆分设计

### relation-config.mjs

职责：

- 从 `view-config.json` 读取 `relations`。
- 保存 relation 配置。
- 规范化 relation 配置。
- 过滤非法配置。
- 提供默认迁移规则，把旧的 `relationTargets` 映射为默认配置。
- 完成迁移后，运行时 relation 判断只能走新的 resolver，不能继续让硬编码 `relationTargets` 参与判定。

建议文件：

```text
C:\Code\Nocturnel\tools\data-editor\src\relation-config.mjs
C:\Code\Nocturnel\tools\data-editor\src\model\relationConfig.ts
```

核心函数：

```ts
function normalizeRelationConfig(value: unknown): Record<string, RelationConfig>;
function buildRelationKey(sourceFile: string, sourceCollection: string, fieldPath: string): string;
function ensureRelationConfig(viewConfig: ViewConfig, key: string): RelationConfig;
```

旧规则迁移要求：

1. 首次加载时，如果 `view-config.json.relations` 为空，可以用当前 `relationTargets` 生成默认配置。
2. 默认配置写入 `view-config.json` 后，后续以配置文件为准。
3. `findRelationTarget()` 只能作为迁移辅助或被删除，不能继续在表格、详情、校验路径中直接决定字段是否 relation。
4. 如果用户取消某个 relation 配置，硬编码旧规则不能再次把它恢复成 relation。

### relation-path.mjs

职责：

- 处理字段路径匹配。
- 支持顶层字段。
- 支持嵌套对象字段。
- 支持数组通配符 `*`。

建议文件：

### relation-index.mjs

职责：

- 按目标文件、collection、key 建立索引。
- 缓存同一个目标的加载结果。

## UI 设计

### 表格单元格

当字段是 relation 时：

- 显示为标签。
- 单值 relation 显示一个标签或 `未设置`。
- 多值 relation 显示多个标签。
- 点击单元格打开 relation popover。
- popover 中可以搜索目标记录。
- 选中后写回 id。
- 单值/多值交互由 `RelationConfig.mode` 决定，不由当前单元格值是否为数组决定。

保存格式：

```json
{
  "skill_id": "skill_slash"
}
```

或：

```json
{
  "skills": ["skill_slash", "skill_guard"]
}
```

### 一级详情面板

一级字段如果是 relation：

- 直接使用 `RelationCellEditor`。
- 不再退回普通文本输入。
- 显示缺失引用提示。

### 二级详情面板

嵌套字段如果匹配 relation path：

- 在二级详情中同样使用 `RelationCellEditor`。
- 字段 path 由当前嵌套路径生成。
- 对数组项统一映射为 `*` 匹配配置。
- 如果当前没有详情字段菜单，则嵌套 relation 的配置先不通过 UI 创建，只通过配置文件或默认配置生效。

示例：

```text
实际路径：effects.0.trigger_skill_id
配置路径：effects.*.trigger_skill_id
```

### 字段菜单

字段菜单保持普通类型切换只包含 `Text / Select`。relation 不作为第三个普通类型出现，而是增加独立操作区：

- `设为关联字段`
- `编辑关联配置`
- `取消关联字段`

只对兼容字段显示：

- 字段值是 `string | null`：可配置为 `single`
- 字段值是 primitive array：可配置为 `multi`
- 字段值是 object / object array：不允许直接配置为 relation，需进入嵌套详情配置内部字段

取消 relation 的行为：

- 只删除当前字段的 relation 配置。
- 不修改业务 JSON/CSV 中的字段值。
- 字段回到原本的 `Text` 或 `Select` 行为。
- 已经保存的 id 字符串仍然保留为普通文本。

嵌套字段配置说明：

- 顶层字段可以从表头字段菜单配置 relation。
- 嵌套字段不能依赖表头字段菜单，因为它们主要出现在一级/二级详情面板中。
- MVP 如要支持嵌套字段配置，需要在详情字段行增加轻量字段菜单。
- 如果暂不做详情字段菜单，则 MVP 只能支持顶层字段配置，嵌套字段 relation 先通过配置文件或默认迁移规则启用。

### 配置弹窗

字段菜单点击 `设为关联字段` 后打开配置弹窗。

弹窗字段：

- 目标文件
- 目标 collection
- 目标主键字段
- 引用模式：单值 / 多值
- 标题字段优先级
- 是否允许缺失引用

交互规则：

- 目标文件从 `/api/files` 获取。
- 目标 collection 从 `loadDocument(targetFile).collections` 获取。
- 目标主键字段从目标 collection 的字段并集获取。
- 默认主键优先选择：
  - `id`
  - `${name}_id`
  - 任何以 `_id` 结尾的字段

配置归属：

- relation 配置属于项目级 schema，保存到 `view-config.json`。
- relation 配置不保存到 `view-configs/<profile>.json`。
- 个人视图配置只保存列宽、隐藏列、换行、顺序等个人偏好。

## 校验设计

### 配置校验

配置错误包括：

- 目标文件不存在
- 目标 collection 不存在
- 目标主键字段不存在

显示状态：

```text
关联配置无效
```

### 引用校验

引用错误包括：

- 单值 id 不存在
- 多值数组中部分 id 不存在
- 目标文件加载失败

显示状态：

```text
未找到引用: skill_old
```

如果目标文件加载失败：

```text
未检查
```

### 类型兼容校验

配置为 `single` 时：

- 兼容：`string | number | null | ""`
- 不兼容：`array | object`

配置为 `multi` 时：

- 兼容：primitive array
- 不兼容：`string | number | object`

不兼容时：

```text
不兼容
```

不自动转换数据。

## 保存设计

### 保存 relation 配置

配置变更写入：

```text
C:\Code\Nocturnel\tools\data-editor\view-config.json
```

保存内容示例：

```json
{
  "relations": {
    "data/enemies.json:enemies:skills": {
      "targetFile": "data/skills.json",
      "targetCollection": "skills",
      "targetKey": "skill_id",
      "mode": "multi",
      "titleFields": ["skill_name", "name", "*_name"],
      "allowMissing": false
    }
  }
}
```

relation 配置是项目级配置，应随仓库同步。它不属于个人视图配置，因为它会影响数据含义、校验和编辑行为。个人视图配置仍然只保存布局偏好，例如列宽、隐藏列、换行、字段顺序和详情字段顺序。

### 保存业务数据

relation 编辑后仍然只写回 id：

```json
{
  "skills": ["skill_slash"]
}
```

不会写入：

```json
{
  "skills": [
    {
      "id": "skill_slash",
      "name": "斩击"
    }
  ]
}
```

### Dirty 状态

需要区分两类 dirty：

- 数据 dirty：修改了业务 JSON/CSV。
- 配置 dirty：修改了 relation 配置。

点击 `Save` 时：

1. 如果数据 dirty，保存业务文件并生成备份。
2. 如果配置 dirty，保存 `view-config.json`。
3. 任一保存失败都不能清空对应 dirty 状态。

## 风险与边界

### 风险一：配置和数据不一致

例如字段配置为 `multi`，但实际数据是字符串。

处理：

- 显示“不兼容”。
- 不自动转换。
- 后续提供手动转换工具。

### 风险二：目标文件很大

目标文件记录很多时，relation option 可能较大。

处理：

- MVP 先内存索引。
- 后续如出现性能问题，再做搜索分页或虚拟列表。

### 风险三：主键改名

主键改名会导致引用方缺失。

处理：

- MVP 只提示。
- 后续做影响分析和批量替换。

### 风险四：嵌套 path 配置难懂

用户不应该手写 `effects.*.skill_id`。

处理：

- UI 自动生成 path。
- 文档中保留 path 规则用于开发和排查。

## 不纳入 MVP 的内容

- 双向 relation
- rollup
- lookup
- relation 浮框中新建目标记录
- 主键改名自动级联
- 删除目标记录时自动清理所有引用
- 多文件批量写入事务

## 最终验收标准

MVP 完成时应满足：

- 可以把任意兼容字段配置为 relation。
- relation 配置保存到 `view-config.json`。
- relation 配置不进入个人 profile。
- relation 不出现在普通 `Text / Select` 类型切换列表中。
- 表格、一级详情、二级详情都能识别 relation。
- relation 选择器能搜索目标记录并写回 id。
- 单值/多值由配置 `mode` 决定。
- 源 JSON/CSV 不保存对象副本。
- 缺失引用有明确提示。
- 取消 relation 只删除配置，不修改业务字段值。
- 取消后刷新不会被旧硬编码规则重新识别为 relation。
- 刷新页面后 relation 配置和显示保持。
- `npm test`、`npm run build`、`npm run test:e2e` 通过。

# Data Editor 当前限制与后续方向

## 概述

本节只记录当前代码已经暴露出来的限制和边界，以及适合进入 `docs/plans/` 的后续主题。

## 当前限制

### 文本与本地化

仓库里仍有一部分界面文案存在编码历史问题，特别是某些组件中的中文字符串。这不影响机制本身，但会影响实际显示质量。

### 顶层编排较重

`src/App.tsx` 目前承担了较多编排职责，包括：

- 文件与配置加载
- 服务恢复状态
- relation/backlink 数据准备
- 保存流程
- 多个对话框状态

这意味着很多横切行为仍集中在一个入口中。

### relation 配置入口仍以字段级为主

当前 relation 的显式配置主要通过字段入口完成。更复杂的嵌套路径维护能力虽然有底层支持，但可发现性仍然不算强。

### 单机使用假设明显

系统默认是本地、单用户、单项目上下文，不包含多人协作、权限控制或远程并发一致性处理。

## 已知边界

- 项目级配置和个人视图配置必须严格分开理解。
- backlink 是派生视图，不是业务字段。
- primary key 候选分析不等于主键正式配置。
- `npm run open` 才是正式打开路径，`npm run dev` 只是开发路径。

## 适合进入 plans 的后续主题

- 文案编码和界面文本统一清理
- `App.tsx` 的编排职责拆分
- 更完整的嵌套 relation 配置体验
- 更强的主键变更批量维护能力
- 更细的测试夹具和端到端覆盖
- 更清晰的错误状态与诊断面板

## 维护规则

从现在开始：

- 正式产品说明更新到 `docs/`
- 执行方案、迁移方案、阶段计划、新功能计划统一放 `docs/plans/`

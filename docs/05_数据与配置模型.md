# Data Editor 数据与配置模型

## 概述

本节描述系统如何理解源文件、如何生成集合、如何通过本机 registry 表达多个项目和数据源，以及项目级和个人级配置各自保存什么。

## 本机项目 registry

Data Editor 用本机 registry 保存可切换项目列表。默认路径：

```text
%APPDATA%\data-editor\projects.json
```

如果设置 `DATA_EDITOR_HOME`，则路径变为：

```text
<DATA_EDITOR_HOME>/projects.json
```

核心结构：

```json
{
  "version": 1,
  "activeProjectId": "nocturnel",
  "projects": [
    {
      "id": "nocturnel",
      "name": "Nocturnel",
      "root": "C:\\Code\\Nocturnel",
      "adapter": "nocturnel",
      "dataSources": [
        { "id": "data", "label": "Data", "kind": "relative", "path": "data" }
      ]
    }
  ]
}
```

`project.id` 和 `dataSource.id` 只允许小写字母、数字、`_` 和 `-`，并且在各自作用域内唯一。

## 数据源

每个项目至少有一个默认数据源：

```json
{ "id": "data", "label": "Data", "kind": "relative", "path": "data" }
```

数据源支持两种路径类型：

- `relative`：相对项目根目录解析。
- `absolute`：按绝对路径解析。

后端文件列表返回虚拟路径，格式为：

```text
<dataSourceId>/<relative-file-path>
```

例如 `balance/items.json` 表示 `balance` 数据源下的 `items.json`。读取和保存时，`src/file-service.mjs` 会重新解析虚拟路径，并检查最终路径仍在对应数据源根目录内。

## 支持的数据格式

当前后端直接支持：

- `.json`
- `.csv`

读写路径由 `server.mjs` 提供，本地保存时会根据扩展名走对应的解析和序列化逻辑。所有文件类 API 都可以通过 `projectId` 指定目标项目；未传入时使用 registry 中的 `activeProjectId`。

## 文件到集合的映射

### JSON

JSON 文件加载后会进入 `DocumentModel`。系统保留原始 root，同时抽象出一个或多个 collection。

常见情况：

- 根是数组：通常直接形成 `$`
- 根是对象：对象中的可编辑数组字段形成多个 collection

### CSV

CSV 会被解析成表格型数据，然后同样通过 `DocumentModel` 暴露给前端。

## 字段显示配置

项目级字段配置保存在 `ViewConfig.fields` 中，包含：

- `type`
- `selectOptions`
- `multiSelectOptions`

这里描述的是字段应该如何显示，不描述字段是否是 relation、backlink 或 primary key。

## 项目级配置

项目级配置仍保存在业务项目自己的 `.data-editor/` 目录：

```text
<project>/.data-editor/view-config.json
<project>/.data-editor/shared-views.json
```

项目级配置结构如下：

```ts
type ViewConfig = {
  fields: Record<string, FieldViewConfig>;
  primaryKeys: Record<string, string>;
  backlinks: Record<string, BacklinkConfig>;
  relations: Record<string, RelationConfig>;
  relationsVersion: number;
};
```

它表达的是项目语义，不是个人偏好。

## 团队共享视图配置

团队共享 view / filter / sort 配置保存在：

```text
<project>/.data-editor/shared-views.json
```

它表达的是团队共享的 collection 视图，不是个人临时草稿。核心结构如下：

```ts
type SharedViewsConfig = {
  version: 1;
  collections: Record<string, {
    views: CollectionView[];
    defaultViewId: string | null;
  }>;
};

type CollectionView = {
  id: string;
  name: string;
  type: "table";
  query: string;
  filters: FilterGroup;
  sorts: SortRule[];
  hidden: string[];
  wrapped: string[];
  order: string[];
  detailOrder: string[];
  widths: Record<string, number>;
};

type FilterGroup = {
  op: "and";
  rules: FilterRule[];
};

type SortRule = {
  id: string;
  field: string;
  direction: "asc" | "desc";
};
```

`collections` 的 key 使用 `<filePath>:<collectionPath>`，例如 `data/items.json:$`。当前共享视图发布链路使用 `CollectionView` 的搜索文本、筛选规则和排序规则；列布局字段作为 schema 预留和兼容字段保留，但当前 UI 的列隐藏、换行、列顺序、详情字段顺序和列宽仍走个人 profile 或浏览器本地偏好，不随 `为所有人保存` 发布。`FilterGroup` 当前固定为单层 `and`，不表达 OR、高级筛选或嵌套条件。`SortRule` 按数组顺序依次应用。

## 个人视图配置

`UserViewProfile` 保存每个用户的布局偏好：

- `sidebarWidth`
- `detailPanelWidth`
- `fileOrder`
- `lastActiveViews`
- `viewDrafts`
- `viewOrderDrafts`
- `viewLayouts`

其中 `viewLayouts` 的结构为：

```ts
type UserViewLayoutState = {
  hidden: string[];
  wrapped: string[];
  order: string[];
  detailOrder: string[];
  widths: Record<string, number>;
};

type UserViewProfile = {
  // ...
  viewLayouts: Record<string, Record<string, UserViewLayoutState>>;
};
```

第一层 key 是 `<filePath>:<collectionPath>`，第二层 key 是共享 view 的 `viewId`。也就是说，个人列布局现在按“collection + view”维度保存，而不是整组 view 共用一套 collection 级布局。

其中 `lastActiveViews`、`viewDrafts`、`viewOrderDrafts` 是共享视图机制的个人状态：

- `lastActiveViews`：记录每个 collection 最近选中的共享视图 id。
- `viewDrafts`：记录每个 collection、每个 view 的个人 draft，例如 `query`、`filters`、`sorts`。
- `viewOrderDrafts`：记录每个 collection 的 view tab 个人排序 draft。

这些 draft 不会自动写入 `.data-editor/shared-views.json`。点击 `为所有人保存` 时，当前 active view 的搜索、筛选、排序 draft 和当前 collection 的 view 顺序 draft 会直接覆盖到团队共享视图配置；点击 `重置` 时丢弃对应 draft。

个人视图配置保存在：

```text
<project>/.data-editor/view-configs/<profile>.json
```

因此不同项目可以拥有同名 profile，但内容彼此独立。

## 本地 localStorage 与 profile draft 边界

未选择 profile 或处于本地模式时，个人 draft 写入浏览器 localStorage。共享视图 draft 的 key 为：

```text
data-editor:shared-view-drafts
```

它保存与 profile draft 同形状的 `lastActiveViews`、`viewDrafts`、`viewOrderDrafts`，但只属于当前浏览器环境。profile 模式下，这些字段写入 `<project>/.data-editor/view-configs/<profile>.json`，可以随 profile 文件在项目内保留。

其他 localStorage key 仍用于本地布局偏好，例如字段隐藏、换行、列宽、列顺序、详情字段顺序、侧栏宽度和 `data-editor:__file-order`。当前这些布局 key 同样按 `filePath + collectionPath + viewId` 区分，不等同于 `.data-editor/shared-views.json`；只有 `为所有人保存` 会更新团队共享视图配置。

## 字段显示类型与字段角色的区别

当前系统明确把这两个层面拆开：

- 显示类型：控件和样式
- 字段角色：数据结构语义

例如：

- 一个字段可以显示为 `Text`，但角色是 `primaryKey`
- 一个字段可以显示为 `Relation`，角色是 `relation`
- `Backlink` 不是普通的持久化业务字段

## 主键候选

系统会分析集合字段，识别 primary key 候选，并给出：

- `configured`
- `unconfigured`
- `candidate-detected`

候选规则当前偏向：

- `id`
- `*_id`

但候选分析本身不等于正式主键配置。正式主键仍以 `ViewConfig.primaryKeys` 为准。

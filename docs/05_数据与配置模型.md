# Data Editor 数据与配置模型

## 概述

本节描述系统如何理解源文件、如何生成集合、如何通过本机 registry 表达多个项目和数据源，以及项目级和个人级配置各自保存什么。

## 本机项目 registry

Data Editor 用本机 registry 保存可切换项目列表。默认路径：

```text
%APPDATA%\data-editor\projects.json
```

如果设置 `DATA_EDITOR_HOME`，则路径变为：

```text
<DATA_EDITOR_HOME>/projects.json
```

核心结构：

```json
{
  "version": 1,
  "activeProjectId": "nocturnel",
  "projects": [
    {
      "id": "nocturnel",
      "name": "Nocturnel",
      "root": "C:\\Code\\Nocturnel",
      "adapter": "nocturnel",
      "dataSources": [
        { "id": "data", "label": "Data", "kind": "relative", "path": "data" }
      ]
    }
  ]
}
```

`project.id` 和 `dataSource.id` 只允许小写字母、数字、`_` 和 `-`，并且在各自作用域内唯一。

## 数据源

每个项目至少有一个默认数据源：

```json
{ "id": "data", "label": "Data", "kind": "relative", "path": "data" }
```

数据源支持两种路径类型：

- `relative`：相对项目根目录解析。
- `absolute`：按绝对路径解析。

后端文件列表返回虚拟路径，格式为：

```text
<dataSourceId>/<relative-file-path>
```

例如 `balance/items.json` 表示 `balance` 数据源下的 `items.json`。读取和保存时，`src/file-service.mjs` 会重新解析虚拟路径，并检查最终路径仍在对应数据源根目录内。

## 支持的数据格式

当前后端直接支持：

- `.json`
- `.csv`

读写路径由 `server.mjs` 提供，本地保存时会根据扩展名走对应的解析和序列化逻辑。所有文件类 API 都可以通过 `projectId` 指定目标项目；未传入时使用 registry 中的 `activeProjectId`。

## 文件到集合的映射

### JSON

JSON 文件加载后会进入 `DocumentModel`。系统保留原始 root，同时抽象出一个或多个 collection。

常见情况：

- 根是数组：通常直接形成 `$`
- 根是对象：对象中的可编辑数组字段形成多个 collection

### CSV

CSV 会被解析成表格型数据，然后同样通过 `DocumentModel` 暴露给前端。

## 字段显示配置

项目级字段配置保存在 `ViewConfig.fields` 中，包含：

- `type`
- `selectOptions`
- `multiSelectOptions`

这里描述的是字段应该如何显示，不描述字段是否是 relation、backlink 或 primary key。

## 项目级配置

项目级配置仍保存在业务项目自己的 `.data-editor/` 目录：

```text
<project>/.data-editor/view-config.json
<project>/.data-editor/shared-views.json
```

项目级配置结构如下：

```ts
type ViewConfig = {
  fields: Record<string, FieldViewConfig>;
  primaryKeys: Record<string, string>;
  backlinks: Record<string, BacklinkConfig>;
  relations: Record<string, RelationConfig>;
  relationsVersion: number;
};
```

它表达的是项目语义，不是个人偏好。

## 团队共享视图配置

团队共享 view / filter / sort 配置保存在：

```text
<project>/.data-editor/shared-views.json
```

它表达的是团队共享的 collection 视图，不是个人临时草稿。核心结构如下：

```ts
type SharedViewsConfig = {
  version: 1;
  collections: Record<string, {
    items: SharedViewItem[];
    defaultViewId: string | null;
  }>;
};

type SharedViewItem =
  | { kind: "view"; view: CollectionView }
  | { kind: "group"; id: string; name: string; views: CollectionView[] };

type CollectionView = {
  id: string;
  name: string;
  type: "table";
  query: string;
  filters: FilterGroup;
  sorts: SortRule[];
  hidden: string[];
  wrapped: string[];
  order: string[];
  detailOrder: string[];
  widths: Record<string, number>;
};

type FilterRule = {
  kind: "rule";
  id: string;
  field: string;
  operator: "is" | "is_not" | "contains" | "does_not_contain" | "is_empty" | "is_not_empty";
  value?: unknown;
};

type FilterGroupNode = {
  kind: "group";
  id: string;
  op: "and" | "or";
  children: FilterNode[];
};

type FilterNode = FilterRule | FilterGroupNode;

type FilterGroup = {
  topLevelRules: FilterRule[];
  advancedRoot: FilterGroupNode | null;
};

type SortRule = {
  id: string;
  field: string;
  direction: "asc" | "desc";
};
```

`collections` 的 key 使用 `<filePath>:<collectionPath>`，例如 `data/items.json:$`。`items[]` 是共享视图结构的唯一真相：

- `kind: "view"` 表示顶层普通视图
- `kind: "group"` 表示顶层视图组，组内 `views[]` 承载具体子视图
- `defaultViewId` 必须始终指向某个真实 view，而不能指向组 id

旧格式 `views[]` 仍可作为读取兼容入口，但新写回统一落到 `items[]`。

当前共享视图发布链路使用 `CollectionView` 的搜索文本、筛选规则和排序规则；列布局字段作为 schema 预留和兼容字段保留，但当前 UI 的列隐藏、换行、列顺序、详情字段顺序和列宽仍走个人 profile 或浏览器本地偏好，不随 `为所有人保存` 发布。`FilterGroup` 现在采用“顶层普通规则 + 唯一高级树”的双层结构：

- `topLevelRules` 对应顶部普通 chips，始终按 `AND` 求值
- `advancedRoot` 对应唯一高级筛选入口 chip，可递归表达 `AND/OR`
- `advancedRoot` 记作第 1 层，最多允许到第 3 层 group
- 第一版不支持把高级树内的某条 rule 直接移回顶部普通 chips

旧格式 `{ op: "and", rules: [...] }` 在加载时会自动升格为 `{ topLevelRules, advancedRoot: null }`。`SortRule` 仍按数组顺序依次应用。

## 个人视图配置

`UserViewProfile` 保存每个用户的布局偏好：

- `sidebarWidth`
- `detailPanelWidth`
- `fileOrder`
- `sidebarTree`
- `lastActiveViews`
- `viewDrafts`
- `viewOrderDrafts`
- `appearance`
- `viewLayouts`
- `collections`

其中 `viewLayouts` 的结构为：

```ts
type UserViewLayoutState = {
  hidden: string[];
  wrapped: string[];
  order: string[];
  detailOrder: string[];
  widths: Record<string, number>;
};

type UserViewProfile = {
  // ...
  sidebarTree: SidebarTreePreferences;
  appearance?: UserAppearancePreferences;
  viewLayouts: Record<string, Record<string, UserViewLayoutState>>;
  collections?: Record<string, UserViewLayoutState>;
};
```

第一层 key 是 `<filePath>:<collectionPath>`，第二层 key 是共享 view 的 `viewId`。也就是说，个人列布局现在按“collection + view”维度保存，而不是整组 view 共用一套 collection 级布局。

其中：

- `sidebarTree` 保存文件树的同级排序和显式展开节点。
- `appearance` 保存个人主题、基础字号，以及未来可能出现的 theme token override。
- `collections` 仍作为旧 collection 级布局结构的兼容字段保留；当前正式布局模型以 `viewLayouts` 为准。

其中 `lastActiveViews`、`viewDrafts`、`viewOrderDrafts` 是共享视图机制的个人状态：

- `lastActiveViews`：记录每个 collection 最近选中的共享视图 id。
- `viewDrafts`：记录每个 collection、每个 view 的个人 draft，例如 `query`、`filters`、`sorts`。
- `viewOrderDrafts`：记录每个 collection 的 view tab 个人排序 draft。

视图分组引入后，还增加了两个和共享结构直接相关的个人状态：

- `structureDrafts`：记录当前 collection 的顶层/组内结构草稿，表达顶层排序、组内排序、拖入组和拖出组。
- 页面上下文中的 `expandedGroupId` 与 `lastActiveViewIdByGroupId`：分别记录当前展开组，以及每个组上次活跃的子视图。

这些 draft 不会自动写入 `.data-editor/shared-views.json`。点击 `为所有人保存` 时，当前 active view 的搜索、筛选、排序 draft 会与当前 collection 的 `structureDrafts` 一起覆盖到团队共享视图配置；点击 `重置` 时丢弃对应 draft。

个人视图配置保存在：

```text
<project>/.data-editor/view-configs/<profile>.json
```

因此不同项目可以拥有同名 profile，但内容彼此独立。

## 本地 localStorage 与 profile draft 边界

未选择 profile 或处于本地模式时，个人 draft 写入浏览器 localStorage。共享视图 draft 的 key 为：

```text
data-editor:shared-view-drafts
```

它保存与 profile draft 同形状的 `lastActiveViews`、`viewDrafts`、`viewOrderDrafts`、`structureDrafts`，但只属于当前浏览器环境。profile 模式下，这些字段写入 `<project>/.data-editor/view-configs/<profile>.json`，可以随 profile 文件在项目内保留。

在视图分组模式下，这里还会保存 `structureDrafts`，用来承载当前浏览器或当前 profile 下尚未发布的共享视图结构改动。

其他 localStorage key 仍用于本地布局偏好，例如字段隐藏、换行、列宽、列顺序、详情字段顺序、侧栏宽度和 `data-editor:__file-order`。当前这些布局 key 同样按 `filePath + collectionPath + viewId` 区分，不等同于 `.data-editor/shared-views.json`；只有 `为所有人保存` 会更新团队共享视图配置。

额外的本地状态还包括：

- `data-editor:__sidebar-tree-prefs`：本地文件树排序与展开状态
- `data-editor:page-context`：按项目分桶的当前文件、collection 与滚动恢复状态
- `data-editor:ui-theme` / `data-editor:ui-font-size`：未进入 profile 模式时的主题与字号 fallback

## 字段显示类型与字段角色的区别

当前系统明确把这两个层面拆开：

- 显示类型：控件和样式
- 字段角色：数据结构语义

例如：

- 一个字段可以显示为 `Text`，但角色是 `primaryKey`
- 一个字段可以显示为 `Relation`，角色是 `relation`
- `Backlink` 不是普通的持久化业务字段

## 主键候选

系统会分析集合字段，识别 primary key 候选，并给出：

- `configured`
- `unconfigured`
- `candidate-detected`

候选规则当前偏向：

- `id`
- `*_id`

但候选分析本身不等于正式主键配置。正式主键仍以 `ViewConfig.primaryKeys` 为准。

# Data Editor 产品概览

## 概述

`data-editor` 是一个本地 Web 数据编辑器，用来查看和修改项目中的 JSON / CSV 数据文件。它的目标不是替代源码编辑器，也不是通用数据库产品，而是给项目型数据提供一个更接近表格数据库的编辑体验，同时保留本地文件、Git diff 和项目结构的可控性。

## 产品定位

- 运行形态：本地 HTTP 服务 + 浏览器前端。
- 主要用户：项目开发者、数据设计人员、需要快速检查或批量维护配置数据的 AI agent。
- 核心场景：浏览数据文件、切换集合、编辑行列、查看嵌套结构、维护 relation / backlink / primary key，并通过自动保存写回源文件。

## 当前已经具备的能力

- 浏览项目内白名单数据文件。
- 加载 JSON 和 CSV，并构建统一的 `DocumentModel`。
- 在主表中编辑顶层字段。
- 在详情面板中编辑嵌套对象和数组。
- 为字段指定显示类型。
- 为字段配置 relation。
- 从 relation 自动派生 backlink 展示列和详情面板引用区。
- 为集合维护 primary key，并分析主键改名影响。
- 自动保存业务数据、项目级视图配置和个人视图配置。
- 服务断开后通过 recovery bridge 尝试恢复。

## 产品不做什么

- 不替代 Notion、doc_sync 或 notion_sync。
- 不负责远程协作或多人同时编辑。
- 不把业务数据迁移成新的持久化格式。
- 不在保存时自动“修复”所有不合法数据。
- 不把历史方案文档继续当作正式产品说明的一部分。

## 当前运行模型

- 前端：React + TypeScript + Vite。
- 本地 API：Node.js HTTP 服务。
- 默认主服务端口：`8787`。
- 默认 recovery bridge 端口：`8791`。
- 项目级配置：`view-config.json`。
- 个人视图配置：通过 `/api/view-profile` 读写。

## 建议理解方式

理解这个产品时，先把它看成三层：

1. 文件和集合层：决定“数据从哪里来”。
2. 视图和字段层：决定“怎么展示和编辑”。
3. 关系和保护层：决定“哪些字段有结构语义，哪些操作需要更强约束”。

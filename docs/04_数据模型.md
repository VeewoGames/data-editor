# Data Editor 数据模型

## 方案概述

本文定义 `data-editor` 内部如何把 JSON/CSV 映射成统一的 database collection，同时保持源文件结构可保存回原形态。

## Document Model

```ts
type DocumentModel = {
  format: "json" | "csv";
  sourcePath: string;
  root: unknown;
  rootKind: "array" | "object";
  collections: CollectionInfo[];
};

type CollectionInfo = {
  path: string;
  label: string;
  rowCount: number;
};
```

## Collection 规则

### root array

源文件：

```json
[
  { "id": 1, "name": "A" }
]
```

collection：

```text
$
```

保存时仍保存为 root array。

### root object

源文件：

```json
{
  "skills": [
    { "id": 1, "skill_id": "skill_a" }
  ],
  "meta": {
    "version": 1
  }
}
```

collection：

```text
skills
```

保存时仍保存为 root object，`meta` 不应丢失。

## 主表字段规则

进入主表的字段：

- `string`
- `number`
- `boolean`
- `null`
- primitive array

进入详情面板的字段：

- object
- object array
- nested array
- mixed array

## 字段显示类型

```ts
type FieldDisplayType =
  | "Text"
  | "Number"
  | "Checkbox"
  | "Select"
  | "Multi-select"
  | "Date"
  | "JSON"
  | "Nested";
```

显示类型不是 schema migration。切换显示类型不修改源数据。

## 兼容性规则

- `Text`：兼容所有值，但复杂值只显示摘要或 JSON。
- `Number`：兼容 number 和可解析数字字符串。
- `Checkbox`：兼容 boolean。
- `Select`：兼容 string、number、null。
- `Multi-select`：兼容 primitive array。
- `Date`：兼容 string 和 number。
- `JSON`：兼容所有值。
- `Nested`：兼容 object 和 array。

## 路径编辑

嵌套编辑通过 path array 定位：

```ts
["effects", 0, "params", "trigger_effect", "type"]
```

所有嵌套修改都应通过统一 callback 进入应用状态，不能由 UI 组件直接保存文件。

## CSV 映射

CSV 被解析为：

```ts
Record<string, string>[]
```

MVP 中 CSV 单元格默认都是 string。后续可以通过字段显示类型把某列解释为 `Number`、`Date`、`Checkbox` 等，但保存时仍序列化为 CSV 文本。

CSV round trip 保证表格语义，不保证字节级保留：

- 原始换行风格
- 空行
- 引号风格
- 注释

## Relation 数据格式规范

### 设计原则

- `Relation` 是编辑器层能力，不是源数据结构升级。
- 业务 JSON 中只保存“目标记录的主键 id”，不保存显示名称副本。
- 编辑器可以把 relation 渲染为标签、弹出选择器、详情跳转或反向引用提示，但保存时只能写回 id。
- relation 配置独立维护，不混入业务 JSON。

### 单值 Relation

单值 relation 的底层格式应为 `string | null`：

```json
{
  "skill_id": "slash",
  "keyword_id": null
}
```

适用场景：

- 一条记录只引用另一张表中的单个目标
- 常见字段名如 `skill_id`、`keyword_id`、`rune_id`

### 多值 Relation

多值 relation 的底层格式应为 `string[]`：

```json
{
  "keywords": ["fire", "burn", "projectile"],
  "runes": ["ignite_core", "ember_mark"]
}
```

适用场景：

- 一条记录引用另一张表中的多个目标
- 顺序有意义时，数组顺序即为显示和保存顺序

### 嵌套结构中的 Relation

嵌套对象和嵌套数组中的 relation 仍然遵循同一规则：只存 id，不存对象副本。

```json
{
  "effects": [
    {
      "trigger_skill_id": "ignite",
      "granted_keywords": ["burn", "projectile"]
    }
  ]
}
```

这类字段在编辑器中可以通过二级详情、标签和 relation 选择器展示，但源数据格式不变。

### 不采用的格式

以下格式不应作为 relation 的保存格式：

```json
{
  "skill": {
    "id": "slash",
    "name": "斩击"
  }
}
```

原因：

- `name` 等显示字段属于冗余副本
- 目标记录改名后容易出现脏数据
- 会把“显示层需求”耦合进“源数据结构”
- 不利于通用编辑器保持 JSON 原形

### 标题显示规则

relation 的可读标题来自目标文件记录本身，而不是当前文件保存的副本。默认规则：

1. 优先使用 `name`
2. 若不存在，则使用 `_name` 后缀字段
3. 若仍不存在，则回退到主键 id

因此，JSON 中保存：

```json
{
  "skill_id": "slash"
}
```

编辑器中可以显示为：

```text
斩击
```

但保存时仍然只写回 `"slash"`。

### 缺失引用与异常状态

- 若 relation 值为空：视为未设置，不报错
- 若 relation 目标文件未加载：显示 `未检查`
- 若 relation id 在目标文件中不存在：显示警告，例如 `未找到引用: slash_old`
- 若 relation 数组中部分值失效：逐项提示，不自动删除

MVP 阶段这些状态属于校验提示，不自动阻止保存。

### 重命名与联动边界

MVP 阶段不要求自动级联修改所有引用方。

也就是说，如果目标表中的主键从：

```json
{ "skill_id": "slash" }
```

改为：

```json
{ "skill_id": "slash_basic" }
```

引用它的其他文件不会自动同步改写，而是会在校验时显示缺失引用警告。

后续如果要支持更接近 Notion 的 relation 体验，可以在编辑器层增加：

- 主键改名影响分析
- 批量联动更新
- 反向引用列表
- 双向 relation 配置

但这些都不应改变源文件 relation 的底层存储格式。

### 当前建议的规范总结

- `Relation(single)` -> `string | null`
- `Relation(multi)` -> `string[]`
- 嵌套 relation -> 仍然只存 id
- 标题显示 -> 来自目标文件，不写回源数据
- relation 配置 -> 独立于业务 JSON
- 无效 relation -> 提示，不擅自修复

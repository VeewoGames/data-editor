# Data Editor 校验与保存机制

## 概述

系统不仅负责“改值”，还要负责识别不兼容状态、区分普通自动保存和结构性命令保存，并在服务异常时尽量恢复。

## 校验类型

当前前端会处理几类校验：

- 必填校验
- 唯一性校验
- relation 引用有效性校验
- 显示类型与实际值兼容性校验

这些校验会以 `ValidationIssue` 的形式附着在具体单元格或字段上。

## relation 校验

relation 校验主要关注：

- 目标记录是否存在
- relation 值的形状是否符合 `mode`
- 目标索引是否成功建立

如果目标索引暂时不可用，系统不会直接崩溃，而是返回中性状态。

## dirty 状态

系统内部明确维护三类与持久化相关的 dirty / pending：

- `dataDirty`
- `viewConfigDirty`
- `profile` autosave dirty

它们分别对应：

- 业务文件修改
- 项目级配置修改
- 命名 view profile 的个人配置修改

shared view draft dirty 继续独立存在，不属于普通 autosave 落盘域。

额外还有一条只属于 shared view 个人模式的临时失败状态：

- `sharedViewDirectSavePending`

它表示“个人模式下最近一次 shared view 直存失败，前端临时保留了一份 fallback draft，等待重试或重置清理”。

## 保存行为

普通业务数据默认自动保存。右上角不提供手动保存按钮；用户编辑数据后，前端通过统一 autosave 调度把待保存文档写回源文件。自动保存状态只用于反馈，不作为全局输入禁用态。

### 业务数据

通过 `POST /api/save` 写回，后端根据扩展名序列化后直接覆盖写入。

当前默认保存策略是 `direct-write`：

- 不生成本地 `.bak` 备份
- Git 作为主要版本回退来源
- 自动保存反馈聚焦在“是否已成功写入”

### 项目级配置

通过 `POST /api/view-config` 保存。relation、primary key 等结构性配置仍属于命令保存或即时配置写入，不并入普通业务数据 autosave 的输入节奏。

### 个人视图配置

通过 `/api/view-profile` 读写。命名 profile 已并入统一 autosave 调度，但仍和 shared view 发布是两回事。

### 团队共享视图配置

shared view 的保存行为分成两种模式：

- `team` 模式：普通 shared view 编辑先写入个人 draft；只有显式点击 `保存团队共享视图` 才会把当前 active view draft 与结构 draft 合并后写回 `<project>/.data-editor/shared-views.json`
- `personal` 模式：普通 shared view 编辑与结构拖拽走串行直存，直接调用 shared views 保存链路写回 `<project>/.data-editor/shared-views.json`

`team` 模式和 `personal` 模式都不改变业务数据 autosave 或 profile autosave 的边界；它们只改变 shared view 的正式配置何时落盘。

### 个人模式直存失败降级

`personal` 模式下，如果 shared view 正式配置写回失败：

- 前端固定反馈 `共享视图自动保存失败`
- 当前 collection 的 fallback draft 会被临时保留，避免用户刚改的 shared view 状态直接丢失
- 设置面板中会出现轻量 `重试共享视图保存` 入口，而不会重新常驻显示团队发布按钮
- 视图筛选区的 `重置` 会清理这份 fallback draft，并同时清掉 pending / retry 状态

## 批量保存

系统支持把多个待保存文档作为一组提交，用于主键改名后批量同步 relation 的场景。这类同步使用 `commandSaving` 表示显式命令保存，允许禁用会造成并发结构变更的按钮。

## 服务恢复

前端会定期探测主服务健康状态，并监听 recoverable request 事件。

当主服务不可达时：

1. 先确认故障是否持续存在。
2. 再检查 recovery bridge 是否还活着。
3. 如果 bridge 可用，尝试 `reopen`。
4. 如果有未保存改动，则进入 `recoveredPendingReload` 等保护状态，而不是静默刷新。

## 关闭行为

`POST /api/shutdown` 会触发主服务停止流程，随后 controller 会协同停止对应服务。`stop.mjs` 也可以通过运行时状态文件直接处理停机。

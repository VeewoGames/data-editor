# Data Editor 技术决策

## 方案概述

本文记录 `data-editor` 当前已确认的技术方向和仍待确认的技术问题。目标是在开发前先把边界讲清楚，避免实现阶段反复改方向。

## 已确认决策

### 使用本地 Web 工具

采用 `tools/data-editor/` 下的本地 Web 工具，而不是 Godot editor plugin。

原因：

- 更适合 Codex Browser 使用。
- 不需要启动 Godot。
- 更容易采用现代 Web 表格和组件生态。
- 更容易支持 CSV/JSON 以外的后续格式。

### 使用 React + TypeScript + Vite

首版直接使用 React + TypeScript + Vite，不先做 vanilla JS MVP。

原因：

- 目标体验接近 Notion database。
- 复杂交互包括字段菜单、popover、详情面板、键盘/focus 管理。
- 后续从 vanilla 迁移到组件化 UI 会产生重写成本。
- TypeScript 有利于约束 document model、field type 和 nested path。

### 使用 TanStack Table

使用 `@tanstack/react-table` 管理表格状态。

原因：

- headless，不强制视觉风格。
- 适合 Notion-like 自定义 UI。
- 支持排序、列可见性、行模型等基础能力。

### 使用 Radix UI

使用 Radix UI primitives 构建菜单、popover、dialog、select、tooltip。

原因：

- 可访问性和 keyboard/focus 基础较好。
- 不强制样式。
- 适合 Notion-like 高密度交互。

### 使用 Tabler Icons filled 系列

使用 `Tabler Icons` 的 filled 系列作为主图标库。

原因：

- 项目需要简洁现代的工具型图标风格。
- filled 图标更适合表达 active、selected、dirty、error 等状态。
- `Tabler Icons` 覆盖面广，后续扩展 CSV/JSON 以外格式时可选图标充足。
- MIT license，适合项目长期使用。

实现时优先使用官方 React package 或可 tree-shaking 的导入方式，避免一次性打包整套图标。

### Node local API 负责文件读写

浏览器前端不直接读写文件。Node server 提供：

- `/api/files`
- `/api/document`
- `/api/save`

保存 API 必须使用 allowlist，只允许写入 `/api/files` 返回的 `.json` / `.csv`。

### 保持源文件形态

JSON root object 文件不迁移为 root array。

原因：

- `skills.json`、`enemies.json`、`traits.json`、`affixes.json` 现有运行时和工具链依赖 wrapper key。
- 编辑器内部可以抽象 collection，但保存必须保持 root shape。

## 边界约束

- React 组件不解析或序列化文件。
- React 组件不直接调用文件系统。
- `DataTable.tsx`、`DetailPanel.tsx`、`NestedEditor.tsx` 不直接调用保存 API。
- codec、document model、file service 不依赖 React。
- 保存前必须创建备份。
- 保存失败不得清除 dirty 状态。

## 待确认技术问题

- 是否需要 `pnpm` / `npm` / `bun` 固定一种包管理器？当前建议使用 `npm`。
- 是否需要把字段显示配置持久化到项目文件？当前 MVP 建议使用 browser localStorage。
- 是否需要增加 `/api/preview-save` 做保存前 diff preview？
- 是否需要在 MVP 中引入虚拟滚动库？当前建议先使用 render limit，后续再加。
- 是否需要使用 CSS modules？当前方案允许 plain CSS，后续根据组件规模决定。

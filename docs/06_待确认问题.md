# Data Editor 待确认问题

## 方案概述

本文列出开发前需要确认或排序的问题。确认后再进入实现，可以减少返工。

## 产品问题

1. MVP 是否需要支持新增/删除行？需要
2. MVP 是否需要支持新增/删除字段？需要
3. root object 中非 collection metadata 是否只读展示，还是需要编辑？MVP 只读展示，不编辑。例如 `{ "skills": [...], "meta": {...} }` 中的 `meta` 只作为文件元信息展示，避免误改非表格数据。
4. CSV 是否需要自定义 delimiter，例如 `;`、tab？MVP 不需要。先支持常规逗号分隔 CSV；后续如果出现 TSV 或分号分隔文件，再扩展 delimiter 设置。
5. 字段显示类型配置是否只保存在 localStorage，还是写入项目配置？本地
6. 是否需要保存前 diff preview？不需要
7. 是否需要字段级数据校验，例如 `rune_id` 必填、`id` 唯一？需要
8. 是否需要跨文件 relation 提示，例如 `enemies.skills[]` 引用 `skills.skill_id`？需要

## 交互问题

1. 主表是否需要一开始就支持列拖拽排序？需要
2. 主表是否需要一开始就支持 pinned columns？不需要
3. 单元格编辑是 click 进入，还是 double click 进入？click
4. `Enter` 行为是保存编辑，还是打开详情？打开详情
5. nested object array 的 mini table 是否需要排序/筛选？MVP 不需要。详情面板里的小表格先用于查看和编辑嵌套数组，不做排序/筛选，避免首版交互过重。
6. JSON fallback 是否允许格式化和压缩？需要格式化，不需要压缩。复杂字段使用 JSON 文本编辑时提供“格式化”按钮，帮助阅读；不提供“压缩”按钮，避免降低可读性。

## 技术问题

1. 是否接受 `npm install` 作为首版启动前置步骤？接受。为了获得更好的 Notion-like 交互体验，可以接受首次使用前执行 `npm install`。
2. 是否需要锁定 Node 版本？需要。锁定一个稳定 Node 版本，优先保证工具长期可用，避免不同机器或未来环境因为 Node 版本差异导致启动失败。
3. 是否需要生成 `package-lock.json` 并纳入 Git？需要。`package-lock.json` 纳入 Git，保证依赖版本稳定可复现。
4. 是否需要在 server 端增加最大文件大小限制？需要。MVP 设置一个较宽松的文件大小上限，超过上限时提示用户，不直接打开，避免误开超大文件导致浏览器卡死或保存风险。
5. 是否需要 Web Worker 解析 JSON/CSV？MVP 不需要。先保证当前 Nocturnel 数据文件体验流畅，后续遇到大文件再加入 Web Worker。
6. 是否需要 Playwright 自动化 UI 测试？需要，但只覆盖关键流程：打开文件、编辑单元格、编辑嵌套字段、保存到临时文件、确认 root 结构不变。

## 建议优先确认

开发前建议先确认：

1. MVP 是否支持新增/删除行。需要
2. 字段显示配置是否只放 localStorage。是
3. 是否需要保存前 diff preview。不需要
4. 是否接受 `npm install` 和 React/Vite 工具链。接受。为了获得更好的 Notion-like 交互体验，首版采用 React + TypeScript + Vite + TanStack Table + Radix UI，并接受 `npm install` 作为启动前置步骤。

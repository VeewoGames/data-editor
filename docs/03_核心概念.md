# Data Editor 核心概念

## 概述

这一节定义项目内最重要的正式术语。后续文档默认沿用这些词，不再重复解释。

## DocumentModel

`DocumentModel` 是前端消费数据文件时的统一结构，核心字段包括：

- `format`：`json` 或 `csv`
- `sourcePath`：源文件相对路径
- `root`：原始根数据
- `rootKind`：`array` 或 `object`
- `collections`：当前文档暴露给编辑器的集合列表
- `metadata`：补充摘要信息
- `rootCollectionKind` / `rootKeyField`：特殊根结构的附加信息

它的职责是把“一个文件”抽象成“一个或多个可编辑集合”。

## Collection

`collection` 是编辑器中的主表数据源。一个文件可以只暴露一个集合，也可以暴露多个集合。

常见情况：

- root array：通常直接映射到 `$`
- root object：内部可编辑数组字段会变成独立 collection

## Row / Record

主表中的一行对应一个 `DataRecord`。它本质上是 `Record<string, unknown>`，编辑器在此基础上叠加显示类型、字段角色和校验状态。

## Field

字段有两组语义：

1. 显示类型：决定控件和表现方式。
2. 字段角色：决定它在系统中的结构职责。

显示类型当前包括：

- `Text`
- `Number`
- `Checkbox`
- `Select`
- `Multi-select`
- `Relation`
- `Backlink`
- `Date`
- `JSON`
- `Nested`

字段角色当前包括：

- `normal`
- `primaryKey`
- `relation`
- `backlink`

## ViewConfig

`ViewConfig` 是项目级配置，决定：

- 字段显示配置
- primary key
- backlinks
- relations
- relations 版本

它描述的是“这个项目的数据应该如何被解释和呈现”，不是单个用户的布局偏好。

## UserViewProfile

`UserViewProfile` 是个人视图配置，保存：

- 侧边栏宽度
- 列隐藏状态
- 换行状态
- 列顺序
- 详情面板字段顺序
- 列宽

它只影响个人使用体验，不改变业务语义。

## Relation

`relation` 表示一个字段的值引用了另一个文件/集合中的主键。它是可编辑的，保存时只写回原始 id 值。

## Backlink

`backlink` 不是业务字段，而是由 relation 派生出来的只读引用视图。它可以出现在主表和详情面板里，但不会写回业务文件。

## Primary Key

`primary key` 用来标识集合中的主记录键。它决定：

- relation 的目标键
- 唯一性保护
- 主键改名影响分析
- backlink 汇总的定位基础

## Dirty 状态

系统内部区分两类脏状态：

- 数据脏：业务文件内容被修改
- 配置脏：视图配置被修改

两者会一起影响保存行为，但不是同一类变更。

# Data Editor 视觉与交互体验

## 方案概述

本文定义 `data-editor` 的视觉风格和常见交互体验优化。目标不是复制 Notion，而是借鉴 Notion database 的克制、清晰、低干扰和高密度编辑体验，让 JSON/CSV 数据编辑更像使用一个本地数据库工作台，而不是传统代码编辑器或企业表格系统。

## 视觉方向

### 关键词

- 克制
- 高密度
- 清晰层级
- 低装饰
- 可扫描
- 轻量反馈
- 编辑优先

### 整体气质

界面应该接近 Notion database / Linear / GitHub settings 这类工具型产品：

- 以数据内容为主，不做营销式 hero、插画、渐变背景。
- 页面背景保持浅色中性。
- 线条、分隔、hover、focus 状态要轻，但必须可见。
- 字体层级不要过大，主表和详情面板优先保证密度和可读性。
- 控件圆角控制在 8px 以内。

## 图标系统

### 图标库

`data-editor` 使用 `Tabler Icons` 作为主图标系统，状态与关键操作优先使用 filled 系列。

原因：

- filled 图标比纯 outline 图标更容易表达 active、selected、dirty、error 等状态。
- `Tabler Icons` 覆盖面广，适合文件、表格、字段类型、排序、筛选、保存、删除等工具型界面。
- filled 系列可以让关键操作在高密度界面中更容易被识别。
- MIT license，适合本地工具长期维护。

### 使用原则

- 默认图标尺寸：16px。
- 表头、字段菜单、行操作等密集区域优先使用 14-16px。
- 顶部工具栏按钮可以使用 16-18px。
- 图标颜色默认继承文字色，不单独制造大面积彩色图标。
- active 状态优先通过背景、文字色和图标色共同表达。
- 危险动作图标只在删除、不可逆操作中使用红色。
- 如果某个语义没有合适的 filled 版本，可以使用 `Tabler Icons` 同库 outline 图标补足，但不能混入其它图标库。
- 不混用多套图标库，除非 `Tabler Icons` 无法覆盖某个明确需求。

### 推荐图标映射

| 功能 | 图标方向 |
|---|---|
| 搜索 | search |
| 保存 | device-floppy |
| 未保存 | circle-dot / alert-circle |
| JSON 文件 | file-code |
| CSV 文件 | table |
| 字段类型 | text-size / number / checkbox / calendar / tags |
| 排序 | sort-ascending / sort-descending |
| 筛选 | filter |
| 添加行 | row-insert-bottom |
| 添加字段 | column-insert-right |
| 删除 | trash |
| 详情面板 | layout-sidebar-right |
| 嵌套字段 | hierarchy / braces |
| 不兼容 | alert-triangle |

## 布局结构

```text
┌─────────────────────────────────────────────────────────────┐
│ Top toolbar                                                  │
├───────────────┬───────────────────────────────┬─────────────┤
│ Sidebar       │ Database table                │ Detail      │
│ File list     │ Rows / columns / cells        │ Properties  │
│ Collections   │                               │ Nested data │
└───────────────┴───────────────────────────────┴─────────────┘
```

### 左侧栏

- 宽度建议：240-280px。
- 文件列表按名称分组或排序。
- 当前文件和当前 collection 要有明确选中态。
- 文件过多时支持搜索或过滤。
- 不要用大卡片；使用紧凑 list item。

### 顶部工具栏

包含：

- 搜索框
- 保存按钮
- dirty 状态
- `Reset view`
- 当前行数摘要
- 文件路径或 collection 名称

工具栏高度建议 44-52px，不要过高。

### 主表

- 表头 sticky。
- 行高建议 32-38px。
- 单元格左右 padding 8-10px。
- 列宽可调整。
- 文字默认单行省略，详情在 tooltip 或详情面板中查看。
- 当前选中行要有明显但不刺眼的背景。
- hover 行背景轻微变化。

### 右侧详情面板

- 宽度建议：360-480px。
- 可折叠或可关闭。
- 展示当前 record 的完整属性。
- 嵌套字段用 section 分组。
- 长文本和 JSON fallback 使用等宽字体或独立编辑区域。

## 色彩与状态

### 基础色

- 背景：接近白色或浅灰。
- 主文字：深灰，不使用纯黑大面积文本。
- 次级文字：中灰。
- 边框：浅灰。
- 选中态：浅蓝或浅灰蓝。
- 危险动作：红色，但只用于删除、不可逆或高风险操作。

### 状态设计

- hover：轻微背景变化。
- focus：清晰 outline，不能只靠颜色。
- dirty：顶部显示未保存状态，例如 `未保存`。
- saving：保存按钮进入 loading 状态。
- saved：短暂显示 `已保存`。
- error：使用 inline message，不使用阻断式弹窗作为默认错误展示。
- incompatible：单元格显示 `不兼容`，并使用轻微红色文字或图标提示。

## 字段类型视觉

### Text

- 普通文本单元格。
- 长文本单行省略，详情面板展示完整内容。

### Number

- 数字右对齐。
- 非数字值切换到 Number 时显示 `不兼容`。

### Checkbox

- 使用 checkbox 或 switch-like checkbox。
- 不兼容值显示 `不兼容`。

### Select

- 使用单个 tag/chip。
- 不同值可以自动分配轻量颜色，但不要过度彩虹化。

### Multi-select

- 使用多个 chips。
- chips 必须支持横向溢出或折叠，不撑高整行。

### Date

- 使用紧凑日期文本。
- MVP 可以先不做 date picker，只做格式化显示和文本编辑。

### JSON / Nested

- 主表显示摘要，例如 `Object(4)`、`Array(3)`。
- 点击后在详情面板展开。

## 常见交互体验优化

### 搜索

- 搜索框固定在顶部工具栏。
- 输入后即时过滤。
- 搜索结果需要显示当前匹配数量。
- 清除按钮使用 `X` 图标。

### 排序

- 点击表头或表头菜单触发排序。
- 排序状态需要图标提示。
- 再次点击可切换 asc / desc / none。

### 列隐藏与重置

- 字段菜单中提供 `Hide column`。
- 顶部工具栏提供 `Reset view`，恢复隐藏列、列宽、自动换行、列顺序等视图状态。

### 列宽调整

- 鼠标悬停表头右侧时显示 resize handle。
- 拖拽时给出实时宽度变化。
- 双击 handle 可以恢复默认宽度。

### 单元格编辑

- click 进入编辑。
- `Enter` 打开详情面板。
- `Esc` 取消当前编辑。
- `Tab` 保存当前编辑并移动到下一个可编辑单元格。
- 编辑中不能因为 table re-render 丢失 focus。

### 行新增

- 主表底部提供 `New row`。
- 新行使用当前 collection 的字段结构生成空值。
- 新行创建后自动选中并打开详情面板。

### 行删除

- 删除行必须二次确认。
- 删除后显示短暂 toast。
- MVP 不做 undo 时，确认文案必须明确。

### 字段新增

- 表头末尾提供 `+` 添加字段。
- 新字段需要选择显示类型。
- 对 JSON 文件，新字段会写入当前 collection 的所有或部分 records；默认只给当前 record 写入空值更安全。
- 是否批量补齐所有行，应作为用户选择。

### 字段删除

- 删除字段风险高。
- 必须提示会从哪些 records 删除该 key。
- MVP 实现字段删除，但必须通过二次确认，并显示影响的 records 数量。
- 如果用户只是想临时不看某个字段，应优先使用“隐藏字段”。

### 嵌套字段编辑

- object array 使用 mini table。
- mixed array 使用 JSON fallback。
- JSON fallback 必须支持格式化。
- JSON parse 错误要 inline 显示行列或错误信息。

### 保存

- 保存按钮只在 dirty 状态时高亮。
- 保存成功显示备份路径摘要。
- 保存失败保留 dirty 状态。
- 保存时不要阻塞浏览整个页面，但保存按钮要 loading。

### 错误提示

- 文件读取失败：显示错误面板。
- JSON parse 失败：显示 parse error，不进入编辑模式。
- 保存失败：显示错误 toast 和详情。
- 不兼容：显示在单元格和字段菜单中，不作为全局错误。

## 键盘交互

MVP 建议支持：

- `Tab`：移动焦点。
- `Enter`：打开详情面板。
- `Esc`：关闭 popover/dialog 或取消编辑。
- `Ctrl+S`：保存。
- `/`：聚焦搜索框。

后续增强：

- 方向键移动单元格。
- `Shift+Enter` 换行。
- `Ctrl+D` 复制上一行值。
- 批量复制粘贴。

## 响应式规则

MVP 主要面向桌面宽屏，不优先支持手机。

建议断点：

- 宽屏：左侧栏 + 主表 + 右详情面板三栏。
- 中等宽度：详情面板可覆盖式打开。
- 小屏：只保证可查看，不承诺完整编辑体验。

## 避免事项

- 不使用大面积渐变背景。
- 不使用装饰性卡片堆叠。
- 不把主表放进卡片中再套卡片。
- 不做营销式首页。
- 不让按钮文字溢出。
- 不让单元格编辑导致行高剧烈跳动。
- 不用颜色作为唯一状态提示。
- 不在主表中展开深层 JSON，避免横向列爆炸。

## 验收标准

- 打开工具后第一屏就是可用的数据编辑界面。
- 用户能在 3 秒内识别当前文件、collection、搜索、保存和详情面板位置。
- 表头、行、单元格、详情面板视觉层级清晰。
- `不兼容`、dirty、saving、error 状态都有明确反馈。
- 主表在 277 条 `runes.json` 数据下保持可读和可编辑。
- 嵌套字段不会撑爆主表。
- 视觉风格克制，接近 Notion database 的工具感，而不是企业后台大表格或 Excel clone。

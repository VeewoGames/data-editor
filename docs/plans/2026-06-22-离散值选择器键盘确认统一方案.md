# 离散值选择器键盘确认统一方案

## 方案概述

### 1. 总体目标和范围

本方案用于统一 `data-editor` 内所有“离散值选择器”的搜索与确认交互，解决当前不同入口在“输入后默认命中哪个候选”“按回车后会发生什么”“选中一个值后搜索词是否清空”上的行为漂移。

本次统一覆盖以下 4 类入口：

- [src/table/OptionFieldEditor.tsx](C:/Code/data-editor/src/table/OptionFieldEditor.tsx)
  - 表格与详情页中的 `Select` / `Multi-select` 字段编辑器
- [src/table/RelationCellEditor.tsx](C:/Code/data-editor/src/table/RelationCellEditor.tsx)
  - 表格与详情页中的 `Relation` 选择器
- [src/components/filters/MultiSelectFilterPopover.tsx](C:/Code/data-editor/src/components/filters/MultiSelectFilterPopover.tsx)
  - 顶部普通筛选 chip 的离散值选择弹层
- [src/components/filters/AdvancedFilterRuleEditor.tsx](C:/Code/data-editor/src/components/filters/AdvancedFilterRuleEditor.tsx)
  - 高级筛选规则里的离散值选择弹层

本次不包含以下范围：

- 字段下拉、操作符下拉、逻辑连接词下拉等普通 `Select` 菜单
- 离散值排序规则、拖拽排序规则、颜色编辑规则
- 模糊排序算法升级
- 旧行为兼容开关

第一版的目标不是引入复杂的搜索排序，而是先把所有离散值选择器收敛到一套清晰、可预测、可测试的键盘确认模型。

### 2. 各阶段任务概要

#### 阶段一：定义统一交互契约

主要工作是把当前各入口分散的键盘与搜索行为收敛成统一规则，包括默认候选、回车确认、无匹配创建、选中后清空搜索词等。预期成果是一份所有组件共同遵守的交互契约，后续实现不再各自解释“回车”的含义。

#### 阶段二：抽离共享交互状态

主要工作是把“搜索词 -> 可见候选 -> 默认候选 -> 确认动作”的状态机抽成共享逻辑，由各组件注入自己的候选集、选中提交方式和是否允许创建。预期成果是减少 `OptionFieldEditor`、筛选弹层和 relation 选择器之间的重复逻辑，避免再次分叉。

#### 阶段三：接入 4 类选择器

主要工作是让表格/详情、普通筛选、高级筛选、relation 选择器都接入共享交互逻辑，并保持各自现有的数据提交边界不变。预期成果是无论用户从哪个入口进入，输入与回车确认的体验都一致。

#### 阶段四：补齐可见高亮与回归测试

主要工作是补上默认候选的视觉高亮，并为单选、多选、可创建、不可创建、relation、筛选等路径补测试。预期成果是用户能够直观看到回车会命中谁，且后续改动不再轻易破坏该交互。

### 3. 整体结构框架

整体结构分三层：

- 交互状态层
  - 负责管理搜索词、过滤结果、默认候选、回车确认语义、确认后清空搜索词
- 组件适配层
  - 各组件只负责把自己的 `options`、`selectedValues`、`toggle` / `commit` / `create` 行为接入共享状态层
- 视图表现层
  - 负责渲染搜索框、已选 chip、候选列表、默认高亮行，以及无匹配提示

该结构的重点是：统一的是“交互规则”，而不是强行把所有 UI 组件合并成一个大组件。不同入口可以保留自己的布局和提交节奏，但必须共用同一套回车确认模型。

## 现状分析

### 1. `OptionFieldEditor` 已有局部能力，但契约不完整

[src/table/OptionFieldEditor.tsx](C:/Code/data-editor/src/table/OptionFieldEditor.tsx) 当前已经支持：

- 根据输入过滤现有选项
- 无匹配时在回车后创建新标签
- 创建后把搜索词清空

但它还缺：

- 输入后默认选中“当前最佳候选”
- 有匹配时直接回车确认该候选
- 通过统一规则决定“回车是确认候选还是创建新值”
- 选中一个已有标签后清空搜索词

换句话说，它已经有一半的能力，但没有完整的统一状态机。

### 2. 筛选弹层只有“搜索 + 点击勾选”，缺少键盘确认模型

[src/components/filters/MultiSelectFilterPopover.tsx](C:/Code/data-editor/src/components/filters/MultiSelectFilterPopover.tsx) 和 [src/components/filters/AdvancedFilterRuleEditor.tsx](C:/Code/data-editor/src/components/filters/AdvancedFilterRuleEditor.tsx) 当前具备：

- 搜索输入
- 候选过滤
- 点击 checkbox / row 选中
- 已选 chip 删除

但它们当前没有：

- 默认候选概念
- 回车确认候选
- 无匹配时创建新值
- 选中后自动清空搜索词

这也是截图中“输入了 `d`，用户仍需要再点一次 `dot` 或点创建”的根因。

### 3. `RelationCellEditor` 不应支持创建，但应统一确认逻辑

[src/table/RelationCellEditor.tsx](C:/Code/data-editor/src/table/RelationCellEditor.tsx) 也是离散值搜索选择器，但它的候选来自 relation 目标记录，因此不能像标签字段那样自由创建。

它应统一的部分是：

- 输入后默认命中第一候选
- 回车直接确认该候选
- 确认后清空搜索词

它不应统一的部分是：

- 无匹配时创建新值

因此本方案必须支持“允许创建”和“不允许创建”两种模式，而不是一套逻辑硬套所有组件。

## 统一交互契约

### 1. 默认候选定义

当输入框中存在任意非空字符时，组件应基于当前过滤结果确定一个默认候选。

第一版统一规则：

- 单选、`Relation`、筛选单值场景：
  - 默认候选 = 当前过滤结果的第一项
- 多选场景：
  - 默认候选优先取“当前过滤结果中的第一项未选值”
  - 若过滤结果都已处于选中状态，则回退为当前过滤结果第一项
- 过滤顺序沿用现有 `filterOptions()` 或组件当前候选顺序
- 第一版不引入新的模糊匹配排序算法

这样做的原因是先保证行为一致，再考虑未来是否需要升级“最符合”的定义。

### 2. 回车确认规则

按下回车时，统一遵循以下规则：

1. 如果存在默认候选，则直接确认该候选。
2. 如果不存在默认候选，且当前组件允许创建新值，则创建当前输入值并立即选中。
3. 如果不存在默认候选，且当前组件不允许创建，则不执行创建，也不改动已选值。

这里“确认候选”的含义是：

- 多选：把该候选加入已选值
- 单选：把该候选设为当前唯一值
- Relation：把该候选提交为 relation 选中值
- 筛选：把该候选加入筛选规则的 `value`

### 3. 搜索词清空规则

一旦发生以下任一动作，搜索词必须立即清空：

- 通过回车确认了默认候选
- 通过点击候选确认了一个值
- 通过回车创建并选中了一个新值

这样做的目的是避免当前这种“已选中一个标签，但旧搜索词还留在输入框里，后续继续操作时上下文混乱”的状态。

搜索词被清空后，组件还必须同步恢复到以下状态：

- 候选列表立即回到非搜索态的全量列表
- 默认候选立即清空，不在空搜索词状态下保留高亮命中
- 已选值保持不变

这样可以保证“确认一次 -> 输入框清空 -> 继续输入下一次”的节奏稳定可预期。

以下情况不清空：

- 仅输入字符但尚未确认
- 无匹配且当前组件不允许创建时按回车
- 删除已选 chip

### 4. 已选值与回车语义

回车语义固定为“确认当前默认候选”，而不是对当前命中值做 toggle。

原因如下：

- 用户输入字符后的主要意图是快速确认，而不是反选
- 如果把回车做成 toggle，多选场景下很容易出现“按一次回车把已选项又取消”的反直觉行为

因此，若默认候选已经处于已选状态：

- 多选场景下，回车不做取消
- 单选场景下，回车不产生额外变化

同时，多选场景为了减少“回车命中已选首项导致无事发生”的概率，应优先把默认候选落到第一项未选值；只有当过滤结果全部已选时，才允许默认候选落在已选项上。

### 5. 开合行为边界

本方案不改变各组件现有的弹层开合策略。

例如：

- 单选字段当前如果选中后仍保持打开，则本方案保持不变
- 筛选弹层和 relation 弹层当前如何保持焦点、如何提交值，也保持现状

本次只统一“搜索、默认候选、确认、清空搜索词”这一层，不额外引入关闭弹层的副作用。

### 6. 焦点恢复规则

为了支持连续的“输入 -> 回车 -> 再输入”快路径，凡是存在搜索输入框的离散值选择器，在以下动作完成后都必须把焦点恢复到搜索输入框：

- 回车确认默认候选
- 点击候选确认一个值
- 回车创建并选中新值

这里的要求是“保持弹层现有开合策略不变，但确认完成后尽量把用户送回输入框”，这样不同入口才能真正共享同一套连续键盘操作体验。

以下动作不强制恢复焦点：

- 删除已选 chip
- 无匹配且当前组件不允许创建时按回车

### 7. 默认候选高亮的视觉优先级

默认候选必须有独立于 `hover` 和 `selected` 的可见态，避免用户不知道回车会命中谁。

视觉规则建议固定为：

- `hover`：表示鼠标当前悬停
- `selected`：表示该值已在当前选择集中
- `default-candidate`：表示当前回车将命中的候选

若同一项同时满足多个状态：

- `selected + default-candidate`
  - 保留“已选”的基础视觉
  - 再叠加一层弱但明确的默认命中标记，例如边框、背景加深或左侧强调条
- `hover + default-candidate`
  - 仍应能识别“这是默认命中项”，不能让 hover 完全覆盖默认态

第一版不强制规定具体像素方案，但必须在 CSS 和测试中把这三种状态明确拆开。

## 组件级落地规则

### 1. `OptionFieldEditor`

该组件既用于 `SelectCellEditor`，也用于 `MultiSelectCellEditor`，是本次可创建标签场景的核心入口。

落地要求：

- 输入后计算默认候选
- 按回车优先确认默认候选
- 无匹配时允许创建新值
- 通过点击选中已有值后清空搜索词
- 通过创建新值后清空搜索词
- 每次确认后恢复输入焦点
- 单选与多选共用同一套确认规则

### 2. `RelationCellEditor`

该组件保留 relation 候选只读来源，不允许用户创建 relation 值。

落地要求：

- 输入后计算默认候选
- 按回车确认默认候选
- 点击候选后清空搜索词
- 每次确认后恢复输入焦点
- 无匹配时仅保留空结果提示，不创建新值

### 3. `MultiSelectFilterPopover`

该组件当前既服务 `Multi-select`，也服务 `Select` 和 `Relation` 的筛选值编辑。

因此它内部必须按字段来源区分是否允许创建：

- `Multi-select` / `Select`
  - 允许从筛选 UI 直接新建未配置值
  - 新建后不仅写入当前筛选规则，也要写回字段配置，成为正式候选
- `Relation`
  - 不允许创建，只能从真实 relation 候选中确认

这里的关键不是按组件名判断，而是按字段值来源判断；同时，确认一个值后必须清空搜索词并把焦点恢复到输入框。

### 4. `AdvancedFilterRuleEditor`

高级筛选的离散值弹层与普通筛选弹层在交互上应完全一致，只是挂载位置不同。

落地要求：

- 与 `MultiSelectFilterPopover` 共用同一套离散值确认逻辑
- 根据字段类型区分允许创建与否
- 选中后清空搜索词
- 每次确认后恢复输入焦点
- 保留高级筛选现有的字段、操作符、rule 替换链路

## 推荐实现方案

### 1. 抽共享 hook，而不是复制事件处理

推荐新增一层共享 hook，例如：

- `useDiscreteValuePicker`
- 或 `useSelectableOptionSearch`

其职责应包括：

- 管理输入值
- 接收外部注入的候选列表、过滤函数，以及可选的默认候选排序策略
- 根据输入值和候选列表计算过滤结果
- 计算默认候选
- 对外暴露 `onEnter`
- 对外暴露“确认某个值后清空输入”的统一方法
- 对外暴露“确认后恢复焦点”的统一回调约定

不建议把逻辑继续分别写在 4 个组件里，否则未来任何一个入口增加键盘行为时都要改多处。

### 2. 允许创建与不允许创建走同一状态机分支

共享层需要显式接受一个布尔能力，例如：

- `allowCreate`

这样：

- `OptionFieldEditor` 传 `true`
- `RelationCellEditor` 传 `false`
- 筛选弹层按字段类型动态决定

避免出现 relation 被意外赋予创建能力，或者筛选弹层与表格字段编辑器行为再次分叉。

### 3. 默认候选高亮必须可见

仅在逻辑层定义“默认候选”还不够，视图层必须明确渲染高亮状态。

高亮要求：

- 与 hover / selected 状态区分开
- 当搜索词为空时，不强制高亮
- 当有搜索词且存在默认候选时，高亮唯一一项
- 多选场景下，若存在未选命中项，优先高亮第一项未选值

否则用户无法预判按回车后的结果，体验仍然不稳定。

## 风险与边界

### 1. “最符合”暂不等于模糊排序

用户的表述是“最符合该字符的标签选项”，但当前仓库没有稳定的模糊匹配排序机制。

为了控制范围，第一版将其具体化为：

- 当前过滤结果首项 = 当前默认候选

这能先满足“直接回车确认”的主要诉求，同时避免在本次方案中引入额外排序争议。

### 2. 筛选值创建能力需按字段来源判断

筛选弹层表面上都是离散值选择，但其候选来源并不完全相同：

- Multi-select / Select 可以来自 view config，并且本方案明确允许从筛选 UI 正式新增选项
- Relation 必须来自真实 relation 目标

因此不能因为它们共用 `MultiSelectFilterPopover` 就把 `Relation` 也一刀切成可创建；允许正式新建只适用于 `Select` / `Multi-select`。

### 3. 筛选中新建选项的语义必须与字段编辑器对齐

本方案明确规定：

- 在 `Select` / `Multi-select` 的筛选弹层中输入一个当前不存在的值
- 若用户通过回车触发创建
- 则该值必须被正式写入字段配置
- 同时加入当前筛选规则

也就是说，筛选弹层里的“创建”不是临时筛选值，不是仅对当前 view 生效的局部状态，而是与字段编辑器中的创建语义保持一致，成为该字段后续所有入口都可见的正式候选。

这项规则有两个直接后果：

- `OptionFieldEditor` 与筛选弹层不能再各自维护不同的创建语义
- 筛选弹层在创建后除了更新 rule，还必须触发字段配置更新链路

### 3. 选中后清空搜索词会改变连续输入节奏

这是用户明确要求的行为，但它会改变一些旧的连续批量勾选节奏。例如用户过去可能输入一次关键字后，连续点多个结果；统一后，每确认一次都需要重新输入搜索词。

本次仍按用户要求执行，因为目标就是让“输入 -> 回车 -> 继续输入”成为默认快路径。

### 4. 回车后的连续输入体验依赖焦点恢复

如果只清空搜索词而不恢复焦点，那么不同入口上的“连续回车/继续输入”体验仍会分叉：

- `OptionFieldEditor` 目前已有显式的 focus restore 习惯
- 筛选弹层和高级筛选弹层当前没有同等强度的焦点恢复约束

因此本方案必须把“确认后恢复输入焦点”提升为显式契约，而不是实现细节。

## 验证策略

需要至少补以下测试：

1. `OptionFieldEditor`
   - 有匹配时回车确认默认候选
   - 无匹配时回车创建新值
   - 点击已有候选后清空搜索词
   - 确认后焦点恢复到输入框
   - 搜索词清空后列表恢复全量，默认高亮消失
   - 单选和多选都覆盖

2. `RelationCellEditor`
   - 有匹配时回车确认默认候选
   - 无匹配时不创建
   - 点击候选后清空搜索词
   - 确认后焦点恢复到输入框

3. `MultiSelectFilterPopover`
   - 普通筛选里输入后默认候选可被回车确认
   - 确认后搜索词被清空
   - 确认后焦点恢复到输入框
   - 搜索词清空后列表恢复全量，默认高亮消失
   - `Select` / `Multi-select` 无匹配时回车会正式创建新选项，并写回字段配置
   - relation 类型无匹配时不创建

4. `AdvancedFilterRuleEditor`
   - 高级筛选离散值弹层的行为与普通筛选一致
   - 选中后清空搜索词
   - 确认后焦点恢复到输入框

## 结论

本需求不应被实现为某一个弹层里的局部快捷键补丁，而应被视为 `data-editor` 全站离散值选择交互契约的一次统一收敛。

正确的落地方向是：

- 抽离共享交互状态
- 明确“默认候选 / 回车确认 / 无匹配创建 / 选中后清空”的统一规则
- 让 `OptionFieldEditor`、`RelationCellEditor`、普通筛选、高级筛选四类入口共同遵守

# 离散值选择器键盘确认统一 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在 `data-editor` 中统一 `OptionFieldEditor`、`RelationCellEditor`、普通筛选离散值弹层、高级筛选离散值弹层的键盘确认模型，实现默认候选、回车确认、确认后清空搜索词、恢复输入焦点，以及筛选中新建正式选项并写回字段配置。

**Architecture:** 先把“搜索词 -> 过滤结果 -> 默认候选 -> 回车动作”下沉为纯逻辑 helper，再分别接入表格/详情选择器与筛选选择器。字段配置写回继续放在 `App.tsx` 这一层，展示组件只发出“需要正式创建选项”的动作请求，不直接保存配置。

**Tech Stack:** React 18, TypeScript/TSX, `.mjs` pure helpers, Node `--test`, Playwright, Radix Popover

## 方案概述

### 1. 总体目标和范围

本实施计划用于把 [2026-06-22-离散值选择器键盘确认统一方案.md](C:/Code/data-editor/docs/plans/2026-06-22-离散值选择器键盘确认统一方案.md) 推进到可执行代码。目标是让所有离散值选择入口都共享同一套确认契约，同时保留 `Relation` 不可创建、筛选 `Select` / `Multi-select` 可正式创建并写回字段配置的边界。

### 2. 各阶段任务概要

阶段一：先抽纯逻辑 helper。  
主要工作是把默认候选、回车动作、清空搜索词、是否允许创建这些规则做成可单测的纯函数；预期成果是交互契约先脱离 UI 固化下来；执行顺序先写 node 测试再实现。

阶段二：接入表格/详情选择器。  
主要工作是改 `OptionFieldEditor` 与 `RelationCellEditor`，补默认高亮、回车确认、确认后清空搜索词与恢复焦点；预期成果是表格和详情先变成稳定参考实现；执行顺序先补 e2e 失败测试，再最小实现。

阶段三：接入筛选与高级筛选，并补字段配置写回。  
主要工作是给 `ViewFilterBar` / `AdvancedFilterPanel` 透传“正式新建选项”动作，在 `App.tsx` 复用现有 `selectOptions` / `multiSelectOptions` 更新链路保存配置，并明确创建后的 options 回流策略，以及 `effectiveOptions -> visibleOptions -> filteredOptions` 的本地渲染链；预期成果是筛选里的创建语义与字段编辑器完全对齐；执行顺序先普通筛选，再高级筛选。

阶段四：补样式和总回归。  
主要工作是补 `default-candidate` 视觉态、确认后列表复位断言、字段配置持久化断言，以及最后的 `test` / `typecheck` / `build` / 定向 Playwright 回归；预期成果是行为、视觉和持久化都有自动化护栏。

### 3. 整体结构框架

实现结构保持三层：

1. 纯逻辑层：`src/table/discrete-value-picker.mjs`
2. 组件层：`OptionFieldEditor`、`RelationCellEditor`、`MultiSelectFilterPopover`、`AdvancedFilterRuleEditor`
3. 应用层：`App.tsx` 负责字段配置正式写回，`ViewFilterBar` / `AdvancedFilterPanel` 负责动作透传

两个强约束必须写死：

- 创建正式选项的保存职责只能在 `App.tsx`，不能散落到弹层组件里。
- `Relation` 一律不可创建，筛选里只有 `Select` / `Multi-select` 可以正式创建并写回字段配置。

- 筛选里的正式创建不能只改 rule，还必须同步落盘字段配置。

```js
export function resolveDefaultCandidate({ filteredOptions, selectedValues, mode }) {
  if (!filteredOptions.length) return null;
  if (mode !== "multi") return filteredOptions[0];
  const selected = new Set(selectedValues.map((item) => String(item)));
  return filteredOptions.find((option) => !selected.has(String(option.value))) ?? filteredOptions[0];
}

export function resolveEnterAction({ search, defaultCandidate, allowCreate }) {
  if (defaultCandidate) return { type: "select", value: String(defaultCandidate.value) };
  const trimmed = search.trim();
  if (!trimmed || !allowCreate) return { type: "noop" };
  return { type: "create", value: trimmed };
}

export function resetSearchStateAfterCommit() {
  return { search: "", highlightedValue: null };
}

export function confirmNextSelectedValues({ mode, selectedValues, value }) {
  const normalized = selectedValues.map((item) => String(item));
  if (mode === "single") return [value];
  if (normalized.includes(value)) return [...selectedValues];
  return [...selectedValues, value];
}
```

```tsx
async function handleCreateFormalFilterOption(input: { field: string; fieldType: "Select" | "Multi-select"; value: string }) {
  const nextOptions = buildFormalOptionList({
    fieldName: input.field,
    fieldType: input.fieldType,
    fieldViewConfigs,
    createdValue: input.value,
  });
  mutateOptionFieldTransaction({
    mutateViewConfigDraft: (draft) => {
      const key = fieldViewConfigKey(selectedPath, collectionPath, input.field);
      if (!key) return;
      const current = ensureFieldViewConfig(draft, key);
      draft.fields[key] = {
        ...current,
        selectOptions: input.fieldType === "Select"
          ? buildOptionConfigFromOptions(nextOptions)
          : current.selectOptions,
        multiSelectOptions: input.fieldType === "Multi-select"
          ? buildOptionConfigFromOptions(nextOptions)
          : current.multiSelectOptions,
      };
    },
  });
  return nextOptions;
}
```

这里把“创建后的 options 回流策略”和渲染链写死，避免实现时再猜：

- 第一优先：创建成功后把 `nextOptions` 放进组件本地 `localOptionsOverride`
- 第二优先：`effectiveOptions` 必须成为 `filteredOptions` 与默认候选计算的新来源，而不是只设置 override 却继续使用旧的 `visibleOptions`
- 第三优先：继续沿用现有 `mergeSelectedOptions()`，保证即使 props 还没回流，新创建值也能先在已选 chip 中可见
- 最终状态：当 `fieldViewConfigs` 通过现有链路回流后，组件可清空 `localOptionsOverride` 或继续保持两者等价

这里需要补一个明确的纯辅助函数，专门负责生成“正式创建后的完整 options 列表”：

```tsx
function buildFormalOptionList({ fieldName, fieldType, fieldViewConfigs, createdValue, baseOptionsOverride }) {
  const currentOptions = baseOptionsOverride ?? optionsForField(fieldName, fieldType, fieldViewConfigs);
  if (currentOptions.some((option) => option.value === createdValue)) return currentOptions;
  return [...currentOptions, { value: createdValue, label: createdValue, color: null }];
}
```

- [ ] **Step 6: 给高级筛选补同等粒度的输入焦点与本地 options override**

```tsx
const [search, setSearch] = useState("");
const inputRef = useRef<HTMLInputElement | null>(null);
const [localOptionsOverride, setLocalOptionsOverride] = useState<MultiSelectOptionView[] | null>(null);

const visibleOptions = useMemo(
  () => mergeSelectedOptions(optionsForField(rule.field, fieldType, fieldViewConfigs, relationFilterOptions), selectedValues),
  [fieldType, fieldViewConfigs, relationFilterOptions, rule.field, selectedValues],
);
const effectiveOptions = localOptionsOverride ?? visibleOptions;
const filteredOptions = useMemo(() => filterOptions(effectiveOptions, search), [effectiveOptions, search]);
const defaultCandidate = resolveDefaultCandidate({
  filteredOptions,
  selectedValues,
  mode: "multi",
});

function focusSearchInput() {
  queueMicrotask(() => focusWithoutScroll(inputRef.current));
}

function confirmAdvancedFilterValue(value: string) {
  const nextValues = selectedValues.includes(value) ? selectedValues : [...selectedValues, value];
  updateRule({ ...rule, operator: discreteOperator(rule.operator), value: nextValues });
  setSearch("");
  focusSearchInput();
}

async function handleAdvancedFilterEnter() {
  const allowCreate = fieldType === "Select" || fieldType === "Multi-select";
  const action = resolveEnterAction({ search, defaultCandidate, allowCreate });
  if (action.type === "select") {
    confirmAdvancedFilterValue(action.value);
    return;
  }
  if (action.type === "create" && onCreateFormalOption) {
    const nextOptions = await onCreateFormalOption({ field: rule.field, fieldType, value: action.value });
    setLocalOptionsOverride(nextOptions);
    confirmAdvancedFilterValue(action.value);
  }
}
```

这一步的强约束是：

- 高级筛选不能只透传 `onCreateFormalOption` 却忘了补 `inputRef` / `focusSearchInput`
- 高级筛选不能只在普通筛选里落 `localOptionsOverride`，自己仍然依赖旧的 `visibleOptions`
- 高级筛选的 `fieldType` 继续在组件内部解析，但其创建、确认、回流、聚焦行为必须与普通筛选等价

### 2. 关键实现边界

- 纯逻辑 helper 不负责保存配置，只负责给出候选和回车动作。
- UI 组件不直接保存字段配置，只发出 `onCreateFormalOption`。
- `App.tsx` 必须复用现有 `mutateOptionFieldTransaction` / `buildOptionConfigFromOptions` 链路保存字段配置，不能新造不存在的 `saveProjectViewConfig` 类接口。
- 表格、详情、筛选、高级筛选都必须走独立的 `confirm...` 语义，不能直接复用 `toggle...`，否则会违反“回车不反选”的契约。
- `ViewFilterBar -> MultiSelectFilterPopover` 需要显式透传 `fieldType` 与 `onCreateFormalOption`；高级筛选链只需要透传 `onCreateFormalOption`，`fieldType` 继续由 `AdvancedFilterRuleEditor` 内部解析。
- 筛选正式创建后的 options 回流策略必须先写死：本地 `localOptionsOverride` 负责即时回显，`fieldViewConfigs` 回流负责最终一致性；不能再留 `syncLocalOptions(...)` 这类未定义占位接口。
- 键盘 Enter 与鼠标点击必须分工明确：Enter 只走 `confirm...`，鼠标点击候选/已选 chip 才保留 `toggle...` 或 remove 语义。
- 普通筛选与高级筛选都必须把 `effectiveOptions` 作为 `filteredOptions`、默认候选和高亮计算的实际来源；不能只设置 override 却仍然消费旧 options。

### 3. 最终验收口径

完成后，用户在四类入口都应得到一致体验：

- 输入字符后能看到明确默认候选
- 直接按回车即可确认默认候选
- `Select` / `Multi-select` 在筛选中无匹配时可正式创建新选项
- `Relation` 无匹配时不会创建
- 每次确认后搜索词清空、焦点回到输入框、列表恢复全量、空搜索态不保留默认高亮

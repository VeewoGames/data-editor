# Tabler 图标接入方案

## 方案概述

### 1. 总体目标和范围

本方案用于在 `data-editor` 的共享视图图标体系中，正式接入 `Tabler S / Tabler L` 两套来源分组，并保持与现有 `Phase A` 图标架构一致，即：

- 继续沿用 `metadata registry / runtime asset registry` 分层
- 继续沿用图标包按需加载模型
- 继续沿用 picker 顶部分组导航
- 不回退到“手工列举零散 Tabler 图标”的旧方式

本次目标如下：

- 将 Tabler 图标正式分成：
  - `Tabler S`
  - `Tabler L`
- 为 Tabler 建立与 Streamline 对齐的本地静态资产来源
- 生成 `tabler-shared-view-icons` 运行时 registry 与类型定义
- 将 Tabler 接入现有 `shared view icon` 的 pack/runtime contract
- 保证 picker 中可浏览、可搜索、可按包加载
- 保持当前“默认仅当前标签搜索，勾选后才做全局搜索”的现有 contract 不变
- 为正式来源包补齐最小可用的卸载保护边界，避免已被使用的图标包被直接卸载
- 为后续“卸载保护”和“性能对比”保留稳定扩展点

本次范围包括：

- `vendor/tabler-svg` 资产目录设计
- Tabler generated registry 生成脚本与产物
- `icons.ts` 中 pack 与分组接入
- `ViewTabs.tsx` 中分组可见性与交互回归
- `ViewTabs.tsx` 中图标包最小卸载保护边界
- node tests 与 e2e 验证补齐

本次不包含：

- 更细的 Tabler 标签语义分类
- 更细的图标包卸载保护诊断文案与完整策略扩展
- 性能基线与对照报告的最终收口
- 替换全部 legacy/builtin 图标为新的 Tabler id
- 自动批量迁移现有 shared view config 中的 legacy Tabler icon id

### 2. 各阶段任务概要

#### 阶段一：资产来源收敛

主要工作：

- 固定 Tabler 官方来源与版本钉住方式
- 建立 `vendor/tabler-svg/filled/**`
- 建立 `vendor/tabler-svg/outline/**`
- 明确 `filled -> Tabler S`、`outline -> Tabler L`

预期成果：

- Tabler 图标来源从“零散运行时代码引用”收敛为稳定静态资产
- 后续任何人都可以按同一来源与版本重复生成资产

执行顺序：

1. 固定官方来源与版本钉住方式
2. 固定目录结构
3. 固定命名规范
4. 固定 family 到 pack 的映射

#### 阶段二：生成链路接入

主要工作：

- 新增 Tabler registry 生成脚本或扩展现有生成脚本
- 输出：
  - `src/generated/tabler-shared-view-icons.mjs`
  - `src/generated/tabler-shared-view-icons.d.ts`

预期成果：

- Tabler 图标集合具备与 Streamline 对齐的 generated 真相源

执行顺序：

1. 先定义生成产物结构
2. 再读取 vendor 目录生成元数据
3. 最后补类型与测试

#### 阶段三：运行时 pack 接入

主要工作：

- 扩展 `SharedViewIconPackId`
- 将 Tabler registry 合并进现有 metadata registry
- 增加：
  - `tabler-filled`
  - `tabler-outline`

预期成果：

- 现有 pack/runtime/fallback 模型可以无差别消费 Tabler

执行顺序：

1. 扩展 pack id
2. 合并 metadata
3. 接入 runtime lazy load

#### 阶段四：picker 导航与搜索接入

主要工作：

- 在 picker 顶部加入：
  - `Tabler S`
  - `Tabler L`
- 保证搜索、当前 tab 搜索、全局搜索都能覆盖 Tabler

预期成果：

- 用户可以在正式来源分组中浏览与选择 Tabler 图标

执行顺序：

1. 分组元数据接入
2. 搜索文本接入
3. e2e 回归

#### 阶段五：验证与回归

主要工作：

- 补 generated registry 测试
- 补 `icons.ts` pack contract 测试
- 补 picker e2e

预期成果：

- Tabler 接入不会破坏已完成的 Phase A 能力

执行顺序：

1. 数据层测试
2. 运行时 contract 测试
3. UI 回归测试

### 3. 整体结构框架

本方案的整体结构分为四层：

1. 资产层
   - 官方来源：Tabler Icons 官方 GitHub tagged snapshot
   - `vendor/tabler-svg/filled/**`
   - `vendor/tabler-svg/outline/**`

2. 生成层
   - `src/generated/tabler-shared-view-icons.mjs`
   - `src/generated/tabler-shared-view-icons.d.ts`

3. 运行时 registry 层
   - 与现有 `streamline` 元数据合并
   - 进入 `sharedViewIconMetadataRegistry`
   - 运行时按包懒加载 SVG

4. picker 交互层
   - `Tabler S / Tabler L` 成为正式顶部来源分组
   - 与 `最近 / 收藏 / Legacy` 共存

## 一、设计原则

### 1. 统一采用“静态资产 + generated registry”链路

Tabler 本次不采用“直接反射 `@tabler/icons-react` 导出组件集合”的方式，而采用：

- 静态 SVG 资产落盘
- generated registry 生成
- 运行时统一懒加载

原因：

1. 与 Streamline 现有链路一致  
   后续维护、测试、性能治理都可以复用现有模式。

2. 来源边界清晰  
   `filled` 与 `outline` 的全集定义稳定，不依赖运行时库内部导出细节。

3. 更适合 pack 管理  
   后续做“按包卸载”“统计包体积”“来源审计”都更直接。

### 2. 不把当前手工 Tabler 运行时图标当作正式全集

当前 [icons.ts](C:/Code/data-editor/src/components/icons.ts) 中大量 `IconXxxFilled` / `IconXxx` 只是：

- 当前运行时组件桥接层
- 一部分 legacy/builtin icon 的实现来源

它们不应继续承担：

- `Tabler S / Tabler L` 全集定义
- picker 正式来源分组定义
- 搜索元数据真相源

### 3. 不破坏 Phase A 已完成合同

Tabler 接入必须复用现有合同：

- pack 状态只存内存
- `Base` 仍默认加载且不可卸载
- 未加载图标继续通过 fallback 路径兜底
- 收藏仍只记录 `SharedViewIconId`，不关心具体来源实现

### 3.1 继承当前搜索 contract，不改变默认交互

当前实现已经固定以下行为：

- 默认只在当前标签页内搜索
- 只有勾选“全局”后，才跨全部图标元数据搜索
- “全局”勾选状态不持久化，重新打开 picker 或刷新页面后回到默认未勾选

Tabler 接入不得改变这组语义。也就是说：

- `Tabler S / Tabler L` 必须同时支持当前标签搜索与全局搜索
- 但**不能**借接入 Tabler 的机会把搜索默认切成全局

### 4. 资产来源必须可重建、可审计、可版本钉住

Tabler 这次必须固定单一官方来源，避免实现时再次做“npm 包还是仓库快照”的决策。

本方案采用：

- **Tabler 官方 GitHub 仓库的 tagged release snapshot**

不采用：

- 运行时临时抓取官网页面
- 仅依赖 `@tabler/icons-react`
- 以 npm 安装后的 package 目录作为正式 vendoring 真相源

原因：

1. 本轮目标是 **全量 vendored 静态 SVG 资产**
   GitHub tagged snapshot 与这个目标天然一致，适合直接落盘与 review。

2. tagged snapshot 更适合 **版本钉住**
   文档、脚本、提交都可以明确记录来源 tag，例如 `tabler-icons@vX.Y.Z`。

3. 比 npm 包目录更稳定
   npm package 适合运行时消费，但未必适合作为“完整静态资产 vendor 真相源”；发布产物裁剪、目录调整、附带文件变化都会增加不必要不确定性。

4. 更利于代码评审和审计
   “这批 SVG 来自哪个官方 tag” 比“来自某次 npm install 后的 node_modules 目录”更容易回查。

## 二、目标目录与产物

### 0. 官方来源与版本策略

本轮固定来源为：

- Tabler Icons 官方 GitHub 仓库
- 使用 **release tag / git tag 对应快照** 作为资产导入源

建议在实施产物中同时记录：

- 来源仓库 URL
- 来源 tag
- 来源上游目录映射
- 导入日期
- 导入脚本命令

建议新增一份导入记录，例如：

- `artifacts/tabler-import/source.json`

至少包含：

- `repository`
- `tag`
- `upstreamFilledPath`
- `upstreamOutlinePath`
- `importedAt`
- `filledCount`
- `outlineCount`

要求：

- 导入脚本必须显式校验 `upstreamFilledPath` 与 `upstreamOutlinePath` 是否存在
- 上游目录不存在、数量为 0、或 family 缺失时必须 **fail fast**

这样后续升级 Tabler 版本时，可以直接对比来源、目录映射与数量变化。

### 1. 资产目录

新增：

- `vendor/tabler-svg/filled/**`
- `vendor/tabler-svg/outline/**`

约束：

- 文件名统一使用 kebab-case
- 不引入额外层级语义目录
- 资产目录只保存可直接消费的 SVG 文件
- 同一来源版本的全量同步必须可重复执行

补充约束：

- `filled` 与 `outline` 各自保留原始 basename
- 不在 vendor 阶段强行把 `filled` / `outline` 文件改成带前缀的新文件名
- family 差异在生成层体现，而不是通过修改原始文件名来体现

### 2. generated 产物

新增：

- `src/generated/tabler-shared-view-icons.mjs`
- `src/generated/tabler-shared-view-icons.d.ts`

建议输出结构：

- `tablerSharedViewIcons`
- `tablerSharedViewIconIds`
- `tablerSharedViewIconSearchTextById`
- `tablerSharedViewIconGroups`
- `TablerSharedViewIconId`
- `TablerSharedViewIconMeta`

### 3. pack id 与 UI label

运行时 pack id：

- `tabler-filled`
- `tabler-outline`

UI label：

- `tabler-filled -> Tabler S`
- `tabler-outline -> Tabler L`

### 4. 仓库体积与产物规模边界

本方案接受以下代价：

- `vendor/tabler-svg/**` 会显著增加 SVG 文件数量
- `src/generated/tabler-shared-view-icons.*` 会增加 generated 代码体积

这是有意接受的 tradeoff，因为：

- 当前优先级是来源稳定、真相源统一、后续包治理容易
- 本轮性能控制依赖“默认仅加载 Base”，而不是依赖“仓库里没有资产”

因此本轮不把“vendor 文件多”当作阻断条件，但要求：

- 默认运行时不 eager 加载全部 Tabler SVG
- 构建与运行时验证要继续覆盖 bundle 变化和加载路径

## 三、ID 与命名规则

### 1. 图标 id 规则

建议生成规则：

- `tablerFilled<PascalName>`
- `tablerLine<PascalName>`

示例：

- `arrow-left` -> `tablerLineArrowLeft`
- `heart`（位于 filled family）-> `tablerFilledHeart`

约束：

- 与现有 builtin id 不冲突
- 与现有 Streamline id 不冲突
- 可以从 id 反推来源 family

### 1.1 filename normalization 与 collision 规则

实现必须固定一套 **确定性的 filename -> id** 规则，并在生成阶段显式检查冲突。

推荐规则：

1. 读取 vendor 中的相对路径与文件 basename
2. basename 按 kebab-case 拆词，再转 PascalCase
3. `filled` family 前缀固定为 `tablerFilled`
4. `outline` family 前缀固定为 `tablerLine`

示例：

- `filled/arrow-left.svg` -> `tablerFilledArrowLeft`
- `outline/arrow-left.svg` -> `tablerLineArrowLeft`

冲突处理规则：

- 如果生成出的 id 与现有 builtin id 或 Streamline id 冲突，生成脚本必须 **fail fast**
- 如果同一 family 内出现重复 basename，生成脚本必须 **fail fast**
- 如果 `filled` / `outline` 由于异常文件名导致生成出重复 id，生成脚本必须 **fail fast**

同时要求输出一份 collision 摘要，建议落到：

- `artifacts/tabler-import/collision-report.json`

本轮不接受“静默跳过冲突文件”。

### 2. searchText 规则

每个生成 icon 至少输出：

- 原始文件名
- 拆词后的空格形式
- family 名
- `tabler`
- `filled` / `outline`
- `tabler s` / `tabler l`

这样可以支持：

- 顶部分组内搜索
- 全局搜索
- 通过来源关键字搜索

### 3. 现有 legacy Tabler icon 的边界

本轮不做自动迁移，但必须明确边界：

1. 当前 `sharedViewLegacyIconRegistry` 中由 Tabler React 组件实现的旧 icon，继续保留在：
   - `Base`
   - `Legacy`

2. 新接入的 `Tabler S / Tabler L` 是正式来源分组，但 **不反向吞并** 旧 id。

3. 本轮不自动把：
   - 最近
   - 收藏
   - 已保存 shared view icon 配置
   中的 legacy Tabler id 改写为新 Tabler id。

4. 本轮之后：
   - 不再向正式来源分组中新增 legacy Tabler id
   - 新的 Tabler 全集只通过 generated registry 进入系统

这样可以避免当前配置失效，同时防止 legacy 继续扩张。

## 四、运行时接入设计

### 1. metadata registry 扩展

在 [icons.ts](C:/Code/data-editor/src/components/icons.ts) 中：

- 导入 Tabler generated registry
- 将其合并进 `sharedViewIconMetadataRegistry`
- 为每个 Tabler icon 标记：
  - `packId`
  - `outputPath`
  - `searchText`

### 2. runtime asset registry 扩展

继续沿用当前模式：

- `import.meta.glob(...)`
- 按 pack id 懒加载 SVG
- 通过 `loadSharedViewIconPack(packId)` 注册真实组件

Tabler 只需要新增：

- `vendor/tabler-svg/**/*.svg` 的 glob
- `tabler-filled`
- `tabler-outline`

实现建议：

- 不再为每个来源新增一套平行的 `registerXxxPack` 分支
- 将当前 Streamline pack 注册逻辑抽成通用 helper：
  - 输入：`packId -> iconIds -> outputPath`
  - 输出：将 SVG 模块注册到 `sharedViewIconRegistry`

原因：

- 当前 [icons.ts](C:/Code/data-editor/src/components/icons.ts) 已经开始承担 pack 运行态核心逻辑
- 如果继续为 Tabler 复制一套注册分支，后续再加来源时会继续膨胀
- 抽成通用 helper 后，Tabler 只是“新增 family 数据”，不是“新增第二套机制”

### 3. pack 分组顺序

本轮接入后，picker 顶部正式顺序应为：

1. `最近`
2. `收藏`
3. `Micro S`
4. `Core S`
5. `Tabler S`
6. `Micro L`
7. `Tabler L`
8. `Legacy`

说明：

- `Legacy` 继续作为过渡池
- 当前内置零散 Tabler icon 暂不强制迁移出 `Legacy`
- 等后续迁移策略稳定后，再决定是否将一部分旧 id 映射到正式 Tabler id

### 4. Base 与 Tabler 的关系

本轮必须固定以下边界：

1. `Base` 只表示“默认始终可用的一小组基础 icon”
2. `Tabler S / Tabler L` 表示“正式来源全集”
3. 即使某个基础 icon 的渲染实现来自 Tabler React，也不意味着它自动属于 `Tabler S / Tabler L`

结论：

- `Base` 是可用性分组，不是来源全集
- `Tabler S / Tabler L` 是来源全集，不是默认可用集合

这样可以避免：

- 同一个 icon 既被当成基础默认项，又被误解为已加载整包
- 搜索、卸载保护、来源展示之间边界混乱

### 5. 卸载保护最小边界

当前代码里非 `base` pack 原则上都可以直接卸载，因此 Tabler 接入不能只补 UI 分组，不补最小保护。

本轮至少要满足：

1. `Base` 仍然不可卸载
2. 如果某个 shared view 当前实际使用了 `tabler-filled` 或 `tabler-outline` 中的图标，则对应 pack 不允许直接卸载
3. 被保护的 pack 在 options 浮框中必须呈现不可卸载状态，而不是点击后 silently fallback

本轮**不要求**一次性做完所有通用卸载策略，但必须保证：

- 当前已被 shared views 使用的正式来源包，不会被直接卸载
- Tabler 接入后不会把用户已选图标打回 fallback icon

## 五、测试与验收

### 1. 数据层测试

新增或扩展：

- generated registry 测试
- 确认 `tabler-filled` / `tabler-outline` 被正确生成
- 确认至少存在代表性 id

### 2. contract 测试

扩展现有 `tests/view-state.test.mjs`：

- `SharedViewIconPackId` 含 `tabler-filled` / `tabler-outline`
- picker 分组含 `Tabler S / Tabler L`
- metadata registry 含 Tabler generated registry

### 3. e2e 测试

扩展现有图标 picker e2e：

- `Tabler S` 可见
- `Tabler L` 可见
- 切换分组后有代表性图标
- 默认未加载时出现 unloaded hint
- 通过 options 加载后从 unloaded 变为 loaded
- 页面刷新后 pack 状态回到仅 `Base`
- “全局”开关默认不勾选
- 勾选“全局”前，当前标签外的 Tabler 图标不可命中
- 勾选“全局”后，可跨组命中 Tabler 图标
- 重新打开 picker 或刷新页面后，“全局”开关回到未勾选
- 已被当前 shared view 使用的 Tabler pack 不可直接卸载
- 全局搜索能搜到 Tabler 图标

### 4. 验收标准

本轮完成标准：

- Tabler 两套正式来源进入仓库
- picker 顶部分组中可见 `Tabler S / Tabler L`
- Tabler 图标可搜索、可加载、可选择
- 默认不加载整包，需经 options 显式加载
- 刷新后非 Base pack 状态不持久化
- 默认搜索范围仍为当前标签，只有显式勾选后才做全局搜索
- 已被当前 shared view 使用的 Tabler pack 不可直接卸载
- 现有 Phase A 测试不回归
- 新增 Tabler 相关测试通过

## 六、推荐实施顺序

推荐按以下顺序落地：

1. 固定官方来源 tag，并生成 `artifacts/tabler-import/source.json`

7. 扩展 picker 分组、搜索与卸载保护测试
8. 跑 focused tests、typecheck、build、图标 picker e2e

## 七、决策结论

本方案采用：

- **全量 vendored 进仓库**
- **静态 SVG 资产 + generated registry**
- **Tabler 官方 GitHub tagged snapshot 作为唯一正式资产来源**
- **接入现有 Phase A pack/runtime contract**

不采用：

- 直接反射 `@tabler/icons-react` 导出集合作为正式真相源
- 以 npm 安装后的 package 目录作为正式 vendoring 真相源

原因：

- 前者更稳定、更可测试、更适合后续性能治理和来源管理
- GitHub tagged snapshot 更容易版本钉住、代码评审和重复生成
- `@tabler/icons-react` 与 npm package 更适合运行时组件消费，不适合作为本轮“全量静态资产 vendor 真相源”

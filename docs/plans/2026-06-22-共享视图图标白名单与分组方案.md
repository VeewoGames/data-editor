# 共享视图图标白名单与分组方案

## 方案概述

### 1. 总体目标和范围

本方案用于收敛共享视图 icon picker 第一版的最终图标池与分组结构，满足以下目标：

- 保留当前默认图标 `borderAll`，不替换默认入口
- 除默认项外，图标候选统一使用 Tabler `Filled` 风格
- icon picker 顶部采用标签页分组，而不是单个平铺大列表
- 分组同时覆盖通用任务场景与游戏项目常见语义

本次范围包括：

- 共享视图 icon picker 的图标白名单
- 图标分组标签设计
- 每组图标候选列表
- 可直接落地到 `ViewTabs` 的数据结构草案

本次不包含：

- emoji、自定义图片、远程图标
- 动态从全部 Tabler 图标自动扫描生成分组
- 组级 icon 持久化 schema
- icon 颜色系统

### 2. 各阶段任务概要

#### 阶段一：白名单收敛

主要工作：

- 固定保留 `borderAll` 作为默认图标
- 从当前安装的 `@tabler/icons-react` 中只挑选 `Filled` 风格图标
- 排除品牌类、噪音类和过于奇怪的图标

预期成果：

- 第一版有一套稳定、可控、足够丰富的图标池

执行顺序：

- 先保留默认项
- 再按语义筛选 filled 图标

#### 阶段二：分组设计

主要工作：

- 设计 icon picker 标签页
- 兼顾通用工作流与游戏项目语义
- 保证每组图标规模可控

预期成果：

- 用户找图标时按语义路径浏览，而不是在大平铺里盲找

执行顺序：

- 先确定一级分类
- 再把图标落到每个分类

#### 阶段三：代码结构草案

主要工作：

- 输出 `sharedViewIconRegistry`
- 输出 `sharedViewIconGroups`
- 输出搜索与最近使用建议

预期成果：

- 后续可以直接按文档抄成实现

执行顺序：

- 先 registry
- 再 groups
- 最后补最近与搜索规则

### 3. 整体结构框架

本方案把 icon picker 的图标系统拆成四层：

1. 默认项：固定保留 `borderAll`
2. 白名单：所有允许选择的 filled 图标
3. 分组：标签页语义分区
4. 交互辅助：`最近` 与搜索

核心边界如下：

- `borderAll` 是系统默认项，不和 filled 风格池做一刀切替换
- 正式候选图标统一使用 `Filled`
- 图标按语义分组，不按几何造型分组
- 游戏项目语义优先进入一级分类，而不是作为杂项附庸

## 设计原则

### 1. 默认项稳定优先

当前共享视图已经默认回退到 `borderAll`。这条链路已经稳定，不应为了追求“全 filled”而把默认值换掉。

所以本方案明确：

- `borderAll` 继续保留
- 在 icon picker 中以“默认”身份单独展示
- 其余正式候选继续全部使用 filled 风格

### 2. 图标要足够多，但不能失控

如果只给 6 到 10 个图标，用户很快就会觉得贫瘠；但如果把几十上百个图标一次性无分组平铺，反而会降低可用性。

因此第一版策略是：

- 提供足够丰富的候选
- 用标签页分组降低认知负担
- 搜索只在白名单内进行

### 3. 语义分组优先于造型分组

用户不会按“圆的、方的、带箭头的”去找图标，而是按用途找：

- 我想给一个战斗系统找图标
- 我想给一个奖励表找图标
- 我想给一个配置或数据视图找图标

因此分组必须按语义组织。

### 4. 游戏项目语义应是一等公民

当前项目以游戏相关内容为主，icon picker 如果只按通用办公语义分组，会导致：

- 常见游戏系统找不到对应图标路径
- 用户被迫在“杂项”里找火焰、奖杯、手柄、幽灵、王冠

所以本方案明确把游戏语义提升为一级分类。

## 顶层标签页设计

建议第一版采用以下标签页：

1. `最近`
2. `常用`
3. `工作流`
4. `数据/配置`
5. `玩法/系统`
6. `战斗/危险`
7. `世界/探索`
8. `奖励/收集`

说明：

- `最近` 不是静态白名单分组，而是运行时记录
- `常用` 负责覆盖最普适的通用图标
- 后 6 组覆盖业务语义

## 搜索与最近使用建议

### 搜索

第一版建议：

- 搜索框常驻在 icon picker 顶部
- 搜索范围只在白名单内
- 搜索匹配：
  - 分组内 label
  - icon id
  - 预定义关键词别名

可加的关键词别名示例：

- `战斗` → `shield` `bolt` `flame` `bow`
- `奖励` → `trophy` `diamond` `coin`
- `系统` → `gamepad` `puzzle` `sparkles`
- `数据` → `table` `database` `fileAnalytics`

### 最近

第一版建议：

- 最近使用记录保存在本地
- 每次选择 icon 后，把该 icon 推到最近列表顶部
- 最多保留 8 到 12 个
- `borderAll` 也允许出现在最近中

## 不建议纳入第一版的图标

以下类型不建议纳入：

- 品牌图标
- 货币细分图标，如 `CoinBitcoinFilled`
- 过强现实语义图标，但对游戏项目帮助不大的办公图标
- 太细碎、相互区分度很低的一串近似图标

## 结论

第一版建议采用：

- `borderAll` 继续保留为默认图标
- 额外补充约 56 个 filled 图标
- 用 8 个标签页做语义分组
- 同时覆盖通用工作流与游戏项目常见语义

这样既不会破坏现有默认链路，也能把 icon picker 做成一个更完整、更像产品功能而不是临时补丁的方案。

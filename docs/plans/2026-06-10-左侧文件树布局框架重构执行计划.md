# 左侧文件树布局框架重构执行计划

> **For agentic workers:** 推荐按任务顺序逐项执行，并在每个任务结束后先跑对应最小验证，再进入下一个任务。

**目标：** 将左侧文件树重构为统一 tree row 布局框架，消除当前 `source/folder/file` 行结构不一致导致的视觉错位，同时保持既有排序、展开、拖拽和个人偏好持久化规则不变。

**架构：** 第一阶段先统一 DOM 骨架和样式契约，在不改变树数据和拖拽业务规则的前提下消除当前框架缺陷；第二阶段再拆分树组件、引入 view-model 和独立类系，完成长期维护收敛。

**技术栈：** React、TypeScript、现有 `sidebar-tree.mjs` 树数据模型、`useSidebarTreeDragReorder` 拖拽 hook、Playwright e2e、Node test。

---

- `src/styles.css`

## 二、执行策略

执行分两大阶段，不能混着做。

### 阶段 A：先止血，统一骨架

目的：

- 不先拆太多文件
- 先把 `source/folder/file` 的 DOM 结构统一
- 先把视觉错位根因消掉
- 保住现有拖拽和个人偏好逻辑

成功标准：

- 三类节点共享同一套 DOM 槽位结构
- `source` 隐藏逻辑不变
- 拖拽规则不变
- 文本列对齐恢复
- 第一阶段结束时仍保留 `.sidebar-file-item` 兼容类与 `data-file-path` 契约
- 第一阶段不引入新的展开状态持久化逻辑，只保证不退化当前行为

### 阶段 B：再收敛，拆组件与样式体系

目的：

- 降低 `Sidebar.tsx` 复杂度
- 建立稳定的 tree row 类名体系和 view-model 层
- 为后续图标、状态位、菜单、计数扩展准备边界

成功标准：

- `Sidebar.tsx` 不再承载大段树行 JSX
- tree row 样式独立于通用 `.sidebar-item` 补丁式覆盖
- 关键布局与交互测试齐备


## 三、具体任务

### 任务 1：补稳定测试选择器并冻结行为基线

**目标：** 先补布局断言所需的稳定选择器，再锁定不能被破坏的现有行为。

**文件：**

- 修改：`src/components/Sidebar.tsx`
- 修改：`tests/data-editor.spec.ts`

**工作项：**

1. 在当前 tree row 里补稳定测试选择器
   - 为 label 槽位补 `data-sidebar-slot="label"` 或等价稳定类名
   - 为 toggle / leading 等关键槽位预留稳定选择器，避免测试绑到第几个 `span`

2. 补一个“文本列对齐”测试辅助函数
   用 `locator.boundingBox()` 或 `evaluate(getBoundingClientRect().x)` 获取：
   - 根层文件 label x
   - folder 子文件 label x
   - `source` 隐藏场景下的根层文件 label x

3. 新增 3 组布局契约断言
   - 根层文件与同深度 folder 子文件 label 起点一致
   - 折叠/展开前后，同一文件 label 起点一致
   - 单 source 隐藏场景下 label 起点仍一致

4. 保留现有 3 条行为测试继续作为安全网
   - `sidebar renders folder tree`
   - `sidebar drag reorder only works among siblings`
   - `single data source hides the source row`

**验证：**

- `npm run test:e2e -- --grep "sidebar renders folder tree|sidebar drag reorder only works among siblings|single data source hides the source row|label alignment"`


### 任务 2：在 `Sidebar.tsx` 内统一 tree row DOM 骨架并同步重写样式契约

**目标：** 先不拆文件，直接把三类节点统一成同一行结构，并同步修正样式，避免中间态 UI 损坏。

**文件：**

- 修改：`src/components/Sidebar.tsx`
- 修改：`src/styles.css`

**工作项：**

1. 把当前 `file` / 非 `file` 两套 JSX 收敛成统一骨架
   固定槽位：
   - `indent`
   - `toggle`
   - `leading`
   - `label`
   - `trailing`

2. 保持这些契约不变
   - 文件节点继续保留 `data-file-path`
   - 文件节点继续保留可测量的 button ref
   - `data-sidebar-node-id` / `data-sidebar-node-kind` 继续保留

3. `source/folder` 图标位改为空槽，不删除槽位

4. `file` 节点的 `toggle` 也改为空槽，不再靠“少渲染一个元素”表达

5. `source` 隐藏逻辑继续沿用当前 render 层裁剪
   - 保留 `rootNodes = orderedTree.length === 1 && orderedTree[0]?.kind === "source" ? orderedTree[0].children ?? [] : orderedTree`

6. 同步重写 tree row 样式契约
   - 停止让 `.sidebar-tree-item` 与 `.sidebar-file-item` 用不同列数表达结构差异
   - 改为统一列模板
   - `indent/toggle/leading/label/trailing` 全部固定存在
   - 空内容只清空视觉，不移除槽位

7. 外层整行 hit area 保持一致
   - hover / selected / focus 背景仍覆盖整行
   - control/content 或槽位重排不能缩小点击热区

8. 第一阶段继续保留兼容契约
   - `.sidebar-file-item` 继续存在
   - `data-file-path` 继续存在
   - 现有 e2e helper 不需要在这一任务同步重写

**验证：**

- `npm run typecheck`
- 跑任务 1 的 layout e2e

## 四、每阶段完成判定

### 阶段 A 完成条件

满足以下全部条件才允许进入阶段 B：

1. 统一 tree row DOM 骨架已落地
2. 当前视觉错位消失
3. `source` 隐藏逻辑不变
4. 同父级拖拽仍正常
5. 跨父级拖拽仍无效
6. 文本列对齐测试已通过

### 阶段 B 完成条件

3. tree 样式已形成独立类系
4. 非树区域未被误伤
5. 所有验证项通过


## 五、风险控制

### 1. 不要一次性改三件事

禁止在同一提交里同时做：

- DOM 骨架重写
- 拖拽命中机制重写
- 组件拆文件

这三件事必须拆开，否则很难定位回归来源。

### 2. `source` 隐藏只能留在 render 层

不要改 `sidebar-tree.mjs` 的真实树结构去“消灭 source”，否则会影响节点 id、偏好持久化和排序边界。

### 3. 第一阶段不要动个人偏好模型

本次框架重构的重点是布局系统，不是改排序/展开偏好的存储格式。
除非发现现有模型无法承载，否则不要扩散到 `view-profile` / `view-state-storage`。

### 4. 本轮不新增展开状态持久化

虽然总体需求里提到个人视图偏好，但当前执行计划的重构目标是布局框架。
本轮只要求：

- 排序相关既有持久化不回退
- 当前展开体验不回退

本轮不要求新增 `expandedNodeIds` 的持久化存储；如要做，应另开任务并单独验证。


## 六、推荐提交边界

建议最少拆成 5 个提交，并保持以下顺序：

1. `test: 补左侧文件树稳定选择器与布局契约回归`
2. `refactor: 统一左侧文件树行骨架并同步收敛样式契约`
3. `fix: 适配左侧文件树拖拽到统一骨架`
4. `refactor: 引入左侧文件树 view-model 并拆分树组件`
5. `refactor: 收敛左侧文件树独立样式体系`


## 七、最终交付标准

最终交付必须满足：

- 当前视觉错位问题消失
- tree row 结构从补丁式实现升级为统一框架
- 既有业务规则和个人偏好持久化不回退
- 后续再改 `source/folder` 图标、文件图标、状态位、菜单时，不会再出现同类错位

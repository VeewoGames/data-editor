# 共享视图图标 Phase A 实施计划

## 方案概述

### 1. 总体目标和范围

本计划用于落地 [共享视图图标收藏与来源分组方案](C:/Code/data-editor/docs/plans/2026-06-24-共享视图图标收藏与来源分组方案.md) 的 Phase A，目标是先把 shared view icon picker 的核心信息架构、收藏链路和按包加载机制跑通，同时把当前“全量 eager 导入全部 Streamline SVG”的性能风险从结构上降下来。

Phase A 本轮只交付以下能力：

- 顶部导航切换为：
  - `最近`
  - `收藏`
  - `Micro S`
  - `Core S`
  - `Micro L`
  - `Legacy`
- 收藏状态进入 `UserViewProfile`
- 未选择 profile 时，收藏入口禁用
- 建立 `Base` 基础保留包
- 建立图标包元数据模型与运行时加载状态
- picker 右上角 options 浮框支持按包加载 / 卸载
- 默认不全量加载所有图标 SVG
- 未加载图标使用统一占位卡片
- 顶部 tab / 菜单 trigger / drag ghost 在图标未加载时统一使用 fallback 占位
- legacy / builtin icon 在过渡期内仍可浏览、可再次选择

本计划包含：

- `profile schema`
- `icons registry`
- `ViewTabs` picker 交互
- `App.tsx` profile 写链与图标包状态入口
- `tests` 与性能基线采集口径

本计划不包含：

- `Tabler S / Tabler L` full set 正式接入
- `Legacy` 过渡组消化完成
- 自动生成 `Base` 名单
- 图标颜色、标签、排序等扩展交互

### 3. 整体结构框架

Phase A 的整体结构分五层：

1. `Profile` 层
   - 收藏状态归属 `UserViewProfile`
   - 无 profile 时禁用收藏

2. `Metadata Registry` 层
   - 保存图标 id、group、packId、searchText、outputPath
   - 支撑分组、搜索、收藏、最近

3. `Runtime Asset Registry` 层
   - 只保存当前已加载图标包的真实渲染组件
   - 默认仅 `Base`

4. `Pack Runtime State` 层
   - 管理已加载包、受保护包、可卸载包

5. `Picker UI` 层
   - tabs、搜索、收藏、占位卡片、options 浮框

核心边界：

- 搜索不等于可渲染
- 收藏不等于 shared view config
- Phase A 包状态只存内存，不落 profile、不落 localStorage、刷新后回到默认仅 `Base`
- `Base` 永远可用，且不可卸载

## 当前约束

- 当前 `src/components/icons.ts` 仍使用 `import.meta.glob(..., eager: true)` 全量导入 Streamline SVG，必须改造。
- 当前 `selectedViewProfileName` 为空时，`mutateSelectedViewProfile()` 直接返回 `false`，因此不能假设无 profile 也能写收藏。
- 当前 `ViewTabs.tsx` 的 `sharedViewIconRegistry[candidate]` 假设所有图标都可直接渲染，Phase A 必须打破这个假设。

## 固定 Contract

### 1. 包状态持久化策略

Phase A 中图标包加载状态采用：

- 纯内存会话态

明确规则：

- 不写 `UserViewProfile`
- 不写 localStorage
- 不写 shared view config
- 页面刷新后恢复到默认仅 `Base` 已加载

原因：

- Phase A 的目标是先验证按包加载的结构与收益
- 过早把包状态持久化，会把“个人偏好”与“性能实验参数”混在一起
- 纯内存态最容易验证默认路径成本是否真的下降

### 2. 未加载 icon 的统一 fallback 规则

只要某个 icon 的所属包当前未加载，以下所有入口都必须走同一套 fallback：

- 顶部 view tab icon
- view 菜单 header icon trigger
- icon picker 卡片
- recent 列表卡片
- drag ghost icon

统一规则：

- 不直接访问缺失的 runtime asset component
- 显示统一 fallback 占位
- 占位尺寸与真实图标容器一致
- 在 picker 卡片场景中额外显示“需先加载”提示与加载动作

这样可以避免同一个未加载 icon 在不同入口表现不一致。

### 3. `Base / Legacy` 静态映射真相源

Phase A 必须提供一个静态真相源，明确：

- 哪些 icon 属于 `Base`
- 哪些现有 builtin / legacy icon 属于 `Legacy`

推荐产物：

- `src/components/icons.ts` 中的显式常量映射

要求：

- `Base` allowlist 为手写常量
- `legacy icon id -> packId` 为手写常量
- 同一个 icon 不能同时属于多个 Phase A pack
- 默认 shared view 常用 icon 不能遗漏

测试要求：

- 映射表需有源码级测试锁定
- 至少覆盖不重复、包含默认项、包含关键高频项三类断言

### 4. 性能记录落盘位置

Phase A 的性能基线和复测结果不能只停留在终端输出，必须有固定落盘位置。

推荐：

- `artifacts/icon-pack-performance/phase-a-baseline.json`
- `artifacts/icon-pack-performance/phase-a-after.json`

至少记录：

- build 时长
- 主 bundle 体积
- dev health ready 时长
- picker 首开时长
- 单包加载时长
- 是否复现 `EMFILE`

## 风险与应对

风险一：只增加包管理 UI，但底层仍然全量 eager 导入  
应对：以 registry 分离为先决条件；未完成分离前不宣称 Phase A 完成。

风险二：`Base` 名单过大，导致默认路径仍然过重  
应对：Phase A 先从最小可用集合开始，后续只增不乱扩。

风险三：未加载占位卡片与真实卡片结构差异过大，导致样式反复调整  
应对：固定卡片尺寸和主交互骨架，占位只替换内容，不改容器尺寸。

风险四：卸载规则遗漏 `recent` / `favorites` / shared views 某一条引用链  
应对：以“引用扫描 + 明确禁止/占位策略”统一处理，不做 silent fallback。

风险五：未加载 fallback 只覆盖 picker，不覆盖 tab/menu/ghost  
应对：把所有渲染入口统一纳入同一 fallback contract，并用测试锁定。

## 执行决策

当前推荐的 Phase A 执行策略：

1. 先固定 `Base / Legacy` 映射与包状态 contract
2. 再做 registry 分离和 `Base`
3. 再打通收藏字段全链路
4. 再接 options 浮框与统一 fallback 占位
5. 最后补 recent / 卸载保护与性能复测

推荐理由：

- 先把 pack 真相源和运行时边界定死，后面的 UI 和收藏测试不容易返工
- 先完成 `Base + registry 分离`，才能真实验证性能收益
- 把 Tabler full set 留到 Phase B，能显著降低当前改动面和风险

# 表格单元格布局框架统一方案

## 方案概述

### 1. 总体目标和范围

本方案目标是统一 data-editor 主表格的单元格布局框架，解决当前标题列、普通文本、Select / Multi-select / Relation chip、wrapped 文本、编辑态浮框之间上下留白和视觉基线不一致的问题。

当前问题不是某个单独 CSS 属性失效，而是表格布局职责分散：`td.data-cell`、文本 renderer、标题按钮、选项 trigger、chip、编辑态浮框都在各自控制高度、padding、line-height 或对齐方式。局部修改 `vertical-align`、`padding` 或编辑态边框只能修复单一截图，不能保证不同内容类型在同一行内长期一致。

本方案覆盖：

- 主表格 body 单元格的 DOM 布局契约。
- Text、标题列、Select、Multi-select、Relation、Backlink、Nested summary、Checkbox、Incompatible、编辑态文本单元格的统一外层布局。
- 单行内容、wrapped 内容、chip 内容、编辑态浮框的垂直对齐规则。
- 对应 E2E 几何回归测试。

本方案不覆盖：

- 字段类型语义调整。
- autosave / save coordinator 机制重构。
- 详情页字段布局。
- 表头布局和列拖拽交互。
- 选项弹层内部视觉。

### 2. 各阶段任务概要

1. **现状基线阶段**
   - 主要工作：用 `rg` 和 DOM/CSS 证据梳理当前 `td.data-cell`、`CellRenderer`、`table-columns.tsx`、`OptionFieldEditor`、`RelationCellEditor`、`TableTextCellEditor` 的布局职责。
   - 预期成果：明确哪些样式属于单元格外层布局，哪些属于内容组件外观，形成改造前问题清单。
   - 执行顺序：先读渲染链路，再读样式，再补失败 E2E。

2. **统一单元格布局契约阶段**
   - 主要工作：引入统一的 `TableCellFrame` / `TableCellContent` 结构，并明确 frame 必须覆盖 `table-columns.tsx` 的所有 body cell render 分支，而不是只放进普通 `CellRenderer`。
   - 预期成果：所有数据单元格共享同一套外层 frame，`td` 不再直接承担内容 padding 和垂直布局职责；标题列、Backlink、Nested summary 等特殊分支不会遗漏。
   - 执行顺序：先定义契约和 CSS，再迁移最简单的普通 Text。

3. **内容类型迁移阶段**
   - 主要工作：依次迁移普通 Text、标题列、Select、Multi-select、Relation、Backlink、Nested summary、Checkbox、Incompatible、编辑态文本。
   - 预期成果：不同内容类型在同一行内使用相同的垂直节奏；内容组件只负责自身展示，不再决定整格上下留白。
   - 执行顺序：先文本和标题，再 token/chip 类，最后编辑态浮框。

4. **wrapped 与单行策略收敛阶段**
   - 主要工作：统一区分 `center`、`top`、`editor` 三类单元格布局策略。
   - 预期成果：单行内容默认垂直居中；wrapped 内容默认顶部对齐；同一行被高内容撑开时，单行内容仍保持上下留白一致。
   - 执行顺序：先用数据属性表达布局模式，再删除分散的 `vertical-align` 和 padding 修补。

5. **编辑态浮框接入阶段**
   - 主要工作：让文本编辑态浮框依附统一 frame 或 `td` positioning context，而不是依赖负 margin 或内容宽高推算。
   - 预期成果：编辑态边框始终贴合整个单元格边界，且不会影响 blur-only 保存策略。
   - 执行顺序：先保持当前手动提交语义，再只替换定位与尺寸 contract。

6. **验证与清理阶段**
   - 主要工作：补充同一行多内容类型的几何 E2E，删除旧样式残留，跑局部 E2E、`npm test` 和必要的浏览器截图验证。
   - 预期成果：用自动化锁定标题、chip、普通文本、wrapped 文本、编辑态浮框的布局一致性。
   - 执行顺序：先局部 E2E，再完整测试，最后清理无效 class。

### 3. 整体结构框架

目标结构如下：

```tsx
<td
  className="data-cell"
  data-cell-kind="data"
  data-column-field={fieldName}
  data-wrap-mode="truncate|wrap"
  data-cell-layout="center|top|editor"
>
  <div
    className="table-cell-frame"
    data-cell-frame-kind="text|title|token|editor|checkbox|nested|backlink|incompatible"
    data-cell-frame-layout="center|top|editor"
  >
    <div className="table-cell-content" data-cell-content-kind="text|title|token|editor|checkbox|nested|backlink|incompatible">
      {content}
    </div>
  </div>
</td>
```

职责分层：

- `td.data-cell`：只负责表格语义、列宽、边框、背景、定位上下文。
- `.table-cell-frame`：统一负责单元格高度、padding、垂直对齐、单行/wrapped 布局模式。
- `.table-cell-content`：统一承载内容组件，限制宽度、溢出和内容布局。
- 具体 renderer：只负责文本、按钮、chip、输入框等内容本身，不再控制整格上下留白。

Frame 包裹位置必须遵守以下约束：

- `TableCellFrame` 不只放在 `CellRenderer` 内部。
- `table-columns.tsx` 中所有 body cell 分支都必须返回 frame。
- `CellRenderer` 可以继续负责字段类型分派，但它返回的内容必须被 frame 包裹，或者由它内部按统一 contract 包裹并覆盖所有字段类型分支。
- 标题列、Backlink、Nested summary 这些不经过普通 `CellRenderer` 的分支必须显式接入 frame。

建议最终 CSS contract：

```css
.data-table td.data-cell {
  background: var(--color-bg-surface);
  border-bottom: 1px solid var(--color-border-subtle);
  border-right: 1px solid var(--color-border-subtle);
  height: 36px;
  max-width: 360px;
  min-width: 56px;
  padding: 0;
  position: relative;
}

.table-cell-frame {
  align-items: center;
  box-sizing: border-box;
  display: flex;
  min-height: 36px;
  padding: 7px 8px;
  width: 100%;
}

.table-cell-frame[data-cell-frame-layout="top"] {
  align-items: flex-start;
}

.table-cell-frame[data-cell-frame-layout="editor"] {
  align-items: stretch;
}

.table-cell-content {
  min-width: 0;
  width: 100%;
}
```

## 一、问题复盘

### 1. 当前修复失败的原因

当前表格中，同一行可能同时出现以下内容：

- 标题列：`title-cell-button`
- 普通文本：`cell-display`
- Select / Multi-select / Relation：`multi-select-trigger` + `chips-cell` + `chip`
- Backlink：`BacklinkCellViewer` 返回文本或 chip flow
- Nested summary：`nested-summary`
- Checkbox：`checkbox-cell`
- Incompatible：`cell-incompatible`
- wrapped 文本：`cell-text-wrap`
- 编辑态文本：`table-text-cell-editor`

这些内容类型各自带有不同布局规则：

- `td.data-cell` 有 `padding-top: 7px` 和 `padding-bottom: 7px`。
- `td.data-cell` 曾经统一 `vertical-align: top`，后续局部改成 `middle`。
- `.multi-select-trigger` 有 `min-height: 28px` 和 `padding: 4px 0`。
- `.chip` 有自己的内边距和 line-height。
- 普通文本只有文字 line box。
- 标题列是按钮，不走普通文本 cell 的 class。
- 编辑态浮框又使用独立定位逻辑。
- Backlink、Nested summary、Checkbox、Incompatible 也有独立 class，当前不共享同一外层布局 contract。

因此，“底部额外留白”不是一个单一元素的 padding，而是多层高度模型叠加后的视觉结果。

### 2. 为什么只改 `vertical-align` 不够

`vertical-align: middle` 只能影响 table-cell 内部 inline/block box 的垂直位置，不能统一以下问题：

- chip trigger 自己的 `min-height` 与内部 padding。
- 标题按钮和普通文本 line-height 不一致。
- 文本字形视觉中心与 DOM box 中心不一致。
- wrapped 列撑高行后，非 wrapped 列不同内容组件的实际 box 高度不同。
- 编辑态浮框依赖 `td` positioning context，而普通态内容不依赖同一 frame。

所以局部 `vertical-align` 修复只能保证某个测试用例的 DOM box 居中，不能保证用户截图里的标题、chip、普通文本在真实页面同一行中视觉一致。

### 3. 当前测试覆盖缺口

已有回归更多关注单一内容类型：

- 普通文本在高行中上下 gap 是否接近。
- 编辑态浮框是否贴合整个 `td`。
- wrapped 文本是否完整换行。

缺少的关键测试是：

- 同一行内标题、chip、普通文本的视觉中心是否接近。
- token cell 与 text cell 是否使用同一外层 frame。
- 标题列、Backlink、Nested summary、Checkbox、Incompatible 是否与普通 cell 使用同一布局 contract。
- 非 wrapped chip 的 trigger 内部 padding 是否制造额外底部留白。

## 二、目标布局契约

### 1. `td` 只保留表格职责

`td.data-cell` 不再直接控制内容上下 padding，也不再用 `vertical-align` 表达不同内容类型的业务布局。

保留职责：

- 表格边框。
- 背景。
- 列宽约束。
- 行高下限。
- 编辑态定位上下文。

不再承担：

- 内容上下 padding。
- 文本/chip 垂直对齐。
- wrapped 和单行内容的内容布局。

`vertical-align` 最终不作为业务布局机制使用。若浏览器默认 table-cell 行为需要保留 fallback，也必须在注释中说明它不是内容对齐的 source of truth。

### 2. `table-cell-frame` 负责整格节奏

统一 frame 是本方案的核心。

它必须满足：

- 所有数据单元格都有。
- 单行内容默认 `align-items: center`。
- wrapped 内容使用 `align-items: flex-start`。
- 特殊内容通过 `layout="center|top|editor"` 显式选择对齐策略，而不是只由 `wrapped` 推导。
- padding 只在这里定义。
- 编辑态浮框可以贴合 frame 或 `td`。

### 3. 内容组件只负责自身外观

内容组件需要收敛为：

- Text：只控制文本截断或换行。
- Title：只控制点击打开详情和标题文本样式。
- Select/Multi-select/Relation：只控制 trigger、chip、popover 交互。
- Chip：只控制颜色、圆角、文本。
- Editor：只控制输入行为和编辑态视觉。

内容组件不应该再通过 `min-height`、额外上下 padding、局部 `align-items` 影响整个单元格的垂直节奏。

## 三、组件改造方案

### 1. 新增 `TableCellFrame`

建议新增文件：

- `src/table/TableCellFrame.tsx`

接口建议：

```tsx
export type TableCellContentKind =
  | "text"
  | "title"
  | "token"
  | "editor"
  | "checkbox"
  | "nested"
  | "backlink"
  | "incompatible";

export type TableCellLayout = "center" | "top" | "editor";

type TableCellFrameProps = {
  kind: TableCellContentKind;
  layout?: TableCellLayout;
  children: React.ReactNode;
};

export function TableCellFrame({ kind, layout = "center", children }: TableCellFrameProps) {
  return (
    <div className="table-cell-frame" data-cell-frame-kind={kind} data-cell-frame-layout={layout}>
      <div className="table-cell-content" data-cell-content-kind={kind}>
        {children}
      </div>
    </div>
  );
}
```

Frame 包裹位置：

- 优先在 `table-columns.tsx` 的 cell render 分支中统一包裹。
- `CellRenderer` 内部可以继续负责字段类型分派，但不要让它成为唯一 frame 入口。
- 如果选择在 `CellRenderer` 内包裹，则必须同步把标题列、Backlink、Nested summary 等非 `CellRenderer` 分支也包裹，不能形成双入口 contract。
- 执行时必须用测试或静态断言确认所有 body cell render 分支都经过 `TableCellFrame`。

### 2. 普通 Text 迁移

当前：

```tsx
<div className="editable-cell cell-display cell-text-content">
  <span>{textValue}</span>
</div>
```

目标：

```tsx
<TableCellFrame kind="text" layout={wrapped ? "top" : "center"}>
  <div className="cell-display cell-text-content">
    <span>{textValue}</span>
  </div>
</TableCellFrame>
```

注意：

- `.cell-display` 不再负责整格 flex 对齐。
- `.cell-text-content` 只负责文本显示。
- 单行截断和 wrapped 展示仍保留。

### 3. 标题列迁移

当前标题列在 `table-columns.tsx` 中独立渲染 `title-cell-button`，没有和普通文本共享 frame。

目标：

```tsx
<TableCellFrame kind="title" layout={columnModel.wrapped ? "top" : "center"}>
  <button className="title-cell-button ...">
    <span className="title-cell-text">{value}</span>
  </button>
</TableCellFrame>
```

要求：

- 标题列默认和普通文本使用同一上下 padding。
- 标题列仍保持整格点击打开详情。
- 标题列可以继续不参与表格文本编辑模式。

### 4. Select / Multi-select / Relation 迁移

当前 token 类单元格的高度由 `.multi-select-trigger` 和 `.chip` 共同决定。

目标：

```tsx
<TableCellFrame kind="token" layout={wrapped ? "top" : "center"}>
  <OptionFieldEditor surface="table" ... />
</TableCellFrame>
```

同时收敛：

```css
.table-cell-content[data-cell-content-kind="token"] .multi-select-trigger {
  min-height: 0;
  padding: 0;
  width: 100%;
}

.table-cell-content[data-cell-content-kind="token"] .multi-select-trigger,
.table-cell-content[data-cell-content-kind="token"] .relation-trigger {
  align-items: center;
  display: flex;
  min-height: 100%;
}

.table-cell-content[data-cell-content-kind="token"] .chips-cell {
  align-items: center;
}

.data-cell[data-wrap-mode="wrap"] .table-cell-content[data-cell-content-kind="token"] .chips-cell {
  align-items: flex-start;
}
```

保留 chip 自身外观 padding，但不让 trigger 额外制造整格上下留白。

点击热区要求：

- trigger 必须继续占满 frame 的可点击宽度。
- 非 wrapped token 的点击区域不能只剩 chip 本身。
- hover 背景应覆盖 frame 内的有效区域。
- popover anchor 仍应来自 trigger，不能改成不可聚焦的普通 div。

### 5. Backlink / Nested / Checkbox / Incompatible 迁移

这些分支不能遗漏，否则新旧布局会共存。

Backlink：

```tsx
<TableCellFrame kind="backlink" layout={wrapped ? "top" : "center"}>
  <BacklinkCellViewer ... />
</TableCellFrame>
```

Nested summary：

```tsx
<TableCellFrame kind="nested" layout="center">
  <button className="nested-summary">...</button>
</TableCellFrame>
```

Checkbox：

```tsx
<TableCellFrame kind="checkbox" layout="center">
  <label className="checkbox-cell">...</label>
</TableCellFrame>
```

Incompatible：

```tsx
<TableCellFrame kind="incompatible" layout="center">
  <div className="cell-incompatible">...</div>
</TableCellFrame>
```

要求：

- Backlink wrapped chip flow 顶部对齐。
- Nested summary、Checkbox、Incompatible 默认居中。
- 这些分支的旧 class 只保留内容外观职责，不再决定整格 padding。

### 6. 编辑态文本迁移

编辑态保持当前 blur-only 提交语义：

- 输入中只更新 draft。
- blur / Enter / 文件切换 flush 时提交。
- Escape 撤销并退出。

布局上改为依附统一 frame：

```tsx
<TableCellFrame kind="editor" layout="editor">
  <TableTextCellEditor ... />
</TableCellFrame>
```

编辑态浮框有两种可选实现：

方案 A：铺满 `td`

```css
.data-cell:has(.table-text-cell-editor:focus-within) {
  position: relative;
}

.table-text-cell-editor:focus-within {
  position: absolute;
  inset: 0;
}
```

方案 B：铺满 `table-cell-frame`

```css
.table-cell-frame:has(.table-text-cell-editor:focus-within) {
  position: relative;
}

.table-text-cell-editor:focus-within {
  position: absolute;
  inset: 0;
}
```

推荐方案 A。原因是用户期望边框贴合整个单元格边界，而不是只贴合内容 padding 区。

Textarea 高度策略必须显式定义：

- 单行 input 编辑态：浮框贴合当前 `td`，不主动撑高行。
- wrapped textarea 编辑态：允许 textarea 根据内容增长，并通过现有行高测量机制更新行高；如果当前虚拟行高系统无法即时扩展，则 textarea 内部允许临时滚动，但不能截断文字。
- 文件切换、关系跳转、关闭编辑模式前仍必须 flush active draft。

## 四、样式收敛方案

### 1. 删除或降级旧样式职责

需要逐步清理：

```css
.data-table td.data-cell {
  padding-bottom: 7px;
  padding-top: 7px;
  vertical-align: middle;
}

.data-table td.data-cell[data-wrap-mode="wrap"] {
  vertical-align: top;
}
```

改为：

```css
.data-table td.data-cell {
  padding: 0;
}
```

`vertical-align` 不再作为内容对齐机制。若保留浏览器 fallback，必须避免继续用它表达 `wrap` / `truncate` 业务差异。

### 2. 建立统一 frame 样式

```css
.table-cell-frame {
  align-items: center;
  box-sizing: border-box;
  display: flex;
  min-height: 36px;
  padding: 7px 8px;
  width: 100%;
}

.table-cell-frame[data-cell-frame-layout="top"] {
  align-items: flex-start;
}

.table-cell-frame[data-cell-frame-layout="editor"] {
  align-items: stretch;
}
```

### 3. 内容类型局部样式

Text：

```css
.table-cell-content[data-cell-content-kind="text"] .cell-display {
  min-width: 0;
  width: 100%;
}
```

Title：

```css
.table-cell-content[data-cell-content-kind="title"] .title-cell-button {
  min-height: 0;
  padding: 0;
  width: 100%;
}
```

Token：

```css
.table-cell-content[data-cell-content-kind="token"] .multi-select-trigger {
  min-height: 0;
  padding: 0;
  width: 100%;
}
```

Editor：

```css
.table-cell-content[data-cell-content-kind="editor"] .table-text-cell-editor {
  width: 100%;
}
```

## 五、验证方案

### 1. 必须新增的 E2E

1. **同一行标题、chip、文本垂直中心一致**

构造一行：

- title：`燃烧之心`
- select/chip：`element`
- text：`火焰技能暴击率+10%`

断言：

```ts
expect(Math.abs(titleCenter - cellCenter)).toBeLessThanOrEqual(2);
expect(Math.abs(chipCenter - cellCenter)).toBeLessThanOrEqual(2);
expect(Math.abs(textCenter - cellCenter)).toBeLessThanOrEqual(2);
```

2. **单行文本在高行内上下留白一致**

同行由 wrapped 长文本撑高，短文本仍保持上下 gap 接近。

3. **chip 在高行内上下留白一致**

同行由 wrapped 长文本撑高，Select chip 的视觉中心仍接近单元格中心。

4. **标题列在高行内上下留白一致**

标题列在高行内不贴顶、不贴底。

5. **wrapped 文本仍顶部对齐**

wrapped 长文本仍从单元格顶部 padding 区开始，不因为统一 frame 被居中。

6. **编辑态浮框贴合整个单元格**

单行、高行、wrapped 三种情况下，编辑态浮框上下左右贴合 `td`。

7. **表格编辑保存语义不回退**

输入中不写盘，blur 后保存，文件切换前 flush。

8. **Backlink / Nested / Checkbox / Incompatible 不回退**

Backlink 文本/chip、Nested summary、Checkbox、Incompatible 在普通行和高行内都遵守 frame 对齐规则。

9. **截图级验证**

至少保留一组人工截图或 Playwright screenshot 辅助验证，覆盖标题、chip、普通文本同一行。几何断言是主验证，截图用于发现 DOM center 与人眼视觉中心不一致的问题。

### 2. 测试选择器建议

为降低测试脆弱性，建议在 frame 上增加数据属性：

```tsx
data-cell-frame-kind="text|title|token|editor|checkbox|nested|backlink|incompatible"
```

## 六、执行建议

### 阶段一：只建立 frame，不同时改所有内容类型

先迁移普通 Text 和标题列，验证：

- 标题和普通文本在普通行、高行内上下对齐。
- wrapped 文本仍顶部对齐。

不要在第一步同时迁移 chip 和编辑态，否则回归范围过大。

### 阶段二：迁移 token 类内容

迁移 Select / Multi-select / Relation 后，重点检查：

- chip 单行居中。
- non-wrapped chip 仍单行截断。
- wrapped chip 仍可多行排列。
- popover 触发区域不缩小。

### 阶段三：迁移其它特殊分支

迁移 Backlink、Nested summary、Checkbox、Incompatible，重点检查：

- Backlink wrapped chip 仍顶部对齐。
- Nested summary 点击详情或展开行为不变。
- Checkbox 点击热区不缩小。
- Incompatible 的图标和文字仍居中。

### 阶段四：迁移编辑态浮框

编辑态需要单独验证：

- 普通行边框贴合。
- 高行边框贴合。
- wrapped 列边框贴合。
- 输入中不 autosave。
- blur 后保存。

### 阶段五：删除旧样式残留

清理目标：

- `td.data-cell` 的内容 padding。
- 分散的 `vertical-align` 修补。
- token trigger 上决定整格高度的 padding/min-height。
- 标题按钮上决定整格高度的 padding/min-height。

清理后必须重跑完整 E2E 子集。

## 七、风险与约束

### 1. `display: table-cell` 与 flex frame 的混合风险

`td` 仍然是 table-cell，内部 frame 用 flex。不要把 `td` 自身改成 flex，否则可能破坏 table layout、列宽、虚拟滚动和 sticky header。

### 2. token 触发区域不能缩小

Select / Multi-select / Relation 的 trigger 仍应占满整格可点击区域。即使去掉 trigger 自身 padding，也要由 frame 提供点击空间。

### 3. wrapped 内容不能被居中

统一 frame 不能让 wrapped 文本垂直居中。wrapped 的语义是完整展示多行内容，应该顶部对齐。

### 4. 编辑态浮框不能依赖负 margin

负 margin 只能修单一 padding 模型。一旦高行、wrapped、标题列、chip 参与，就会再次失效。编辑态应依赖 `td` 或 frame 的确定定位上下文。

### 5. 不做长期兼容双框架

项目仍处于早期阶段，重构时不做防御性兼容。迁移完成后应删除旧布局 class 的整格职责，只保留必要的内容外观 class。

### 6. 不能只迁移 `CellRenderer`

标题列、Backlink、Nested summary 不完全经过普通 `CellRenderer`。如果执行时只改 `CellRenderer`，会形成部分单元格新框架、部分单元格旧框架的混合状态，问题会继续复现。

### 7. 几何测试不能替代截图判断

DOM box 居中不等于文字字形视觉居中。方案执行时应同时使用几何断言和至少一次截图核对，尤其是中文文本、英文 chip、标题链接混排场景。

## 八、完成判定

满足以下条件才视为完成：

1. `td.data-cell` 不再直接负责内容上下 padding。
2. Text、Title、Select、Multi-select、Relation、Backlink、Nested、Checkbox、Incompatible、编辑态文本都经过统一 frame。
3. 同一行内标题、chip、普通文本视觉中心一致。
4. wrapped 内容顶部对齐。
5. 编辑态浮框贴合整个单元格边界。
6. 输入中不 autosave、blur 后保存的语义不回退。
7. 相关 E2E 和 `npm test` 通过。
8. 删除或降级旧的局部修补样式，避免新旧布局语义共存。

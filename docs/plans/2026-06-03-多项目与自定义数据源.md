# 多项目与自定义数据源方案

## 方案概述

### 总体目标和范围

本方案目标是让 Data Editor 从“启动时绑定单一项目目录”的工具，升级为“启动一次后可在 UI 中自由切换多个本地项目”的本机数据编辑器。每个项目可以配置一个或多个数据源目录，数据源既可以是项目内相对路径，也可以是本机绝对路径。

本阶段范围包含：

- 本机私有项目注册表，用于保存多个本地项目和数据源配置。
- 后端项目管理 API，用于读取、创建、修改、删除、激活项目。
- 多数据源文件枚举、读取、保存和备份。
- UI 项目切换器和项目配置入口。
- 旧 `--project` / `--root` 启动方式的兼容迁移。

本阶段不包含：

- 团队共享项目 manifest 的完整机制。
- 远程数据源、权限系统或多人协作。
- 任意路径临时读写。所有读写路径必须来自已注册项目的数据源。

### 各阶段任务概要

第一阶段：建立本机项目注册表和项目上下文模型。

主要工作是新增 registry 文件读写、项目定义校验、默认项目生成，以及将 `projectContext` 从单 `dataRoot` 扩展为多 `dataSources`。预期成果是后端能稳定表达多个项目和多个数据源，但 UI 仍可先沿用旧入口。

第二阶段：改造文件服务和 API。

主要工作是让 `/api/files`、`/api/document`、`/api/save`、`/api/view-config`、`/api/view-profiles` 等接口都能基于 `projectId` 解析当前项目上下文。预期成果是同一个服务进程内可以访问不同项目，但读写仍受数据源边界约束。

第三阶段：实现 UI 项目切换和项目管理。

主要工作是在左侧栏顶部加入项目切换器，新增项目管理弹窗，支持添加项目、编辑项目根目录、添加数据源、切换当前项目。预期成果是用户无需重启服务即可切换不同本地 JSON / CSV 数据集合。

第四阶段：迁移、文档和验证。

主要工作是兼容旧命令、补充 README 和 docs，增加单元测试与端到端测试。预期成果是旧用法不破坏，新用法有明确文档和测试覆盖。

执行顺序为：注册表模型 -> 多数据源文件服务 -> API -> 前端状态切换 -> 项目管理 UI -> 文档与验证。

### 整体结构框架

整体结构分为五层：

- `project-registry`：负责本机项目清单的持久化、校验、默认值和迁移。
- `project-context`：根据项目定义生成运行期上下文，包括项目根目录、数据源、配置路径、备份路径和文件策略。
- `file-service`：只处理已授权数据源内的 JSON / CSV 文件，提供文件枚举、读取、保存和备份。
- `server`：提供项目管理 API 和按 `projectId` 分派的数据 API。
- `App / Sidebar / Toolbar`：负责项目选择、数据源分组展示、未保存切换拦截和当前工作区刷新。

## 背景

当前 Data Editor 已经从业务项目内工具逐步拆成独立可运行工具，正式启动入口是：

```powershell
npm run open -- --project C:\Code\Nocturnel --adapter nocturnel
```

现有实现的关键限制是：服务启动时只创建一个 `projectContext`，文件枚举固定从该项目下的 `data/` 目录开始，UI 中也没有项目切换入口。这导致用户如果有多个本地项目，或者同一项目的数据文件分散在多个目录，就只能反复用不同参数启动服务，体验和状态管理都不理想。

## 目标

本方案的目标是：

- 启动一次 Data Editor 后，可以在 UI 中切换不同本地项目。
- 每个项目可以自定义一个或多个数据源目录。
- 数据源支持项目内相对路径和本机绝对路径。
- 每个项目保留独立的 `.data-editor/` 语义配置和备份；服务进程 runtime、controller state 和服务日志归属 Data Editor 本机全局目录。
- 旧 `--project` / `--root` 用法继续可用，并能自动迁移为本机 registry 中的项目。
- 所有读写仍受 allowlist 和数据源边界保护，不能通过前端请求直接读写任意路径。

## 非目标

- 不在第一阶段引入团队共享项目配置作为 source of truth。
- 不支持远程 URL、数据库、云盘 API 或 Notion API 作为直接数据源。
- 不实现多用户权限、锁机制或并发编辑协调。
- 不把外部数据源目录作为项目级配置、备份、运行时状态或日志的存放位置。
- 不做旧机制防御式兼容。项目处于早期阶段，内部数据模型可以清理为新框架，但旧 CLI 用法需要保留用户入口兼容。

## 推荐方案

采用“本机私有项目注册表”作为第一阶段方案。

注册表放在 Data Editor 自己的本机配置目录中，默认路径为：

```text
%APPDATA%\data-editor\projects.json
```

测试和高级场景可通过环境变量覆盖：

```text
DATA_EDITOR_HOME=C:\Users\lans\.data-editor
```

该方案的理由是：

- 多项目路径通常是个人机器状态，不适合写入业务仓库。
- 可以避免 Windows / macOS / Linux 的本地路径差异污染共享配置。
- 能快速满足本地多个项目自由切换的主要需求。
- 后续仍可在此基础上增加“从项目 manifest 导入”的能力。

## 数据模型

### Registry

注册表保存所有本机项目和当前激活项目：

```json
{
  "version": 1,
  "activeProjectId": "nocturnel",
  "projects": [
    {
      "id": "nocturnel",
      "name": "Nocturnel",
      "root": "C:\\Code\\Nocturnel",
      "adapter": "nocturnel",
      "dataSources": [
        {
          "id": "data",
          "label": "Data",
          "path": "data",
          "kind": "relative"
        },
        {
          "id": "balance",
          "label": "Balance",
          "path": "D:\\GameData\\balance",
          "kind": "absolute"
        }
      ],
      "filePolicy": {
        "includeExtensions": [".json", ".csv"]
      }
    }
  ]
}
```

### ProjectDefinition

后端内部建议使用以下结构：

```ts
type ProjectDefinition = {
  id: string;
  name: string;
  root: string;
  adapter: string;
  dataSources: DataSourceDefinition[];
  filePolicy: {
    includeExtensions: string[];
  };
};
```

### DataSourceDefinition

数据源定义如下：

```ts
type DataSourceDefinition = {
  id: string;
  label: string;
  path: string;
  kind: "relative" | "absolute";
};
```

相对路径以项目 `root` 为基准。绝对路径直接作为数据源根目录，但仍必须经过路径规范化和边界检查。

### 标识符规则

`project.id` 和 `dataSource.id` 必须满足：

- 只允许小写字母、数字、短横线和下划线。
- 不能包含 `/`、`\`、`:`、空白字符或 URL 转义字符。
- `project.id` 在 registry 内唯一。
- `dataSource.id` 在同一个项目内唯一。
- 同一个项目内不能出现解析后根目录完全相同的数据源。

如果项目 root 或数据源 path 当前不存在，registry 仍保留配置，但 API 返回该项目或数据源的错误状态，UI 以不可用状态展示，不自动删除用户配置。

## 路径模型

前端和 API 不直接传真实绝对路径，而是使用虚拟路径：

```text
<dataSourceId>/<innerPath>
```

示例：

```text
data/runes.json
data/items/weapons.json
balance/weapon_table.json
```

选择 `/` 作为数据源前缀分隔符，是为了避开当前 relation、backlink、primary key key 中已经使用的 `:` 分隔模型。默认数据源 `data` 下的路径仍保持旧语义，例如旧路径 `data/runes.json` 在新模型下天然表示 `data` 数据源内的 `runes.json`。

后端解析流程：

1. 根据 `projectId` 找到项目定义。
2. 根据虚拟路径第一段找到 `dataSourceId`。
3. 将数据源根目录解析为绝对路径。
4. 将 `innerPath` 通过 `resolveInsideRoot(dataSourceRoot, innerPath)` 解析。
5. 检查扩展名是否属于 `includeExtensions`。
6. 文件读取和保存只允许发生在解析后的数据源范围内。

这样可以同时满足 UI 展示、跨数据源区分和安全边界。

## 后端 API

### 项目管理

新增 API：

```text
GET    /api/projects
POST   /api/projects
POST   /api/project-update
POST   /api/project-delete
POST   /api/project-activate
```

`GET /api/projects` 返回：

```json
{
  "activeProjectId": "nocturnel",
  "projects": []
}
```

`POST /api/project-activate` 请求体为 `{ "projectId": "nocturnel" }`，只更新 registry 中的 `activeProjectId`，不重启服务。

采用扁平 API 路径是为了贴合当前 `server.mjs` 的简单 pathname 分发方式。后续如果引入轻量 router，再考虑 REST 风格的 `PATCH /api/projects/:id`。

### 数据文件

现有数据 API 增加 `projectId`：

```text
GET  /api/files?projectId=nocturnel
GET  /api/document?projectId=nocturnel&path=data/runes.json
POST /api/save
```

`POST /api/save` 请求体：

```json
{
  "projectId": "nocturnel",
  "path": "data/runes.json",
  "root": []
}
```

`/api/files` 返回值建议增加数据源信息：

```json
[
  {
    "path": "data/runes.json",
    "displayPath": "runes.json",
    "dataSourceId": "data",
    "dataSourceLabel": "Data",
    "size": 1024,
    "modifiedAt": "2026-06-03T12:00:00.000Z"
  }
]
```

### 视图配置

视图配置仍按项目保存：

```text
<project>/.data-editor/view-config.json
<project>/.data-editor/view-configs/<profile>.json
```

相关 API 增加 `projectId`：

```text
GET  /api/view-config?projectId=nocturnel
POST /api/view-config
GET  /api/view-profiles?projectId=nocturnel
GET  /api/view-profile?projectId=nocturnel&name=Lans
POST /api/view-profile
```

写入型 API 的请求体必须带 `projectId`。

## 前端交互

### 项目切换入口

项目切换器放在左侧栏顶部，替代当前简单标题区域。左侧栏顶部展示当前项目名和项目菜单按钮，菜单中提供：

- 切换项目
- 添加项目
- 编辑当前项目
- 打开项目管理

项目切换属于 workspace 级导航，会刷新文件树、collection、视图配置和当前文档，因此放在 Sidebar 比放在 Toolbar 更符合职责边界。

### 数据源展示

文件列表按数据源分组：

```text
Nocturnel

Data
  runes.json
  skills.json

Balance
  weapon_table.json
```

文件选择后，`selectedPath` 保存虚拟路径，例如 `data/runes.json`。UI 可用 `displayPath` 展示更友好的名称。

### 项目管理弹窗

项目管理弹窗支持：

- 新增项目：填写项目名称、项目根目录、默认 adapter。
- 编辑项目：修改名称、根目录、adapter。
- 添加数据源：填写数据源名称、路径、路径类型。
- 删除数据源：删除前检查当前选中文件是否属于该数据源。
- 删除项目：禁止删除最后一个项目；删除当前激活项目后自动切换到剩余第一个项目。

第一阶段不做系统文件选择器集成，路径用文本输入，后续可接入原生目录选择能力。

### 未保存切换拦截

切换项目或删除当前数据源前，如果当前文档或视图配置有未保存改动，必须出现阻断确认：

- 保存后切换
- 放弃改动并切换
- 取消

确认切换后，前端清空当前文档级状态，包括：

- `selectedPath`
- `model`
- `collectionPath`
- `selectedRowIndex`
- relation indexes
- validation issues
- primary key sync plan
- 当前文件 dirty state

然后按新 `projectId` 重新加载 files、view config、view profiles。

## 配置和备份归属

即使某个数据源是绝对路径，项目语义配置和文件备份仍归属项目 `root`：

```text
<project>/.data-editor/view-config.json
<project>/.data-editor/view-configs/
<project>/.data-editor/backups/
```

服务进程 runtime、controller state 和服务日志归属 Data Editor 本机全局目录：

```text
%APPDATA%/data-editor/runtime/
%APPDATA%/data-editor/logs/
```

如果设置了 `DATA_EDITOR_HOME`，则改为：

```text
<DATA_EDITOR_HOME>/runtime/
<DATA_EDITOR_HOME>/logs/
```

这样 controller bridge 和主服务生命周期不再绑定某一个 active project。UI 切换项目只改变当前数据上下文，不改变服务进程身份。

备份文件名包含虚拟路径，避免不同数据源下同名文件冲突：

```text
data__runes.json.2026-06-03T12-00-00-000Z.bak
balance__weapon_table.json.2026-06-03T12-00-00-000Z.bak
```

## 兼容和迁移

### 旧启动命令

旧命令继续可用：

```powershell
npm run open -- --project C:\Code\Nocturnel --adapter nocturnel
npm run open -- --root C:\Code\Nocturnel
```

启动时迁移规则：

- 如果 registry 不存在，自动创建 registry。
- 如果命令行传入 `--project` 或 `--root`，且 registry 中没有该 root 对应项目，则自动注册一个项目。
- 自动注册项目的默认数据源为 `data`，路径为 `data`。
- 如果命令行传入的项目已存在，则将其设置为 active project。
- 如果没有命令行项目，则使用 registry 中的 `activeProjectId`。
- 如果 registry 为空，UI 显示添加项目空状态。

### ViewConfig Key

当前 relation、backlink、primary key 等配置大量使用文件路径参与 key。引入虚拟路径后，默认数据源继续使用旧的 `data/...` 路径语义，非默认数据源使用 `<dataSourceId>/<innerPath>`。

迁移策略：

- 旧 key 中的 `data/runes.json` 不需要迁移，仍表示默认 `data` 数据源内的 `runes.json`。
- 新增数据源的 key 使用同一套路径形态，例如 `balance/weapon_table.json:$:weapon_id`。
- 禁止在 `dataSource.id` 中使用 `:`，避免破坏现有 relation key 的 `sourceFile:sourceCollection:fieldPath` 分隔模型。
- 不引入长期双写。加载时只做必要的规范化，保存后统一写入新模型。

## 安全边界

所有文件读写必须满足：

- `projectId` 必须存在于 registry。
- `dataSourceId` 必须存在于对应项目。
- `projectId` 和 `dataSourceId` 必须通过标识符规则校验。
- `innerPath` 必须解析在数据源根目录内。
- 扩展名必须属于项目 `filePolicy.includeExtensions`。
- 前端不能通过请求体临时传入任意绝对路径。
- 删除项目或数据源只修改 registry，不删除磁盘上的业务文件。

绝对路径数据源是允许的，但它必须先被用户显式注册到本机 registry。

## 风险和应对

### Relation Key 迁移风险

风险：旧 view-config 中的 relation、backlink、primary key key 使用 `:` 分隔 source file、collection 和 field path；如果虚拟路径也使用 `:`，会直接破坏解析。

应对：虚拟路径使用 `/` 分隔数据源和内部路径，默认数据源继续保持 `data/...` 语义；`dataSource.id` 禁止包含 `:`、`/` 和 `\`。

### App 状态复杂度风险

风险：`App.tsx` 已承担较多编排职责，直接追加项目切换逻辑会进一步膨胀。

应对：项目切换相关状态和动作应抽成独立 workspace 逻辑，`App.tsx` 只消费 active project、files、view config 和切换回调。

### 绝对路径安全风险

风险：自定义绝对路径容易引入任意文件读写问题。

应对：绝对路径只能来自 registry；API 请求只接受 `projectId` 和虚拟路径；后端每次读写都重新解析并检查边界。

### 多实例端口风险

风险：当前 controller bridge 会校验 project root 和端口，多项目 UI 切换后服务不再等同于单 project root。

应对：controller identity 应从“单项目身份”调整为“toolRoot + registry home + port + mode”，active project 不参与端口占用判断。

同时，`runtime-state.mjs`、`recovery-bridge.mjs`、`service-lifecycle.mjs`、`open.mjs`、`stop.mjs` 必须统一使用 registry home 作为运行时状态目标，不能再用 active project root 作为 controller/service state 的存储根。

- 从项目内 `data-editor.project.json` 导入项目模板。
- 支持团队共享 manifest 和本机私有 override 合并。
- 接入原生目录选择器，减少手输路径。
- 为常用项目增加 pin、recent、搜索排序。
- 为数据源增加只读标记，支持查看但禁止保存。

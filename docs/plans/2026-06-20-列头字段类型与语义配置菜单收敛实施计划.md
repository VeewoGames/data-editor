# 列头字段类型与语义配置菜单收敛实施计划

## 方案概述

### 1. 总体目标和范围

本实施计划用于收敛 `data-editor` 列头菜单中“字段类型切换”和“语义配置”两组能力的显示逻辑，解决当前 `displayType` 同时承担“基础字段类型”“当前渲染态”“relation/backlink 覆盖态”的职责，导致菜单按钮出现条件混乱、文案语义不稳定、配置态被隐藏的问题。

本次实施的核心目标有五个：

- 把字段的**基础类型**、**语义角色**、**菜单能力**明确拆层。
- 让列头菜单不再直接依赖当前 `displayType` 拼业务判断，而是统一消费 capability。
- 收紧 `Document` / `Relation` / `Backlink` 等叠加语义与基础字段类型之间的边界。
- 把标题字段、主键字段、relation 字段的准入边界明确收窄到你刚确认的产品规则。
- 用自动化测试把菜单显隐矩阵固化下来，避免后续再回到“加一个特判修一个洞”的状态。

本次实施范围包含：

- 列模型新增 `baseDisplayType`、`effectiveDisplayType`、`capabilities`。
- 抽取字段菜单 capability 计算逻辑。
- `ColumnHeader` 菜单显隐规则改为 capability 驱动。
- `Document` 与 `Relation` 互斥约束收口。
- 列头菜单相关单测与 e2e 覆盖补齐。

本次实施不包含：

- 新增 `Multi-select`、`Number`、`Date` 等新的字段类型转换能力。
- 改造 relation / document 的底层数据模型。
- 重做列头菜单视觉样式。
- 扩展 nested 字段路径级的字段类型编辑能力。

### 2. 各阶段任务概要

阶段一：梳理字段状态模型。  
主要工作是把当前列模型中混用的 `displayType` 拆成基础类型和有效渲染类型，并保留当前 cell 渲染行为不变。预期成果是 UI 逻辑不再需要从覆盖态里反推基础字段类型。执行顺序先列模型，再表格接线层。

阶段二：建立 capability 矩阵。  
主要工作是把“可切换字段类型”“可设标题”“可设主键”“可配置 relation”“可配置 document”统一收进一处，形成稳定输出。预期成果是菜单层只负责渲染，不再自行推导业务规则。执行顺序先纯函数，再接入列模型。

阶段三：重写列头菜单判定。  
主要工作是让 `ColumnHeader` 按 capability 渲染中间类型切换区和语义配置区，并重新整理分组和 separator。预期成果是同一字段在不同配置态下菜单保持可预测。执行顺序先 props，再菜单分组，再按钮 disabled/hidden 策略。

阶段四：补互斥规则与动作保护。  
主要工作是补 `Document` / `Relation` 互斥、`Document` 不可设为主键等保护，并在 action 层做最终兜底。预期成果是不会再出现“UI 看不见但配置仍残留”或“同字段叠两套语义”的脏状态。执行顺序先 capability，再 action guard，再清理逻辑。

阶段五：补测试与验证。  
主要工作是补列模型单测、capability 单测和列头菜单 e2e；预期成果是基础类型、叠加角色、菜单显隐三层矩阵具备自动化护栏。执行顺序先纯函数单测，再组件/模型单测，最后 e2e。

### 3. 整体结构框架

本次实现采用四层结构：

1. **基础字段类型层**
   - 表示字段在 `viewConfig.fields[*].type` 上的真实类型。
   - 只包括持久化的基础类型，不包含 relation/backlink 这种叠加语义。

2. **字段语义角色层**
   - 表示字段当前承担的角色，如 `normal`、`relation`、`backlink`、`primaryKey`、`document`。
   - 角色来源于 `viewConfig.relations`、`viewConfig.documentFields`、`viewConfig.primaryKeys` 等语义配置。

3. **能力层**
   - 根据基础字段类型、角色、nested/backlink 等约束，统一输出 capability。
   - capability 是列头菜单唯一的业务判断输入。

4. **展示层**
   - `ColumnHeader` 只负责根据 capability 决定显示、禁用或隐藏哪些菜单项。
   - `CellRenderer` 继续根据 `effectiveDisplayType` 决定渲染路径。

这四层之间必须遵守两个硬边界：

- `effectiveDisplayType` 只服务于 cell 渲染，不再反向决定菜单能力。
- 菜单能力必须来自统一 capability 计算函数，禁止在 `ColumnHeader` 内继续散落 `displayType === ...` 这类特判。

- `Document` 已在设计文档中被定义为独立字段类型，不是 `Relation` 的特例，也不是 `Text` 的显示插件。
- `Backlink` 是派生列，当前不支持字段类型编辑。
- `Nested` 字段当前不支持列头层面的字段类型转换。
- 现有产品范围内，字段类型切换仍只允许 `Text`、`Select`、`Document` 这三种基础类型。

  - relation 只允许配置在**非标题、非主键的 `Text` 字段**上。
  - 标题字段只允许 `Text`。
  - 主键字段只允许 `Text`。
  - 存量 `Document + Relation` 双配置状态在首次加载时自动清理，不提供显式修复入口。

## 目标能力矩阵

本次实施采用如下 capability 基线：

| 基础字段类型 | canChangeType | canBeTitle | canBePrimaryKey | canConfigureRelation | canConfigureDocument |
| --- | --- | --- | --- | --- | --- |
| `Text` | 是 | 是 | 是 | 视当前是否为标题/主键而定 | 是 |
| `Select` | 是 | 否 | 否 | 否 | 否 |
| `Document` | 是 | 否 | 否 | 否 | 是 |
| `Number` | 否 | 否 | 否 | 否 | 否 |
| `Checkbox` | 否 | 否 | 否 | 否 | 否 |
| `Multi-select` | 否 | 否 | 否 | 否 | 否 |
| `Date` | 否 | 否 | 否 | 否 | 否 |
| `JSON` | 否 | 否 | 否 | 否 | 否 |
| `Nested` | 否 | 否 | 否 | 否 | 否 |
| `Backlink` | 否 | 否 | 否 | 否 | 否 |

额外硬约束：

- `Document` 与 `Relation` 必须互斥。
- `Document` 不允许被设为主键。
- `Select`、`Document`、`Number`、`Checkbox`、`Multi-select`、`Date`、`JSON`、`Nested` 都不允许被设为标题字段。
- `Select`、`Document`、`Number`、`Checkbox`、`Multi-select`、`Date`、`JSON`、`Nested` 都不允许被设为主键字段。
- relation 只允许配置在**当前既不是标题字段、也不是主键字段**的 `Text` 字段上。
- `Relation` 不作为持久化基础类型写入 `fieldConfig.displayTypes`。
- `Backlink` 和 `Nested` 始终禁止字段类型切换与语义配置。
- 首次加载时若同一字段同时存在 `Document` 与 `Relation` 配置，自动清理为单一有效状态，不展示修复入口；清理优先保留 `Document`，移除 relation 配置。

## 关键决策落点

### 1. Relation 配置字段是否继续显示基础类型切换区

结论：**不继续显示。**

原因：

- 你已确认 relation 只能配置在非标题、非主键的 `Text` 字段上。
- relation 一旦成立，该字段就进入明确的结构语义态，不再作为普通基础类型编辑入口。
- 菜单层继续暴露基础类型切换区，会让“这是 Text 字段的普通编辑”与“这是结构化 relation 字段”两种心智并存，继续制造歧义。

因此，relation 配置态字段在列头菜单中：

- 不显示中间“字段类型切换区”；
- 只显示 relation 相关动作（编辑关联配置 / 取消关联字段）；
- 若用户需要改回普通 `Text` 或改成 `Document`，必须先取消 relation，再使用基础字段类型切换。

### 2. 标题字段为什么只允许 Text

结论：**标题字段只允许 `Text`。**

业务理由：

- 标题字段承担的是集合主展示文本，不应混入枚举值、文档引用或结构语义。
- `Text` 是唯一不带额外语义、可稳定承载自由命名内容的基础字段。
- 这样可以避免标题字段再叠 relation/document/primary key 语义，收紧“主展示文本”职责边界。

### 3. 主键字段为什么只允许 Text

结论：**主键字段只允许 `Text`。**

业务理由：

- 当前产品里的主键不只是唯一值，还承担 relation/document 引用锚点、路径迁移、主键改名分析等跨模块语义。
- 统一到 `Text` 能避免 number/select/document 等不同值域在去重、重写、显示、迁移上的分歧。
- 这也是最利于后续 relation/document 规则收口的最小模型。

已有其他类型主键配置的处理：

- 首次加载时保留其已配置状态，但菜单层不再允许新增这类配置。
- 若后续用户主动切换主键到别的字段，则只能选 `Text` 字段。
- 本次计划不做批量迁移旧主键类型，只防止继续新增。

### 4. 旧的非 Text 标题字段配置如何处理

结论：**与旧的非 `Text` 主键配置采用同一策略。**

处理规则：

- 首次加载时保留其已配置状态，不做静默迁移。
- 菜单层不再允许新增非 `Text` 标题字段。
- 若用户后续主动切换标题字段，则只能选择 `Text` 字段。

### 5. Relation 配置态字段的菜单 contract

结论：**relation 配置态字段进入专用结构菜单，不再展示普通字段类型/语义配置入口。**

具体 contract：

- 不显示中间“字段类型切换区”；
- 不显示“设为标题字段”；
- 不显示“设为主键ID”；
- 只显示 relation 相关动作（编辑关联配置 / 取消关联字段）；
- 仍保留与结构无关的通用动作，例如排序、筛选、换行、复制字段文本、隐藏、移动、重置列宽、删除字段。

### 6. Document + Relation 双配置自动清理发生在哪一层

结论：**在 view-config 归一化/加载阶段清理。**

原因：

- 这是配置真相层的脏状态，不应等到 UI 打开菜单时才临时修复。
- 提前在加载阶段清理，后续列模型、capability、菜单与渲染层都可以只面对单一有效状态。
- 这也最利于测试：给定旧配置输入，归一化输出即为唯一真相。

落实要求：

- 清理逻辑必须进入 view-config 归一化或等价的加载入口。
- 归一化结果保留 `Document`，移除 relation 配置。
- 补单测直接锁定该归一化行为。

  - 存量 `Document + Relation` 双配置加载后被自动清理。

验收结果：

- 菜单显隐矩阵在自动化测试中被明确定义。
- 未来新增字段类型或语义配置时，必须先更新 capability 测试。

## 风险与处理

### 风险 1：旧逻辑默认把 `displayType` 当唯一真相

风险说明：
现有多个调用点可能隐式依赖“relation 配置后 `displayType === Relation`”这一行为。

处理策略：

- 第一轮保留 `effectiveDisplayType` 给 cell 渲染与兼容路径使用。
- 菜单层先切走，不一次性替换所有消费者。

### 风险 2：菜单规则收紧后测试大面积失效

风险说明：
部分既有 e2e 可能默认断言某些菜单项总是可见。

处理策略：

- 先补 capability/列模型单测作为真值。
- 再逐个修正 e2e 断言，改为按字段类型与角色判断。

### 风险 3：互斥清理影响现有存量配置

风险说明：
如果历史上已存在同时配置 `Document` 和 `Relation` 的字段，动作层强制互斥可能改变当前行为。

处理策略：

- 实施前先搜索测试数据和默认配置中是否存在双配置状态。
- 若无正式依赖，直接采用“新规则覆盖旧状态”的早期项目策略。
- 首次加载自动清理 relation，保留 `Document`，并补测试锁住该行为。

### 风险 4：旧项目存在非 Text 的标题或主键配置

风险说明：
你已确认新规则收窄为“标题只允许 Text”“主键只允许 Text”，但旧 view-config 可能已经存在其他类型字段被配置为标题或主键。

处理策略：

- 首次加载不主动迁移这类旧配置，避免无提示改写用户项目。
- 菜单层不再允许新增这类配置。
- 如后续用户主动改配，只允许选择 `Text` 字段。

### 风险 5：双配置清理如果放在 UI 层会导致不同模块看到不同真相

风险说明：
如果 `Document + Relation` 双配置只在菜单或列模型层临时处理，其他依赖 view-config 的模块仍可能读到脏状态。

处理策略：

- 明确把自动清理放进 view-config 归一化/加载阶段。
- 归一化后的单一真相再流向列模型、capability、渲染和菜单。
- 为归一化入口补独立测试，避免 UI 测试替代配置层测试。

## 验证计划

最小验证集：

- `npm run typecheck`
- 相关模型/纯函数单测
- 定向 e2e：列头菜单字段类型与语义配置场景

重点验证场景：

1. 普通 `Text` 字段打开列头菜单。
2. `Select` 字段打开列头菜单。
3. `Document` 字段打开列头菜单。
4. 已配置 relation 的字段打开列头菜单。
5. 自动生成的 backlink 列打开列头菜单。
6. 切换 `Text -> Document`，再检查 relation/document 配置动作与文案。
7. 非 `Text` 字段不显示“设为标题”“设为主键ID”。
8. 标题字段与主键字段不显示“设为关联字段”。
9. 首次加载带 `Document + Relation` 双配置的测试数据，确认 relation 被自动清理。

## 实施后预期结果

完成本计划后，列头菜单将形成以下稳定契约：

- 基础字段类型决定“能不能变成什么类型”。
- 字段语义角色决定“额外能做哪些结构性配置”。
- capability 是菜单唯一判断输入。
- `Relation` / `Document` / `Backlink` 的边界在代码和 UI 上保持一致。
- 标题字段、主键字段、relation 字段的准入边界收敛为你已确认的单一规则：三者都以 `Text` 为唯一合法基础字段，其中 relation 还必须额外满足“非标题、非主键”。

这会把当前“中间几个按钮出现条件比较混乱”的问题，收敛为一套可推导、可测试、可维护的规则体系，而不是继续堆特判。

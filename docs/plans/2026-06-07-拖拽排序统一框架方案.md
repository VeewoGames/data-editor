# 拖拽排序统一框架方案

## 方案概述

### 总体目标和范围

本方案的目标是为 `data-editor` 建立一套统一的拖拽排序框架，优先解决当前多个排序入口在交互模型、响应速度、位置预览和动画稳定性上的不一致问题。目标体验收敛为三条：

1. 及时响应：拖动开始后，位置变化能够快速反映，不需要过大的位移阈值。
2. 位置预览：拖动过程中始终能明确看到“如果现在松手会落到哪里”。
3. 动画流畅：拖动过程、跨越多个项、大范围扫动和松手收尾都不出现闪烁、错乱或明显抖动。

本轮范围聚焦“**一维列表重排**”这类拖拽排序，优先覆盖以下入口：

- [src/table/OptionFieldEditor.tsx](C:/Code/data-editor/src/table/OptionFieldEditor.tsx)
- [src/components/Sidebar.tsx](C:/Code/data-editor/src/components/Sidebar.tsx)
- [src/components/ViewTabs.tsx](C:/Code/data-editor/src/components/ViewTabs.tsx)
- [src/detail/DetailPanel.tsx](C:/Code/data-editor/src/detail/DetailPanel.tsx)

其中列拖拽排序虽然同样属于拖拽交互，但它带有横向滚动、slot 计算、auto-scroll 和表头 ghost，复杂度更高，因此本方案不要求第一阶段就把 [src/table/column-dnd.mjs](C:/Code/data-editor/src/table/column-dnd.mjs) / [src/table/DataTable.tsx](C:/Code/data-editor/src/table/DataTable.tsx) 并入统一框架。列拖拽可以在统一内核稳定后再评估是否接入，或只复用部分底层能力。

本方案不继续采用当前“拖拽过程中直接改真实列表顺序”的模型，而是统一切换到“**投影式拖拽 + 放手提交**”模型：

- 拖拽过程中不提交真实顺序
- 拖拽过程中不依赖真实 DOM 重排来做预览
- 只在 drop 时一次性提交最终顺序

### 各阶段任务概要

#### 第一阶段：抽统一拖拽排序内核

主要工作：

- 新增共享 drag engine / projection / geometry / animation 抽象
- 定义统一的拖拽状态模型、命中规则和 drop 提交流程
- 明确共享层只统一“拖拽状态机、投影计算和提交协议”，不强制统一所有入口的 DOM 结构
- 先不迁移现有组件，只建立最小可用内核

预期成果：

- 一套可复用的一维排序拖拽内核
- 一组明确的输入输出边界
- 纯逻辑测试覆盖阈值、方向、dropIndex 和 projectedOrder

执行顺序：

1. 定义状态与类型
2. 拆分几何判定与投影算法
3. 建立 drop 提交接口

#### 第二阶段：先迁移高痛点入口

主要工作：

- 先将 [src/table/OptionFieldEditor.tsx](C:/Code/data-editor/src/table/OptionFieldEditor.tsx) 迁到新内核
- 再迁 [src/components/Sidebar.tsx](C:/Code/data-editor/src/components/Sidebar.tsx)

预期成果：

- 解决当前选项排序拖拽中的闪烁、方向手感不一致和松手重渲染问题
- 验证统一框架在纵向列表上的可行性

执行顺序：

1. 迁移 `OptionFieldEditor`
2. 定向回归和浏览器验证
3. 迁移 `Sidebar`

#### 第三阶段：统一中复杂度排序入口

主要工作：

- 迁移 [src/components/ViewTabs.tsx](C:/Code/data-editor/src/components/ViewTabs.tsx)
- 让 [src/detail/DetailPanel.tsx](C:/Code/data-editor/src/detail/DetailPanel.tsx) 适配统一内核，而不是整套重做

预期成果：

- 视图标签和详情字段排序与选项排序共享同一套交互语义
- 统一阈值、预览、ghost、placeholder 和动画策略

执行顺序：

1. `ViewTabs`
2. `DetailPanel`
3. 跨入口手感一致性回归

#### 第四阶段：评估列拖拽接入策略

主要工作：

- 分析 [src/table/column-dnd.mjs](C:/Code/data-editor/src/table/column-dnd.mjs) 是否能直接接入统一内核
- 若不能，明确它与一维排序拖拽共享哪些底层能力

预期成果：

- 形成“全项目统一拖拽框架”的最终边界
- 决定列拖拽是完全迁入，还是只复用局部工具层

执行顺序：

1. 评估
2. 决策
3. 单独实施或保留独立

### 整体结构框架

统一拖拽排序框架建议拆成四层：

1. `Drag Engine`
2. `Projection Layer`
3. `Commit Layer`
4. `Animation Layer`

这四层职责分离，核心原则是“交互判定、视觉预览、真实数据提交”不能再绑在同一份即时重排状态上。

## 现状问题与根因

当前仓库里的拖拽排序并没有使用同一个框架，而是多套自写实现并存：

- [src/components/ViewTabs.tsx](C:/Code/data-editor/src/components/ViewTabs.tsx) 自己维护 pointer drag、drop target 和 ghost
- [src/components/Sidebar.tsx](C:/Code/data-editor/src/components/Sidebar.tsx) 自己维护 preview order 和 `elementFromPoint(...)`
- [src/detail/DetailPanel.tsx](C:/Code/data-editor/src/detail/DetailPanel.tsx) 自己维护 placeholder、ghost 和 mousemove
- [src/table/OptionFieldEditor.tsx](C:/Code/data-editor/src/table/OptionFieldEditor.tsx) 自己维护 pointermove、阈值、本地重排和动画
- [src/table/column-dnd.mjs](C:/Code/data-editor/src/table/column-dnd.mjs) 只服务于列拖拽

这导致几个直接问题：

- 同一项目内拖拽响应手感不一致
- 上下方向阈值和判定逻辑不一致
- 预览方式不一致，有的是真重排，有的是 placeholder，有的是 drop marker
- 动画策略不一致，有的用 class、有的用 transform、有的用 ghost
- 问题一旦出现，只能在某个组件里继续做补丁式修补

当前最核心的架构问题是：部分拖拽实现，尤其是 `OptionFieldEditor`，在拖拽过程中直接修改真实列表顺序。这样会把下面三层混在一起：

- 交互判定
- 视觉预览
- 数据提交

一旦拖拽过程直接改真实 DOM 顺序，就会出现一系列链式副作用：

- 命中区域跟着移动
- 动画对象跟着变化
- 下一次判定基于变化后的 DOM 几何
- 父层配置回写后再触发更大范围重渲染

这正是闪烁、抖动、方向手感异常和松手时二次跳动的根因。

需要特别说明的是，[src/detail/DetailPanel.tsx](C:/Code/data-editor/src/detail/DetailPanel.tsx) 虽然也属于自写实现，但它已经具备明显的投影式特征：

- `startOrder`
- `restOrder`
- `targetIndex`
- placeholder
- ghost

因此它不应被视为与 `OptionFieldEditor`、`Sidebar` 同等级的问题源，更适合作为统一框架的参考实现或适配对象。

## 统一框架设计

### 一、统一状态模型

所有一维列表拖拽都应统一到下面这组状态：

- `baseOrder`
  拖拽开始时的真实顺序
- `draggingId`
  当前被拖拽的 item id
- `dragStartIndex`
  拖拽开始时索引
- `dropIndex`
  当前投影落点
- `pointerStart`
  拖拽起点坐标
- `pointerCurrent`
  当前指针坐标
- `dragging`
  是否已经越过阈值进入真实拖拽态

派生结果：

- `projectedOrder`
- `ghostPosition`
- `placeholderIndex`
- `itemOffsets`
- `dragDirection`

关键约束：

- `baseOrder` 在拖拽过程中保持不变
- `projectedOrder` 只用于渲染
- 真实顺序只在 `drop` 时提交

### 一点五、入口接入矩阵

统一框架不要求所有入口使用完全相同的视觉呈现，而是统一“拖拽状态机、投影计算、命中判定和提交协议”。各入口的推荐接入方式如下：

| 入口 | 当前特点 | 接入统一层 | 视觉实现策略 |
| --- | --- | --- | --- |
| [OptionFieldEditor](C:/Code/data-editor/src/table/OptionFieldEditor.tsx) | 弹层内纵向选项列表，当前问题最集中 | `Core + Projection + Animation` | siblings offset 为主，ghost 可选 |
| [Sidebar](C:/Code/data-editor/src/components/Sidebar.tsx) | 主列表实时预览，靠 `previewFileOrder` 驱动 | `Core + Projection` | projected list render 为主，ghost 可选 |
| [ViewTabs](C:/Code/data-editor/src/components/ViewTabs.tsx) | 横向标签条，已有 ghost / drop marker | `Core + Projection + Animation` | ghost + drop marker |
| [DetailPanel](C:/Code/data-editor/src/detail/DetailPanel.tsx) | 已具备 placeholder + ghost 投影模型 | `Core` 优先，Presentation 适配 | 保留 placeholder + ghost |

这张矩阵的约束是：

- 必须共享：状态机、阈值、命中、投影顺序、drop 提交协议
- 可选共享：ghost、placeholder、sibling offset 的渲染 helper
- 不共享：每个入口的外壳 DOM、按钮结构、视觉皮肤

### 二、统一事件生命周期

所有一维拖拽统一按以下生命周期处理：

#### 1. `pointerdown`

- 记录起点和源 item
- 不立即进入 dragging
- 不改顺序

#### 2. `pointermove`

- 超过统一阈值后进入 dragging
- 指针更新节流到一帧一次
- 根据当前指针位置计算 `dropIndex`
- 更新投影状态，不改真实顺序

#### 3. `pointerup`

- 若 `dropIndex` 有效且顺序发生变化，则生成 `nextOrder`
- 只在这一步调用外层 `onDrop(nextOrder)`
- 清空 drag state

#### 4. `pointercancel` / escape

- 放弃投影状态
- 不提交顺序

### 三、统一位置预览模型

统一采用“**投影优先**”的模型，框架层统一支持 `ghost`、`placeholder`、`sibling transform` 这三类能力，但不要求每个入口必须三者同时使用。

拖拽过程中：

- 被拖拽项不再依赖原列表节点真实挪动
- 原列表可以用 placeholder 占位，也可以由 projected list 直接表达目标位置
- 其它项根据 `dropIndex` 让位
- ghost 是否单独渲染，由入口视觉决定

这套模型的好处：

- 命中判定不再被真实 DOM 重排干扰
- 视觉预览明确
- 松手时只需提交最终顺序，不会再多打一轮预览动画

### 四、统一命中与阈值策略

统一框架层应负责：

- 拖拽启动阈值
- 上下方向的命中提前量
- 当前 item 与目标 item 的插入判定

建议默认策略：

- 启动阈值：`4~6px`
- 向上拖：更早触发插入
- 向下拖：稍晚触发插入

注意：这些都应成为共享参数，而不是散落在每个组件里的硬编码。

建议内核参数：

- `startThresholdPx`
- `forwardTriggerRatio`
- `backwardTriggerRatio`
- `axis`

### 五、统一动画层

动画层不能再由各组件自行堆叠 `requestAnimationFrame`、`style.transform` 和 “跳过下一次动画” 这类逻辑。

统一原则：

- ghost 的移动单独处理
- sibling 的让位统一用 transform transition 或 WAAPI
- placeholder 只负责稳定布局，不承担动画语义
- drop 后只允许一次 settle 动画
- 不允许旧动画残留到下一轮拖拽判定

推荐做法：

- 行位移采用统一 transition token
- ghost 优先采用独立定位层，但不是强制
- 每次重算 offsets 时，旧动画先取消，再播放新动画

## 推荐代码组织

建议新增一个共享目录：

- [src/drag/](C:/Code/data-editor/src/drag/)

内部建议拆分为：

- `sortable-types.ts`
  统一状态、事件、结果类型
- `sortable-geometry.ts`
  rect 采集、命中判定、方向阈值
- `sortable-projection.ts`
  从 `baseOrder + draggingId + dropIndex` 计算 `projectedOrder`
- `sortable-animation.ts`
  统一 ghost 与 sibling 动画调度
- `useSortableProjection.ts`
  React hook 封装

组件层职责缩减为：

- 提供 item id 列表
- 提供 item ref
- 提供 handle
- 渲染 placeholder / ghost / item
- 在 drop 时提交最终顺序

共享层边界明确如下：

- 必须共享：
  - 拖拽启动判定
  - pointer / mouse 输入适配
  - `dropIndex` 计算
  - `projectedOrder` 生成
  - drop 提交协议
- 可选共享：
  - ghost 渲染 helper
  - placeholder 渲染 helper
  - sibling offset 动画 helper
- 不共享：
  - 每个入口的容器结构
  - 每个入口的菜单、按钮、字段块、标签壳层 DOM

## 风险与边界

### 风险一：统一框架过早试图吞并列拖拽

列拖拽与一维纵向列表排序的几何模型不同，第一阶段不建议硬并。

### 风险二：共享层过大，组件接入成本过高

共享层应提供“排序内核”和“投影工具”，而不是强制所有视觉都长得一样。

### 风险二点五：把 `DetailPanel` 当成问题实现整体重写

`DetailPanel` 当前已经接近投影式拖拽，若把它当成重灾区重做，收益小于成本。更合理的路径是让它优先适配共享核心，而不是先推翻视觉实现。

### 风险三：拖拽框架与现有外层持久化时机冲突

必须明确：

- 拖拽过程只投影
- drop 时生成最终顺序
- 外层持久化仍按现有页面语义执行

### 风险四：动画层与布局层职责不清

动画层只负责“怎么动”，不负责“排到哪里”；排序算法只负责“排到哪里”，不负责“怎么动”。

### 风险五：输入事件模型不统一

当前 [src/detail/DetailPanel.tsx](C:/Code/data-editor/src/detail/DetailPanel.tsx) 使用 `mousedown / mousemove / mouseup`，其它入口大多已经是 pointer 事件。若不在方案里明确统一输入层，共享 drag engine 会卡在事件适配阶段。

## 推荐决策

推荐采用以下策略：

1. 先统一“一维列表排序内核”，不要一开始就吞并列拖拽。
2. 先迁移 `OptionFieldEditor + Sidebar`，验证统一模型。
3. 验证通过后，再迁 `ViewTabs + DetailPanel`。
4. 列拖拽单独评估，决定是完全迁入还是仅共享底层几何工具。

本方案的核心不是继续调阈值、调动画、调提交时机，而是把拖拽排序从“真实列表实时重排”重构成“投影式拖拽 + 放手提交”的统一模型。只有这样，及时响应、位置预览和动画流畅这三个目标才能同时成立。

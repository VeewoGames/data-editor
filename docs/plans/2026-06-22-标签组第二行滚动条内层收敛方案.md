# 标签组第二行滚动条内层收敛方案

## 方案概述

### 1. 总体目标和范围

本方案用于解决 `ViewTabs` 中“标签组第二行”横向滚动条看起来位置过高、底部留白明显的问题。当前 UI 中，用户体感上的空白主要不是来自标签行内容本身，而是来自第二行内部横向滚动容器承担了过大的滚动槽占位，同时外层容器又额外参与了高度兜底，导致“滚动条没有真正贴底、收紧外层也不明显”。

本次方案的总体目标有四个：

- 把第二行标签组的滚动条垂直空间从外层 `.view-tabs-group-row` 收敛到内层 `.view-tabs-group-tabs`。
- 让第二行的视觉留白主要由内层滚动容器自己控制，而不是由外层 `min-height` / `padding` 间接兜底。
- 在不影响全局大 scrollbar hit area 策略的前提下，为标签组第二行提供局部例外，得到更贴近当前交互密度的滚动槽占位。
- 把搜索框、标签行、右侧加号按钮的垂直对齐关系重新拉齐，避免“滚动条下去了，但行内控件悬空或错位”。

本次方案范围包含：

- `src/components/ViewTabs.tsx` 对应第二行结构的布局职责梳理。
- `src/styles.css` 中 `.view-tabs-group-row`、`.view-tabs-group-tabs`、`.view-tabs-group-create`、`group-tab-search` 相关布局样式调整。
- 针对第二行滚动槽占位的静态样式约束测试。
- 针对正式 `dist` 资源的重建和页面实测验证。
- 针对搜索框展开态、窄窗口和正式页 computed style 的专项核查。

本次方案不包含：

- 重做顶层视图标签行 `.view-tabs-top-level` 的布局。
- 修改全局 scrollbar 20px hit area 的总体策略。
- 改造标签组的交互语义、拖拽行为或搜索行为。
- 调整表格区、详情区、textarea 等其它滚动容器的滚动条策略。

### 2. 各阶段任务概要

阶段一：恢复稳定基线。  
主要工作是撤回前一轮仅压缩外层 `.view-tabs-group-row` 的临时收紧调整，确保第二行回到结构方案前的稳定状态。预期成果是后续效果只由结构方案本身决定，不再掺杂旧方案残留。执行顺序先样式回退，再重建 `dist`，再确认正式页面读到的是真实基线值。

阶段二：明确外层与内层职责。  
主要工作是重新定义外层 `.view-tabs-group-row` 与内层 `.view-tabs-group-tabs` 的垂直布局边界：外层只负责搜索框、滚动标签区、创建按钮三块的整体排布和边框关系；内层负责标签流和横向滚动条占位。预期成果是滚动槽相关空间不再由外层 `min-height` 间接兜底。执行顺序先职责收敛，再冻结外层基线值，再删除/弱化外层的滚动条补偿逻辑。

阶段三：把滚动条空间下沉到内层。  
主要工作是给 `.view-tabs-group-tabs` 提供独立的垂直空间策略，例如局部 scrollbar 高度例外、局部 padding/margin、必要时的 `scrollbar-gutter`/容器对齐修正，让横向滚动槽在视觉上更贴底。预期成果是第二行底部留白明显减少，且变化直接可见。执行顺序先仅调整内层横向 scrollbar 占位，再观察外层是否仍需额外高度。

阶段四：重新对齐搜索框与加号按钮。  
主要工作是修正 `.group-tab-search` 和 `.view-tabs-group-create` 在新垂直空间分配下的对齐方式，避免出现标签行下沉后搜索框和加号仍停留在旧基线的问题。预期成果是第二行三块内容仍在统一基线上，用户不会看到一边贴底一边悬空。执行顺序先搜索框，再右侧按钮，再整体 row 的 `align-items` / 自适应高度，并补搜索框展开态检查。

阶段五：验证与固化。  
主要工作是补样式约束测试、正式页截图核查和实际运行值读取，确认改变的是 `.view-tabs-group-tabs` 的滚动槽占位，而不是只在外层“数字变了但视觉不变”。预期成果是后续若有人重新把滚动槽责任抬回外层，测试能立刻暴露回退。执行顺序先静态测试，再构建，再正式页面实测。

### 3. 整体结构框架

本次实现采用“两层布局、三块内容”的结构框架：

1. **外层行容器：`.view-tabs-group-row`**
   - 负责第二行整体边框、左右内边距、搜索框区域、标签滚动区、创建按钮区域的三列布局。
   - 不再承担主要的滚动槽高度补偿职责。
   - 保留稳定基线：`padding-top: 8px`、`padding-bottom: 2px`、`.has-horizontal-scroll { min-height: 44px }` 作为结构方案前基线；除非内层方案验证后证明仍有必要，否则不主动继续压缩外层。
   - `has-horizontal-scroll` 状态在结构方案里只表达“当前内层可横向滚动”，不再驱动额外的外层高度策略。

2. **内层滚动容器：`.view-tabs-group-tabs`**
   - 负责标签项排列、横向滚动和滚动条本身的垂直占位。
   - 这是本方案的主调整对象，滚动条的视觉贴底与底部留白改善必须从这里产生。
   - 若全局 20px hit area 对此场景过重，应只在此容器局部降级，不影响其它滚动区域。
   - 局部例外的 contract 必须明确为：只修改 `.view-tabs-group-tabs` 的**横向 scrollbar 高度**，例如 `.view-tabs-group-tabs::-webkit-scrollbar { height: 12px|14px; }`；禁止修改 `width`、禁止修改通用 `thumb` 规则、禁止改成容器级 `scrollbar-width: auto/thin` 这种会影响双轴的做法。

3. **辅助控件区：搜索框与创建按钮**
   - 搜索框 `group-tab-search` 必须以第二行内容区为基线，而不是以旧的外层补偿高度为基线。
   - 创建按钮 `.view-tabs-group-create` 必须和标签滚动区保持统一垂直中心，不能因内层下沉而出现“漂浮”。

这套结构必须遵守两个硬边界：

- 第二行的横向滚动条占位真相只能收敛到 `.view-tabs-group-tabs`，禁止继续依赖 `.view-tabs-group-row` 的额外 `min-height` / `padding-bottom` 做主补偿。
- 外层 `.view-tabs-group-row` 只表达布局容器语义，内层 `.view-tabs-group-tabs` 只表达滚动内容语义；后续样式和测试必须按这条边界继续演进。

### 4. 职责映射表

| 区域 | 唯一职责 | 明确不负责 |
| --- | --- | --- |
| `.view-tabs-group-row` | 三列布局、边框、左右内边距、整体基线 | 主滚动槽占位、局部 scrollbar 尺寸决策 |
| `.view-tabs-group-tabs` | 标签流、横向滚动、横向 scrollbar 占位 | 搜索框/加号按钮对齐 |
| `.group-tab-search` | 搜索框自身尺寸与展开态输入布局 | 滚动槽高度补偿 |
| `.view-tabs-group-create` | 创建按钮的垂直对齐和点击区 | 滚动槽高度补偿 |

## 现状与问题归因

### 1. 已验证的事实

通过正式 `http://127.0.0.1:8787/` 页面和本地自动化实测，当前“第二行没有明显变化”的根因已经明确：

- 外层 `.view-tabs-group-row` 的收紧数值可以命中，但视觉收益很弱。
- 真正占据底部留白的是内层 `.view-tabs-group-tabs` 继承的全局横向 scrollbar 高度。
- 因为全局策略把横向 scrollbar hit area 放大到 `20px`，第二行这种高度较紧的标签流场景会直接感知到滚动槽体积。
- 只压外层 `padding-top` / `min-height`，会被内层滚动槽占位抵消，导致“样式值变了，但肉眼感觉没变”。
- 当前正式页基线已验证为：`.view-tabs-group-row.has-horizontal-scroll` 读取到 `min-height: 44px`、`padding-top: 8px`、`padding-bottom: 2px`；因此后续效果判断必须以正式页 computed style 为准，而不是仅以源码推断。

### 2. 为什么结构方案比外层微调更有效

外层微调本质上是在调整“包裹盒子有多高”，而不是在调整“滚动条到底占哪一层空间”。只要滚动槽仍由内层大尺寸 scrollbar 占据，外层再减几像素，效果都会被吞掉。

结构方案之所以更有效，是因为它直接改动问题源头：

- 把滚动条空间下沉到真正滚动的容器。
- 可以对 `.view-tabs-group-tabs` 单独做局部 scrollbar 例外。
- 搜索框和加号按钮可以围绕新的滚动基线重新对齐。

因此，从因果链条看，结构方案不是“更复杂但也许更好”，而是“更对症的唯一有效方向”。

## 推荐实现方向

### 方案 A：内层局部 scrollbar 例外 + 外层只保留轻量容器职责

这是本方案推荐的默认落地路径。

核心动作：

- 外层 `.view-tabs-group-row` 保持当前稳定基线，不再继续承接主滚动槽补偿。
- 内层 `.view-tabs-group-tabs` 承担横向滚动条的垂直空间。
- 对 `.view-tabs-group-tabs` 单独降低横向 scrollbar 占位高度，使其明显小于全局 `20px`。
- 搜索框与创建按钮围绕新的内层基线重新对齐。
- `has-horizontal-scroll` 只作为“内层当前存在横向滚动”的状态开关，不再驱动外层高度变化。

优点：

- 命中问题源头，视觉收益明显。
- 改动边界仍局限在 `ViewTabs` 第二行，不会波及表格主滚动区。
- 便于在正式页中快速验证变化是否有效。

风险：

- 需要同步修正搜索框与加号按钮的对齐，不是单点数值改动。
- 若只降 scrollbar 高度而不补对齐，可能出现标签贴底但左右控件浮在中线的问题。
- 搜索框展开态若未单独校验，可能出现默认态对齐正常、展开态输入框高度错位的问题。

### 方案 B：保持全局 scrollbar 尺寸不变，仅通过内层 padding / margin 重新布槽

这是保守备选方案，不推荐作为第一选择。

核心动作：

- 维持 `.view-tabs-group-tabs` 继承全局 20px scrollbar。
- 通过内层 padding-bottom、负 margin、对齐方式等局部技巧，让视觉上滚动条更贴底。

优点：

- 不需要引入局部 scrollbar 例外，策略一致性更强。

缺点：

- 仍然背着 20px 滚动槽体积，节省空间的上限有限。
- 容易出现“数字变化明显，视觉改善有限”的情况，和前一轮问题相同。

## 推荐结论

推荐采用 **方案 A：内层局部 scrollbar 例外 + 外层职责收敛**。

理由有三点：

- 它直接作用于当前已验证的根因：`.view-tabs-group-tabs` 的大 scrollbar 占位。
- 它允许第二行保持更紧凑的视觉密度，同时不牺牲表格主区的大 hit area。
- 它的风险是局部且可验证的，主要集中在搜索框和创建按钮对齐，而不是全局交互回归。

## 推荐实施步骤

推荐按“两步法”执行，而不是一开始同时大改三块布局：

### 第一步：只改内层横向 scrollbar 占位

- 保持 `.view-tabs-group-row` 的当前基线值不动。
- 只对 `.view-tabs-group-tabs::-webkit-scrollbar` 提供局部横向高度例外。
- 禁止改 `width`、禁止改通用 thumb、禁止先碰搜索框和加号按钮。

目标是先验证：第二行问题是否主要由内层 scrollbar 占位引起。

### 第二步：仅在第一步确实有效但对齐失衡时，再补搜索框和加号按钮

- 修正 `.group-tab-search` 的默认态和展开态垂直对齐。
- 修正 `.view-tabs-group-create` 的垂直居中。
- 必要时再微调 `.view-tabs-group-row` 的 `align-items` 或局部子项 `align-self`。

这样可以最大限度减少“多个变量同时变化导致无法判断收益来源”的问题。

## 实施前后的验证口径

结构方案实施后，验证不应再停留在“外层 `min-height` 有没有变化”，而应改成以下口径：

1. 正式 `8787` 页面中，`.view-tabs-group-tabs` 的 scrollbar 占位是否明显缩小。
2. 第二行底部白带/留白是否肉眼可见地减少。
3. 搜索框、标签流、创建按钮是否仍在一致的垂直基线上。
4. 搜索框展开态下，输入框是否仍与标签行处于合理基线，不出现额外高度跳变。
5. 窄窗口/长标签场景下，第二行是否仍稳定，不出现标签裁切、按钮悬空或搜索框错位。
6. 正式 `dist` 页面里，能否通过 computed style 直接读到新的 `.view-tabs-group-tabs` scrollbar 相关效果，而不是仅在源码里存在。
7. 当前目标环境为桌面运行时（Chromium/Electron）；本方案不要求 Firefox 等其它引擎达到完全一致的滚动条方向控制能力。

只有以上验证同时满足，才能说明结构方案真正有效。

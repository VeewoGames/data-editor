# Streamline 标签自动补全策略指南

## 概述

### 1. 总体目标和范围

本指南的目标是把 `micro-solid` 图标 tags 的主生产路线，从“持续访问官网详情页抓取”切换为“基于已采集官方 tags 的自动补全”。输入限定为当前仓库内已经稳定存在的三类信号：

- 图标名称：`name` / `slug` / `itemId`
- 图标图像：本地 `SVG` 资产
- 官方样本：manifest 中已成功回填的官方 `tags`

本指南覆盖：

- 为什么当前样本规模已经足以支撑自动补全
- 自动补全应该采用什么分层策略
- 如何把策略落成可批处理的脚本链路
- 如何评估质量并决定哪些结果自动接受、哪些结果进入人工复核
- 后续其他大批量标签任务如何复用这一套方法

本指南不覆盖：

- 继续把官网详情页抓取做成高吞吐主路径
- 重新设计 shared-view 搜索消费模型
- 引入新的远程真值源或依赖不确定的第三方站点接口

### 2. 各阶段任务概要

第一阶段：建立自动补全的输入面和真值面  
主要工作：把当前 manifest 中已采集的官方 tags 视为训练样本，把名称和 SVG 视为统一输入  
预期成果：得到一份稳定、可离线批处理的样本库  
执行顺序：先做

第二阶段：设计候选标签生成策略  
主要工作：组合名称检索、视觉近邻、受控词表约束、标签共现规则  
预期成果：得到一条不依赖官网访问的候选标签生成链  
执行顺序：在输入面确认后进行

第三阶段：设计质量门禁和回填流程  
主要工作：定义自动接受、人工复核、拒绝回填三档规则  
预期成果：让批处理既能跑得快，又不会把低质量自由发挥 tags 直接写入 manifest  
执行顺序：在候选生成策略之后

第四阶段：沉淀为可复用实施指南  
主要工作：固化目录结构、脚本职责、评估方法、运行顺序  
预期成果：后续其他图标族或其他标签任务可以直接套用  
执行顺序：最后

### 3. 整体结构框架

```text
已标注官方样本库
  -> 名称特征层
  -> 图像特征层
  -> 候选标签召回层
  -> 置信度融合层
  -> 自动接受 / 人工复核 / 拒绝
  -> manifest 回填
  -> registry 再生成与搜索验收
```

## 当前证据

基于 `artifacts/streamline-export/micro-solid-full.manifest.json` 当前状态：

- `total = 1904`
- `labeled = 859`
- `unlabeled = 1045`
- `uniqueTags = 3276`
- `avgTagsPerLabeled = 14.22`

这说明当前不是“样本太少，无法泛化”的问题，而是“已有样本足够多，但主策略仍然把官网访问当瓶颈”。

同时，当前未标注项都已有本地 SVG：

- `inbox.svg`
- `inbox-open.svg`
- `inbox-post.svg`
- `incognito-mode.svg`
- `indent-increase-1.svg`

因此，新策略不缺输入面。

## 为什么要换策略

当前官网详情页路线的问题不是正确性，而是吞吐量和稳定性：

- `MCP` 在大量样本上稳定表现为“推进 pending 指针，但 tags 为空”
- 浏览器 fallback 虽然能补洞，但只能低频、串行、小批次推进
- 对剩余 `1000+` 个空缺 tags 来说，这条路径过慢

所以官网抓取不应该再是主生产路线，而应退化为：

- 小规模真值补样路径
- 争议样本的人工复核路径
- 自动补全效果不达标时的局部校验路径

## 新主策略

### 策略总述

新主策略应改为：

1. 用现有 `859` 个官方已标注图标建立“标签示例库”
2. 对每个空缺图标，分别从“名称相似”和“图像相似”两侧召回近邻样本
3. 把近邻样本的官方 tags 作为候选池
4. 用规则和置信度融合，筛出最终建议 tags
5. 高置信度自动回填，中置信度进复核队列，低置信度保留空缺

### 为什么必须是“名称 + 图像”双路

只看名称不够，因为很多图标名过于通用：

- `inbox`
- `image-top`
- `bookmark`
- `warning`

只看图像也不够，因为图形近似常常跨语义：

- 布局类图标和媒体类图标可能共享相似的矩形/线段结构
- 表情类图标的外轮廓相似度很高，但情感语义差异大

因此必须双路：

- 名称路负责语义边界
- 图像路负责同名变体和近义图形 disambiguation

## 建议的自动补全流水线

### 第 1 层：受控词表

先从已标注样本中抽出受控词表，而不是允许模型自由生成任意 tag。

建议构建三层词表：

- 核心词表：直接来自现有 `3276` 个唯一官方 tags
- 家族词表：按 slug 前缀或语义簇聚合，例如 `image-*`、`inbox-*`、`chat-bubble-*`
- 共现词表：统计高频 tag 搭配，例如 `image + picture + photo + gallery`

规则：

- 默认只从受控词表中选 tag
- 只有当名称 token 明确出现、且近邻样本支持时，才允许新增词表外 tag

### 第 2 层：名称近邻召回

对每个待补图标，从名称侧召回近邻样本。

建议特征：

- `itemId`
- `slug`
- `name`
- slug token 切分
- 前缀家族，例如 `image-picture`

建议评分：

- 完全同前缀高权重
- 同主名词高权重，例如 `inbox-*`
- 同修饰词低一档，例如 `open / scan / top / right`

例如：

- `inbox-open` 的名称近邻应优先召回 `inbox`、`inbox-post`、`inbox-tray-*`
- `image-picture-landscape-2` 的名称近邻应优先召回 `image-picture-landscape-1`、`image-picture-gallery`

### 第 3 层：图像近邻召回

对每个待补图标，从图像侧召回近邻样本。

这里不要直接依赖网页截图，直接使用本地 SVG。

建议两种实现层级：

#### 方案 A：低成本稳定版

从 SVG 中抽结构特征：

- path 数量
- viewBox 比例
- 横竖主方向
- 圆/矩形/折线占比
- 连通块数量
- 关键几何 token

优点：

- 快
- 完全离线
- 易批处理

缺点：

- 只能抓粗结构

#### 方案 B：推荐主力版

把 SVG 渲染为统一尺寸 PNG，再做图像 embedding 或视觉相似检索。

建议流程：

- SVG -> 64x64 / 96x96 PNG
- 提取视觉 embedding
- 对已标注样本建近邻索引
- 对待补样本查 top-k 视觉近邻

优点：

- 对同名不同形、同形不同名的识别能力更强
- 更适合后续其他图标族复用

缺点：

- 需要额外渲染与索引步骤

结论：

- 长期主力应选方案 B
- 方案 A 可作为无模型依赖 fallback

### 第 4 层：候选标签融合

候选标签不是直接取最近一个样本，而是融合多邻居结果。

建议来源权重：

- 名称近邻 tags：高权重
- 图像近邻 tags：中高权重
- 名称 token 直出词：高权重但需白名单
- 全局高频泛词：低权重，必要时压制

建议去噪规则：

- 过泛 tag 降权：如 `symbol`、`design`、`interface`
- 只在单个邻居出现且与名称无关的 tag 降权
- 与当前家族前缀不一致的 tag 降权
- 视觉相似但语义偏移的 tag 需要至少两个来源支持才保留

### 第 5 层：自动接受阈值

建议输出三档：

- `auto_accept`
  - 名称近邻和图像近邻一致性高
  - 候选 tags 大多来自同语义簇
  - 没有明显离群 tag
- `review_required`
  - 候选结果大体合理，但包含 2-4 个不稳定 tag
  - 名称路和图像路部分冲突
- `reject_or_skip`
  - 候选高度分散
  - 主要命中泛词
  - 图像和名称都找不到足够近邻

### 当前第一版门禁实测

基于当前仓库内第一版离线实现，对 `micro-solid` 全量未标注项实跑后的结果为：

- `auto_accept = 29`
- `review_required = 647`
- `reject = 369`

这组分布比早期“1045 全部 auto_accept”的粗放结果更可用，说明门禁已经从“只要有近邻就放行”收紧为：

- 没有词汇证据的项直接拒绝
- 只有单个强近邻或存在语义冲突风险的项进入人工复核
- 只有至少两个强词汇近邻共同支持的项才自动接受

当前第一版的关键规则是：

- `SVG` 结构特征只能加分，不能单独驱动建议
- `open`、`circle`、`left`、`right` 一类泛修饰词不参与高权重语义判定
- 自动接受需要多个强词汇近邻共同支持，而不是单个近邻高分即可放行

因此，当前策略已经适合作为后续大批量标签任务的默认起点：

- 高吞吐的离线候选生产
- 受控的自动接受窗口
- 可追踪的人工复核池

## 推荐的实施顺序

### 阶段一：先做“离线评估版”，不要直接回填 manifest

先在已标注样本上做 holdout 验证：

1. 从当前 `859` 个已标注样本里抽一部分作为验证集
2. 暂时隐藏它们的 tags
3. 用自动补全策略重新预测
4. 和原官方 tags 对比

建议指标：

- tag precision
- tag recall
- top-k candidate recall
- auto_accept 覆盖率
- auto_accept 的人工 spot check 准确率

只有 holdout 表现稳定后，才开始回填真实空缺项。

### 阶段二：对真实空缺项生成“建议文件”

不要第一步就回写 manifest，先生成独立建议文件，例如：

`artifacts/streamline-export/micro-solid-generated-tags.json`

每项至少包含：

- `itemId`
- `name`
- `outputPath`
- `suggestedTags`
- `evidence`
  - `nameNeighbors`
  - `imageNeighbors`
  - `nameTokens`
- `confidence`
- `decision`
  - `auto_accept`
  - `review_required`
  - `reject`

### 阶段三：只把高置信度结果写回 manifest

建议默认只写回：

- `decision = auto_accept`
- 且 `confidence >= 阈值`

人工复核项保持在建议文件里，不直接污染 manifest。

### 阶段四：写回后继续沿现有 registry 闭环验收

写回 manifest 后，沿用现有链路：

- regenerate shared-view registry
- 检查 `searchText`
- 用 `ViewTabs.tsx` 的真实消费逻辑做 query 验收

这一步不需要新发明第二套搜索模型。

## 脚本职责建议

建议新策略拆成 5 个脚本，而不是一个大脚本：

1. `build-streamline-tag-knowledge.mjs`
   - 从已标注 manifest 建词表、共现关系、名称索引

2. `build-streamline-image-index.mjs`
   - 从 SVG 渲染或抽特征，建立图像近邻索引

3. `suggest-streamline-tags.mjs`
   - 对空缺样本生成建议 tags 和置信度

4. `evaluate-streamline-tag-suggestions.mjs`
   - 在 holdout 样本上做离线评估

5. `apply-streamline-tag-suggestions.mjs`
   - 只把高置信度建议写回 manifest

这样后续复用时，不需要重新理解整条链。

## 对当前项目的直接建议

### 推荐主策略

推荐采用：

- 名称近邻检索
- SVG 渲染后的视觉近邻检索
- 受控词表约束
- 高置信度自动写回

而不是：

- 继续把官网详情页抓取当主生产路线
- 直接让模型自由生成 tags 后无约束写回

### 为什么这个策略适合当前项目

因为当前项目已经具备：

- 足够大的官方已标注样本库
- 本地完整 SVG 资产
- 已有 manifest / registry / searchText 消费闭环

缺的不是更多网页访问，而是把这些现有资产组织成一条可离线复用的生成链。

## 后续复用规则

以后碰到其他大规模标签补全任务，优先按下面顺序判断：

1. 是否已有一批高质量人工或官方标签样本
2. 是否已有稳定名称字段
3. 是否已有本地图像资产
4. 是否已有下游消费面可做真实 query 验收

如果四项都满足，就优先用“样本检索 + 图像近邻 + 受控词表”的自动补全路线，而不是优先回到远程抓取。

## 当前建议结论

对 `micro-solid`，建议正式切换为：

- 官网抓取：降级为局部真值补样和争议复核路径
- 自动补全：升级为主生产路线

下一步最值得做的不是继续加大官网访问节奏，而是先落一版：

- `holdout 评估`
- `建议文件`
- `高置信度回填`

如果你同意，我下一步就不再继续跑批次抓取，而是直接开始把这套自动补全链路拆成具体脚本和评估入口。

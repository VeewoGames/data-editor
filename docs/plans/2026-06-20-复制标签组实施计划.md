# 复制标签组实施计划

## 方案概述

### 1. 总体目标和范围

本实施计划用于把“复制标签组”从方案文档推进到可落地代码。目标是在现有视图组菜单中新增“复制组”入口，并实现深复制整组的能力：复制组节点、复制组内所有子视图、复制当前可见的筛选/排序/查询状态、复制列布局配置，并在完成后自动切换到新组。

本次实施范围包含：

- `shared-view-structure.mjs` 新增组复制 helper。
- `App.tsx` 新增组复制动作编排。
- `ViewTabs.tsx` 新增“复制组”菜单项和事件透传。
- 定向测试补齐到结构层、状态层和交互层。

本次不包含：

- 批量复制多个组。
- 复制组后的批量重命名流程。
- 额外的回滚事务框架。

### 2. 各阶段任务概要

阶段一：补结构层 helper。  
主要工作是新增复制组纯函数，接收当前 resolved 组快照并输出新组 config、视图 id 映射和草稿复制结果；预期成果是不用触碰 UI 就能验证复制语义；执行顺序先测试后实现。

阶段二：补应用层动作编排。  
主要工作是在 `App.tsx` 中把“保存结构”“复制 viewDrafts”“复制 viewLayouts”“切换到新组”串起来；预期成果是点击一次菜单即可完成整组复制；执行顺序先 profile/local 双分支，再接成功/失败提示。

阶段三：补菜单入口。  
主要工作是在 `ViewTabs.tsx` 中新增“复制组”按钮，并接到 `App.tsx` handler；预期成果是用户能在组菜单里触发完整复制；执行顺序先 props，再菜单项。

阶段四：补测试与收尾。  
主要工作是补 `shared-view-structure.test.mjs`、`view-state.test.mjs`、`data-editor.spec.ts`；预期成果是复制来源、复制结果和激活行为具备自动化护栏；执行顺序先单测，再交互测试。

### 3. 整体结构框架

实现结构保持三层：

1. 结构层：负责“复制哪个组、复制哪些视图、生成哪些新 id、插入到哪里”。
2. 应用层：负责“保存 config、同步草稿、同步布局、更新 active 状态、给用户反馈”。
3. 展示层：负责“菜单入口和点击事件”。

执行时必须遵守两个边界：

- 复制来源以当前 resolved 组快照为准，不以旧 config 为准。
- 布局复制和内容草稿复制不能分离成两个独立用户动作，必须在同一个 handler 内连续执行。

这里再补三条强约束，避免实现时跑偏：

- `App.tsx` 必须先把目标组内每个视图与其 `viewDrafts` 合并成最终快照，再把这个快照传给 helper；helper 不负责自己推断 draft merge 规则。
- 生成 `nextConfig` 时，整份 collection 的 top-level items 必须以当前 resolved top-level 结构为基底，而不是以旧 `sharedViewsConfig.collections[collectionKey].items` 为基底。
- profile 模式下批量复制布局时，必须以上一轮 `copyViewLayoutState(...)` 的返回结果作为下一轮输入，不能每轮都从原始 profile 重新开始。

## 前置事实

### 已有可复用能力

- [src/view/shared-view-structure.mjs](C:/Code/data-editor/src/view/shared-view-structure.mjs) 已有 `createViewGroupConfig`、`createViewInGroupConfig`、`deleteViewGroupConfig` 等结构 helper。
- [src/App.tsx](C:/Code/data-editor/src/App.tsx) 已有 `handleDuplicateSharedView(viewId)`，能复用 `copyViewLayoutState(...)`。
- [src/components/ViewTabs.tsx](C:/Code/data-editor/src/components/ViewTabs.tsx) 已有组菜单、单视图“创建副本”入口和 `onDuplicateView` 透传模式。

### 需要新增的核心能力

- 组级 duplicate helper。
- 一次返回 `sourceViewId -> targetViewId` 映射。
- 一次返回应复制到新视图下的 `viewDrafts` 快照。
- 组菜单中的 `onDuplicateGroup` 事件链路。

### 当前实现对本次任务的直接约束

- `resolvedCollectionViews.topLevelItems` 反映的是当前结构层真相，但其中的 view 内容本身不自动合并 `viewDrafts`。
- `handleDuplicateSharedView(viewId)` 的现有做法已经证明：复制当前可见视图内容时，必须先 `mergeSharedViewWithDraft(...)`。
- `copyViewLayoutState(...)` 返回的是新的 profile/local state 结果，因此批量复制时必须串行累积，不能把每次调用都当成无副作用原地修改。

### Task 1：补结构层组复制 helper

涉及文件：

- [src/view/shared-view-structure.mjs](C:/Code/data-editor/src/view/shared-view-structure.mjs)
- [tests/shared-view-structure.test.mjs](C:/Code/data-editor/tests/shared-view-structure.test.mjs)

执行内容：

- 新增 `duplicateViewGroupConfig(...)`。
- 输入建议：
  - `sharedViewsConfig`
  - `collectionKey`
  - `groupId`
  - `resolvedTopLevelItems`
  - `resolvedGroupSnapshot`
- 输出建议：
  - `config`
  - `group`
  - `firstViewId`
  - `sourceToTargetViewIdMap`

实现要求：

- 从 `resolvedTopLevelItems` 中定位目标组，而不是从原始 config 中直接取组。
- 复制来源使用 `resolvedGroupSnapshot`，保证组内每个视图的 `query / filters / sorts` 都已经是“当前界面可见版本”。
- 生成 `nextConfig` 时，以当前 `resolvedTopLevelItems` 为整份 collection 结构基底，再把复制出的新组插入原组后方。
- 新组插在原组后面。
- 新组名按 `原组名 副本` 做唯一化。
- 子视图名保持原名。
- 子视图 id 全量重新生成。

验收标准：

- helper 能在不依赖 React 状态的情况下完整产出复制结果。
- 返回映射数量等于源组子视图数量。
- 返回的 `firstViewId` 等于新组数组中的第一个视图 id。
- 返回的新组结构已经包含正确的 `query / filters / sorts` 快照，不依赖 helper 内部再做 draft merge。

### Task 3：在 App 层新增复制组动作

涉及文件：

- [src/App.tsx](C:/Code/data-editor/src/App.tsx)
- 可能需要少量调整 [src/view-state-storage.mjs](C:/Code/data-editor/src/view-state-storage.mjs) 复用现有布局复制函数

执行内容：

- 新增 `handleDuplicateSharedViewGroup(groupId: string)`。
- 在调用 helper 之前，先由 `App.tsx` 构造两个输入：
  - `resolvedTopLevelItems`
  - `resolvedGroupSnapshot`

`resolvedGroupSnapshot` 的构造规则必须写死：

1. 从 `resolvedCollectionViews.topLevelItems` 找到目标组。
2. 对组内每个 view：
   - 读取 `draftSource.viewDrafts?.[activeCollectionKey]?.[view.id]`
   - 调用现有 `mergeSharedViewWithDraft(...)`
3. 产出“当前界面实际看到的组快照”后再传入 helper。

动作顺序必须固定：

1. `setCommandSaving(true)`，清状态。
2. 先构造 `resolvedGroupSnapshot`。
3. 调 helper 生成 `nextConfig` 和 `sourceToTargetViewIdMap`。
3. `saveSharedViews(nextConfig, activeProjectId)`。
4. `setSharedViewsConfig(nextConfig)`。
5. 基于 `sourceToTargetViewIdMap` 和当前 collection 的 `viewDrafts`，在 `App.tsx` 中显式构造 `duplicatedViewDraftsById`：
   - 只有源视图存在 draft 时才复制
   - target key 使用目标 view id
6. 更新 shared draft state：
   - 把 `duplicatedViewDraftsById` 合入当前 collection 的 `viewDrafts`
   - `viewOrderDrafts` 不新增、不改写，保持当前 state
   - 更新 `lastActiveViews[collectionKey] = firstViewId`
7. 逐个执行布局复制：
   - profile 模式：在 `mutateSelectedViewProfile(...)` 内循环调用 `copyViewLayoutState({ mode: "profile", ... })`
   - local 模式：循环调用 `copyViewLayoutState({ mode: "local", ... })`
8. profile 模式下必须采用串行累积：
   - 第一轮以上一时刻的 draft profile 为输入
   - 后续每一轮都以上一轮 `copyResult.profile` 为下一轮输入
   - 最终一次性回写 `draftProfile.viewLayouts / draftProfile.collections`
9. 更新 page context：
   - `expandedGroupId = result.group.id`
   - `lastActiveViewIdByGroupId[result.group.id] = firstViewId`
10. 设置成功提示 `已复制视图组`
11. 失败时根据阶段区分提示：
   - 如果 config 保存失败：直接报错
   - 如果 config 已保存但布局复制阶段失败：提示“视图组已复制，但部分布局复制失败”
12. 即使发生“结构已保存但布局复制失败”，仍保留切换到新组的行为，因为结构已经真实存在，不能把用户留在旧组造成错觉。
13. `finally` 中恢复 `commandSaving`

验收标准：

- 点击一次可以完成结构复制、草稿复制、布局复制和激活切换。
- 复制后当前 active view 是新组第一个子视图。
- 页面第二行会自动展开到新组。
- 如果源组存在未保存内容草稿，复制后的新组看到的是草稿态，而不是旧落盘态。
- 如果源组存在未保存结构顺序，复制后的新组顺序与当前界面一致。

## 风险控制

### 风险 1：复制来源不对

控制方式：

- 强制 helper 从 `topLevelItems` 复制，而不是从 `sharedViewsConfig.collections[collectionKey].items` 直接复制。
- 强制 `App.tsx` 在调用 helper 前先构造 merge-draft 后的 `resolvedGroupSnapshot`。

### 风险 2：只复制了结构，没复制草稿

控制方式：

- `App.tsx` 基于 `sourceToTargetViewIdMap` 显式构造 `duplicatedViewDraftsById`。
- `App.tsx` 在更新 `updateSharedViewDraftState(...)` 时一并合入。

### 风险 3：只复制了内容，没复制布局

控制方式：

- 逐个走 `copyViewLayoutState(...)`，不要自己手写布局对象拷贝。
- profile 模式下按累积结果串行复制，不要每轮从原始 profile 重新开始。

### 风险 4：半完成状态误报成功

控制方式：

- 对“config 已保存但布局复制失败”的场景输出单独提示。
- 后续如果用户反馈需要更强一致性，再单独评估回滚或批量事务化，不在本轮扩 scope。

# 共享视图图标收藏与来源分组方案

## 方案概述

### 1. 总体目标和范围

本方案是 `data-editor` 共享视图图标体系的上位方案文档，负责定义长期目标、信息架构主轴、稳定数据边界，以及不同下位文档的职责分工。

这份文档不再承担具体执行真相源职责。Phase A 的执行收口、Tabler 的正式供给链、以及已经接受的运行时 contract，分别以下列文档为准：

- [共享视图图标 Phase A 实施计划](C:/Code/data-editor/docs/plans/2026-06-24-共享视图图标Phase-A实施计划.md)
- [Tabler 图标接入方案](C:/Code/data-editor/docs/plans/2026-06-24-Tabler图标接入方案.md)
- [.claw/truth/adr/shared-view-icon-phase-a.md](C:/Code/data-editor/.claw/truth/adr/shared-view-icon-phase-a.md)
- [.claw/truth/shared-view-tabler-icon-supply-chain.md](C:/Code/data-editor/.claw/truth/shared-view-tabler-icon-supply-chain.md)

本方案当前负责回答四个问题：

1. 共享视图图标体系最终想收敛成什么样的导航与分层结构。
2. 收藏、搜索、来源分组、图标包加载分别属于哪一层职责。
3. 哪些边界已经是 accepted contract，不应再被后续实现轮次随意重开。
4. 哪些后续事项仍然存在，但应进入下位实施文档或独立执行轮次，而不是继续堆在总方案里。

本方案范围包括：

- 顶层导航目标
- 收藏归属与存储边界
- 图标来源与运行时分层
- 图标包加载 / 卸载的长期约束
- 文档层级划分与后续治理边界

本方案不包含：

- 具体实现步骤、文件改动清单、脚本命令
- 某一轮 Phase A 或 Tabler 执行的逐项验收细节
- 阶段性回归日志、临时验证噪声、一次性排障记录

### 2. 各阶段任务概要

#### 阶段一：上位信息架构收敛

主要工作：

- 把 shared view icon picker 的导航主轴收敛为“来源 + 风格 + 用户偏好”
- 固定收藏、最近、来源分组、兼容分组之间的职责边界

预期成果：

- 一套长期稳定的信息架构，不再把语义组、来源组、实验组混在一起

执行顺序：

1. 先固定偏好分组与来源分组职责
2. 再固定兼容层与过渡层边界

#### 阶段二：数据与运行时 contract 收敛

主要工作：

- 固定收藏归属 `UserViewProfile`
- 固定 `metadata registry / runtime asset registry` 分层
- 固定 pack 加载与保护边界

预期成果：

- 后续实现只在既定 contract 上追加，不回退到单一 eager registry 或双轨存储

执行顺序：

1. 先固定数据归属
2. 再固定运行时分层
3. 最后固定 pack 行为

#### 阶段三：来源扩展与兼容治理

主要工作：

- 把 Streamline、Tabler、Legacy 兼容池放回同一体系中理解
- 明确哪些属于正式来源，哪些只是过渡兼容

预期成果：

- 未来继续扩来源时，不需要重新定义整套 picker 结构

执行顺序：

1. 先固定正式来源
2. 再处理兼容来源的长期去留

#### 阶段四：文档与执行分层治理

主要工作：

- 把总方案、实施计划、truth、ADR 的职责分开
- 避免下位执行细节反向污染上位方案

预期成果：

- 总方案保持可读、可复用、可继续作为后续轮次入口

执行顺序：

1. 上位方案只保留长期结构
2. 下位文档承接具体执行与 accepted truth

### 3. 整体结构框架

共享视图图标体系长期按五层理解：

1. 用户偏好层
   - `最近`
   - `收藏`

2. 正式来源层
   - `Micro S`
   - `Core S`
   - `Tabler S`
   - `Micro L`
   - `Tabler L`

3. 兼容过渡层
   - `Legacy` 或等价兼容池
   - 用于承接尚未迁移完成的 builtin / legacy icon

4. 元数据层
   - 保存 icon id、group、packId、searchText、outputPath
   - 支撑搜索、分组、收藏、最近、兼容判断

5. 运行时资产层
   - 只保存当前已加载 pack 的真实渲染组件
   - 支撑 fallback、按包加载、卸载保护、刷新恢复

## 一、文档职责与当前边界

### 1. 本文档是总方案，不是执行清单

当前共享视图图标体系已经经历过至少两轮正式执行收口：

- Phase A：收藏归属、分层 registry、基础 pack 与会话态加载
- Tabler：正式 tagged snapshot 来源、generated registry、受保护 pack 合同

- “下一步需要在哪个文件新增什么字段”

- “某个脚本建议怎样命名”
- “某轮回归具体跑了哪些命令”

这些内容若仍有价值，应以下位文档为准，而不是回写进总方案。

### 2. 当前 accepted contract 已经拆到下位文档

后续若要判断某条规则是否已经接受，应优先看：

- 实施计划：执行范围与阶段边界
- truth：可复用事实与真实链路
- ADR：长期决策与后果

而不是把总方案里的旧措辞当成最新真相源。

### 3. 本文档当前的正确职责

本文档保留：

- 长期目标
- 稳定分层
- 文档层级

## 二、已接受的长期合同

### 1. 收藏归属用户 profile，不走 localStorage 兜底

这条边界已经接受：

- 收藏状态属于 `UserViewProfile.favoriteSharedViewIconIds`
- 收藏不进入 shared view config
- 无 profile 时收藏入口禁用
- 不引入 localStorage 兼容写入

这意味着：

- “当前视图选了什么图标”是共享配置事实
- “我收藏了哪些图标”是用户偏好事实

### 2. 图标体系长期采用 metadata / runtime 双层

这条边界已经接受：

- `metadata registry` 负责图标可搜索、可分组、可收藏、可进入最近
- `runtime asset registry` 只负责当前已加载图标的真实渲染

因此：

- 搜索可见不等于当前可渲染
- 未加载图标必须走统一 fallback / 占位路径
- 后续做性能治理、包管理或来源扩展，都要在这两层上追加，而不是回到单一 eager registry

### 3. pack 加载状态是会话态，`Base` 永远保底

当前 accepted contract：

- `Base` 默认加载且不可卸载
- 非 `Base` pack 默认按需加载
- pack 加载状态不写 profile、不写 localStorage、不写 shared config
- 刷新后回到默认保底集合

这条规则的意义是：

- 保证基础图标始终可用
- 让运行时成本与长期偏好解耦
- 后续性能评估可以基于同一默认路径进行

### 4. 被当前共享视图结构引用的 pack 必须自动装载且不可卸载

当前 accepted contract：

- 当前结构正在使用的 icon pack 会被自动装载
- 包管理 UI 中这类 pack 显示为“已使用”
- “已使用”状态禁止卸载

这不是通用锁系统，而是最小保护边界，用于避免：

- 已保存图标在 reload 后掉回 fallback
- 用户误卸载当前视图正在引用的 pack

### 5. Tabler 已经是正式来源，不再是零散手工补丁

当前 accepted contract：

- `Tabler S / Tabler L` 属于正式来源层
- 正式来源固定为官方 GitHub tagged snapshot
- 本地真相源固定为 `vendor/tabler-svg/filled` 与 `vendor/tabler-svg/outline`
- 正式 generated truth 固定为 `src/generated/tabler-shared-view-icons.*`

因此，本方案不再把“是否要正式接入 Tabler”当成待决策事项。

### 6. 搜索 contract 已经固定

当前 accepted contract：

- 默认只在当前标签页内搜索
- 勾选全局后才跨全部图标元数据搜索
- 全局搜索勾选状态不持久化
- 关闭并重新打开 picker 后回到默认未勾选

这条规则已经不属于开放设计问题。

## 三、来源分组与兼容层的长期语义

### 1. 正式目标导航

长期目标态的主导航顺序仍然是：

1. `最近`
2. `收藏`
3. `Micro S`
4. `Core S`
5. `Tabler S`
6. `Micro L`
7. `Tabler L`

这是上位信息架构目标，不依赖某一轮具体实施节奏。

### 2. `Legacy` 属于兼容层，不属于长期主导航目标

`Legacy` 的职责是兼容过渡，不是长期正式来源。

它存在的原因是：

- 当前仍有 builtin / legacy icon 需要稳定渲染与再次选择
- 不能让已有 shared view 配置在迁移前失去来源归宿

因此，`Legacy` 可以在当前实现中继续存在，但它的文档语义应始终保持为：

- 兼容层
- 过渡层
- 非最终主导航目标

### 3. 正式来源扩展要走同一模式

未来若继续接入新的正式图标来源，必须继续沿用：

- 可审计静态资产
- generated metadata truth
- packId 归属
- metadata / runtime 双层消费

而不是回到手工 import 一批运行时组件充当全集定义。

## 四、当前剩余治理议题

以下议题仍然存在，但不应混回总方案的执行细节里：

### 1. `Legacy` 的长期消化路径

仍需继续回答：

- 哪些 legacy icon 需要长期保留
- 哪些可以迁移到正式来源
- 何时可以把 `Legacy` 从当前兼容可见状态进一步收紧

这属于后续兼容治理轮次，不属于本总方案中的开放实现步骤。

### 2. 性能指标的持续复测

“默认不全量加载 + 双层 registry”已经是 accepted contract，但性能是否持续受控，仍需要在后续实现轮次里按统一口径复测。

这类内容应沉淀在：

- 实施计划
- truth
- 性能产物

而不是继续在总方案里维护一整套阶段性采集脚本说明。

### 3. 更细的包管理诊断能力

例如：

- 更完整的卸载原因提示
- 当前引用 pack 的统计与定位
- 兼容池引用诊断

这些都是可能的后续增强，但不影响现有上位结构是否成立。

## 六、修订后的结论

经过本轮复核，这份文档的正确定位已经收敛为：

- 它继续存在，而且有价值
- 但它是“共享视图图标体系总方案”，不是“下一轮实施清单”
- 它必须让位给已经 accepted 的 Phase A / Tabler / truth / ADR 文档，而不是重新覆盖它们

后续继续维护这份文档时，应优先检查：

1. 这段内容是在描述长期结构，还是在重复某轮执行细节。
2. 这条规则是不是已经在 truth / ADR 中被接受。
3. 这条内容是否仍属于总方案层，而不是应迁移到实施计划或事实沉淀层。

只要保持这三条检查，这份文档就还能继续作为共享视图图标体系的上位入口使用。

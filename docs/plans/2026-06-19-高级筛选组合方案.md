# 高级筛选组合方案

## 概述

### 1. 总体目标和范围

本方案的目标是在 `data-editor` 现有 shared view / local draft / filter chips 机制上，引入一版与 Notion 高级筛选组合语义对齐的筛选能力。重点不是第一版就做像素级复刻，而是先把“真实生效的筛选结构、编辑入口、保存链路、执行结果”做正确。

本方案覆盖以下范围：

- 顶部普通筛选 chips 与唯一高级筛选入口 chip 的组合语义
- 高级筛选树的数据模型、归一化、执行器与持久化结构
- 高级筛选面板的交互和组件拆分
- shared view / local draft / reload / save for everyone 的一致性
- 回归测试、兼容迁移和实现风险

本方案不包含以下范围：

- 像素级 1:1 复刻 Notion 视觉样式
- 多个高级筛选入口 chip
- 高级树内部拖拽排序
- 把高级树内的 rule 移回顶部普通 chips
- 权限、协作冲突检测、审计记录

### 2. 各阶段任务概要

第一阶段：模型与执行层收口  
主要工作是升级 `filters` 数据模型、补充递归执行器、兼容旧格式归一化。预期成果是现有 view 能无损迁移，新旧筛选都能被同一套执行逻辑正确求值。

第二阶段：交互与状态操作收口  
主要工作是保留顶部普通 chips，同时引入唯一高级筛选入口 chip 与高级面板，并补齐“合并到高级筛选中”“添加规则/分组”“与/或切换”“删除/复制/转换成分组”等操作。预期成果是用户可以完整编辑高级筛选结构，且语义与顶部 chips 一致。

第三阶段：持久化、回归与文档收口  
主要工作是打通 shared view draft / save / reload，补全测试和文档。预期成果是高级筛选在保存、刷新、切换 view 后行为稳定，不破坏现有 shared views 语义。

### 3. 整体结构框架

整体结构采用“顶层普通规则 + 唯一高级筛选树”的双层模型：

```text
top_level_rule_1
AND top_level_rule_2
AND ...
AND advanced_filter_root
```

其中：

- 顶部普通 chips 都是真实生效的筛选条件
- 顶部普通 chips 之间固定是 `AND`
- 顶部高级筛选入口 chip 只能有一个
- 进入高级筛选树的规则不再显示在顶部
- 高级筛选树内部支持 `AND/OR`、分组和最多 3 层嵌套
- 整体结果是“所有顶部规则成立”且“高级筛选树成立”

## 一、需求语义收敛

### 1. 顶部 chips 不是摘要，而是运行态筛选条件

顶部普通 filter chips 不是单纯展示，它们本身就是当前 view 的生效条件。用户点击 chip 编辑、删除，或者通过菜单把它“合并到高级筛选中”，都会直接改变当前 view 的筛选结果。

这意味着第一版不能把顶部 chips 理解成高级筛选树的扁平摘要，也不能在进入高级筛选后把顶部 chips 退化成只读说明。

### 2. 顶部普通 chips 的语义固定为 AND

只要某个筛选条件还留在顶部，它就属于顶层普通规则集合。顶层普通规则之间没有可切换连接词，始终是固定 `AND`。

例如顶部存在：

- `owner: player`
- `dev_status: 不是 已废弃/草稿`

其真实含义就是：

```text
owner = player
AND dev_status != [已废弃, 草稿]
```

### 3. 高级筛选入口 chip 只能有一个

顶部只能存在一个高级筛选入口 chip。它代表唯一一棵高级筛选树，而不是多个并列高级组。

这条约束的意义是：

- 避免顶层同时出现多个高级组带来的额外复杂度
- 让交互结构与数据模型更容易收口
- 更符合“先把主流程做对”的第一版目标

### 4. 合并到高级筛选中的规则不再显示在顶部

当某个顶部普通 rule 执行 `合并到高级筛选中` 后：

- 它从顶部普通 chips 区移除
- 它被加入唯一高级筛选树
- 它不再以普通 chip 的形式重复显示在顶部

顶部此时只显示：

- 剩余的普通 chips
- 唯一高级筛选入口 chip

### 5. 高级筛选树本身也是真实生效条件

高级筛选树不是草稿区，也不是“待确认条件池”。只要高级筛选树存在，它就是当前 view 的真实生效条件。

因此，整个筛选系统的真实执行语义是：

```text
match(row) =
  matchAllTopLevelRules(row)
  AND
  matchAdvancedRoot(row)
```

## 二、现状约束与改造目标

### 1. 当前实现的限制

当前仓库中的筛选模型和执行器都只支持单层 `AND`：

- [src/api/client.ts](/C:/Code/data-editor/src/api/client.ts:124) 中 `FilterGroup` 固定为单层 `{ op: "and"; rules: FilterRule[] }`
- [src/view/filtering.mjs](/C:/Code/data-editor/src/view/filtering.mjs:16) 中 `applyViewFilters()` 通过 `rules.every(...)` 执行
- [src/view/shared-view-normalize.mjs](/C:/Code/data-editor/src/view/shared-view-normalize.mjs:96) 中只会归一化平铺 `rules`
- [src/components/ViewFilterBar.tsx](/C:/Code/data-editor/src/components/ViewFilterBar.tsx:62) 只负责平铺 chips
- [src/components/filters/FilterActionMenu.tsx](/C:/Code/data-editor/src/components/filters/FilterActionMenu.tsx:4) 目前只有删除操作

### 2. 当前实现不能直接承载的能力

要支持本方案，当前实现至少缺少：

- 顶层普通规则与高级树并存的表达能力
- 高级树的递归节点结构
- `AND/OR` group 的递归执行
- “合并到高级筛选中”的状态迁移
- 高级树编辑 UI

### 3. 改造原则

本次改造应遵循以下原则：

- 不做防御性兼容旧交互，旧单层 `AND` 直接升格到新模型
- 不引入多个高级入口 chip
- 顶部普通 chips 与高级树共享同一份 `filters` 状态
- 旧 view 的结果必须在迁移后保持不变
- 必须解除“同字段只能有一条 rule”的现有限制

## 三、数据模型设计

### 1. 目标模型

建议把 `CollectionView.filters` 升级为如下结构：

```ts
type FilterOperator =
  | "is"
  | "is_not"
  | "contains"
  | "does_not_contain"
  | "is_empty"
  | "is_not_empty";

type FilterRuleNode = {
  kind: "rule";
  id: string;
  field: string;
  operator: FilterOperator;
  value?: unknown;
};

type FilterGroupNode = {
  kind: "group";
  id: string;
  op: "and" | "or";
  children: FilterNode[];
};

type FilterNode = FilterRuleNode | FilterGroupNode;

type ViewFilters = {
  topLevelRules: FilterRuleNode[];
  advancedRoot: FilterGroupNode | null;
};
```

### 2. 设计理由

该模型直接表达了产品语义：

- `topLevelRules` 对应顶部普通 chips
- `advancedRoot` 对应顶部唯一高级筛选入口 chip
- 顶层普通 rules 固定 `AND`
- 高级树内部可递归表达 `AND/OR`

这样可以避免两类常见错误建模：

- 把顶部 chips 当成高级树的摘要
- 把多个高级组提升为顶层并列实体

### 3. 节点约束

第一版建议加上以下约束：

- `advancedRoot` 最多 3 层嵌套，与 Notion 对齐
- `advancedRoot` 为空时记为 `null`
- `topLevelRules` 内只允许 `rule`，不允许顶层 group
- `advancedRoot` 必须是 `group`
- 每个节点 `id` 唯一

深度计数规则明确如下：

- `advancedRoot` 记作第 1 层
- `advancedRoot` 的直接子 group 记作第 2 层
- 第 2 层 group 的子 group 记作第 3 层
- 超过第 3 层时禁止继续创建子 group

## 四、执行语义设计

### 1. 顶层执行公式

对任意一行 `row`，筛选结果定义为：

```ts
match(row) =
  topLevelRules.every((rule) => matchRule(row, rule))
  && (advancedRoot ? matchGroup(row, advancedRoot) : true);
```

### 2. group 递归语义

```ts
function matchGroup(row, group) {
  if (group.op === "and") return group.children.every((child) => matchNode(row, child));
  return group.children.some((child) => matchNode(row, child));
}
```

### 3. rule 语义

`rule` 的字段判断应尽量复用当前 `matchesFilterRule()` 的逻辑，包括：

- 文本 `contains / does_not_contain`
- `is / is_not`
- `is_empty / is_not_empty`
- `Select / Multi-select / Relation / Checkbox` 等字段差异

第一版不应顺手修改 rule 的比较语义，只扩展“组合方式”。

### 4. 空树与空组处理

建议规则如下：

- `advancedRoot === null`：视为 `true`
- 编辑中的空 group 可以暂时存在
- 规范化或保存前空 group 要被裁掉
- 若裁掉后 `advancedRoot` 无 children，则写回 `null`
- 若用户尝试在第 3 层 group 下继续创建子 group，UI 直接禁用该动作或明确拒绝，不做自动裁剪

## 五、交互设计

### 1. 顶部筛选栏结构

顶部筛选栏应包含：

- `+ 筛选`
- 顶部普通 filter chips
- 唯一高级筛选入口 chip（存在时）
- 右侧的 `重置` / `为所有人保存`

### 2. 顶部普通 chips

顶部普通 chips 行为如下：

- 点击 chip：编辑这条 rule
- 点击 `...`：
  - `删除筛选`
  - `合并到高级筛选中`

第一版必须支持同字段出现多条 rule，不再沿用当前“一个字段只能添加一次”的限制。

例如以下结构应被视为合法：

- `skill_category 是 general`
- `skill_category 不是 summon`

它们可以同时存在于顶部，或同时存在于高级树内。

### 3. 高级筛选入口 chip

顶部高级筛选入口 chip 只能有一个：

- 它代表唯一的 `advancedRoot`
- 点击后打开高级筛选面板
- 文案第一版建议显示规则总数，例如：
  - `1 条规则`
  - `3 条规则`

第一版不要求把高级树完整摘要压缩到 chip 文案里。

### 4. 合并到高级筛选中

执行 `合并到高级筛选中` 时：

- 若 `advancedRoot === null`
  - 创建默认 `advancedRoot`
  - 默认 `op = "and"`
  - 将当前 rule 移入 `advancedRoot.children`
- 若 `advancedRoot !== null`
  - 将当前 rule 加入 `advancedRoot.children` 末尾
- 当前 rule 从 `topLevelRules` 中删除
- 自动打开高级筛选面板，便于用户继续编辑

注意：这里是“移动”，不是“复制”。

### 5. 高级筛选面板

高级筛选面板是唯一高级树的主编辑器，负责：

- 渲染 `advancedRoot`
- `添加筛选规则`
- `添加筛选分组`
- `AND/OR` 切换
- 删除节点
- 复制节点
- rule 转换成 group

其组织方式应为树状列表，而不是继续复用当前小型 chip popover。

### 6. group 内部交互

每个 group 容器应包含：

- 当前 group 的 `op` 选择器
- children 列表
- `添加筛选规则`
- `添加筛选分组`
- group 级操作菜单

group 内第一条 child 前显示“当”，后续 child 前显示可切换的连接语义入口，与 Notion 交互接近即可，不要求像素级一致。

插入规则明确如下：

- 点击某个 group 内的 `添加筛选规则`：新 rule 加到该 group 的 `children` 末尾
- 点击某个 group 内的 `添加筛选分组`：新 group 加到该 group 的 `children` 末尾
- 当 `advancedRoot === null` 且首次执行 `合并到高级筛选中` 时：被移动的 rule 作为 `advancedRoot.children[0]`

### 7. rule 行交互

每条 rule 行应包含：

- 连接词位置
- 字段选择器
- operator 选择器
- value 编辑区
- `...` 菜单

第一版 `...` 建议支持：

- `移除`
- `创建副本`
- `转换成分组`

节点操作语义明确如下：

- `创建副本`
  - 复制当前节点
  - 副本插入在原节点后面
  - 副本必须生成新的唯一 `id`
- `转换成分组`
  - 仅对 rule 生效
  - 用当前 rule 原地替换成一个新 group
  - 新 group 默认继承父 group 当前的 `op`
  - 原 rule 成为该 group 的第一个 child
- `移除`
  - 从当前父 group 的 `children` 中删除该节点
  - 若删除后父 group 变空，则按空 group 规则继续裁剪

第一版不支持把高级树内的 rule 直接移回顶部普通 chips。若用户要恢复为顶部普通 chip，只能删除后重新从顶部添加。

## 六、状态操作 API 设计

为了避免在 React 组件中内联递归修改逻辑，建议把树操作抽到单独 helper 中。

### 1. 顶层操作

建议提供以下 helper：

- `createEmptyViewFilters()`
- `addTopLevelRule(filters, rule)`
- `updateTopLevelRule(filters, ruleId, updater)`
- `removeTopLevelRule(filters, ruleId)`
- `mergeTopLevelRuleIntoAdvancedRoot(filters, ruleId)`

### 2. 高级树操作

建议提供以下 helper：

- `createAdvancedRoot()`
- `addRuleToGroup(filters, groupId, rule)`
- `addGroupToGroup(filters, groupId, group)`
- `updateGroupOp(filters, groupId, op)`
- `removeNodeFromAdvancedRoot(filters, nodeId)`
- `duplicateNodeInAdvancedRoot(filters, nodeId)`
- `convertRuleToGroup(filters, ruleId)`
- `replaceNodeInAdvancedRoot(filters, nodeId, nextNode)`
- `canCreateChildGroup(filters, groupId)`

### 3. 通用查询操作

建议提供以下 helper：

- `findNodeById(filters, nodeId)`
- `countAdvancedRules(filters)`
- `getAdvancedDepth(filters)`
- `isAdvancedRootEmpty(filters)`

### 4. 规范化操作

建议提供以下 helper：

- `normalizeViewFilters(value)`
- `pruneEmptyGroups(filters)`
- `enforceMaxAdvancedDepth(filters, maxDepth = 3)`
- `dedupeFilterNodeIds(filters)`

这些 helper 还应提供稳定的插入与计数约定：

- 新增节点默认插入目标 group 的 `children` 末尾
- 复制节点默认插入原节点后
- 规则总数只统计 `rule` 节点，不统计 `group`

这些 helper 应放在纯逻辑文件里，供：

- UI 组件
- normalize 逻辑
- 测试
- 执行器

共同使用。

## 八、持久化与兼容迁移

### 1. 持久化位置

新模型仍然挂在 `CollectionView.filters` 上，不新开独立字段。原因是：

- shared view / local draft 现有结构无需引入平行配置
- 保存、重置、reload 的责任边界更清晰
- 旧逻辑更容易迁移

### 2. 旧格式兼容策略

旧格式：

```ts
{ op: "and"; rules: FilterRule[] }
```

在 normalize 时自动升格为：

```ts
{
  topLevelRules: old.rules.map(rule => ({ kind: "rule", ...rule })),
  advancedRoot: null,
}
```

这样可以保证：

- 旧 view 的行为不变
- 不会突然生成一个高级组
- 用户只有明确执行“合并到高级筛选中”后才出现高级入口 chip

### 3. 规范化规则

`normalizeViewFilters()` 建议保证：

- 过滤非法节点
- 递归裁掉空 group
- group 深度不超过 3
- `topLevelRules` 只保留 rule
- `advancedRoot` 非空时必须为 group
- 旧 `{ op: "and"; rules }` 始终升格为仅包含 `topLevelRules` 的新结构
- 同字段多条 rule 合法，不做按字段名去重

### 4. draft / save / reload 语义

高级筛选与现有 shared view 机制保持一致：

- 当前修改先进入个人 draft
- 点击 `重置` 丢弃当前 draft
- 点击 `为所有人保存` 覆盖 shared view 配置
- reload / 切换 view / 切换文件后恢复 draft 或 shared view 的最终结构

第一版不改变现有 shared views 的协作边界。

另外，以下状态链路必须与新模型一起改：

- dirty 判定
- `重置` 是否显示
- `为所有人保存` 是否显示
- `Save for everyone` 后是否仍保留 draft
- active view reload 后的 filter 结构恢复

这些链路不能继续依赖旧 `FilterGroup` 的 `op + rules[]` 判等。

- 旧 `{ op: "and"; rules }` 正常升格
- 非法 node 被忽略
- 空 `advancedRoot` 归一化为 `null`
- 顶部 rules 中不会混入 group
- 同字段多条 rule 不会被误去重

### 3. 状态操作测试

至少覆盖：

- `合并到高级筛选中` 会把 rule 从顶部移到高级树
- 首次合并会创建 `advancedRoot`
- 高级树删空后高级入口 chip 消失
- rule 转 group 后结构合法
- 复制节点后 `id` 不冲突
- 复制节点后插入在原节点后
- 第 3 层 group 下不能继续创建子 group
- 高级树内 rule 不支持直接移回顶部普通 chips

### 4. UI / E2E 测试

至少覆盖：

- 顶部普通 chip 与高级入口 chip 同时生效
- 已进入高级树的 rule 不再显示在顶部
- `AND/OR` 切换后结果变化正确
- `为所有人保存` 后 reload 一致
- 旧 view 打开后仍能编辑和保存
- 同字段多条规则可以创建、保存、reload 后仍存在
- dirty 圆点、`重置`、`为所有人保存` 的显示与新模型一致

## 十、风险与取舍

### 1. 最大风险

最大风险不是样式，而是状态模型分叉。如果继续让“顶部 quick filter”和“高级筛选树”各自维护一套独立状态，后续一定会出现：

- 显示与执行不一致
- 保存与 reload 后结构漂移
- 某条 rule 同时出现在顶部和高级树

所以本方案明确要求它们共享同一份 `CollectionView.filters`。

### 2. 第一版刻意不做的内容

以下内容不建议第一版进入范围：

- 多个高级筛选入口 chip
- 顶层普通规则支持 `OR`
- 高级树拖拽排序
- 复杂摘要 chip 文案
- 高度拟真的 Notion 样式复刻

### 3. 本方案的推荐落地顺序

推荐实现顺序：

1. 升级 `filters` 类型、normalize 和执行器
2. 同步改 dirty 判定、draft/save/reload 的旧 `FilterGroup` 依赖点
3. 实现树操作 helper
4. 改 `ViewFilterBar`，支持同字段多规则、唯一高级入口 chip 和“合并到高级筛选中”
5. 落 `AdvancedFilterPanel` 与递归 editor
6. 补 shared view draft/save/reload 回归
7. 补测试和文档

## 十一、最终建议

本需求应按“唯一高级筛选树 + 顶部普通 rules 固定 AND”的模型实现，而不是把顶部 chips 视为高级树摘要，也不是把高级组做成多个顶层并列入口。

第一版的最优解不是视觉复刻，而是先确保以下四点全部成立：

- 顶部 chips 和高级树都是真实生效条件
- 进入高级树的 rule 不再在顶部重复显示
- shared view 的 draft/save/reload 语义不变
- 旧单层 `AND` view 能无损迁移

只要这四点收稳，后续再补更细的 Notion 交互和视觉都不会推翻基础模型。

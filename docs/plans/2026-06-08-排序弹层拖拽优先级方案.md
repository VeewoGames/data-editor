# 排序弹层拖拽优先级方案

## 方案概述

### 总体目标和范围

本方案的目标是为 `SortPopover` 增加“通过拖拽排序规则位置来改变排序优先级”的能力，并保持当前 shared view / local draft 的 `sorts` 数组语义不变。用户在排序弹层中调整两条或多条排序规则的上下顺序后，表格排序优先级应立即按新的数组顺序生效。

本轮范围只覆盖 [src/components/sort/SortPopover.tsx](C:/Code/data-editor/src/components/sort/SortPopover.tsx) 这一处排序弹层，以及与之直接相关的轻量拖拽复用层、样式和测试。不在本轮范围内的内容包括：

- 不重做全项目拖拽框架
- 不迁移 `ViewTabs`、`DetailPanel`、列拖拽等其它入口
- 不改变 `SortRule` 数据结构
- 不新增“保存优先级”之类独立配置字段

### 各阶段任务概要

第一阶段：确认复用边界并泛化现有 hook。  
主要工作是复核现有一维拖拽内核和弹层内列表拖拽接法，优先把现有 `useOptionFieldDragReorder` 泛化成共享 hook，而不是新增一份平行实现。预期成果是形成“底层复用共享 drag engine、中层复用共享 list reorder hook、上层只接排序行”的清晰边界。执行顺序是先读现有 `one-dimensional-dnd` 和 `useOptionFieldDragReorder`，再做最小泛化。

第二阶段：接入 `SortPopover` 拖拽排序。  
主要工作是为每条排序规则增加左侧拖拽把手，接入纵向投影式拖拽预览，并在松手时提交新的 `sorts` 顺序。预期成果是用户能在弹层中稳定地拖动排序规则上下移动，影响排序优先级。执行顺序是先接交互，再补视觉状态。

第三阶段：补齐验证与回归。  
主要工作是增加纯逻辑测试和 E2E 交互测试，确认拖拽后排序优先级、生效结果和现有字段/方向选择、删除排序行为不冲突。预期成果是该能力可回归、可维护。执行顺序是先测顺序提交，再测真实表格排序结果。

### 整体结构框架

本轮建议采用“三层接入”：

1. 底层拖拽内核：继续复用 [src/drag/one-dimensional-dnd.mjs](C:/Code/data-editor/src/drag/one-dimensional-dnd.mjs)
2. 中间轻量 hook：优先由现有 `useOptionFieldDragReorder` 泛化得到的共享纵向列表重排 hook
3. 组件接入层：`SortPopover` 负责渲染排序行、拖拽把手和最终 `onChangeSorts(nextSorts)`

核心原则是：不重起新框架，不复制一套拖拽算法，而是在现有共享内核上补一层适合 `SortPopover` 的可复用外壳。

## 现状分析

### 当前已有的可复用能力

[src/drag/one-dimensional-dnd.mjs](C:/Code/data-editor/src/drag/one-dimensional-dnd.mjs) 已经具备本轮所需的大部分底层能力：

- `createOneDimensionalDragSession`：统一拖拽启动、预览、提交、取消生命周期
- `createVerticalProjection`：统一纵向列表投影模型
- `projectVerticalDrag`：根据 pointer 位置计算 `dropIndex` 和 `projectedOrder`
- `arrayMove`：根据目标位置生成新顺序

这说明当前项目并不缺“拖拽框架底层”，缺的是一个能直接服务弹层内纵向重排列表、且不和现有实现分叉的轻量接入层。

### 当前最接近本需求的实现

[src/table/useOptionFieldDragReorder.ts](C:/Code/data-editor/src/table/useOptionFieldDragReorder.ts) 是目前最接近 `SortPopover` 需求的现成样板。它已经具备：

- pointer 事件驱动
- 纵向投影式拖拽
- ghost / placeholder 预览
- 拖拽过程不直接提交真实顺序
- 松手后一次性提交最终顺序

它的问题不在交互模型，而在实现边界过于绑定 `OptionFieldEditor`：

- 数据 id 固定为 `option.value`
- DOM ref 语义固定为 option row
- 提交逻辑固定写回 option 列表

因此它适合作为本轮的**首选泛化来源**，不适合作为 `SortPopover` 直接硬接的现成 API。

### 当前不建议直接复用的实现

[src/components/ViewTabs.tsx](C:/Code/data-editor/src/components/ViewTabs.tsx) 和 [src/detail/DetailPanel.tsx](C:/Code/data-editor/src/detail/DetailPanel.tsx) 也有拖拽排序，但不适合作为这次的首选复用层：

- `ViewTabs` 是横向标签条，命中和视觉模型不同
- `DetailPanel` 仍是 `mouse` 事件模型，不是当前更一致的 pointer 方案
- 两者都比 `OptionFieldEditor` 离“弹层内纵向小列表”更远

## 方案对比

### 方案 A：在 `SortPopover` 内直接复制一份拖拽逻辑

优点：

- 实现速度最快
- 改动范围最小

缺点：

- 和 `OptionFieldEditor` 再次分叉
- 未来其它弹层如果也要排序，又会复制第三份、第四份
- 不符合当前项目已经存在共享 drag engine 的结构方向

### 方案 B：复用共享 drag engine，优先泛化现有 hook 为共享纵向列表重排 hook

优点：

- 最大化复用现有底层能力和现有弹层拖拽接法
- 不需要启动全项目拖拽重构
- 和 `SortPopover` 的需求边界高度贴合
- 后续若还有弹层内小列表排序，可以继续复用
- 避免留下 `OptionFieldEditor hook` 与 `SortPopover hook` 两套平行实现

缺点：

- 比方案 A 多一层抽象工作
- 需要控制好 API 边界，避免又抽成过重框架

### 方案 C：直接按统一框架文档重构拖拽体系后再接 `SortPopover`

优点：

- 结构最完整
- 长期一致性最好

缺点：

- 本轮范围明显过大
- 会把“给排序弹层增加拖拽优先级”升级成全项目级重构
- 风险和验证成本都不成比例

## 推荐方案

推荐采用方案 B：**复用共享 drag engine，优先泛化现有 `useOptionFieldDragReorder` 为共享纵向列表重排 hook，再接入 `SortPopover`**。

推荐理由：

- 现有底层能力已经足够，不需要重造算法层
- 现有最接近的 `useOptionFieldDragReorder` 已经证明“弹层内纵向拖拽”模型成立，应优先演进它，而不是再复制一份
- `SortPopover` 本身只是一个弹层内的短列表，最合适走“轻量适配”，不该演变成大重构
- 与 [docs/plans/2026-06-07-拖拽排序统一框架方案.md](C:/Code/data-editor/docs/plans/2026-06-07-拖拽排序统一框架方案.md) 的方向一致：共享核心，组件层保持自己的 DOM 和视觉
- 能为后续类似“弹层内小列表重排”提供一个清晰、低成本的接入模板

## 详细设计

### 一、数据模型与提交语义

本轮不改变 `SortRule` 结构，仍然直接使用当前的 `sorts: SortRule[]` 表达排序优先级。

关键语义保持如下：

- `sorts[0]` 是最高优先级
- `sorts[1]` 是第二优先级
- `applyViewSorts(rows, sorts)` 继续按数组顺序逐条比较
- 拖拽本质上只是重排 `sorts` 数组顺序

因此拖拽提交后的唯一输出就是：

```ts
onChangeSorts(nextSorts)
```

不新增额外同步层，不新增优先级字段，不引入中间映射。

### 二、通用 hook 边界

建议优先将现有 [src/table/useOptionFieldDragReorder.ts](C:/Code/data-editor/src/table/useOptionFieldDragReorder.ts) 泛化为一个共享轻量 hook，例如：

- `src/components/useVerticalListDragReorder.ts`

然后让 `OptionFieldEditor` 和 `SortPopover` 都基于这层工作。这个 hook 只负责“纵向列表拖拽排序”的通用交互，不绑定排序、选项、多选等业务概念。

建议输入边界：

- `order: string[]`
- `itemRefs`
- `onCommitOrder(nextOrder: string[])`
- 可选的拖拽阈值或视觉参数

建议输出边界：

- `draggingId`
- `dragPreview`
- `beginDrag(id, event)`
- `cancelDrag()` 或等价清理能力（若现有实现已内聚在事件生命周期中，则不强求单独暴露）

它应复用：

- `createOneDimensionalDragSession`
- `createVerticalProjection`
- `projectVerticalDrag`
- `arrayMove`

它不应负责：

- 渲染具体行内容
- 决定每行字段/方向/删除按钮结构
- 决定最终业务对象如何从 `id[]` 映射回对象数组

如果泛化后共享 hook 的输入输出仍然被 `OptionFieldEditor` 业务字段强耦合，则说明泛化失败；此时才退回到在 `SortPopover` 局部实现一层薄包装，但这应作为备选，不应作为首选。

### 三、`SortPopover` 接入方式

[src/components/sort/SortPopover.tsx](C:/Code/data-editor/src/components/sort/SortPopover.tsx) 的改造建议如下：

1. 每个排序行左侧新增单独拖拽把手  
把手是唯一允许启动拖拽的区域，避免与字段 select、方向 select、删除按钮抢 pointer。

2. 每条排序规则继续以 `sort.id` 作为稳定拖拽 id  
这样不会额外发明临时 key，也能与当前删除/修改逻辑保持一致。

3. 拖拽过程中只做投影预览，不立即提交 `sorts`  
也就是视觉上显示新位置，但真正 `onChangeSorts(nextSorts)` 只在 pointer release 时触发。

4. 拖拽完成后按 `id` 顺序重建 `nextSorts`

5. 字段和方向的修改逻辑保持现状  
拖拽只改顺序，不改 `field` 和 `direction` 本身的行为。

6. 拖拽开始后要抑制后续 click  
避免松手时误触发 `Select.Trigger`、按钮聚焦或其它点击副作用。该策略应与现有 `ViewTabs` / `Sidebar` 的 suppress click 思路保持一致。

### 四、视觉与交互建议

本轮视觉目标不是做复杂动画，而是保证“可控、稳定、可见”。

本轮应明确采用唯一视觉模型：**placeholder + ghost**。

含义如下：

- 原始列表中的被拖拽行在拖拽开始后退出正常占位
- 原位置或投影位置由 placeholder / gap 表达
- 一个独立 ghost 跟随 pointer 垂直移动

本轮不采用“只改 projected order、不渲染 ghost”的简化模型，也不同时保留多套可选视觉分支，以避免实现和测试发散。

建议保留下面几项：

- 左侧把手图标使用现有 `dragHandle`
- 被拖拽行显示 pressed / dragging 状态
- 列表中显示明确 placeholder / projected gap
- 弹层内 ghost 跟随 pointer 垂直移动

建议暂不引入：

- 跨列表拖拽
- 惯性动画
- 自动滚动
- 键盘排序交互

由于排序弹层通常只有 1 到 4 条规则，这里优先强调稳定性和点击区隔离，不追求过度动画。

### 五、测试策略

需要补两类验证。

第一类：纯逻辑或轻逻辑测试

- 拖拽后 `sorts` 顺序正确变化
- 拖动未越过阈值时不提交
- 拖回原位时不产生无效变更
- `orderedIds -> nextSorts` 的映射保持稳定，不丢项、不重复

建议补一个独立 helper，例如：

- `reorderSortRulesById(sorts, orderedIds)`

把“按拖拽结果重建 `nextSorts`”从组件事件中拆出来单测，降低 E2E 定位成本。

第二类：E2E 测试

- 在排序弹层中新增两条排序规则
- 通过左侧把手把第二条拖到第一条之前
- 验证排序行显示顺序已变化
- 验证真实表格结果按新的优先级生效
- 验证字段 select、方向 select、删除按钮仍正常工作
- 至少覆盖两个打开入口：
  - 主“排序”按钮打开的弹层
  - 已存在 sort chip 打开的弹层

真实排序结果的验收条件需要写死为可执行断言：

1. 选用一个 fixture，使至少两行记录满足“第一排序字段相同、第二排序字段不同”。
2. 初始 `sorts = [A, B]` 时，断言前两行顺序为结果 X。
3. 拖拽为 `sorts = [B, A]` 后，断言前两行顺序切换为结果 Y。
4. X 与 Y 必须不同，且变化能够证明优先级确实已交换，而不是只改了弹层 UI。

## 实施边界

本轮允许修改的文件类型包括：

- `SortPopover` 组件
- 由 `useOptionFieldDragReorder` 泛化出的共享轻量拖拽 hook
- 排序弹层样式
- 排序相关测试

本轮不建议顺手修改的内容包括：

- `ViewTabs` 拖拽
- `DetailPanel` 拖拽
- `OptionFieldEditor` 的既有行为
- 全局 drag framework 文档或目录大搬迁

## 风险与规避

### 风险一：拖拽把手和其它控件点击冲突

规避方式是严格限定只有左侧把手可启动拖拽，字段/方向下拉和删除按钮继续只负责自身点击行为。

### 风险二：把当前需求做成过重抽象

规避方式是只泛化“纵向列表拖拽重排”的薄 hook，不在本轮引入新的大目录或全项目统一 presentation 层。

### 风险三：拖拽预览与弹层 portal/层级干扰

规避方式是尽量让 ghost 和 placeholder 保持在当前 `SortPopover` 内容层内，不新增跨 portal 的复杂叠层。

### 风险四：`sort.id` 在字段切换时变化导致拖拽态不稳定

当前 `field` 修改会通过 `createSortId(...)` 生成新 id。规避方式是：

- 拖拽会话只基于拖拽开始时的 `id` 快照排序
- drop 时再按最新 `sorts` 和 `orderedIds` 重建结果
- 避免在拖拽中并发触发字段变更；正常使用下这两者不会并发发生

### 风险五：弹层关闭/切换入口时拖拽清理不完整

排序弹层既可能由主按钮打开，也可能由 sort chip 打开。规避方式是确保拖拽生命周期与当前弹层实例绑定，关闭弹层时清理 pointer session、ghost 和 placeholder，不能把拖拽中间态泄漏到下一次打开。

## 最终建议

最终建议是：

1. 不新造拖拽框架
2. 继续复用现有 `one-dimensional-dnd` 作为底层
3. 优先把 `useOptionFieldDragReorder` 泛化为共享纵向列表重排 hook，并尽可能让 `OptionFieldEditor` 也回到这层
4. 用左侧把手把 `SortPopover` 接入这层
5. 采用唯一的 `placeholder + ghost` 视觉模型
6. 通过可执行的排序结果断言验证“拖拽后排序优先级真实生效”

这样既符合当前项目的框架演进方向，也能把这次需求控制在合理范围内，不把一个弹层交互改动演变成全局拖拽重构。

# 视图分组实施计划

## 方案概述

### 1. 总体目标和范围

本计划用于把当前扁平共享视图导航，升级为“顶层视图 + 视图组”的两层结构，并落地以下能力：

- 顶层第一行同时承载普通视图与视图组。
- 点击视图组后，在第二行展开该组内的具体视图标签。
- 顶层 `+` 只负责创建顶层普通视图或顶层视图组。
- 组内第二行 `+` 只负责向当前组内创建子视图。
- 支持把普通视图拖入组、从组中拖出、组内排序、顶层排序。
- 保持“真正的工作对象始终是具体视图”，组只负责收纳与导航。

本次范围包含数据模型、状态模型、迁移链路、`ViewTabs` 交互、草稿保存/重置、拖拽规则、测试与文档更新。

本次不包含组级筛选、组三层嵌套、组级权限或组级独立页面语义。

### 2. 各阶段任务概要

阶段一：状态与迁移打底。
主要工作是补齐 `structureDrafts`、组展开状态和路径迁移；预期成果是新增状态面可以稳定持久化和迁移；执行顺序先测后改。

阶段二：共享视图 schema 升级。
主要工作是把 collection 级配置从 `views[]` 升级为 `items[]`；预期成果是新旧共享视图配置都能稳定读取，且新写回只产出新结构；执行顺序先 normalize 再 save。

阶段三：`view-state` 桥接与解析收敛。
主要工作是让现有 shared view 读写逻辑基于 `items[]` 工作，并为后续组交互准备统一解析层；预期成果是 duplicate、rename、delete、save draft 等能力不再依赖旧 `views[]`；执行顺序先修红测，再补真正的组级 helper。

阶段四：两层 UI 与双 `+` 交互落地。
主要工作是改造 `App.tsx` 与 `ViewTabs.tsx`，引入顶层 `+`、组内 `+`、第二行展开和组内 active 恢复；预期成果是用户可以稳定创建顶层视图、顶层组和组内子视图；执行顺序先静态结构，再接入状态与动作。

阶段五：拖拽与保存语义闭环。
主要工作是补齐入组、出组、组内排序、顶层排序、结构草稿保存/重置；预期成果是结构修改先进入草稿，只有显式保存才写入共享配置；执行顺序先纯逻辑测试，再接 UI 拖拽。

阶段六：验证、文档与收尾。
主要工作是补齐单测、Playwright、说明文档；预期成果是旧扁平实现可被新结构完整替换；执行顺序先跑定向测试，再做端到端验证。

### 3. 整体结构框架

整体结构分三层：

1. 存储层：`.data-editor/shared-views.json` 使用 `collections[collectionKey].items[]` 作为唯一共享视图结构真相。
2. 状态层：本地 `shared view drafts` 负责内容草稿和结构草稿，`page context` 负责当前展开组和每个组上次激活的子视图。
3. 渲染层：`App.tsx` 负责解析 active view / expanded group / dirty state，`ViewTabs.tsx` 只消费 resolved 结构并发出交互事件。

核心边界如下：

- 组不是视图，不承载查询、筛选、排序和列布局。
- active 对象始终是具体视图，不是组。
- 顶层 `+` 与组内 `+` 是两个独立入口，职责不能混用。
- 结构修改先写 `structureDrafts`，显式保存后才进入共享配置。

## 数据与状态约束

共享配置约束：

- `defaultViewId` 必须始终指向某个真实 view，而不能指向 group。
- `group.id` 只在结构层使用，不能复用为 active view id。
- `items[]` 是唯一共享结构真相，旧 `views[]` 只作为读取兼容入口。

本地草稿约束：

- `viewDrafts` 只挂具体 view。
- `viewOrderDrafts` 在桥接期仍表示扁平 view 顺序；进入结构层后逐步退化为兼容字段。
- `structureDrafts` 专门描述顶层/组内结构变化。

页面上下文约束：

- `expandedGroupId` 表示当前 collection 下正在展开的组。
- `lastActiveViewIdByGroupId[groupId]` 表示该组上次活跃的具体视图。
- 页面刷新时先恢复组，再恢复组内视图；若目标视图已失效，则回落到该组第一个子视图。

## 风险与收敛原则

- 不再继续在 `App.tsx` 或 `ViewTabs.tsx` 中临时拼 group 语义，必须先建立独立解析层。
- 不做“组级共享内容草稿”，避免把组错误地演化成视图。
- 不长期保留 `views[]` 与 `items[]` 两套写路径，桥接完成后统一收口到 `items[]`。
- 拖拽与结构保存必须走结构草稿，不允许直接改共享配置再假装是草稿。

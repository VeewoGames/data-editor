# 复制标签组方案

## 方案概述

### 1. 总体目标和范围

本方案用于在现有“顶层视图 + 视图组”结构上新增“复制标签组”能力，入口放在组的浮框菜单中。复制操作的目标不是仅复制组外壳，而是深复制整个组及组内所有视图，使用户可以快速基于一整套现有标签组继续调整。

本次范围包含：

- 组菜单新增“复制组”操作入口。
- 复制时生成一个新的组节点，并复制组内全部子视图。
- 复制组时完整复制每个子视图的 query、filters、sorts，以及 profile/local 下的列布局配置。
- 新组名称追加“副本”，组内子视图名称保持原名。
- 复制成功后自动切换到新组并展开，激活新组中的第一个子视图。
- 补齐数据层 helper、`App.tsx` 动作编排、`ViewTabs` 菜单、测试与文档。

本次不包含：

- 批量复制多个组。
- 组级筛选、组级排序、组三层嵌套。
- 复制运行态 UI 状态，例如滚动位置、筛选条展开状态、临时输入焦点。

### 2. 各阶段任务概要

阶段一：补齐组级复制的纯数据 helper。  
主要工作是新增“复制组结构”函数，输出新组、新视图 id 映射和默认激活目标；预期成果是可以独立于 UI 深复制 group item；执行顺序先纯逻辑、后接保存。

阶段二：接入应用层保存流程。  
主要工作是在 `App.tsx` 中编排 shared view config 保存、布局状态复制和新组激活；预期成果是用户点击菜单后可以真正生成完整副本；执行顺序先 profile 布局，再 local 布局。

阶段三：补齐菜单入口与交互反馈。  
主要工作是给组的浮框菜单增加“复制组”项，并在成功后自动展开新组；预期成果是功能可达且反馈一致；执行顺序先 UI 入口，再接动作。

阶段四：测试与文档收尾。  
主要工作是补结构测试、状态测试和必要的交互测试；预期成果是组复制的命名、结构、布局继承和激活行为有自动化护栏；执行顺序先单测，再端到端。

### 3. 整体结构框架

整体实现分三层：

1. 结构层：`shared-view-structure.mjs` 负责基于当前 resolved 组快照复制组节点、生成新 group/view id、保持组内顺序，并返回旧视图到新视图的映射。
2. 应用层：`App.tsx` 负责调用 helper 保存新 shared view config，并逐个复制每个子视图的内容草稿与布局状态，再更新当前激活组和激活视图。
3. 展示层：`ViewTabs.tsx` 只负责提供“复制组”入口，不直接处理复制逻辑。

核心边界如下：

- 组复制是“深复制”，组 id 和组内每个视图 id 都必须重新生成。
- 组内视图名称保持原名，只有组名追加“副本”。
- 复制范围包括视图内容配置和列布局配置，但不包括临时 UI 状态。
- 复制完成后，active 对象仍然是具体视图，不是组。
- 复制来源必须以用户当前看到的组内容和顺序为准，而不是只以已落盘的 shared view config 为准。

## 需求收敛

### 用户确认后的最终语义

- 菜单入口位于组浮框中，与“重命名组”“在组内创建视图”“删除组”同级。
- 菜单文案建议使用“复制组”。
- 新组名称规则：`原组名 副本`。
- 组内子视图名称保持原名，不追加“副本”。
- 新组中的子视图数量、顺序、筛选条件、排序、查询、列布局要与原组一致。
- 复制成功后自动切换到新组，展开该组，并默认激活新组中的第一个子视图。

### 复制范围定义

每个子视图需要复制的内容包括：

- `query`
- `filters`
- `sorts`
- 当前会话里该视图已经合并到界面上的内容状态
- `viewDrafts` 中该视图的共享内容草稿（如果存在）
- `viewLayouts` 中该视图的 `hidden / wrapped / order / detailOrder / widths`

明确不复制的内容包括：

- 当前表格滚动位置
- detail panel 打开状态
- 筛选条是否临时展开
- 当前菜单打开状态

## 现状分析

### 已有能力

当前代码已经具备两类可复用基础：

1. 单视图复制链路。  
`handleDuplicateSharedView` 已能基于一个 source view 创建新 view，并通过 `copyViewLayoutState` 复制对应布局。

2. 视图组结构解析链路。  
`shared-view-structure.mjs` 已负责 group/view 的结构解析、创建组、组内新增视图、排序和展开恢复。

这意味着“复制组”不需要另起一套机制，应该建立在现有“复制视图 + 结构 helper”之上。

### 当前实现约束

现有渲染链路已经说明，用户当前看到的组内容不一定等于 `sharedViewsConfig` 原始内容：

- 视图内容层面，`App.tsx` 会先把 `activeSharedView` 与 `draftSource.viewDrafts` 合并后再参与渲染。
- 结构层面，`resolveSharedViewStructure(...)` 会优先应用 `structureDrafts`，因此当前组内顺序可能已经偏离已落盘 config。

因此，复制组如果只读取 `sharedViewsConfig`，会出现两类偏差：

- 复制出的筛选条件/查询/排序落回旧版本，而不是当前草稿态。
- 复制出的组内顺序不是当前界面所见顺序。

### 现有缺口

当前系统缺少以下能力：

- 没有组级 duplicate helper。
- 没有“旧视图 id -> 新视图 id”批量映射输出。
- `ViewTabs` 组菜单没有“复制组”项。
- `App.tsx` 没有“一次复制整个组并逐个复制内容草稿与布局”的动作编排。

## 推荐实现方案

### 方案选择

推荐采用“结构深复制 + 批量复用单视图布局复制”的实现方式。

原因：

- 与现有 `handleDuplicateSharedView + copyViewLayoutState` 机制一致。
- 布局复制逻辑不分叉，后续维护成本最低。
- 可以把“组复制”拆成“复制结构”和“复制每个视图布局”两个边界清晰的动作。

不推荐直接在多个模块散落处理组复制副作用，否则容易出现 shared view config 已复制、但 layout 没跟上的半完成状态。

## 详细设计

### 1. 数据层：新增复制组 helper

建议在 `src/view/shared-view-structure.mjs` 新增类似 `duplicateViewGroupConfig(...)` 的 helper。

输入建议包括：

- `collectionKey`
- `groupId`
- `topLevelItems`
- `viewDraftsById`
- `sharedViewsConfig`

输出建议包括：

- `config`：复制完成后的新 shared view config
- `group`：新复制出来的 group 节点
- `sourceToTargetViewIdMap`：原组内视图 id 到新视图 id 的映射
- `firstViewId`：新组中第一个子视图 id，供激活使用
- `duplicatedViewDraftsById`：应复制到新视图 id 下的内容草稿快照

helper 职责：

- 在当前 `topLevelItems` 中找到目标组，而不是只从原始 config 中查找。
- 生成新组 id。
- 生成新组名：`原组名 副本`，并做唯一化处理。
- 逐个复制组内视图结构：
  - 新视图 id 唯一化
  - 子视图 name 原样保留
  - 当前 resolved 视图上的 `query / filters / sorts` 原样复制
- 逐个生成 `sourceViewId -> targetViewId` 映射。
- 如果源视图存在内容草稿，则同步生成目标视图草稿快照。
- 把新组插入到原组后面，保持用户空间局部性。

这里要明确：helper 的复制来源应以“当前 resolved group snapshot”为准。也就是说，即使用户还没有保存结构草稿或视图草稿，只要界面上已经显示了那套内容，复制组就应复制当前界面状态。

### 2. 应用层：新增组复制动作

建议在 `src/App.tsx` 新增 `handleDuplicateSharedViewGroup(groupId: string)`。

动作顺序建议如下：

1. 基于当前 `resolvedCollectionViews.topLevelItems` 与当前 collection 的 `viewDrafts` 调用 `duplicateViewGroupConfig(...)`，生成新 config、id 映射和目标草稿。
2. 保存新的 `sharedViewsConfig`。
3. 先把 helper 返回的 `duplicatedViewDraftsById` 合入当前 collection 的 `viewDrafts`。
4. 针对映射中的每一对 `sourceViewId -> targetViewId`：
   - 如果当前是 profile 模式，复用 `copyViewLayoutState({ mode: "profile", ... })`
   - 如果当前是 local 模式，复用 `copyViewLayoutState({ mode: "local", ... })`
5. 更新 `lastActiveViews[collectionKey]` 为新组第一个子视图 id。
6. 更新 page context：
   - `expandedGroupId = 新组 id`
   - `lastActiveViewIdByGroupId[新组 id] = 新组第一个子视图 id`
7. 设置成功提示，例如“已复制视图组”。

这里的默认激活目标明确固定为“新组数组中的第一个子视图”，不再额外读取源组最近活跃子视图，避免行为歧义。

### 3. 视图草稿与布局复制策略

这里需要区分三类状态：

1. 共享视图内容配置  
这部分直接由 `duplicateViewGroupConfig` 在结构层复制，因为它属于当前 resolved `CollectionView` 自身内容。

2. 共享内容草稿  
这部分不能忽略。若源视图在 `viewDrafts` 下存在尚未保存的内容草稿，复制组时必须把该草稿复制到目标视图 id 下，否则复制结果会退回到旧视图内容。

3. 布局状态  
这部分不能只靠结构层复制，必须在 `App.tsx` 中通过现有 `copyViewLayoutState` 逐个复制。

这样可以保证：

- 当前界面上可见的视图内容被完整复制。
- profile/local 两条布局持久化路径继续复用现有规则。
- 不额外引入新的布局存储分叉。

### 4. 菜单层：新增“复制组”入口

在 `src/components/ViewTabs.tsx` 中：

- 给 `ViewTabsProps` 新增 `onDuplicateGroup(groupId: string)`。
- 在组浮框菜单中，插入“复制组”按钮。
- 建议位置放在“在组内创建视图”下方、“删除组”上方。
- 行为与现有菜单项保持一致：点击后先关闭 popover，再触发复制动作。

菜单结构建议如下：

- 重命名组
- 在组内创建视图
- 复制组
- 分割线
- 删除组

### 5. 命名与唯一性规则

规则建议如下：

- 新组名默认 `原组名 副本`
- 若冲突则按现有 unique name 规则继续生成，例如 `原组名 副本 2`
- 子视图名称保持原名
- 子视图 id 全部重新生成，不能复用原 id
- 复制组只面向非空组；当前产品模型下所有组都至少包含一个子视图，因此无需额外设计空组复制语义

原因是：

- 用户明确要求组内视图名不要额外加“副本”。
- 同名子视图位于不同组内，对用户理解是可接受的。
- 真正的唯一标识应由 view id 保证，而不是靠展示名。

## 测试方案

### 单元测试

建议补在 `tests/shared-view-structure.test.mjs`：

- 复制组后会生成一个新组，并插入到原组后面。
- 新组名会追加“副本”并做唯一化。
- 新组内子视图数、顺序、query、filters、sorts 与原组一致。
- 新组内子视图 id 全部不同于原组。
- 新组内子视图 name 保持原名。
- 返回的 `sourceToTargetViewIdMap` 完整可用。

建议补在 `tests/view-state.test.mjs`：

- `App.tsx` 存在 `handleDuplicateSharedViewGroup`。
- 组复制动作会调用 shared view helper。
- 组复制动作会把当前 collection 的 `viewDrafts` 一并复制到目标视图 id。
- 组复制动作会逐个调用 `copyViewLayoutState`。
- 复制完成后会更新 `lastActiveViews` 和 `expandedGroupId`。
- 若存在 `structureDrafts`，复制来源仍然以当前 resolved 顺序为准。

### 交互测试

建议补在 `tests/data-editor.spec.ts`：

- 打开组菜单可以看到“复制组”。
- 点击后出现新组。
- 新组自动展开并激活第一个子视图。
- 新组子视图的筛选条件与原组一致。
- 新组子视图的未保存内容草稿与原组当前界面状态一致。
- 新组子视图的列顺序和列宽与原组一致。

## 风险与护栏

### 风险 1：结构复制成功，但布局未复制

根因是 shared view config 和 view layout state 处于两条不同持久化链路。

护栏：

- 组复制动作必须把“保存 config”和“复制布局”放在同一个 handler 中。
- 单测要覆盖每个映射视图都执行了布局复制。

### 风险 2：组名唯一了，但视图 id 映射不完整

根因是组复制是批量动作，任何一个子视图漏映射都会导致布局丢失。

护栏：

- helper 返回 `sourceToTargetViewIdMap`，不要在 `App.tsx` 再临时扫描新组推断。
- 测试中显式校验映射 key 数量和组内视图数量一致。

### 风险 3：结构复制了，但内容草稿仍然指向旧视图

根因是当前界面所见视图内容可能来自 `viewDrafts`，而不是 shared config 落盘值。

护栏：

- helper 或 `App.tsx` 必须显式生成 `duplicatedViewDraftsById`。
- 测试中要覆盖“源视图存在未保存筛选草稿时，目标视图打开后仍能看到该草稿”。

### 风险 4：复制后没有切到新组，用户误以为失败

根因是复制动作如果只保存结构、不切激活态，视觉反馈很弱。

护栏：

- 复制完成后默认切到新组并展开。
- 成功提示文案明确写“已复制视图组”。

### 风险 5：组复制中途失败，产生半完成状态

根因是 shared config 保存与布局复制属于两条链路，无法天然事务化。

护栏：

- 方案默认接受“结构先保存、布局随后补齐”的执行顺序，但必须把所有布局复制放在同一个 handler 中连续完成。
- 一旦布局复制阶段失败，状态提示中要明确是“视图组已复制，但部分布局复制失败”，避免用户误判为完全成功。
- 若后续需要更强一致性，再单独评估是否引入批量复制 helper 或回滚策略；本轮先不扩 scope。

## 实施顺序建议

1. 先补 `shared-view-structure.mjs` 的组复制 helper 和对应测试。
2. 再在 `App.tsx` 中接入 `handleDuplicateSharedViewGroup`，同时补 `viewDrafts` 和 `viewLayouts` 的批量复制。
3. 然后在 `ViewTabs.tsx` 中补菜单入口。
4. 最后补 `data-editor.spec.ts` 的交互覆盖。

这样可以先锁定纯结构行为，再接入 UI，排错成本最低。

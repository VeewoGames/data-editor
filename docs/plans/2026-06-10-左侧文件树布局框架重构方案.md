# 方案概述

## 1. 总体目标和范围

本方案的目标，是把左侧 `Files` 区从“文件列表样式上叠加树行为”的实现，重构为“原生树节点布局框架”，解决当前已经暴露出的视觉错位问题，并为后续的图标、折叠、拖拽、计数、菜单、状态点等扩展建立稳定的结构基础。

本次方案只覆盖左侧文件树的展示和交互框架，包括：

- 树节点统一布局协议
- 节点渲染分层
- 样式系统重构原则
- 拖拽命中面与视觉结构的解耦
- 对应的测试和回归策略

本方案不改变以下既有业务规则：

- 左侧第一层按 `data source` 分组显示
- `data source` 只有一层时默认隐藏
- 文件拖拽排序只允许在原文件夹内进行
- 同层的文件夹和文件支持拖动排序
- 排序和展开状态属于每个用户自己的本地/个人视图偏好
- 刷新后保留

## 2. 各阶段任务概要

### 阶段一：统一树节点布局协议

主要工作：

- 定义所有树节点共用的行结构和槽位语义
- 明确 `source`、`folder`、`file` 之间允许变化的字段，以及不允许变化的结构
- 终止当前“不同节点类型使用不同 DOM 子元素数量”的实现方式

预期成果：

- 一套可被所有节点类型共享的树行协议
- 后续 UI 调整不再依赖占位元素和隐式列数对齐

执行顺序：

- 先定义协议，再进入组件和样式改造

### 阶段二：拆分树控制区与内容区

主要工作：

- 把树的缩进、折叠控制和节点内容展示解耦
- 让层级结构只影响控制区，不影响文本列起点
- 建立统一的树行内容容器

预期成果：

- 删除某类图标时不会再导致文本列整体漂移
- `source`、`folder`、`file` 的文本列坐标统一

执行顺序：

- 在树节点协议确定后，重构渲染结构

### 阶段三：重构交互命中模型

主要工作：

- 把拖拽、选中、折叠的命中区域绑定到统一的树行语义
- 降低交互逻辑对具体图标、占位节点和 DOM 顺序的依赖
- 保持当前“仅同父级可重排”的业务约束

预期成果：

- 视觉结构变化不再引起拖拽或点击行为错乱
- 树行交互和树行视觉布局脱钩

执行顺序：

- 先完成结构重构，再适配交互层

### 阶段四：收敛样式体系与测试

主要工作：

- 用统一的 tree row token 和类名体系管理树节点样式
- 回收历史上从 `.sidebar-item` 继承并被局部覆盖的脆弱样式路径
- 补齐布局契约测试与关键 e2e 回归测试

预期成果：

- 样式规则集中且稳定
- 后续功能变更有明确回归保护

执行顺序：

- 在布局和交互收敛后统一补齐

## 3. 整体结构框架

推荐把左侧文件树分成三层：

1. Tree data layer
   负责 `source` / `folder` / `file` 树结构、节点父子关系、排序结果、展开状态和持久化数据。

2. Tree view-model layer
   负责把树节点映射成统一的渲染输入，明确每一行应该如何显示，而不是由 JSX 临时推断。

3. Tree render layer
   负责按统一布局协议渲染节点行，不再允许不同类型节点偷偷使用不同列模型。

此外，本方案采用“两步走”执行策略：

1. 第一步先统一 tree row 的 DOM 骨架、点击命中面和布局协议，在不改变既有排序与拖拽业务规则的前提下消除当前错位根因。
2. 第二步再把树渲染职责拆分到独立组件和样式体系中，完成结构清理和长期维护优化。

这样可以先止住当前框架缺陷，再降低一次性大重构带来的回归风险。


# 一、问题根因分析

当前视觉错位并不是单个图标删错，而是左侧树的布局框架存在结构性问题。

## 1.1 缺少统一的树行布局协议

当前 `source`、`folder`、`file` 三类节点虽然都被视为 sidebar row，但它们的 DOM 结构并不一致：

- `source/folder` 行：`indent + toggle + label`
- `file` 行：`indent + spacer + file icon + label`

这意味着它们并没有共享一套稳定的列语义，只是被放在同一个视觉容器里勉强共存。一旦某类节点图标被移除、某类节点新增状态位、或者某类节点增加菜单入口，列对齐就会失真。

## 1.2 视觉结构和交互结构强耦合

当前折叠箭头、拖拽命中、选中区域、图标位置、文本列位置，都直接依赖于具体子节点顺序。这样会导致一个问题：只要视觉结构调整，交互命中逻辑也更容易被连带破坏。

## 1.3 当前实现依赖占位元素和覆盖式样式

当前样式机制本质上是：

- 基础层使用 `.sidebar-item`
- 树节点层通过 `.sidebar-tree-item` / `.sidebar-file-item` 覆盖 grid
- 再借助 `indent`、`toggle-spacer` 等占位节点维持对齐

这种方式的问题是，布局正确性依赖于以下条件同时成立：

- 子元素个数正确
- 子元素顺序正确
- 不同节点类型的列数恰好兼容
- 样式覆盖顺序稳定

只要其中一项变化，视觉结构就会出现错位。


# 二、目标框架设计

## 2.1 建立统一的 Tree Row 槽位模型

所有树节点都必须共享同一个行骨架，不允许因为节点类型不同而改变列模型。

推荐固定为五个槽位：

1. `indent`
2. `toggle`
3. `leading`
4. `label`
5. `trailing`

它们的职责如下：

- `indent`：只表达层级，不承载图标或交互
- `toggle`：只负责折叠箭头；无子节点时保留空槽
- `leading`：主图标位；文件可显示文件图标，`source/folder` 可为空
- `label`：主文本位
- `trailing`：计数、状态点、菜单等扩展位

这套槽位模型必须对三类节点保持完全一致，只允许槽位内容不同，不允许槽位结构不同。

这里必须增加一个强约束：

- 所有树节点必须渲染同一套 DOM 骨架
- 某个槽位即使没有内容，也只能渲染为空槽，不能删除该槽位
- 不允许通过“不同节点渲染不同数量的 `span`”来维持视觉结果

也就是说，统一槽位模型不是视觉建议，而是必须落地到 DOM 结构上的硬约束。只有这样，grid 坐标系、点击热区、拖拽命中和文本起始列才能稳定。

## 2.2 `source` / `folder` / `file` 的映射规则

基于统一槽位模型，节点展示规则如下：

- `source`
  - `indent`：根据深度计算
  - `toggle`：显示或隐藏折叠符号
  - `leading`：为空
  - `label`：显示 source 名称
  - `trailing`：预留给未来计数或菜单

- `folder`
  - `indent`：根据深度计算
  - `toggle`：显示或隐藏折叠符号
  - `leading`：为空
  - `label`：显示文件夹名称
  - `trailing`：预留给未来计数或菜单

- `file`
  - `indent`：根据深度计算
  - `toggle`：为空槽
  - `leading`：显示 `json/csv` 图标
  - `label`：显示文件名
  - `trailing`：预留给未来状态位

这样即使去掉 `source/folder` 图标，文本列也不会整体左移或右移，因为它们所在的槽位体系根本没有变化。

同时，本方案将“文本列对齐”定义为明确的框架目标：

- 根层文件与同深度 folder 子文件的文本起始列必须一致
- `source` 隐藏场景下，根层文件与展开后的 folder 子文件文本起始列仍必须一致
- 去掉 `source/folder` 图标后，文件文本列不能因为槽位缺失而漂移


# 三、渲染结构重构方案

## 3.1 拆分控制区和内容区

建议每个树节点行拆成两个明确区域：

- `tree-row-control`
- `tree-row-content`

职责如下：

- `tree-row-control`
  - 负责 `indent + toggle`
  - 表达树语义
  - 不承担业务内容展示

- `tree-row-content`
  - 负责 `leading + label + trailing`
  - 表达节点内容
  - 不承担树层级定位

推荐结构示意：

```tsx
<button className="tree-row">
  <span className="tree-row-control">
    <span className="tree-indent" />
    <span className="tree-toggle" />
  </span>
  <span className="tree-row-content">
    <span className="tree-leading" />
    <span className="tree-label" />
    <span className="tree-trailing" />
  </span>
</button>
```

这个拆分的核心价值是：

- 缩进和折叠位不会再被图标策略影响
- 内容区内部可以自由演进，而不会破坏树语义布局
- 文件图标、状态点、菜单入口都能单独调整

这里还需要明确交互契约：

- 外层 `tree-row` 仍然是统一的点击、选中、hover 和聚焦命中面
- `tree-row-control` 与 `tree-row-content` 只是内部布局分层，不改变整行的可点击范围
- 不能因为把缩进和折叠拆到控制区，就让 hover/selected 背景只覆盖内容区

也就是说，“控制区/内容区解耦”是布局解耦，不是交互命中面的拆分。

## 3.2 禁止按节点类型分叉 JSX 结构

当前树节点渲染存在两套 JSX 路径：

- `file` 一套
- 非 `file` 一套

这类分叉是导致结构漂移的根源之一。改造后应统一为：

- 树遍历层只产出 view model
- 节点行只渲染一套统一模板
- 节点差异通过 view model 字段表达，而不是通过 JSX 分叉表达

推荐 view model 结构：

```ts
type TreeRowViewModel = {
  id: string;
  kind: "source" | "folder" | "file";
  depth: number;
  expandable: boolean;
  expanded: boolean;
  toggleVisible: boolean;
  leadingIcon: "json" | "csv" | null;
  label: string;
  trailing: null | { kind: string; value?: string };
  selectable: boolean;
  draggable: boolean;
};
```

统一模板只读这个结构，不再自己猜测该渲染几个 `span`。

## 3.3 `source` 隐藏策略

当前业务要求是：当只有一层 `data source` 时，默认隐藏该层。

这里必须明确隐藏发生的层级，否则后续排序、展开状态和测试边界会混乱。

本方案建议：

- Tree data layer 仍然保留真实的 `source` 节点
- Tree view-model layer 仍然允许生成 `source` row model
- Tree render layer 根据“是否只有单一 source 且应隐藏”来决定是否裁掉这一层，只渲染其 children

这样做的好处是：

- 不破坏树数据的真实结构
- 排序、展开状态、节点 id 和偏好持久化仍然以完整树为准
- UI 层只做显示裁剪，不引入额外数据分叉

因此，“隐藏 source”应被定义为 render 层策略，而不是数据层结构改写。


# 四、样式系统方案

## 4.1 缩进规则

缩进不能再通过“整行 padding-left 随深度递增”的方式表达。
推荐规则是：

- 整行外层 `padding` 固定
- `indent` 宽度由 `depth * step` 决定
- `toggle` 宽度固定
- 内容区起点固定

这样层级变化只影响树控制区宽度，不会影响内容区列结构。

## 4.2 Token 化样式变量

建议为树节点布局引入明确 token：

- `--tree-indent-step`
- `--tree-toggle-width`
- `--tree-leading-width`
- `--tree-row-gap`
- `--tree-row-min-height`
- `--tree-content-gap`

这些 token 应统一由 tree row 样式使用，避免继续把树节点样式建立在 `.sidebar-item` 的通用列表样式上再做局部覆盖。

## 4.3 类名体系

建议建立独立的 tree row 类名体系：

- `.tree-row`
- `.tree-row-control`
- `.tree-row-content`
- `.tree-indent`
- `.tree-toggle`
- `.tree-leading`
- `.tree-label`
- `.tree-trailing`
- `.tree-row--file`
- `.tree-row--folder`
- `.tree-row--source`
- `.tree-row--selected`
- `.tree-row--dragging`

目标是让树样式成为一个自洽系统，而不是旧 sidebar list 样式上的补丁层。


# 五、交互框架方案

## 5.1 选中与折叠

选中和折叠应基于统一行语义处理：

- 行本身是主交互面
- 折叠状态从节点数据决定
- 折叠箭头只作为视觉和点击入口，不应承担布局补偿职责

## 5.2 拖拽命中重构

当前拖拽命中逻辑主要依赖 `.sidebar-file-item`。未来应改为依赖统一 tree row 语义：

- 通过 `data-node-id` / `data-node-kind` 识别节点
- 通过 view model 判断是否可拖拽
- 通过树数据层判断是否同父级
- 通过树行位置决定预览顺序

这样可以确保：

- 删除图标不影响拖拽
- 增加 trailing 内容不影响拖拽
- 树节点内部 DOM 调整不影响拖拽

## 5.3 拖拽迁移顺序

当前实现中，拖拽逻辑不仅依赖树数据，还依赖这些现有契约：

- 文件行 DOM ref
- `data-file-path`
- `.sidebar-file-item`
- 文件按钮的矩形测量结果

因此不能先大幅拆结构，再回头重接拖拽。推荐迁移顺序如下：

1. 先统一 tree row 的 DOM 骨架，但保留现有文件行可测量 ref 和 `data-file-path` 契约
2. 在统一骨架上适配 `useSidebarTreeDragReorder`
3. 等拖拽回归通过后，再考虑把命中逻辑进一步从 `.sidebar-file-item` 迁移为更纯粹的 tree row 语义

也就是说，第一次重构的重点是稳定骨架，不是一次性重写全部拖拽命中机制。


# 六、组件职责拆分建议

建议将当前 Sidebar 树部分拆为以下职责边界：

## 6.1 `Sidebar.tsx`

职责：

- 只负责左侧区域整体拼装
- 负责 Files / Collections / Metadata 等分区的组合
- 不再持有复杂树渲染细节

## 6.2 `SidebarTree.tsx`

职责：

- 负责树遍历
- 负责展开态和节点序列渲染
- 负责树拖拽 hook 的接线

## 6.3 `SidebarTreeRow.tsx`

职责：

- 负责单一树行的统一渲染
- 只接受 view model，不自己推断树结构
- 统一渲染 `indent/toggle/leading/label/trailing`

## 6.4 `sidebar-tree-view-model.ts`

职责：

- 负责把 `source/folder/file` 节点映射成统一 row model
- 把节点差异从 JSX 中剥离到配置层

这样的拆分可以减少 `Sidebar.tsx` 继续膨胀，也能让树布局框架的边界更清晰。

需要额外明确一个边界：

- `Files` 树区域使用独立 tree row 类系和组件结构
- `Collections`、`Metadata` 等非树区域继续复用现有 `.sidebar-item` / `.metadata-row` 体系

本次重构不应把整个 sidebar 的所有分区一并拖入 tree row 体系，否则会扩大回归范围。


# 七、测试与回归方案

## 7.1 单测

需要补齐以下测试：

- `source/folder/file` 是否都生成统一槽位语义
- `source/folder` 的 `leading` 是否为空
- `file` 的 `leading` 是否按扩展名映射正确
- `depth` 是否只影响控制区宽度，不影响内容区结构

## 7.2 组件或 e2e 测试

需要覆盖以下场景：

- 单 `data source` 时隐藏 source 行，但文件文本仍和 folder 子项对齐
- 去掉 `source/folder` 图标后，文件文本列不漂移
- 折叠/展开不影响文件文本起始列
- 同父级拖拽仍正常
- 跨父级拖拽仍无效
- 选中态和 hover 态在三类节点上表现一致

除了行为验证，还应增加可执行的布局验证：

- 比较根层文件与目标 folder 子文件的 `.tree-label` 左边界坐标
- 比较 `source` 隐藏前后，同一层级文件文本起始列是否一致
- 比较折叠/展开前后，同一文件的文本起始列是否变化

不应只依赖肉眼截图判断“看起来对齐”，否则后续样式轻微漂移很难稳定发现。

## 7.3 回归关注点

回归时应优先关注：

- 根层文件
- 多层 folder 场景
- `source` 隐藏场景
- 长文件名截断
- 拖拽中 ghost 和预览态
- 选中行背景和文本对齐


# 八、验收标准

本方案落地后的验收标准应明确为以下可验证结果：

1. 所有 `source`、`folder`、`file` 节点共享同一套 tree row DOM 骨架。
2. 某个槽位无内容时只能为空槽，不能删除该槽位。
3. 根层文件与同深度 folder 子文件的文本起始列一致。
4. 单 `data source` 隐藏场景下，文本列对齐规则仍成立。
5. 选中态、hover 态和点击命中面覆盖整行，而不是只覆盖内容区。
6. 同父级文件拖拽仍然生效，跨父级拖拽仍然无效。
7. 展开/折叠不会改变同一节点文本列起始位置。
8. `Collections`、`Metadata` 等非树区域视觉与交互不被 tree row 重构误伤。
9. 个人视图偏好中的排序和展开状态继续可持久化并在刷新后保留。


# 九、实施原则

## 9.1 推荐方案

推荐执行“框架级统一 Tree Row 重构”，而不是继续局部修补当前样式。

原因：

- 当前问题已经证明现有布局系统并不稳定
- 继续微调 `padding`、`grid-template-columns`、占位节点，只能修当前截图，不能修框架问题
- 项目仍处早期，做结构性收敛的成本较低，后续收益更高

## 9.2 不推荐方案

不推荐以下路径：

- 仅通过删除或新增 `spacer` 修正文本位置
- 继续在 `.sidebar-item` 基础上叠加更多局部覆盖
- 继续让不同节点类型使用不同 JSX 结构但试图通过 CSS 对齐

这些方案本质上仍然保留了当前问题的根因，后续只要视觉需求再变化，错位问题还会重复出现。


# 十、最终建议

本次应把左侧文件树正式提升为一个独立的树形布局系统，而不是继续把它当作“文件列表的一个变体”处理。

最关键的落地点只有两个：

1. 所有节点必须共享统一的 tree row 槽位模型
2. 树控制区和节点内容区必须解耦

只要这两个基础没有建立，任何对图标、折叠、拖拽、状态位的微调，都会继续在视觉和交互上产生连锁问题。

因此，后续执行应优先完成树行框架重构，再去做具体视觉调整。

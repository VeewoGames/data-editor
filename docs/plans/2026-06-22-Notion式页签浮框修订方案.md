# Notion式页签浮框修订方案

## 方案概述

### 1. 总体目标和范围

本方案的目标，是把当前共享视图页签与视图组的操作浮框，从“菜单项驱动”的结构，修订为更接近 Notion 的“属性编辑头部 + 动作列表”结构：

- 浮框顶部直接展示“当前图标 + 重命名输入框”
- 点击顶部图标后，打开独立的图标选择浮框
- 主浮框下半部分继续承载筛选、复制、删除等动作项

本方案固定以下实现约束：

- 保持当前入口语义：只有当前 active tab 或 active group 再点击一次才打开主浮框
- `borderAll` 继续作为默认图标，不替换
- icon picker 搜索默认全局覆盖整个白名单，不限制在当前标签页
- 视图组 header 左侧图标按钮第一版禁用，只保留结构占位，不打开 picker

本次范围包括：

- `ViewTabs` 视图页签浮框和视图组浮框的信息架构调整
- 视图重命名交互从“菜单项触发”改为“顶部输入框直接编辑”
- 图标选择交互从“菜单内嵌 icon grid”改为“独立次级 Popover”
- 第一版 icon picker 直接包含搜索框
- icon picker 图标白名单、默认项和最近分组规则收敛
- 对应状态流、焦点行为和测试策略修订

本次不包含：

- 视图类型切换能力本身，`显示为` 仍保持禁用预留
- emoji、自定义上传图片、远程图标资源、图标颜色系统
- 全局菜单系统重构

### 2. 各阶段任务概要

#### 阶段一：浮框信息架构修订

主要工作：

- 把当前“重命名”/`重命名组` 菜单项移除
- 在主浮框顶部增加 header 区域
- header 左侧显示当前图标按钮，右侧显示名称输入框
- 视图菜单和视图组菜单都统一为这套 header 结构
- 视图组图标按钮第一版明确为禁用态结构占位

预期成果：

- 用户打开页签浮框后，首先看到的是“当前页签属性”
- 图标和名称不再埋在菜单动作之间

执行顺序：

- 先固定 header 结构
- 再迁移原有 rename / icon 交互

#### 阶段二：交互模型切换

主要工作：

- 名称输入框改为浮框打开即直接可编辑
- 点击图标按钮时，打开独立 icon picker 浮框
- icon picker 第一版直接带搜索输入框
- icon picker 采用 2 字标签页分组，并内置足够数量的 filled 图标白名单
- 搜索规则固定为全局搜索白名单
- icon picker 选中后立即写回共享视图 icon，并关闭 picker

预期成果：

- 交互路径更接近 Notion
- “图标编辑”和“名称编辑”都成为一级属性编辑动作

执行顺序：

- 先完成主浮框内的 header 可用性
- 再接二级图标浮框
- 最后补焦点与关闭策略

#### 阶段三：动作区收敛

主要工作：

- 保留 `显示为` 禁用预留入口
- 保留 `编辑视图`、`拷贝视图链接`、`创建视图副本`、`删除视图`
- 视图组菜单保留组内创建、复制组、删除组等组级动作
- 移除当前无实际价值的 `来源 / 团队共享视图` 展示区
- 清理当前菜单中与新 header 重复的视觉元素

预期成果：

- 浮框结构分层清晰：上方属性、下方动作
- 不再出现“顶部已经有图标和名字，下面还有重命名和 icon 区块”的重复

执行顺序：

- 先删重复项
- 再调顺序和分隔线

#### 阶段四：验证与回归

主要工作：

- 更新 `ViewTabs` 结构断言
- 更新 Playwright 交互路径
- 验证顶层视图、组内视图、视图组都能走统一 header 交互

预期成果：

- 新交互不会破坏现有 shared view icon 存储链路
- rename / icon picker / duplicate / reload 都有测试保护

执行顺序：

- 先单测结构
- 再 e2e 验证真实操作流

### 3. 整体结构框架

修订后的浮框建议分成三层：

1. 主触发层：页签按钮本身
2. 主浮框层：header 属性编辑区 + 动作菜单区
3. 次级浮框层：带搜索框的 icon picker

核心边界如下：

- 图标编辑和名称编辑属于“视图属性编辑”
- `显示为` 仍属于未来“视图类型编辑”
- icon picker 是 header 的子交互，不再作为主菜单正文的一部分
- icon 数据链路保持现状，仍写入 shared view config，不进入 draft
- 视图和视图组共用统一的 header 式浮框骨架，只在动作区按对象类型分流
- icon picker 采用固定白名单和固定分组，不走动态全量图标扫描
- `最近`、默认项、搜索范围都采用固定规则，不交由实现时临时决定

## 当前实现与用户目标的差异

当前实现位于 [src/components/ViewTabs.tsx](C:/Code/data-editor/src/components/ViewTabs.tsx:434) 和 [src/components/ViewTabs.tsx](C:/Code/data-editor/src/components/ViewTabs.tsx:581)，其结构更接近“动作菜单”：

- `重命名` 是一个菜单项，点击后才切换为输入框
- `重命名组` 也是一个菜单项，和视图菜单分成另一套结构
- `更换图标` 不是独立浮框，而是直接把 icon grid 内嵌进当前菜单
- `来源 / 团队共享视图` 占一整行，但没有真实操作价值
- 图标选择区和动作区混在同一层里

用户截图对应的目标交互则更接近 Notion：

- 主浮框顶部就是图标和名称输入框
- 图标按钮点击后，再弹一个带搜索框的独立图标选择浮框
- 下方动作区保持菜单语义

所以这次不是简单换皮，而是一次交互模型切换：

- 从“菜单项触发编辑”
- 改成“浮框打开即进入属性编辑上下文”

## 需求拆解

### 1. 顶部 header

目标状态：

- 左侧：当前视图 icon 按钮
- 右侧：当前视图名称输入框

视图组对应规则：

- 左侧：当前组 icon 按钮
- 右侧：当前组名称输入框

说明：

- 视图组当前没有持久化 group icon 数据链路，本轮 header 结构应统一，但组图标按钮固定为禁用态占位，不打开 picker，也不接入真实存储。
- 也就是说，“视图组菜单统一成 header 式结构”是本轮硬要求；“组图标可持久化修改”不是本轮范围。

含义：

- icon 和名称是当前页签最核心的两个身份属性
- 它们应该在浮框打开时即直接可见，而不是先点某个动作项

### 2. 图标选择器改为次级浮框

目标状态：

- 点击 header 左侧 icon 按钮
- 在按钮附近打开带搜索框的 icon picker
- 选中图标后立即生效

这和当前实现的本质区别是：

- 当前：图标候选列表属于主菜单正文
- 目标：图标候选列表属于图标按钮的二级上下文

第一版强制要求：

- icon picker 顶部必须有搜索输入框
- 搜索范围仅覆盖本轮允许的受控图标子集
- 标签页名称只能用 2 个字
- 搜索执行范围固定为全局白名单，不因当前标签页变化而缩小

### 3. 名称输入框改为直接编辑

目标状态：

- 浮框打开后即可编辑名称
- 不再需要“重命名”菜单项作为前置动作

固定行为：

- `Enter` 保存并保留主浮框
- `Escape` 回退未提交修改；若 icon picker 打开则优先关闭 picker
- `blur` 自动保存
- 但点击 icon 按钮或 icon picker 内部元素，不视为普通 `blur` 提交触发源

### 4. 下方仍保留动作列表

视图主浮框下半部分保留动作型操作：

1. `显示为`
2. `编辑视图`
3. `拷贝视图链接`
4. `创建视图副本`
5. `删除视图`

其中：

- `显示为` 继续保持禁用预留
- `重命名` 菜单项应删除
- 当前内嵌 icon grid 应删除
- `来源 / 团队共享视图` 整块应删除

视图组主浮框下半部分保留动作型操作：

1. `在组内创建视图`
2. `复制组`
3. `删除组`

其中：

- `重命名组` 菜单项应删除
- 视图组动作区不再单独维持旧式菜单头部

## 推荐交互定义

### 主浮框打开

- 保持当前入口语义：只有当前 active tab 或 active group 再点击一次才打开主浮框
- 打开主浮框时，header 立即可见
- 输入框展示当前 view name
- 图标按钮展示当前 resolved icon
- 输入框默认获得焦点

### 点击图标按钮

- 打开次级 icon picker
- 主浮框保持打开
- 输入框不丢失上下文
- icon picker 顶部显示搜索框，默认聚焦搜索框
- 搜索结果默认全局过滤所有标签页图标，不限制在当前标签页

### 选择图标

- 立即调用现有 `onUpdateViewIcon(viewId, iconId)`
- icon picker 关闭
- 主浮框保持打开
- header 中的 icon 立即刷新

### 2. 状态管理

当前状态模型里，`renamingViewId` / `renamingGroupId` 的语义都是“是否显式进入重命名模式”。

修订后更合理的语义是：

- 只要某个 view menu 打开，就存在对应的 title draft
- rename 不再依赖单独的“切换到重命名模式”

因此更适合把状态改成：

- `openMenuViewId`
- `renameDraft`
- `iconPickerOpenForViewId`
- `openMenuGroupId`
- `groupRenameDraft`
- `iconPickerSearchQuery`
- `recentIconIds`

而不是继续沿用“菜单项式 rename 模式”。

### 3. Popover 焦点管理

这是本次实现的主要复杂点。

需要明确三件事：

1. 打开 icon picker 时，主浮框不能被 Radix 当作 outside click 关闭
2. 选 icon 后，主浮框应保留
3. `Escape` 的关闭层级要可控：
   - 若 icon picker 打开，优先关闭 picker
   - 再按一次才关闭主浮框

建议直接采用：

- 主浮框与 icon picker 都使用 Radix `Popover`
- icon picker 作为 header icon trigger 的嵌套 Popover
- 主浮框 `onInteractOutside` 对 icon trigger 和 picker content 做白名单放行

### 4. 样式层

当前样式主要围绕：

- `.view-tab-menu-content`
- `.view-tab-icon-option-list`
- `.view-tab-rename-form`

修订后需要新的样式分层：

- header 行布局
- icon trigger 按钮
- title input 的 hover / focus / active 态
- icon picker 的搜索区与 icon grid 容器
- 2 字标签页的横向切换样式
- 视图组 header 与视图 header 的共用样式

## 风险与边界

### 1. blur 提交与点击图标冲突

如果输入框 `blur` 就立即保存，而点击 icon 按钮会先让输入框失焦，就容易出现：

- 先触发 rename 保存
- 再打开 icon picker

这不一定错，但要保证它是稳定且可预期的，不然会让交互发飘。

### 2. 双浮框关闭顺序

若不控制好 `onInteractOutside` / `onOpenChange`，容易出现：

- 点击 icon 按钮时主菜单直接关闭
- 选 icon 后两层浮框都消失
- icon picker 打开时无法继续编辑标题

### 3. 组内视图、顶层视图与视图组一致性

这次只是交互层修订，数据层不应再分叉：

- 顶层视图和组内视图都必须走同一个 view menu header
- icon picker 也应复用同一套逻辑
- 视图组至少要复用同一套 header 外壳和输入框行为

## 推荐落地策略

建议按下面顺序实现：

1. 先保留现有数据层和 `onUpdateViewIcon()` 链路不动
2. 先统一视图菜单和视图组菜单的 header 结构
3. 第一版直接实现带搜索框的 icon picker
4. 把 2 字标签页分组和白名单一起落地
5. 再补双 Popover 焦点稳定与 e2e

原因是：

- 当前 icon 持久化链路已经通了，不应该再被交互重构牵着一起改
- 真正有风险的是浮框和焦点管理，不是 icon 数据本身
- 视图组这轮只统一 header 结构，不扩展 group icon schema，可以避免范围继续膨胀

## 建议验收标准

### 结构验收

- 主浮框顶部存在 icon 按钮和 title input
- 不再存在“重命名”菜单项
- 不再在主菜单正文直接渲染 icon grid
- 不再存在“来源 / 团队共享视图”整行
- 视图组菜单也存在 header 式结构
- 视图组图标按钮为禁用态占位，不打开 picker
- 默认项 `borderAll` 固定展示在搜索框下方、标签页上方

### 交互验收

- 点击页签打开主浮框
- 点击当前 active 视图组打开组浮框
- 在输入框内可直接改名
- 点击 icon 按钮后打开带搜索框的独立 icon picker
- icon picker 顶部存在 2 字标签页分组
- 搜索时不受当前标签页限制
- 搜索无结果时显示空态
- 选中图标后主浮框保持打开，icon 立即变化

### 数据验收

- 顶层视图改 icon 后刷新仍保留
- 组内视图改 icon 后刷新仍保留
- duplicate 后 icon 继承行为不回退
- `显示为` 仍然不接入任何 icon 状态
- 移除 `来源 / 团队共享视图` 后，不影响现有共享视图主链路

## 结论

你的需求不是简单的“把更换图标挪到上面”，而是把当前视图页签浮框，从“动作菜单”修订成“属性编辑器”。

最核心的变化是三点：

1. 视图菜单和视图组菜单都统一成 header 式结构
2. 图标和名称上移到 header
3. 图标选择器改成带搜索框的独立次级浮框
4. 图标池改为“保留 `borderAll` 默认项 + Filled 白名单 + 2 字标签页分组”
5. 主菜单正文只保留动作项，并移除无用的 `来源 / 团队共享视图`

这套方案和当前已经打通的 shared view icon 数据链路并不冲突，主要是 `ViewTabs` 交互层重构。下一步如果进入实现，重点应放在双 Popover 焦点管理、搜索框交互和视图组 header 统一，而不是再次改动 icon 存储模型。

# 详情面板条目级 Codex 自动化方案

## 方案概述

### 1. 总体目标和范围

本方案目标是在 `data-editor` 的条目详情面板右上角增加一组可配置的自动化按钮，用于针对当前条目发起 Codex 中的特定自动化处理，并为后续扩展为更多业务动作保留稳定结构。

本次方案范围包括：

- 详情面板右上角按钮区的接入方式。
- 条目级自动化动作的配置模型。
- 前端到后端的受控执行链设计。
- 传递给 Codex 的条目上下文 payload 边界。
- MVP 与后续阶段的拆分、风险和验证建议。

本次方案不包括：

- 直接实现完整 Codex 自动化执行链。
- 让前端直接执行任意 skill 名或任意本地命令。
- 第一版就支持自动修改文件并回写到当前编辑器。
- 将该能力泛化成任意第三方 AI/脚本平台接入框架。

### 2. 各阶段任务概要

第一阶段：收敛当前接入点与边界。

主要工作是确认详情面板头部按钮区、条目上下文来源、现有 API 能力和项目配置宿主。预期成果是明确该需求不是单纯的前端加按钮，而是需要一条新的受控执行链。

第二阶段：设计配置模型与执行链。

主要工作是定义 `entryActions` 一类项目级配置、`actionId -> executor` 的后端映射、条目 payload 和接口草案。预期成果是前端只负责展示和发起，后端负责执行控制与安全边界。

第三阶段：设计详情面板接入与状态反馈。

主要工作是定义右上角按钮展示规则、loading/失败/成功反馈、与现有 `commandSaving` 等状态的关系。预期成果是 UI 接入最小化，不破坏现有编辑和导航行为。

第四阶段：拆分实施路线。

主要工作是划分 MVP、二期增强、验证策略和非目标。预期成果是后续实现可以按阶段推进，而不是一次把“调用、执行、回写、同步”全部绑死。

执行顺序为：现状边界确认 -> 配置模型设计 -> 执行链设计 -> UI 接入设计 -> MVP 拆分 -> 验证与后续路线。

### 3. 整体结构框架

整体结构建议分为四层：

- 项目动作配置层：声明当前项目、当前数据文件或当前 collection 允许出现哪些条目级动作。
- 前端动作展示层：在 `DetailPanel` 中根据当前条目上下文渲染动作按钮并发起请求。
- 服务端执行编排层：校验 action、构造 payload、调用受控的 Codex executor，并返回状态。
- Codex 自动化执行层：按 `actionId` 映射到固定流程或固定 skill 组合，不让前端直接控制底层执行面。

## 背景

你当前的需求是：在条目详情面板的右上角增加一些按钮，用于调用 Codex 中的特定 skill，对当前条目做自动化处理。

从当前仓库实现看，这个需求有一个天然接入点，但没有现成执行链：

- 详情面板右上角已经有现成按钮区，位于 `src/detail/DetailPanel.tsx` 的 `detail-nav`。
- `DetailPanel` 已经能拿到当前条目的 `row`、`rowId`、`sourcePath`、`collectionPath`、`titleField`、`sourceRowIndex` 等关键上下文。
- 当前后端 API 只覆盖项目管理、文件读写、视图配置、重建和关闭，没有“执行 Codex 自动化”的 endpoint。
- 当前仓库中与 Codex 相关的内容主要是 `codex-plugin/skills/open-data-editor/SKILL.md` 这一类插件模板，不是应用内稳定可调的业务 API。

因此，这个需求的本质不是“再加几个 icon-button”，而是“在 detail panel 上接入一条受控、可配置、可扩展的条目级自动化能力”。

## 现状证据与约束

### 1. 前端已有稳定接入点

`src/detail/DetailPanel.tsx` 当前头部结构已经包含：

- 文档面板切换按钮
- 上一条/下一条按钮
- 关闭详情按钮

这意味着条目级自动化按钮可以直接复用现有 `detail-nav` 区域，不需要新增独立工具栏或重构详情面板主体结构。

### 2. 条目上下文已经基本齐备

`src/App.tsx` 构造的 `detailSnapshot` 已经包含：

- `row`
- `rowId`
- `sourceRowIndex`
- `sourcePath`
- `collectionPath`
- `titleField`
- `primaryKeyField`

这意味着第一版不需要为“拿到当前条目是谁”额外重做状态模型，只需要把这些上下文按受控方式序列化给后端。

### 3. 当前后端没有执行 Codex 自动化的业务接口

`server.mjs` 目前提供的是：

- `/api/projects`
- `/api/files`
- `/api/document`
- `/api/save`
- `/api/view-config`
- `/api/shared-views`
- `/api/view-profile`
- `/api/health`
- `/api/rebuild`
- `/api/shutdown`

它没有任何与 skill、automation、command executor 直接对应的 endpoint。因此如果前端新增按钮，后端也必须补一条新接口。

### 4. 当前项目注册模型不适合直接承载业务动作

`src/project-registry.mjs` 与 `src/api/client.ts` 中的项目定义目前主要承载：

- `id`
- `name`
- `root`
- `adapter`
- `dataSources`
- `filePolicy`

这些字段更像运行环境注册，而不是项目业务工作流配置。把条目级动作直接塞进 registry，会把团队共享配置和个人本机状态混在一起。

### 5. 项目模板示例更适合做共享动作配置宿主

`examples/nocturnel/data-editor.project.json` 已经在表达业务项目的：

- 数据目录
- 配置路径
- title field 策略
- primary key 策略
- relation 策略

因此，条目级自动化动作如果是“项目工作流的一部分”，更适合作为项目级共享配置进入这一层，而不是进入本机 registry。

## 目标

本方案的目标是：

- 让 `DetailPanel` 可以基于当前条目渲染一组项目定义的自动化按钮。
- 每个按钮只对应一个受控的 `actionId`，不让前端直接传任意 skill 名。
- 后端根据 `actionId` 构造条目上下文并发起 Codex 自动化执行。
- 第一版优先做到“发起自动化 + 反馈状态 + 可追踪失败”，而不是直接自动改回当前条目。
- 为后续加入更多业务动作保留统一结构。

## 非目标

- 不让用户在前端输入任意命令或任意 skill 名后直接执行。
- 不在第一版处理自动回写、冲突合并、编辑器局部热刷新这一整套复杂链路。
- 不把本轮方案做成通用 agent 平台或任意 AI provider 抽象层。
- 不把条目级自动化与现有保存流程 `commandSaving` 完全耦合成同一个状态机。

## 推荐方案

推荐采用“项目级动作配置 + 服务端受控 executor + 详情面板轻量接入”的结构。

核心原则是：

- 前端只认识动作定义，不认识底层 skill 细节。
- 后端只接受白名单 `actionId`，不接受任意 skill 输入。
- 项目配置声明“哪些条目在什么条件下可以显示什么动作”。
- Codex 侧执行链由服务端维护，不暴露给前端。

这条路线比“前端直接写死调用 skill”更稳，原因有三点：

- skill 名、调用方式、会话组织方式都属于运行时细节，不应成为前端协议的一部分。
- 同一个动作未来可能从“开 Codex 会话”变成“走本地脚本 + Codex 审阅”，`actionId` 能保住前端协议稳定。
- 可以明确限制哪些动作允许在什么项目、什么文件、什么 collection 上出现，避免把编辑器变成任意命令入口。

## 配置模型设计

### 1. 第一版配置真值源

第一版推荐先扩展现有 `project-registry` / `projects.json` / `/api/projects` 这一条运行时真配置链来承载 `entryActions`，而不是第一步就把运行时真值切到 `data-editor.project.json`。

原因是：

- 当前编辑器运行时真正稳定存在的项目配置链，只有 `project-registry -> server -> client`。
- `examples/nocturnel/data-editor.project.json` 目前只是项目模板示例，不是运行时真实配置源。
- 如果第一版直接依赖新的项目模板加载链，功能会先卡在“动作配置从哪里读取”，而不是卡在条目动作本身。
- 先走现有 registry/API，能最小成本打通“配置读取 -> 前端展示 -> 后端校验 -> 动作发起”。

因此，第一版的推荐分层是：

- 运行时真值：现有项目 registry 中的 `entryActions`
- 长期共享目标：未来正式的 `data-editor.project.json`
- 后续迁移路线：支持把项目模板中的 `entryActions` 导入或同步到运行时 registry

这条路线的含义不是否定项目级共享配置，而是先把“项目级共享配置”拆成两个阶段：

- 第一阶段：先在当前可运行链路里落地
- 第二阶段：再把 `data-editor.project.json` 做成团队共享模板和导入源

### 2. 配置 ownership 与优先级

为避免和现有配置面混淆，第一版明确约束如下：

- `projects.json`
  承载条目级动作的运行时真值。
- `view-config.json`
  不承载条目动作，只继续负责字段、主键、relation、document 等编辑器配置。
- `shared-views.json`
  不承载条目动作，只继续负责共享视图定义。
- `view profile`
  不承载条目动作，只继续负责个人布局、模式和偏好。
- `data-editor.project.json`
  第一版仅作为未来共享模板目标，不作为运行时直接真值源。

### 2. 推荐结构

建议新增一层 `entryActions`：

```json
{
  "entryActions": [
    {
      "id": "nocturnel-skill-review",
      "label": "复核词条",
      "icon": "automation",
      "targets": {
        "files": ["data/skills.json"],
        "collections": ["skills"]
      },
      "payload": {
        "includeRow": true,
        "includeNeighbors": false
      }
    }
  ]
}
```

字段建议：

- `id`
  动作稳定标识，前后端协议主键。
- `label`
  详情面板按钮文案。
- `icon`
  复用现有图标系统中的图标 id，不在配置里传原始 SVG。
- `targets.files`
  哪些文件可显示。
- `targets.collections`
  哪些 collection 可显示。
- `payload`
  控制要附带哪些上下文。

第一版不建议把 `executor` 细节写进前端消费的动作配置里。

原因是：

- 前端不需要知道具体如何调用 Codex。
- 运行时 executor 细节属于服务端实现，不应该成为前端协议的一部分。
- 如果后续 executor 更换实现，前端配置不应跟着变。

### 3. 为什么不直接配置 `skillName`

不推荐：

```json
{ "skillName": "some-skill" }
```

理由：

- 把 Codex 内部能力命名暴露成前端协议，后续变更成本高。
- skill 可能需要额外上下文、前置步骤、会话模板，不一定是单一名称可表达。
- 未来一个动作可能组合多个 skill 或切换到底层实现。

更稳的做法是：

- 配置层存 `actionId`
- 后端维护 `actionId -> executorSpec`

## 执行链设计

### 1. 总体数据流

建议数据流如下：

1. `DetailPanel` 根据当前 `detailSnapshot` 与项目动作配置，决定展示哪些按钮。
2. 用户点击按钮。
3. 前端向新接口 `POST /api/entry-actions/run` 发请求。
4. 后端校验：
   - `actionId` 是否存在
   - 当前项目是否允许该动作
   - 当前 `sourcePath` / `collectionPath` 是否匹配目标范围
5. 后端构造条目上下文 payload，并写入结构化 handoff 文件。
6. 后端调用固定本地包装器，由包装器负责发起真实的 Codex 自动化执行。
7. 前端收到同步结果：
   - 已发起
   - 已拒绝
   - 执行失败
8. 第一版仅展示状态和追踪信息，不自动改写当前条目。

### 2. 第一版 executor 落地方式

第一版推荐采用“两段式执行”：

- 第一段：后端生成结构化 handoff 文件
- 第二段：后端调用固定本地包装器

推荐原因：

- 当前仓库没有现成稳定的“应用内直接调用 Codex skill”的协议。
- 只做 handoff 文件，用户感知不够强，难以确认动作是否真的发起。
- 直接把前端或服务端绑死在某个 Codex 内部调用方式上，后续变更成本高。
- “handoff 文件 + 包装器”可以把输入稳定下来，同时把真实调用面封装在可替换层里。

第一版建议流程：

1. 服务端生成 `runId`
2. 服务端把本次动作请求写入 `.data-editor/runtime/entry-actions/<runId>.json`
3. 服务端调用固定包装器，例如：

```text
node scripts/run-entry-action.mjs <runId>
```

4. 包装器读取 handoff 文件，再决定如何发起 Codex 自动化
5. 服务端同步返回 `started + runId`

这样做的好处是：

- handoff 文件可审计、可重放、可调试
- 包装器可替换，不影响前端协议
- 第一版就能做“执行面真实性”验证

### 3. 推荐接口草案

建议新增：

```http
POST /api/entry-actions/run
```

请求体建议：

```json
{
  "projectId": "nocturnel",
  "actionId": "nocturnel-skill-review",
  "sourcePath": "data/skills.json",
  "collectionPath": "skills",
  "rowId": "fireball",
  "sourceRowIndex": 14
}
```

成功响应建议：

```json
{
  "ok": true,
  "status": "started",
  "runId": "ea_20260701_001",
  "displayMessage": "已在 Codex 中发起复核流程"
}
```

失败响应建议：

```json
{
  "ok": false,
  "status": "rejected",
  "errorCode": "ACTION_NOT_ALLOWED",
  "displayMessage": "当前条目不允许执行该动作"
}
```

### 4. 最小接口契约

第一版最小契约建议如下：

请求字段：

- `projectId`
- `actionId`
- `sourcePath`
- `collectionPath`
- `rowId`
- `sourceRowIndex`

服务端校验项：

- `projectId` 必须存在，且建议只允许当前活动项目调用
- `actionId` 必须存在于当前项目 registry 的 `entryActions`
- `sourcePath` 必须命中 `targets.files`
- `collectionPath` 必须命中 `targets.collections`
- `sourcePath` 必须属于当前项目允许的数据源边界
- `rowId` 和 `sourceRowIndex` 至少有一个存在

返回状态建议：

- `started`
  已成功生成 handoff 文件并成功调用包装器
- `rejected`
  请求格式合法，但不满足动作范围或安全校验
- `error`
  运行时异常，例如写文件失败、包装器启动失败

失败原因枚举建议：

- `UNKNOWN_PROJECT`
- `UNKNOWN_ACTION`
- `ACTION_NOT_ALLOWED`
- `PATH_OUT_OF_SCOPE`
- `MISSING_ROW_TARGET`
- `HANDOFF_WRITE_FAILED`
- `EXECUTOR_START_FAILED`

### 5. 本地安全边界

新增 `POST /api/entry-actions/run` 后，它的敏感度高于普通读写接口，因此第一版方案必须明确以下安全边界：

- 只允许已注册项目调用，不允许任意自定义项目根
- 只允许命中当前项目 `entryActions` 白名单的 `actionId`
- 只允许命中当前项目数据源边界内的 `sourcePath`
- 建议限制为当前活动项目调用，避免跨项目误触发
- 后端不得接受任意 skill 名、任意命令行参数或任意脚本路径

第一版不额外引入更重的本地鉴权体系，但要把“项目白名单 + 路径白名单 + actionId 白名单”作为硬边界。

### 6. 条目上下文 payload

服务端传给 Codex 的 payload 建议包含：

- `projectRoot`
- `projectId`
- `adapter`
- `sourcePath`
- `collectionPath`
- `rowId`
- `sourceRowIndex`
- `title`
- `primaryKeyField`
- `row`

可选字段：

- `neighborRows`
- `viewConfigExcerpt`
- `relationContext`

第一版建议只发最小必要集，不默认塞整份文档模型或整文件全文，避免：

- payload 过大
- 暴露过多无关状态
- 让执行面误把周边上下文当真相

### 7. 条目标识规则

第一版不把条目标识设计成“永久实体标识”，而是设计成“当前编辑器中这一次选中的目标条目”。

因此，第一版统一使用下面这组组合作为目标标识：

- `sourcePath`
- `collectionPath`
- `sourceRowIndex`

若当前数据天然存在稳定主键，则额外附带：

- `rowId`
- `primaryKeyField`

这条规则的含义是：

- `rowId` 是增强信息，不是第一版硬前提
- 没有稳定 `rowId` 的根数组文件、mixed-root JSON 条目也允许发起动作
- 第一版不支持 nested detail 子结构直接作为独立动作目标；动作目标只针对主详情面板选中的顶层条目

这样做的原因是：

- 不是所有数据都天然有稳定主键
- 第一版不做自动回写，因此没有必要先解决“长期实体身份”问题
- 当前更重要的是稳定表达“用户现在点中的这一条”

## 详情面板接入设计

### 1. UI 接入位置

接入位置建议直接放在 `detail-nav` 中，与现有图标按钮并列。

原因：

- 当前用户心智就是“右上角是当前条目的直接操作”。
- 不需要再做新的二级工具栏。
- 视觉和交互影响面最小。

### 2. 展示规则

按钮展示建议同时满足以下条件：

- 当前有选中条目。
- 当前项目存在 `entryActions` 配置。
- 当前 `sourcePath` 和 `collectionPath` 命中某个动作的 `targets`。
- 当前不处于该动作自己的运行中状态。

不建议简单按“项目启用即全显示”，否则会在无关条目上制造噪音。

### 3. 状态反馈

第一版建议提供三类状态：

- `idle`
  默认可点击。
- `running`
  当前动作已发起，按钮临时 disabled，并显示转圈或 active 状态。
- `error`
  当前动作发起失败，弹出 toast 或行内提示。

成功后不建议立刻显示“已完成并已更新条目”，因为第一版并不承诺结果已经回写编辑器。

### 4. 第一版异步生命周期边界

第一版只处理“发起动作”这一瞬时流程，不处理完整异步生命周期。

明确边界如下：

- 服务端生成 `runId`，用于日志、handoff 文件和包装器追踪
- 前端只在当前页面会话内维护 `running` 态
- 页面刷新后，不恢复 `running` 状态
- 第一版不做结果轮询
- 第一版不做取消
- 第一版不做历史面板
- 同一条目同一动作在当前页面会话内应禁用重复点击
- 是否允许同一条目多个不同动作并发，第一版建议不支持，统一按“当前页面一次只发起一个条目动作”处理

这部分必须在实现和验收里当作明确 scope cut，而不是隐含行为。

### 5. 与现有状态的关系

不建议把条目级自动化直接并入 `commandSaving`。

推荐关系：

- `commandSaving` 继续只表示现有保存/同步类命令。
- `entryActionRunning[actionId]` 单独维护每个动作的运行状态。

原因是：

- 自动化发起并不等于当前文档保存。
- 一个条目动作失败，不应污染整个编辑器的保存状态。
- 后续如果允许多个只读类动作并存，独立状态更容易扩展。

## 实施拆分

### MVP

MVP 建议只做：

- 基于现有 registry/API 的 `entryActions` 配置读取。
- `DetailPanel` 右上角动作按钮渲染。
- `POST /api/entry-actions/run`。
- 后端 `actionId` 白名单校验和 payload 构造。
- 结构化 handoff 文件生成。
- 固定本地包装器启动。
- 前端状态反馈。

MVP 明确不做：

- 自动回写当前条目。
- 自动刷新当前文件内容。
- 多步骤审批流。
- 动作执行历史面板。
- 结果轮询。
- 跨刷新状态恢复。
- nested detail 子结构动作。
- 任意 skill 名直连。

### 二期

二期可考虑：

- 动作执行历史与最近结果追踪。
- 动作参数化输入。
- 执行完成后的手动“应用结果”或 diff 审核。
- 与文档字段、relation 上下文的更深联动。

### 三期

三期才考虑：

- 自动回写文件。
- 冲突检测与合并。
- 执行后局部刷新当前条目。
- 更复杂的多 action 编排。

## 风险与边界

### 1. 直接暴露 skill 名的风险

如果前端直接携带 skill 名：

- 协议不稳定。
- 安全边界模糊。
- 项目配置会与 Codex 内部实现强耦合。

因此必须通过 `actionId` 做受控映射。

### 2. 自动回写过早引入复杂度

一旦第一版就要求自动回写，会立刻引入：

- 当前编辑器脏状态与外部写入冲突。
- 条目级刷新与整文件刷新策略。
- 失败回滚和部分成功。
- 当前选中行位置保持。

这会让方案从“发起自动化”升级为“分布式编辑协调”，不适合作为首批目标。

### 3. 配置层宿主混乱的风险

如果把动作写在本机 registry：

- 团队无法共享。
- 不同机器行为漂移。
- 仓库评审和版本管理失效。

因此第一版虽然把运行时真值放在 registry，也必须把长期路线明确为“未来由项目模板导入或同步”，而不是让 registry 长期单独演化。

## 验证建议

第一轮实现完成后，至少验证：

1. 在命中 `targets` 的条目上，详情面板显示正确动作；不命中时不显示。
2. 点击按钮后，请求体只包含允许字段，没有把无关全局状态打包出去。
3. 后端能拒绝未知 `actionId` 和越界条目。
4. 动作发起失败时，不会污染现有编辑、保存和行切换状态。
5. 同一条目连续触发时，按钮状态能正确防抖或禁用。
6. 动作成功发起后，能在 `.data-editor/runtime/entry-actions/` 生成可核对的 handoff 文件。
7. 固定包装器能留下可核对的启动记录或结果记录，证明按钮点击确实触发了真实执行面。

## 推荐决策结论

本轮推荐直接定下五件事：

1. 第一版运行时真值先使用现有 registry/API 承载 `entryActions`。
2. 长期共享目标再收敛到 `data-editor.project.json` 模板导入或同步。
3. 前后端协议使用 `actionId`，不直接暴露 `skillName`。
4. 第一版 executor 采用“handoff 文件 + 固定本地包装器”的两段式落地方式。
5. MVP 只做“发起 Codex 自动化 + 返回状态反馈”，不做自动回写编辑器。

## 后续实施建议

如果下一轮进入实现，建议按这个顺序开工：

1. 先补配置读取与类型定义。
2. 再补服务端 `entry-actions` endpoint 和 executor 映射。
3. 然后接入 `DetailPanel` 右上角按钮区。
4. 最后做最小验证和文档更新。

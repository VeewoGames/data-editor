# 表格所有 cell 纵向对齐统一方案

## 方案概述

### 1. 总体目标和范围

本方案目标是彻底统一 data-editor 主表格所有 cell 类型在普通行、top-layout 行、自动换行列启用场景、编辑态行中的纵向对齐体系，消除当前标题列、普通文本、数字、ID、Select / Multi-select / Relation chip、Backlink、Nested summary、Checkbox、不兼容占位态在同一表格中出现“有的顶部对齐、有的仍垂直居中”的问题。

当前问题不是单个 CSS 属性错误，而是纵向对齐职责分散在多层结构中：

- 行级：`data-row-layout="top|center"` 决定整行主布局。
- frame 级：`TableCellFrame` 的 `layout="center|top"` 决定单元格容器行为。
- 内容级：`.table-text-cell-editor`、`.title-cell-button`、`.chips-cell`、`.multi-select-trigger`、`.nested-summary` 等内部又各自声明 `align-items`、`min-height`、`padding`。
- 测量级：wrapped 行高依赖 `getBoundingClientRect().height` 缓存，不同内容结构会把几何差异放大到真实滚动和重排路径中。

本方案先明确沿用当前主表的表级判定语义：只要当前视图存在 wrapped 字段，所有可见行统一进入 `top` 布局模式，而不是在本次治理中额外引入逐行高行判定。逐行判定如果未来需要，会作为独立方案处理。

只要这些层级没有统一的单一真相来源，就会持续出现“某类字段修好了，另一类字段又回退”的问题。

本方案覆盖：

- 主表格 body 中所有可见 cell 类型的纵向对齐规范。
- `Title`、`Text`、`Number`、`ID`、`Select`、`Multi-select`、`Relation`、`Backlink`、`Nested summary`、`Checkbox`、`Incompatible`、文本编辑态的统一对齐 contract。
- 普通行、top-layout 行、编辑开关前后、焦点浮框激活前后的纵向几何一致性。
- 对应 Playwright E2E 回归矩阵。

本方案不覆盖：

- 详情页字段布局。
- 表头区域和列拖拽布局。
- 弹层内部视觉风格。
- autosave / save coordinator 架构。
- 字段类型语义与数据模型。

### 2. 各阶段任务概要

1. **规范收敛阶段**
   - 主要工作：定义统一的纵向对齐规则，明确 top-layout 模式的判定粒度，并消除例外类型的模糊定义。
   - 预期成果：形成可执行的单一布局规范，后续实现不再靠截图逐个修补。
   - 执行顺序：第一步，先定规则，再改代码。

2. **布局层级梳理阶段**
   - 主要工作：梳理行级、frame 级、内容级、测量级四层职责，并找出当前冲突点。
   - 预期成果：明确哪一层是纵向对齐的唯一真相来源，哪些内部组件必须停止自行决定垂直位置。
   - 执行顺序：规则确定后立刻进行。

3. **统一 frame contract 阶段**
   - 主要工作：收敛 `TableCellFrame`、`table-cell-content` 与 `table-cell-content-main` 的对齐职责，让所有 body cell 走同一纵向布局协议。
   - 预期成果：外层容器统一负责纵向对齐，内容组件只负责内容表现。
   - 执行顺序：在改具体字段类型前先完成。

4. **按渲染路径迁移阶段**
   - 主要工作：依次迁移 `text-shell`、`title-button`、`token-trigger`、`special-summary`、`checkbox`、编辑态文本。
   - 预期成果：不同字段类型即使业务语义不同，只要落在同一渲染路径，就遵守同一套纵向对齐语义。
   - 执行顺序：先文本壳层，再 token，再特殊分支，最后编辑态。

5. **测高与回归锁定阶段**
   - 主要工作：统一 wrapped 行测高来源，补齐“同一行多类型混排”的 E2E。
   - 预期成果：切换编辑开关、切 wrapped、切字段类型时，不再发生纵向对齐回退。
   - 执行顺序：实现收敛后立即补齐。

### 3. 整体结构框架

目标结构如下：

```tsx
<tr data-row-layout="center|top">
  <td className="data-cell" data-column-field={fieldName} data-wrap-mode="truncate|wrap">
    <div
      className="table-cell-frame"
      data-cell-frame-kind="title|text|token|editor|checkbox|nested|backlink|incompatible"
      data-cell-frame-layout="center|top"
    >
      <div className="table-cell-content" data-cell-content-kind="...">
        <div className="table-cell-content-main">
          {content}
        </div>
        {issueNode}
      </div>
    </div>
  </td>
</tr>
```

纵向对齐职责必须固定为：

- `tr[data-row-layout]`：只表达“当前表格是否进入 top-layout 模式”。本方案中它不是逐行真实测高结果，而是视图级 wrapped 模式开关的投影。
- `.table-cell-frame[data-cell-frame-layout]`：作为单元格纵向对齐的唯一真相来源。
- `.table-cell-content` / `.table-cell-content-main`：只负责内容排布和 accessory 排列，不再重写纵向基线。
- 各种内容组件：只负责文本、chip、按钮、输入框的局部外观，不再通过 `align-items: center` 把自己重新拉回垂直居中。

## 一、问题定义

### 1. 当前用户可见问题

当前表格中，只要当前视图启用了会导致 top-layout 的 wrapped 字段，例如 `description` 打开自动换行，所有可见行都会切到 top-layout；但各列内部内容并不会稳定遵守“顶部对齐”：

- 标题列有时顶部对齐，有时仍接近垂直居中。
- `skill_id`、`description_zh` 这类单行文本列常常仍停留在中间。
- 数字列如 `ap_cost`、`cooldown`、`mana_cost` 在高行中经常看起来偏中线。
- token / chip 列与普通文本列在同一行中的顶部起点不一致。
- 编辑开关打开后，一部分文本列进入输入壳层，纵向起点会再次发生微调。

这与之前约定的规则冲突：表格一旦进入 top-layout 模式，所有 cell 类型都应使用统一的顶部对齐节奏。

### 2. 为什么这不是简单样式问题

如果只是某个 `td` 少写了 `vertical-align: top`，它应该只影响某一类 cell，且能通过局部补丁立即收口。当前不是这样：

- 行级已经有 `data-row-layout="top"`，但它当前来自表级 `hasWrappedField`，不是逐行测高。
- 一部分 frame 已经在高行里 `align-items: flex-start`。
- 但内部内容组件依然存在 `align-items: center`、`min-height`、`padding`、不同 `line-height`。
- wrapped 行高测量是真实参与虚拟滚动和缓存的，不是纯视觉差异。

这说明问题横跨多个层级，是一个系统性布局 contract 没有统一完成的问题。

### 3. 外部语义依据

本方案的布局判断基于 CSS 标准语义，而不是经验式截图修补：

- `align-items` 决定 flex/grid 直接子项在交叉轴上的对齐方式，`center` 会把子项居中，`start` / `flex-start` 会把子项贴到起始边。[MDN align-items](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Properties/align-items)
- `vertical-align` 只适用于 inline / inline-block / table-cell 语义，不适合作为当前多层 flex 容器体系里的主业务布局机制。[MDN vertical-align](https://developer.mozilla.org/en-US/docs/Web/CSS/Reference/Properties/vertical-align)

结论是：只要我们的真实布局容器是 flex 层级，就必须把纵向对齐真相收敛到 `align-items` 所在的容器，而不是继续依赖 table-cell 语义或内部子节点各自居中。

## 二、目标对齐规范

### 1. 统一规则

本方案建议采用以下统一规则：

1. 只要当前视图存在 wrapped 字段，整张表进入 top-layout 模式，所有可见数据 cell 默认进入顶部对齐。
2. 顶部对齐的定义是“内容起点一致”，不是“视觉大致靠上”。
3. 编辑开关不能改变纵向对齐规则，只能改变交互能力。
4. `Text`、`Title`、`Number`、`ID` 在 top-layout 中必须共享同一组顶部 inset。
5. `Select`、`Multi-select`、`Relation`、`Backlink chip` 在 top-layout 中也必须共享同一组顶部 inset。
6. `Nested summary`、`Incompatible`、`Checkbox` 在 top-layout 中一并统一到顶部对齐，不保留例外。
7. `issue accessory` 不参与定义主内容纵向真相，只能作为 trailing accessory 挂靠在内容区，不得通过自身高度把主内容重新拉回中线。
8. 任意对齐修复都必须落到统一 contract，不允许继续为某一类组件单独打补丁。

### 2. 两种布局模式与一个交互态

本方案最终只保留两种真实布局模式：

- `center`
  - 仅用于普通低行，并且只在当前视图没有 wrapped 字段时使用。
- `top`
  - 用于 top-layout 模式下的所有数据 cell，是主规则。

`editor` 不再被视为布局模式，而是交互态标记：

- `editor`
  - 只表示当前 cell 处于文本编辑交互链路中。
  - 它不能再拥有独立纵向几何语义，必须继承 `center` 或 `top` 其中之一。

### 3. 统一的顶部 inset 规则

最终必须统一的不只是 `align-items`，还包括以下节奏参数：

- `padding-top`
- `padding-bottom`
- `min-height`
- 文本 `line-height`
- chip trigger 的额外高度
- 标题按钮的点击区域内边距
- issue icon 与主内容的并排方式

只要这些参数散落在不同组件里，就无法真正做到“所有 cell 顶部对齐”。

## 三、现状根因拆解

### 1. 行级规则已存在，但没有压过内部组件

当前主表格已经有：

- `tr[data-row-layout="top"]`
- `.table-cell-content`
- `.table-cell-content-main`

并且 top-layout 情况下，这些层已经会切到 `align-items: flex-start`。

但问题在于，内部内容组件并没有把这个规则当作最终真相：

- `.table-text-cell-editor` 默认仍是 `align-items: center`
- `.title-cell-button` 有自己的一套对齐规则
- `.chips-cell` 与 token trigger 仍带有独立的对齐与高度约束
- 某些按钮型内容既要保持点击热区，又把自身当作布局壳层

所以外层说“高行要 top”，内层又说“我自己还是 center”，最后用户看到的就是不统一。

### 2. Text 与 Editor 的统一骨架修复了大问题，但放大了残余差异

前一轮修复已经把普通文本和编辑态骨架统一，解决了“多行编辑塌成单行”和“大幅跳高”的问题。  
但这也让系统更清晰地暴露出剩余问题：

- Text 内部仍是 `span`
- Editor 内部是 `input` / `textarea`
- Number、ID、token、title 并未完全收敛到同一套内部定位语义

因此现在剩下的问题不再是大幅度布局错位，而是不同类型组件在 top-layout 中的纵向起点没有统一。

### 3. 当前最典型的不一致来源

按当前结构，容易出问题的类型有：

- **单行文本型字段**
  - 例如 `skill_id`、`description_zh`
  - 外层已经 top，但内部仍可能因为统一编辑壳层的 `align-items: center` 看起来居中
- **数字型字段**
  - 例如 `ap_cost`、`cooldown`、`mana_cost`
  - 视觉上比 wrapped 文本更容易暴露中线偏移
- **token 型字段**
  - trigger、chip、自身最小高度和 padding 会制造不同起点
- **标题字段**
  - `title-cell-button` 兼有按钮和文本双重职责，最容易和普通文本脱节

## 四、实施方案

### 1. 收敛纵向对齐真相层

本方案建议把纵向对齐真相固定在 `TableCellFrame` 层，而不是继续让内容组件自行决定。

约束如下：

- `tr[data-row-layout]` 只负责告诉 frame：当前表格是否处于 top-layout 模式。
- `TableCellFrame` 根据表格布局模式和渲染路径生成最终 `data-cell-frame-layout`。
- `.table-cell-content` 与 `.table-cell-content-main` 只做宽度、accessory、内容流排布，不再主动居中。
- 所有内容组件默认继承 frame 结果，不再声明新的纵向真相。

### 2. 统一 frame 推导规则

建议在 `table-columns.tsx` 中统一生成 frame meta，明确每类渲染路径的布局含义：

- `title-button`
  - 普通模式：`center`
  - top-layout：`top`
- `text-shell`
  - 普通模式：`center`
  - top-layout：`top`
- `token-trigger`
  - 普通模式：`center`
  - top-layout：`top`
- `special-summary`
  - 包括 `nested`、`backlink`、`incompatible`
  - 普通模式：`center`
  - top-layout：`top`
- `checkbox`
  - 普通模式：`center`
  - top-layout：`top`
- `editor-state`
  - 不再拥有独立布局模式
  - 只继承当前 cell 的 `center` 或 `top`

关键点是：top-layout 进入后，不保留纵向居中的例外种类。

### 3. 统一内容容器 contract

内容层建议统一为：

```css
.table-cell-content {
  display: flex;
  width: 100%;
  min-width: 0;
  gap: 6px;
}

.table-cell-content-main {
  display: flex;
  width: 100%;
  min-width: 0;
}
```

top-layout 场景下：

- `.table-cell-content`
- `.table-cell-content-main`

都不应该再额外写 `align-items: center`。  
是否贴顶只由 frame 决定。

### 4. 收敛 text-shell 渲染路径

`Text`、`Number`、`ID` 应按同一条 `text-shell` 路径统一纵向节奏：

- 普通模式可视觉居中
- 一旦表格进入 top-layout，就统一继承 frame 顶部对齐
- 内部不得再通过单独的 flex 垂直居中抵消外层规则

特别是当前只读文本也使用 `.table-text-cell-editor` 外壳后，必须小心这层的默认 `align-items: center`。  
如果它继续作为文本型内容壳层存在，就需要在 top-layout 只读态下显式切到顶部对齐。

### 5. 收敛 Title

标题列必须被视为文本型字段，而不是“按钮例外”：

- 点击热区要保留整格打开详情
- 但按钮内部不能再决定自己是否垂直居中
- 标题文本的顶部 inset 必须和普通文本一致

这意味着 `title-cell-button` 的视觉按钮语义与表格布局语义要彻底分离：

- 表格布局由 frame 控制
- 标题按钮只负责交互和文本装饰

### 6. 收敛 token-trigger 渲染路径

`Select`、`Multi-select`、`Relation`、`Backlink chip` 必须遵守：

- trigger 的点击区域不再通过额外上下 padding 改变整格纵向节奏
- chip 自身可以有内边距，但不能改变外层内容起点
- top-layout 中 token 起点必须和普通文本起点一致

对 token 类字段，常见错误是：

- 用 trigger 的最小高度维持低行舒适性
- 结果在 top-layout 里继续垂直居中

本方案要求把“低行舒适性”限制在普通模式，top-layout 统一服从 top。

### 7. 收敛 Nested / Backlink / Incompatible / Checkbox

这几类特殊分支必须显式纳入统一 contract，不能默认为“反正用户不常看”：

- `Nested summary`
  - 视作文本型摘要，服从高行 top
- `Backlink`
  - 文本型 backlink 与 chip 型 backlink 都服从高行 top
- `Incompatible`
  - 作为提示信息块，服从高行 top
- `Checkbox`
  - 在本方案中不再保留例外，top-layout 中统一顶部对齐

### 8. `issue accessory` 单独收口

`issue accessory` 必须作为 trailing accessory 明确建模，不能继续隐式跟着不同内容组件漂移：

- 它不参与决定主内容的纵向真相。
- 普通模式下可与主内容共同居中。
- top-layout 下必须 `self-start`，并贴齐主内容顶部节奏。
- 它不能通过自身高度把 `.table-cell-content-main` 重新挤回中线。

### 9. 编辑态浮框与布局参与节点分离

编辑态必须满足两条：

1. 参与行高和顶对齐的是布局节点  
2. 焦点边框、阴影、浮框感由覆盖层提供

不能再让焦点浮框自己决定：

- 纵向起点
- 实际布局高度
- 行高来源

否则每次切编辑态都会重新引入对齐漂移。

### 10. 目标态判定函数

为避免规则再次分散，目标态应明确收敛成下面这条判定链：

```ts
tableLayoutMode = hasWrappedField ? "top" : "center";
cellRenderPath = resolveRenderPath(columnModel, displayType, value, interactiveState);
cellFrameLayout = tableLayoutMode;
cellFrameKind = resolveFrameKind(cellRenderPath);
issueAccessoryLayout = tableLayoutMode === "top" ? "self-start" : "center";
```

解释：

- `tableLayoutMode`：本方案沿用当前实现，仍是表级判定，不在此次治理中改成逐行测高。
- `cellRenderPath`：按真实渲染路径，而不是按业务字段名分组。
- `cellFrameLayout`：不再给 `editor` 单独一套纵向布局。
- `issueAccessoryLayout`：必须显式声明，避免它继续隐式影响主内容。

## 五、回归测试方案

### 1. 必补 E2E 场景

至少覆盖以下场景：

1. 同一 top-layout 行内 `Title / Text / Number / ID` 顶部 inset 一致
2. 同一 top-layout 行内 `Text / Select / Multi-select / Relation` 顶部 inset 一致
3. 同一 top-layout 行内 `Backlink / Nested summary / Incompatible / Checkbox` 顶部 inset 一致
4. 编辑开关前后，同一 top-layout 行内所有字段类型的顶部起点不漂移
5. wrapped 打开后，非 wrapped 列不再垂直居中
6. 文本编辑态、只读态、失焦态纵向起点一致
7. top-layout 中 issue icon 不把主内容挤偏，且自身与主内容同起点

### 2. 几何断言建议

断言应以几何为主，不只看 class：

- `topGap`
- `bottomGap`
- `contentRect.top - cellRect.top`
- `chipRect.top - cellRect.top`
- `buttonRect.top - cellRect.top`
- 编辑态 frame / textarea / cell 三者几何关系

容差建议：

- 正常文本 / chip / title 顶部起点差：`<= 2px`
- 编辑态 frame 与 cell 边界差：`<= 1px` 或 `<= 2px`

### 3. 场景数据建议

测试数据至少要包含：

- 一列长 wrapped 描述
- 一列标题
- 一列单行普通文本
- 一列数字
- 一列单值 select
- 一列多值 token
- 一列 relation
- 一列 nested summary 或 backlink

否则无法覆盖这次要治理的真正系统边界。

## 六、实施风险与取舍

### 1. 风险

主要风险包括：

- 统一顶对齐后，低行里原本“舒服”的 token 视觉会变得更紧，需要重新微调 padding
- 标题列的整格点击区域和顶部对齐容易互相牵制
- 编辑态输入框如果处理不好，会再次引入行高抖动
- 虚拟滚动测高如果不一起收敛，会出现“初始看着对，滚动后又错”

### 2. 建议取舍

本项目处于早期阶段，建议直接清理旧机制，不做防御性兼容：

- 不保留旧的“局部居中例外”样式
- 不保留旧的 `vertical-align` 业务语义
- 不同时维持两套纵向对齐框架

如果 `Checkbox` 最终决定保留例外，也必须以“单一例外”形式显式写入规范，而不是让它继续隐式漂移。

---

## 七、执行建议

### 1. 推荐执行顺序

1. 先写失败测试，锁定真实不一致场景
2. 收敛 frame 真相层
3. 统一 `Text / Title / Number / ID`
4. 统一 `Token 类`
5. 统一 `Nested / Backlink / Incompatible / Checkbox`
6. 收敛编辑态与测高
7. 清理旧样式残留

### 2. 是否需要单独执行计划

需要。  
本方案已经明确问题边界和目标规则，但执行时仍有较多跨文件改动、顺序依赖和测试矩阵，建议再补一份对应的“执行方案”文档，把任务拆成可逐步落地的变更序列。

---

## 八、结论

“表格所有 cell 类型纵向对齐统一”已经是一个明确的系统性布局治理问题，不适合继续按截图逐条修补。  
当前最正确的方向不是继续微调某个组件的 padding，而是把纵向对齐收敛成：

- 行级只表达高行状态
- frame 级作为唯一真相来源
- 内容级只负责外观
- 测量级只读取统一节点

完成后，主表格里不应再存在“描述列顶对齐了，但 ID / 数字 / token 仍看起来在中线”的情况。

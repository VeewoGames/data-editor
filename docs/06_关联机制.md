# Data Editor 关联机制

## 概述

`data-editor` 当前的关联机制由三个部分组成：

1. `relation`：可编辑的正向引用
2. `backlink`：由 relation 派生的只读反向引用
3. `primary key`：关系网络中的目标锚点

这三者共同定义了系统中的跨文件引用语义。

## Relation

`RelationConfig` 的核心结构：

```ts
type RelationConfig = {
  targetFile: string;
  targetCollection: string;
  targetKey: string;
  mode: "single" | "multi";
  titleFields: string[];
  allowMissing: boolean;
};
```

它表达的是：某个源字段引用了哪个目标文件、哪个集合、哪个主键。

### 关键事实

- relation 是显式配置出来的，不依赖字段名猜测。
- 单值 / 多值模式由 `mode` 决定，不由当前单元格值的形状临时推断。
- 保存时业务文件只写回原始 id 或 id 数组。

## Relation key

系统使用统一的 relation key 来标识一个 relation 字段。它由源文件、源集合和字段路径组成。

这使得顶层字段和嵌套字段都能用统一机制处理。

## Backlink

`backlink` 是派生视图，不是业务字段。它当前表现为：

- 主表中的只读显示列
- 详情面板中的引用来源列表

`BacklinkConfig` 当前包括：

```ts
type BacklinkConfig = {
  sourceRelation: string;
  displayMode: "list";
};
```

### 关键事实

- backlink 来源于 relation，不反向写回源数据。
- backlink 列可能处于 `active` 或 `missing-source` 状态。
- backlink 主要服务于“谁引用了当前记录”的可见性。

## Primary Key

主键配置独立保存于 `ViewConfig.primaryKeys`。

它承担的职责包括：

- 唯一标识集合中的记录
- 作为 relation 的目标键
- 作为 backlink 汇总的匹配基础
- 支持主键改名影响分析和同步计划

## 主键改名影响

系统目前已经支持：

- 分析某个 primary key 改名后会影响哪些 relation
- 汇总受影响记录
- 生成同步计划
- 在适用范围内执行同步保存

这部分能力主要由 `relationMaintenance` 相关模块提供。

## relation / backlink / primary key 的边界

- relation：可写，属于编辑行为
- backlink：只读，属于派生视图
- primary key：受保护，属于结构定义

这三个概念不能再混成同一种“字段类型”理解。

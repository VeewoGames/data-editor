# Data Editor 产品定义

## 方案概述

`data-editor` 是面向 Codex 使用场景的本地 Web 数据编辑器，用于以接近 Notion database 的交互体验查看、筛选、编辑和保存 Nocturnel 项目中的 JSON/CSV 数据文件。它不是 Godot editor plugin，也不是 Notion 同步工具；它运行在本地 HTTP server 上，由 Codex Browser 打开，并通过受限文件 API 读写项目内允许的数据文件。

首版目标是把 `data/*.json` 和后续 `.csv` 数据变成统一的 database view：顶层非嵌套字段进入主表，嵌套 object/array 进入详情面板，字段显示类型由用户指定，不兼容时明确显示 `不兼容`。保存时保持源文件结构，不迁移 Nocturnel 现有 JSON root 形态。

## 产品定位

- **产品名称：** `data-editor`
- **运行形式：** Codex 本地 Web 工具
- **主要用户：** Nocturnel 数据设计/开发过程中的人类编辑者，以及辅助检查数据结构的 Codex agent
- **核心体验目标：** 像 Notion database 一样管理 JSON/CSV 数据，但保留本地文件的可控性和 Git diff 可审查性
- **首版技术路线：** React + TypeScript + Vite + TanStack Table + Radix UI + Node.js local API

## 用户问题

当前直接查看或编辑 JSON/CSV 数据存在这些问题：

- 嵌套 JSON 很难在普通文本编辑器里快速理解。
- `runes.json`、`skills.json`、`traits.json` 等文件结构不同，手工编辑容易漏字段或改错结构。
- 纯表格工具不适合处理 `effects[].params.trigger_effect` 这类深层规则对象。
- 普通 CSV/JSON preview 缺少字段类型、筛选、详情面板和不兼容提示。
- 如果把 root object 文件强行改为 root array，会破坏现有 Godot 运行时、同步配置和工具链读取契约。

## 产品原则

- **本地优先：** 默认只读写本机项目文件，不接入远端服务。
- **源结构保持：** UI 内部可以抽象成 collection，但保存时必须保持原始 JSON root shape。
- **通用结构优先：** 不按文件名硬编码专属视图；默认使用同一套 database model。
- **用户可指定解释方式：** 字段显示类型由用户切换，不由工具强行转换数据。
- **不兼容可见：** 当值不能用当前显示类型表达时显示 `不兼容`，不静默改值。
- **嵌套数据不摊平失控：** 顶层非嵌套字段进入主表，嵌套 object/array 进入详情面板。
- **保存可审计：** 保存前创建备份，保存后可以通过 Git diff 审查。
- **UI 与数据逻辑分离：** React 组件不得持有文件解析、序列化和保存逻辑。

## 非目标

- 不替代 `notion_sync` 或 `doc_sync`。
- 不作为游戏运行时依赖。
- 不把 `skills.json`、`enemies.json`、`traits.json`、`affixes.json` 迁移为 root array。
- 不在 MVP 中实现完整公式系统、多人协作、Notion 远端同步。
- 不承诺 CSV 字节级 round trip，只保证表格语义。

## 首版成功标准

- Codex 可以启动本地服务并打开 `http://127.0.0.1:8787`。
- 六个目标 JSON 文件可以被加载、查看和编辑：
  - `data/skills.json`
  - `data/enemies.json`
  - `data/traits.json`
  - `data/affixes.json`
  - `data/keywords.json`
  - `data/runes.json`
- root array 文件直接显示为 `$` collection。
- root object 文件显示内部 array 字段作为 collection，例如 `skills`、`enemies`。
- 主表只显示顶层 primitive 与 primitive array 字段。
- 嵌套字段可以在详情面板查看和编辑。
- 字段类型切换不会自动修改底层数据。
- 保存只允许写入白名单 `.json` / `.csv` 文件。
- 保存前自动备份。

# 用户长期偏好

## 语言
- 默认使用中文交流和分析。
- 项目文档（方案、报告、计划等）用中文撰写（包括文件名）；代码标识、文件路径、命令、API 名称保持原文。

## 行为规则
- 基于项目的任务默认先用 `rg` 或等效搜索定位相关入口，再按调用链和职责边界定向读取关键文件，建立证据链后再下结论或改代码。
- 不要把整个仓库整体塞进上下文，不要基于未验证猜测直接回答。

## 文档规范
每个方案文档需包含概述，内容如下：
1. 总体目标和范围
2. 各阶段任务概要（主要工作、预期成果、执行顺序）
3. 整体结构框架

## 技术重构
项目处于早期阶段，重构时不做防御性兼容。旧机制清理干净，只用新框架。

## Git 提交
当用户表达提交意图时：
1. 先检查 `git status` 和 `git diff`
2. 用中文撰写详细 commit message
3. 执行提交
4. 提交后询问是否 push，不自动推送

## 决策原则
遇到需要决策的事情，不得自行行动。应向用户说明：
- 决策内容和背景
- 推荐方案及理由
- 等待用户回复后执行

## data-editor 服务收尾
当任务中启动过本地服务、临时端口、Browser、Playwright，或执行过 `stop` / `shutdown` / 进程清理时，结束前必须运行：

```powershell
npm run service:finalize
```

最终回复必须包含：
- `8787/api/health` 结果
- `8791/health` 结果
- 正式 URL：`http://127.0.0.1:8787/`
- 临时进程和目录清理结果

Browser 必须停留在 `http://127.0.0.1:8787/`，不得停留在临时验证端口。
